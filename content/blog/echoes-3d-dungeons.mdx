## Entering the Third Dimension

When I first toggled Echoes of the Spire from its top-down 2D view into a 3D first-person perspective, the dungeon stopped being a map and became a *place*. Corridors had height. Shadows fell across stone floors. Enemies didn't just approach — they loomed. That visceral difference convinced me to build out 3D mode as a full-featured rendering path, not just a tech demo.

This post is a deep technical dive into how the 3D systems work: the DungeonGenerator that creates navigable 3D environments, the rendering pipeline that makes them look good at 60fps, the EnemyManager that brings them to life, and the WeaponSystem that puts destruction in the player's hands.

## The DungeonGenerator: From 2D Maps to 3D Worlds

The 3D dungeon starts life as a 2D floor plan. The procedural generation system I described in a previous post outputs a data structure of rooms, corridors, and connections. The DungeonGenerator's job is to **extrude this flat map into three-dimensional geometry**.

Each tile in the 2D grid becomes a 3D cell. Floor tiles get a ground plane. Wall tiles get vertical surfaces with configurable height. Door tiles get archway geometry. The generator walks the grid and builds a mesh for each distinct surface:

```typescript
function extrudeFloorPlan(grid: TileGrid): Mesh[] {
  const meshes: Mesh[] = [];
  
  for (let x = 0; x < grid.width; x++) {
    for (let y = 0; y < grid.height; y++) {
      const tile = grid.get(x, y);
      
      if (tile.type === 'floor') {
        meshes.push(createFloorQuad(x, y));
        meshes.push(createCeilingQuad(x, y, WALL_HEIGHT));
      }
      
      if (tile.type === 'wall') {
        // Only create wall faces adjacent to floor tiles
        for (const dir of CARDINAL_DIRS) {
          const neighbor = grid.get(x + dir.x, y + dir.y);
          if (neighbor?.type === 'floor') {
            meshes.push(createWallQuad(x, y, dir, WALL_HEIGHT));
          }
        }
      }
    }
  }
  
  return mergeMeshes(meshes); // Combine into fewer draw calls
}
```

### Mesh Merging and Draw Call Optimization

A naive implementation would create thousands of individual mesh objects — one per floor quad, one per wall face. Each mesh would require its own draw call, and on WebGL, **draw calls are expensive**. The GPU is fast at rendering triangles, but each draw call has significant CPU-side overhead.

My solution is aggressive **mesh merging**. All floor quads that share the same material get combined into a single mesh. All wall quads of the same type merge together. The entire dungeon floor typically renders in **four to six draw calls**: floor, ceiling, walls, doors, decorative elements, and dynamic objects.

This merging happens during generation, not at runtime. When the player enters a new floor, there's a brief loading moment (masked by the "INITIALIZING..." screen) during which the generator builds and merges all geometry. Once loaded, the floor renders with minimal per-frame overhead.

## The Rendering Pipeline

The 3D renderer is built on raw **WebGL 2** — not Three.js or Babylon.js or any framework. I made this choice for control and performance. Frameworks are excellent for general-purpose 3D, but game rendering has specific needs (batched sprites, custom shaders, tight performance budgets) that benefit from direct GPU access.

The per-frame rendering pipeline has these stages:

1. **Camera Update** — Calculate view and projection matrices from the player's position and look direction.
2. **Frustum Culling** — Determine which rooms and objects are visible from the current viewpoint. Rooms outside the view frustum are skipped entirely.
3. **Static Geometry Pass** — Render the merged dungeon meshes (floors, walls, ceilings) with the dungeon shader, which applies texturing and lighting.
4. **Dynamic Object Pass** — Render enemies, items, projectiles, and interactive objects. These are sorted by distance for correct transparency ordering.
5. **Particle Pass** — Render all active particle systems as additive-blended point sprites.
6. **Weapon Pass** — Render the player's weapon in screenspace (a separate render pass that ensures the weapon is always in front of the world geometry).
7. **UI Overlay** — Composite the HUD elements (health, abilities, minimap) over the 3D scene using a 2D Canvas overlay.

### Lighting System

The dungeon uses a hybrid lighting approach. **Ambient light** provides baseline visibility so the player is never in complete darkness (unless a Lights Out modifier is active). **Point lights** are placed at strategic locations — torches, energy crystals, doorways — and are evaluated per-fragment in the dungeon shader.

I limit the number of active point lights to **eight** per frame, selecting the nearest lights to the player. This keeps the shader relatively simple while still creating dramatic lighting variation. Light colors use the game's purple/blue palette: violet torches, cyan energy well-springs, magenta boss room markers.

```glsl
// Simplified fragment shader for dungeon walls
vec3 computeLighting(vec3 normal, vec3 fragPos, vec3 baseColor) {
  vec3 result = baseColor * ambientIntensity;
  
  for (int i = 0; i < NUM_ACTIVE_LIGHTS; i++) {
    vec3 lightDir = lights[i].position - fragPos;
    float dist = length(lightDir);
    float attenuation = 1.0 / (1.0 + 0.07 * dist + 0.017 * dist * dist);
    float diff = max(dot(normal, normalize(lightDir)), 0.0);
    result += baseColor * lights[i].color * diff * attenuation;
  }
  
  return result;
}
```

## EnemyManager: AI in 3D Space

The EnemyManager handles spawning, behavior updates, and spatial queries for all enemies on the current floor. In 3D mode, enemy AI gains additional behaviors that don't exist in 2D:

**Line-of-sight awareness.** Enemies don't just know where the player is — they check whether they can see the player by casting a ray through the dungeon geometry. Enemies behind walls won't aggro until you enter their room or they hear combat sounds propagate through corridors.

**Vertical awareness.** Some dungeon rooms have elevation changes — raised platforms, sunken pits. Melee enemies need pathfinding that accounts for height, and ranged enemies need to adjust their aim arc based on relative elevation.

**Positional audio cues.** In 3D mode, enemy sounds are spatialized using the Web Audio API's PannerNode. You can hear enemies growling to your left, footsteps approaching from behind, projectiles whizzing past your ear. This spatial audio transforms enemy encounters from visual-only to multi-sensory experiences.

### Enemy Types in 3D

I designed enemy types to exploit the unique properties of 3D space:

- **Lurkers** hide behind corners and ambush when the player passes. Their AI relies on line-of-sight obstruction for positioning.
- **Floaters** ignore floor geometry and approach from above, requiring the player to look up — something that rarely happens in a first-person game.
- **Warpers** teleport short distances, appearing at unexpected positions behind the player's back. Their sound design uses a distinctive spatial audio cue to give fair warning.

## PlayerController: First-Person Movement

The PlayerController translates player input into 3D world movement. Mouse movement controls the camera (look direction), and WASD keys move relative to the current facing. This is standard FPS controls, but implementing them in a browser requires handling several edge cases.

**Pointer Lock API** is essential for mouse look. Without it, the mouse cursor hits the edge of the browser window and stops. I request pointer lock when the game starts and handle the various browser-specific quirks (lock request rejection, lock loss on tab switch, re-acquisition flow).

**Movement collision** uses a capsule collider swept through the dungeon geometry. I check for collision against wall planes and floor edges, sliding the player along walls rather than stopping them dead. This sliding behavior is crucial for smooth movement — getting stuck on wall corners is a classic first-person game frustration.

## WeaponSystem: Putting Destruction in Your Hands

The weapon renders as a viewmodel — a 3D model drawn in screen space in front of the camera. It has procedural idle sway (gentle bobbing synchronized to movement), attack animations (swings, thrusts, or casting gestures depending on class), and impact effects when attacks connect.

Weapon timing integrates tightly with the combat feel systems from 2D mode. Hitstop pauses the weapon animation mid-swing. Screen shake displaces the camera. Particles spawn from the weapon's world-space tip position. The weapon isn't just cosmetic — it's the primary channel for communicating combat impact to the player.

The WeaponSystem also handles **ranged weapon rendering** for classes that use projectile attacks. Projectiles are spawned in 3D space from the weapon's muzzle point, travel through the world with proper collision detection, and produce impact effects on contact with enemies or walls.

## Performance: The Eternal Battle

Running a 3D game in a browser at a stable 60fps is a constant performance negotiation. My optimization toolkit includes geometry instancing for repeated dungeon elements, texture atlasing to minimize bind state changes, frustum and occlusion culling to skip invisible geometry, and a frame budget monitor that dynamically reduces particle counts and shadow quality if framerate drops below a threshold.

The result is a 3D dungeon crawler that runs smoothly on mid-range hardware — which, for a browser game, is the sweet spot. High-end machines get the full visual experience. Lower-end machines gracefully degrade. And players who prefer guaranteed performance can always switch to 2D mode with a single toggle.

Building 3D dungeons in the browser taught me that platform limitations are creative challenges in disguise. WebGL isn't OpenGL. JavaScript isn't C++. But within those constraints, you can build something that feels genuinely immersive — one carefully optimized draw call at a time.
