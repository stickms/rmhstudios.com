## Every Millisecond Counts

A rhythm game that drops frames is a rhythm game that lies to you. If the game renders at an inconsistent frame rate, notes stutter on screen, the visual timing drifts from the audio, and players miss beats that they should have hit. In a genre where the difference between "Perfect" and "Good" is 30 milliseconds, a single dropped frame (16.7ms at 60fps) can ruin a play.

Performance optimization in Slice It! isn't a luxury — it's a core gameplay requirement. Here's how we keep the game running at a rock-solid 60 frames per second while simultaneously processing real-time audio, rendering hundreds of notes, managing game state, and broadcasting multiplayer data.

## Canvas Rendering Architecture

Slice It!'s gameplay runs entirely on an **HTML5 Canvas** element using the 2D rendering context. We chose Canvas over WebGL for simplicity and compatibility — the 2D context is universally supported and sufficient for our rendering needs.

The rendering pipeline follows a strict structure:

```typescript
function gameLoop(timestamp: number): void {
  const deltaTime = timestamp - lastFrameTime;
  lastFrameTime = timestamp;
  
  // 1. Update game state
  updateNotes(deltaTime);
  updateParticles(deltaTime);
  updateScore();
  
  // 2. Clear canvas
  ctx.clearRect(0, 0, canvas.width, canvas.height);
  
  // 3. Render layers (back to front)
  renderBackground(ctx);
  renderLanes(ctx);
  renderNotes(ctx);
  renderHitZone(ctx);
  renderParticles(ctx);
  renderHUD(ctx);
  
  // 4. Request next frame
  requestAnimationFrame(gameLoop);
}
```

### Layer Caching

The background, lane lines, and HUD elements don't change every frame (or change infrequently). Redrawing them from scratch 60 times per second is wasteful. We use **offscreen canvases** to cache static or semi-static layers:

```typescript
const backgroundCache = new OffscreenCanvas(width, height);
const bgCtx = backgroundCache.getContext('2d')!;

// Render background once (or when it changes)
function renderBackgroundToCache(): void {
  bgCtx.clearRect(0, 0, width, height);
  drawGradient(bgCtx);
  drawLaneGuides(bgCtx);
}

// In the main loop, just blit the cached canvas
function renderBackground(ctx: CanvasRenderingContext2D): void {
  ctx.drawImage(backgroundCache, 0, 0);
}
```

This single optimization reduced our per-frame render time by roughly 25%. The background is expensive — it involves gradients and multiple geometric draws — but it only needs to be regenerated when the canvas resizes or the visual theme changes.

## Note Rendering Optimization

At peak density, Slice It! can have 50–100 notes visible on screen simultaneously. Each note needs to be drawn, positioned, scaled (notes grow slightly as they approach the hit zone), and colored based on its type and proximity to the hit zone.

### Object Pooling

Creating and destroying note objects every frame generates garbage collection pressure, which causes periodic stutters. We use an **object pool** for notes:

```typescript
class NotePool {
  private pool: GameNote[] = [];
  private active: Set<GameNote> = new Set();
  
  acquire(): GameNote {
    const note = this.pool.pop() || new GameNote();
    note.reset();
    this.active.add(note);
    return note;
  }
  
  release(note: GameNote): void {
    this.active.delete(note);
    this.pool.push(note);
  }
  
  getActive(): ReadonlySet<GameNote> {
    return this.active;
  }
}
```

Notes are pre-allocated and recycled. When a note scrolls off screen or is hit by the player, it's returned to the pool rather than garbage collected. New notes are drawn from the pool rather than constructed fresh. This eliminated our GC-related frame drops entirely.

### Spatial Culling

Notes that haven't entered the visible area of the canvas yet don't need to be rendered. We maintain a sorted list of upcoming notes and only begin rendering them when their calculated Y position brings them on screen. For a long song with thousands of notes, this means we're typically only processing 50–100 notes per frame instead of thousands.

```typescript
function getVisibleNotes(currentTime: number, viewWindow: number): GameNote[] {
  const startTime = currentTime - 0.5;  // recently passed notes
  const endTime = currentTime + viewWindow;
  
  return activeNotes.filter(
    note => note.time >= startTime && note.time <= endTime
  );
}
```

## Audio Processing Efficiency

The audio engine runs frequency analysis every frame via the `AnalyserNode`. The FFT data drives visual effects — background color pulsing, particle intensity, subtle lane glow effects that respond to the music.

### Throttled Analysis

While the `AnalyserNode` can provide data every frame, we don't always need it at that frequency. Visual effects that respond to bass energy only need updates at 20–30Hz — the human eye can't distinguish bass pulses at 60Hz anyway. We throttle visual effect updates to reduce processing:

```typescript
let lastEffectUpdate = 0;
const EFFECT_UPDATE_INTERVAL = 33; // ~30Hz

function updateVisualEffects(timestamp: number): void {
  if (timestamp - lastEffectUpdate < EFFECT_UPDATE_INTERVAL) return;
  lastEffectUpdate = timestamp;
  
  analyserNode.getFloatFrequencyData(frequencyData);
  const bassEnergy = computeBandEnergy(frequencyData, 0, 10);
  const trebleEnergy = computeBandEnergy(frequencyData, 100, 512);
  
  backgroundPulse = lerp(backgroundPulse, bassEnergy, 0.3);
  particleIntensity = lerp(particleIntensity, trebleEnergy, 0.2);
}
```

### Web Workers for Heavy Lifting

The beatmap generation pipeline — FFT processing, onset detection, feature extraction — runs in a **Web Worker**. This ensures that the computationally expensive pre-processing phase never blocks the main thread. The game loop continues to render smoothly (showing a loading animation) while the worker crunches the audio data in the background.

```typescript
const beatmapWorker = new Worker('/workers/beatmap-generator.js');

beatmapWorker.postMessage({
  type: 'generate',
  audioData: channelData,
  sampleRate: audioBuffer.sampleRate,
});

beatmapWorker.onmessage = (event) => {
  if (event.data.type === 'complete') {
    const beatmap = event.data.beatmap;
    startGame(beatmap);
  }
};
```

## Memory Management

Memory management is critical for a web application that handles large audio files. A five-minute song at 44.1kHz stereo occupies roughly 50MB of decoded PCM data. In a session where a player browses and plays multiple songs, memory can accumulate rapidly.

### Buffer Lifecycle

We follow a strict lifecycle for audio buffers:

1. **Load:** Decode audio into an `AudioBuffer` when the player selects a song
2. **Play:** Use the buffer for playback and analysis
3. **Cache:** After playback, move the buffer to an LRU cache (max 5 songs)
4. **Evict:** When the cache is full, the oldest buffer is dereferenced for garbage collection

### Typed Array Reuse

Many of our processing functions accept typed arrays (`Float32Array`, `Uint8Array`) for frequency and waveform data. Rather than allocating new arrays each frame, we pre-allocate them once and reuse them:

```typescript
// Allocated once at initialization
const frequencyData = new Float32Array(analyserNode.frequencyBinCount);
const waveformData = new Float32Array(analyserNode.fftSize);

// Reused every frame — no allocation
function updateAudioVisualization(): void {
  analyserNode.getFloatFrequencyData(frequencyData);
  analyserNode.getFloatTimeDomainData(waveformData);
  // process data...
}
```

## Particle System Optimization

Hit effects, combo celebrations, and ambient visual elements all use particles. Our particle system is designed for efficiency:

- **Fixed-size particle array** — pre-allocated, never grows
- **Ring buffer allocation** — new particles overwrite the oldest dead particles
- **Batch rendering** — all particles of the same type are drawn in a single pass with shared fill style
- **Simple physics** — linear velocity and constant fade, no complex forces

The particle system can handle 500+ active particles without measurable frame impact.

## Profiling and Monitoring

We built a lightweight **performance monitor** that tracks frame times, memory usage, and audio processing latency in real time. During development, this overlay is always visible. In production, it's hidden behind a developer shortcut.

When frame time exceeds 16ms (indicating a dropped frame), the monitor logs the spike and records what was happening in the game at that moment. This data has been invaluable for identifying and fixing performance regressions.

## The Result

After months of optimization, Slice It! maintains a steady 60fps on mid-range hardware — a 2020 laptop with integrated graphics can run the game smoothly. On high-end hardware, we're consistently under 8ms per frame, leaving ample headroom.

Performance isn't sexy. Nobody plays a rhythm game and thinks, "Wow, the object pooling is really efficient." But they *feel* it. They feel the smoothness, the responsiveness, the absence of stutters. Good performance is invisible — and that invisibility is the highest compliment.
