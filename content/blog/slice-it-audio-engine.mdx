## The Heart of the Game

If Slice It! were a human body, the audio engine would be its heart. Every note the player hits, every beat they feel, every moment of synchronization between music and gameplay — it all flows from the audio engine. Getting this right wasn't just important. It was existential. A rhythm game with bad audio handling is a broken game, full stop.

I want to walk you through exactly how Slice It!'s audio engine works, the challenges I faced building it in the browser, and the lessons I learned along the way.

## Why the Web Audio API

The **Web Audio API** is one of the most powerful yet underappreciated browser APIs available. It provides a modular, graph-based system for audio processing. You create nodes — source nodes, analyser nodes, gain nodes, filter nodes — and connect them into a signal chain. The browser handles the low-level audio rendering on a separate thread, giving you surprisingly high performance.

Here's a simplified version of the core audio setup in Slice It!:

```typescript
const audioContext = new AudioContext();
const sourceNode = audioContext.createBufferSource();
const analyserNode = audioContext.createAnalyser();
const gainNode = audioContext.createGain();

sourceNode.connect(analyserNode);
analyserNode.connect(gainNode);
gainNode.connect(audioContext.destination);

analyserNode.fftSize = 2048;
const bufferLength = analyserNode.frequencyBinCount;
const dataArray = new Float32Array(bufferLength);
```

This chain gives us three crucial capabilities: playback through the source node, real-time frequency analysis through the analyser, and volume control through the gain node.

## Decoding Audio: The First Hurdle

Before any analysis can happen, we need to decode the audio file into a format the Web Audio API can work with. This means converting MP3s, WAVs, OGGs, and other formats into raw PCM data stored in an `AudioBuffer`.

```typescript
async function loadAudio(file: File): Promise<AudioBuffer> {
  const arrayBuffer = await file.arrayBuffer();
  return audioContext.decodeAudioData(arrayBuffer);
}
```

Sounds simple, right? In practice, this is where the first wave of browser inconsistencies hits. Different browsers support different codecs. Safari historically had issues with certain MP3 encodings. Firefox handled OGG better than Chrome in some edge cases. We ended up building a format detection layer that checks the file header and warns users if their file might cause issues.

The decoded `AudioBuffer` contains the raw float sample data for each channel. This is where the real fun begins.

## Handling Stereo and Mono

Not all audio files are created equal. Some are mono (one channel), some are stereo (two channels), and occasionally you encounter files with even more channels. The audio engine needs to handle all of these gracefully.

For beatmap generation, we typically work with a **mono mixdown** — a single channel that represents the combined audio. This simplifies the analysis pipeline and ensures consistent results regardless of the source format.

```typescript
function getMixdown(buffer: AudioBuffer): Float32Array {
  if (buffer.numberOfChannels === 1) {
    return buffer.getChannelData(0);
  }
  
  const left = buffer.getChannelData(0);
  const right = buffer.getChannelData(1);
  const mono = new Float32Array(left.length);
  
  for (let i = 0; i < left.length; i++) {
    mono[i] = (left[i] + right[i]) / 2;
  }
  
  return mono;
}
```

However, we also preserve the original channel data for playback. Players hear their music exactly as it was recorded — stereo panning, spatial effects, and all. The mono mixdown is only used internally for analysis.

## Real-Time Frequency Analysis

The `AnalyserNode` is the workhorse of our audio processing pipeline. During playback, we poll it every animation frame to get the current frequency spectrum:

```typescript
function getFrequencyData(): Float32Array {
  analyserNode.getFloatFrequencyData(dataArray);
  return dataArray;
}
```

This gives us a snapshot of the audio's frequency content at any given moment. Low frequencies (bass, kick drums) occupy the lower bins. Mid frequencies (vocals, guitars) sit in the middle. High frequencies (hi-hats, cymbals, sibilance) live at the top.

We use this data for real-time visual effects during gameplay — the background pulses with the bass, particles shimmer with the treble. But the heavy lifting of beatmap generation happens during a **pre-processing phase** before the song starts, which I'll cover in a dedicated post.

## The AudioContext Resume Problem

One of the most infuriating challenges in browser audio development is the **autoplay policy**. Modern browsers require a user gesture (click, tap, keypress) before an `AudioContext` can start producing sound. This is a sensible security measure — nobody wants websites blasting audio without permission — but it adds complexity.

We handle this with a state machine that tracks whether the audio context is running, suspended, or closed. Every user interaction that should produce audio first checks the context state and resumes it if necessary:

```typescript
async function ensureAudioContext(): Promise<void> {
  if (audioContext.state === 'suspended') {
    await audioContext.resume();
  }
}
```

This seems trivial, but forgetting to handle it causes silent failures — the game appears to work, but no sound plays. We added a visible indicator that warns players if audio initialization fails and provides a button to manually resume.

## Timing Precision

In a rhythm game, timing is everything. The difference between a "Perfect" hit and a "Good" hit might be 30 milliseconds. The difference between "Good" and a miss might be 80 milliseconds. At these margins, any drift between the audio clock and the game clock is catastrophic.

We use `audioContext.currentTime` as our single source of truth for timing. This high-resolution clock is driven by the audio hardware and is far more accurate than `Date.now()` or `performance.now()` for audio-related timing. All game events — note spawning, hit detection, score calculation — are referenced against this clock.

```typescript
function getCurrentBeat(): number {
  const elapsed = audioContext.currentTime - songStartTime;
  return elapsed * (bpm / 60);
}
```

This approach eliminates the drift that would occur if we used separate clocks for audio and gameplay. The game and the music are literally synchronized at the hardware level.

## Memory Management

Audio buffers can be **large**. A five-minute song at 44.1kHz stereo is roughly 50MB of raw PCM data. When players are browsing the song library and previewing tracks, we can easily accumulate hundreds of megabytes of decoded audio.

We implemented an **LRU (Least Recently Used) cache** for audio buffers. The cache holds the most recently played songs, automatically evicting the oldest entries when memory pressure increases. Songs that are queued for multiplayer get pinned in the cache so they can't be evicted mid-session.

## Lessons Learned

Building a real-time audio engine in the browser taught me several hard lessons:

1. **Never trust browser consistency.** Test on every major browser and version. Audio behavior varies wildly.
2. **The AudioContext clock is sacred.** Never use wall-clock time for audio synchronization.
3. **Memory management is not optional.** Audio data is heavy. Budget for it early.
4. **Users will upload anything.** Corrupted files, unusual sample rates, mono recordings of live concerts with clipping distortion — your engine needs to handle it all gracefully.

The audio engine is the foundation that everything else in Slice It! is built upon. Getting it right took weeks of iteration, but the result is a system that handles real-time audio processing with the precision a rhythm game demands.
