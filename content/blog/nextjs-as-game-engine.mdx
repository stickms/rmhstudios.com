## A Framework, Not an Engine — And That's the Point

When people ask what engine we use at RMH Studios, we tell them Next.js. The reaction is usually confusion. Next.js is a web framework. It's for dashboards and e-commerce sites and blogs. It's not a game engine.

And that's exactly why it works for us.

Traditional game engines — Unity, Unreal, Godot — are extraordinary pieces of software. They handle rendering pipelines, physics simulations, asset management, and platform-specific compilation. If you're building a 3D console game or a polished mobile title, these engines are the right choice. We have enormous respect for what they enable.

But we're not building console games. We're building **browser games** — interactive experiences that live at a URL, load in seconds, and work on any device with a modern browser. For that use case, Next.js gives us something no traditional engine can: **the entire web platform as our runtime**.

## The Next.js 15 Moment

In October 2024, Vercel released **Next.js 15**, and it validated many of the architectural bets we'd already made. The release brought stable React Server Components, improved partial prerendering, and significant performance gains in both cold starts and hydration.

For a game studio, these aren't abstract improvements. Server Components mean we can render game menus, leaderboards, and content pages on the server while keeping game logic entirely on the client. Partial prerendering means a player's first paint is near-instant, even if the game's interactive layer takes a moment to hydrate. And the performance improvements in Next.js 15 directly translate to faster load times for our games.

Vercel's growth has been remarkable — the company raised $250 million in 2024 at a $3.25 billion valuation, and the framework now powers hundreds of thousands of production sites. That means the infrastructure we depend on is well-funded, actively maintained, and improving at a pace that benefits our games without any engineering effort on our part.

## One Codebase, Two Jobs

Here's the architectural insight that makes Next.js work as our "engine": **the same codebase serves both the marketing site and the games themselves**.

When you visit rmhstudios.com, you're looking at a Next.js application. When you click "Play" on any of our games, you're still in the same application. The marketing pages use Server Components for fast, SEO-friendly rendering. The game pages use Client Components for interactive, stateful gameplay. The transition between "website" and "game" is seamless because there is no transition — it's all one thing.

This single-codebase approach gives us concrete advantages:

- **Shared authentication.** One login works across all games and the community platform. No separate account systems, no OAuth redirects to external services.
- **Shared state.** Player profiles, achievements, and preferences persist across games because they all live in the same application state.
- **Shared infrastructure.** One deployment pipeline, one set of environment variables, one monitoring dashboard. When we push an update to Slice It!, we're deploying the same way we'd push a blog post.
- **API routes as game servers.** Next.js API routes handle matchmaking, leaderboard updates, and game state persistence. For real-time games like RMHbox, we layer WebSocket connections on top, but the REST backbone is just Next.js routes.

## What Traditional Engines Do Better (And What They Don't)

We want to be honest about the tradeoffs. Traditional game engines excel at things we either don't need or handle differently:

**Rendering pipelines.** Unity and Unreal have sophisticated rendering systems that handle lighting, shadows, post-processing, and LOD management. For our 3D games like Neon Driftway, we use Three.js through React Three Fiber — which gives us a declarative, component-based approach to 3D rendering that fits naturally into our React architecture. It's not as powerful as Unreal's renderer, but it runs in a browser tab without a download.

**Physics engines.** Godot's built-in physics is excellent. For Laundry Sort, we built custom physics using JavaScript. It's more work, but it gives us precise control over the feel of the game, and the physics run identically on every device because they're deterministic JavaScript, not platform-dependent native code.

**Asset pipelines.** Traditional engines have sophisticated asset import and optimization systems. We use standard web tooling — image optimization through Next.js's built-in Image component, code splitting through dynamic imports, and audio management through the Web Audio API. The web platform's asset handling is less specialized but more flexible.

**Platform export.** Unity can export to iOS, Android, PlayStation, Switch, and more. We export to one platform: the browser. And the browser runs on all of those devices. We trade native performance for universal access — a tradeoff we're comfortable with for the kinds of games we build.

## The Server Components Revolution

React Server Components, now stable in Next.js 15, changed how we think about game architecture. Before RSC, every component shipped JavaScript to the client. A leaderboard component that just displays data? Client-side JavaScript. A game description page? Client-side JavaScript. The "game" parts of our app were competing for bandwidth and parse time with components that didn't need to be interactive at all.

With Server Components, we moved roughly **60% of our component tree** to server-only rendering. Game pages now ship dramatically less JavaScript because everything outside the game canvas — navigation, descriptions, social features, settings panels — renders on the server. The client bundle contains only what it needs: the game itself.

For a game like Signal Forge, which has a complex deckbuilding UI alongside the actual roguelike gameplay, this separation is transformative. The card collection browser, the deck history, the strategy guides — all server-rendered. The combat system, the card animations, the procedural generation — all client-side. Each part uses the rendering strategy that makes sense for it.

## API Routes as Game Infrastructure

Next.js API routes are our backend. Full stop. Here's what they handle:

- **Matchmaking** for multiplayer games like RMHbox and RMH Type
- **Leaderboard** persistence and ranking calculations
- **Save state** management for games like Signal Forge and Echoes of the Spire
- **Achievement** tracking across the entire game portfolio
- **Analytics** for gameplay telemetry and debugging

Each API route is a serverless function that scales automatically. When RMHbox has a traffic spike during a weekend game night, the matchmaking routes scale up without any manual intervention. When traffic drops, they scale down. We never think about server capacity.

For real-time multiplayer, we extend beyond API routes into persistent WebSocket connections. But even there, Next.js handles the initial connection setup, authentication, and room management through standard routes before handing off to the WebSocket layer.

## Middleware as Game Logic

One underappreciated Next.js feature is **middleware** — code that runs on every request before it reaches a page or API route. We use middleware for:

- **Session validation** to ensure authenticated game sessions
- **Rate limiting** to prevent leaderboard manipulation
- **A/B testing** for gameplay experiments
- **Geographic routing** to connect players to nearby game servers

This is infrastructure that would require a separate service in a traditional game architecture. In Next.js, it's a single file that runs at the edge.

## The Counterargument

The most common criticism we hear is: "You're limited by the browser." And that's true — we are. We can't access GPU compute shaders the way a native application can. We can't guarantee 120fps on all devices. We can't ship a 50GB open world.

But we can do something no traditional engine enables: **give someone a URL and have them playing a game in three seconds**. No download. No install. No app store approval. No platform fee. Just a link, a click, and a game.

For the kinds of experiences we build — party games, narrative thrillers, rhythm games, competitive typing — that instant accessibility matters more than raw rendering power. And Next.js, with its performance optimizations, server rendering, and edge infrastructure, makes that accessibility feel effortless.

## The Web Is the Platform

We're not anti-engine. We're pro-web. Unity and Godot are excellent tools for the problems they solve. Next.js is an excellent tool for the problem we solve: delivering interactive entertainment to anyone with a browser.

The web platform gets more powerful every year. WebGPU is arriving. The Web Audio API continues to mature. WebAssembly enables near-native performance for compute-heavy tasks. And Next.js keeps evolving to take advantage of all of it.

At RMH Studios, we bet on the web — and Next.js is how we build on that bet. Every game, every app, every community feature ships from one codebase, through one framework, to every device on the planet. That's not a limitation. That's a superpower.
