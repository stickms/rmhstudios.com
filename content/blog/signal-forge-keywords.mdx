## Why Keywords Matter

In any card game, keywords are the atoms of strategic depth. Each keyword is a discrete rule that, when combined with other keywords, creates emergent complexity. A card with **Echo** and **Growing** behaves fundamentally differently from a card with **Echo** and **Exhaust**. Multiply that by 20+ keywords across 60+ cards, and you have a combinatorial explosion of strategic possibilities — exactly what a roguelike deckbuilder needs.

When I started building Signal Forge's keyword system, I set three design goals:

1. **Every keyword should be understandable in one sentence.**
2. **Every keyword should create at least one interesting decision.**
3. **Every keyword should synergize with at least three other keywords.**

Let me walk you through the system and the engineering behind it.

## The Keyword Taxonomy

Signal Forge's keywords fall into several categories. Understanding these categories helps players build mental models for how cards will interact — even with keywords they haven't seen before.

### Offensive Keywords

- **Piercing** — Damage bypasses shields. Simple, devastating against shielded enemies.
- **AOE** — Affects all enemies. In multi-enemy encounters, this is your crowd control.
- **Chain** — If this card kills or triggers a condition, play another card free. Enables explosive turns.
- **Multihit** — Strikes multiple times. Each hit is smaller, but effects like Bleed trigger per hit.
- **Growing** — Gains permanent stats each time it's played. The investment keyword.

### Defensive Keywords

- **Sustain** — Generates persistent healing or shielding over time. The long game keyword.
- **Retain** — Stays in hand between turns instead of being discarded. Reliability.
- **Echo** — Plays a weaker copy of itself automatically. Two effects for one card play.

### Resource Keywords

- **Exhaust** — Removed from deck after use. Powerful but finite.
- **Siphon** — Steals resources or stats from the enemy. Your gain is their loss.
- **Leech** — Heals you for a portion of damage dealt. Aggression meets sustain.

### Status Keywords

- **Bleed** — Applies stacking damage-over-time to the enemy. Death by a thousand cuts.
- **Freeze** — Reduces enemy actions or delays their next turn. Tempo control.
- **Vulnerable** — Increases damage taken by the target. The setup keyword.
- **Weak** — Reduces damage dealt by the target. Defensive debuff.

### Chaos Keywords

- **Wildcard** — Produces a random effect from a curated pool. Embrace the noise.
- **Glitch** — Interacts with the corruption system. High risk, high reward.

## The Card Class: 803 Lines of Game Logic

The `Card` class is the single largest piece of code in Signal Forge, and for good reason — it needs to handle every possible keyword interaction, stat modification, upgrade path, and edge case. At 803 lines, it's a beast, but it's a *well-organized* beast.

Here's the conceptual architecture:

```typescript
class Card {
  // Identity
  id: string;
  name: string;
  waveformType: 'Pulse' | 'Sine' | 'Saw' | 'Noise';
  rarity: 'common' | 'uncommon' | 'rare';
  
  // Base stats
  cost: number;
  damage: number;
  shield: number;
  
  // Keywords
  keywords: Keyword[];
  
  // State
  timesPlayed: number;
  isUpgraded: boolean;
  isCorrupted: boolean;
  
  // The core execution method
  play(target: Enemy, gameState: GameState): PlayResult {
    // 1. Check resource costs
    // 2. Apply pre-play triggers (Retain check, etc.)
    // 3. Calculate modified damage (Growing, Vulnerable, etc.)
    // 4. Apply damage (Piercing check, Multihit loop, etc.)
    // 5. Apply status effects (Bleed, Freeze, Weak, etc.)
    // 6. Apply self-effects (Sustain, Leech, Shield, etc.)
    // 7. Post-play triggers (Echo, Chain, Exhaust, etc.)
    // 8. Update play count for Growing
    // 9. Return result for battle log
  }
}
```

### The Keyword Resolution Pipeline

The trickiest engineering challenge was **keyword ordering**. When a card has multiple keywords, the order in which they resolve matters enormously. Consider a card with both **Multihit** and **Leech**:

- If Leech applies per hit, it's much stronger (4 hits × small heal = significant sustain).
- If Leech applies once for total damage, it's weaker but simpler.

I chose the per-hit resolution because it creates more interesting strategic decisions. But that meant building a resolution pipeline that processes keywords in a defined order:

```typescript
const KEYWORD_RESOLUTION_ORDER = [
  'pre-play',    // Retain, cost modifications
  'targeting',   // AOE, single-target
  'damage-calc', // Growing, Vulnerable, Weak
  'damage-app',  // Piercing, Multihit
  'on-hit',      // Bleed, Freeze, Leech, Siphon
  'post-play',   // Echo, Chain, Exhaust
  'cleanup',     // Glitch corruption check
];
```

Each keyword registers itself into one or more phases of this pipeline. When a card is played, the engine walks through each phase in order, executing all keywords registered to that phase. This architecture made it possible to add new keywords without refactoring existing ones — a critical feature when you're iterating on game design weekly.

## Design Philosophy: Simple Atoms, Complex Molecules

The biggest lesson from building 20+ keywords is this: **individual keywords should be simple; complexity should emerge from combinations.**

Take **Echo** — it just plays a weaker copy of the card. Dead simple. But combine it with:

- **Bleed** → Double Bleed application per turn
- **Growing** → The echo also increments the play count
- **Multihit** → Multiple hits, each echoed, each triggering on-hit effects
- **Sustain** → Persistent echo effect that generates value every turn

None of these combos required special-case code. They emerged naturally from the keyword resolution pipeline processing each keyword independently. That's the mark of a good system — **emergent complexity without emergent bugs** (well, mostly without bugs).

### The "Kitchen Sink" Problem

Early on, I made the mistake of designing cards with too many keywords. A card with Echo + AOE + Piercing + Growing + Leech sounds amazing on paper, but in practice:

1. It's overwhelming to read.
2. It's impossible to balance.
3. It removes decision-making — the card is obviously always the best play.

I now cap most cards at **2-3 keywords**, with rare exceptions for Rare-tier cards that can have up to 4. This constraint forces each card to have a clear identity: *this card does X and Y, and it does them well.*

## Testing 20+ Keywords

Testing combinatorial systems is a nightmare if you do it manually. I built an automated test suite that generates every possible keyword pair and validates that they resolve without errors:

```typescript
// Pseudocode for keyword combination testing
for (const kw1 of ALL_KEYWORDS) {
  for (const kw2 of ALL_KEYWORDS) {
    const testCard = createCard({ keywords: [kw1, kw2] });
    const result = testCard.play(dummyEnemy, dummyState);
    assert(result.resolved === true);
    assert(result.errors.length === 0);
  }
}
```

This caught dozens of edge cases I never would have found through manual playtesting. For example: **Chain + Exhaust** had a bug where a Chained card that Exhausted mid-chain would try to play a card that no longer existed in the deck. The automated tests caught it in seconds.

## Extending the System

One of the joys of a well-designed keyword system is how easy it is to add new keywords. Recently I added **Siphon**, which steals a portion of the enemy's shield for the player. Implementation took about 30 minutes:

1. Define the keyword's phase (on-hit).
2. Write the resolution function (check enemy shield, transfer portion to player).
3. Add tooltip text and battle log messages.
4. Write unit tests for Siphon × every other keyword.
5. Create a few cards that use it.

The pipeline architecture handled everything else. No refactoring, no cascading changes, no breakage. That's the payoff of investing in good architecture upfront.

## What's Next for Keywords

I'm currently prototyping two new keywords: **Overclock** (spend extra resources for amplified effects) and **Cascade** (trigger effects that ripple to adjacent cards in hand). Both are still in the experimental phase, but early testing is promising.

The keyword system is, in many ways, the soul of Signal Forge. Every card is defined by its keywords, and every run is shaped by which keyword combinations you discover and exploit. Building this system has been the most technically challenging and creatively rewarding part of the entire project.
