## The Arms Race

Anti-cheat in gaming has become an arms race, and like most arms races, the escalation has produced collateral damage.

**Riot Games' Vanguard**, the kernel-level anti-cheat for Valorant, runs at the highest privilege level of your operating system — Ring 0, the same level as your device drivers. It loads at boot, before you even open the game, and monitors system activity continuously. Riot argues this is necessary to detect sophisticated cheats that operate at the kernel level.

**Easy Anti-Cheat (EAC)** and **BattlEye**, the two most widely deployed third-party anti-cheat solutions, also operate with elevated system privileges, scanning running processes, loaded DLLs, and memory regions for known cheat signatures.

The security community has raised legitimate concerns about this approach. Kernel-level anti-cheat represents a significant attack surface — any vulnerability in the anti-cheat driver is a vulnerability in the operating system itself. In 2023, researchers demonstrated proof-of-concept attacks exploiting anti-cheat drivers, and multiple EAC and BattlEye vulnerabilities have been publicly disclosed over the years.

Beyond security, there's a philosophical issue: **players are being asked to grant game companies the highest level of system access to play a video game.** For many players, especially those with security expertise, this tradeoff feels disproportionate.

## Browser Games: A Different Threat Model

Browser games face a fundamentally different cheating landscape, and it starts with one unavoidable reality: **the client-side code is visible**.

In a native game, the executable is compiled to machine code. Reverse engineering it requires disassemblers, debuggers, and significant expertise. In a browser game, the game logic is JavaScript — viewable in the browser's DevTools with a right-click. Even minified and obfuscated JavaScript can be read and modified by anyone with moderate technical skill.

This means that many traditional anti-cheat strategies simply don't apply:

- **Code obfuscation** slows down casual inspection but doesn't stop determined cheaters. JavaScript obfuscation tools (Terser, javascript-obfuscator) make code harder to read, but the runtime behavior is still inspectable through breakpoints and console logging.
- **Client-side validation** can be bypassed. Any check that runs in the browser can be disabled by modifying the JavaScript. If the client says "this score is legitimate," the client can be made to lie.
- **Memory scanning** isn't possible. Browsers sandbox JavaScript execution and don't expose process memory to web applications. This prevents both cheats and anti-cheat from operating at the memory level.

For some studios, this leads to despair: "browser games can't be secured." We disagree. They just need a different philosophy.

## Server-Authoritative Architecture

The foundation of our anti-cheat strategy is a principle borrowed from networked game design: **the server is the authority on game state**.

In practice, this means:

### The server validates every meaningful action

When a player in RMH Type submits a typing speed, the server doesn't just record the number the client reports. The server has the original text, the timestamps of each keystroke (sent via WebSocket), and the rules for what constitutes a valid input sequence. It independently calculates the words-per-minute from the raw keystroke data.

When a player in RMHbox scores a point in a minigame, the server has the game state, the physics simulation (or a simplified model of it), and the rules for scoring. The client's reported score is a hint, not a verdict.

### The server enforces rate limits

Rate limiting isn't just for API endpoints — it's a critical anti-cheat tool. If a player's client reports 200 actions per second in a game where human input maxes out at roughly 15 actions per second, something is wrong. Our game servers enforce per-player rate limits on:

- Input frequency (keypresses, clicks, movement commands)
- State change requests (item pickups, ability activations)
- Network message volume (total WebSocket messages per second)

Rate limits are set with generous margins above humanly possible input rates, so legitimate players are never affected. But automated scripts and modified clients that flood the server with inputs are caught immediately.

### The server runs authoritative simulations

For competitive games, the server runs its own simulation of the game state. Client inputs are fed into the server simulation, and the server's state is canonical. If the client and server disagree — for example, if the client claims the player is at position X but the server's simulation has them at position Y — the server's state wins, and the client is corrected.

This is computationally expensive. Running server-side simulations for every active game session requires significant server resources. We accept this cost because the alternative — trusting the client — makes competitive integrity impossible.

## What We Don't Do

We're equally deliberate about what we *don't* do:

### We don't install anything on your system

Our games run in a browser tab. They don't install kernel drivers, background processes, or persistent monitoring software. When you close the tab, the game is gone. This is a fundamental advantage of the browser as a gaming platform, and we consider it non-negotiable.

### We don't use fingerprinting for tracking

Browser fingerprinting — collecting device characteristics (screen resolution, installed fonts, GPU model, etc.) to create a unique identifier — is a technique used by some anti-cheat systems to identify banned users who create new accounts. We don't do this because we believe fingerprinting is a privacy violation. We use standard authentication (accounts, session tokens) and accept that determined evaders will create new accounts.

### We don't pretend cheating is impossible

Here's the honest part: **we cannot prevent all cheating in browser games.** A sufficiently motivated cheater can modify the client-side code, automate inputs, or reverse-engineer our network protocol. Our server-authoritative architecture catches the vast majority of cheating attempts, but we can't claim 100% coverage.

We think this honesty is important because the alternative — claiming your anti-cheat is impenetrable — is a lie that erodes trust when it's inevitably disproven.

## The Practical Toolkit

Beyond server authority, we use several practical techniques:

- **Statistical anomaly detection.** We log gameplay metrics (reaction times, accuracy patterns, score distributions) and flag statistical outliers for review. A typing speed of 250 WPM on RMH Type isn't impossible (the world record is around 300 WPM for short bursts), but it's unusual enough to warrant verification.

- **Replay validation.** For leaderboard-eligible scores, we store the complete input sequence and replay it server-side to verify that the inputs produce the claimed result. This is where our deterministic physics engine (discussed in our Laundry Sort physics post) pays dividends — deterministic simulation means replays are exact.

- **Timing analysis.** Human input has characteristic timing patterns — variable intervals between actions, reaction time distributions that follow known curves. Automated inputs tend to have unnaturally consistent timing. We analyze input timing patterns as one signal (not a sole determinant) in our cheat detection.

- **Graceful degradation.** When we detect suspected cheating, we don't ban immediately. We remove the player's scores from public leaderboards and flag the account for review. False positives happen, and permanent bans based on automated detection alone are unjust.

## The Transparency Principle

We publish our approach to anti-cheat (you're reading it) because we believe **security through obscurity is a weak foundation**. Our server-authoritative architecture doesn't depend on cheaters not knowing how it works. It depends on the mathematical reality that the server has information the client doesn't, and the server's validation logic runs in an environment the cheater can't modify.

This transparency serves several purposes:

- **It sets expectations.** Players know that leaderboards are server-validated, that scores are auditable, and that we take competitive integrity seriously.
- **It builds trust.** When we say a leaderboard is fair, players can understand *why* we believe that, not just take our word for it.
- **It invites scrutiny.** If our approach has weaknesses, we'd rather have the community identify them than discover them through exploitation.

## The Broader Lesson

The kernel-level anti-cheat trend in gaming reflects an industry that has chosen escalation over architecture. Rather than designing systems where the server is authoritative, studios have tried to make the client tamper-proof — a goal that is theoretically impossible on hardware the player owns.

Browser games, by their nature, can't pursue the escalation path. The client is JavaScript. It's readable. It's modifiable. This constraint, which might seem like a weakness, is actually a forcing function for better architecture: **if you can't trust the client, you build systems that don't need to.**

That's the approach we've taken. It's not perfect. No anti-cheat is. But it's honest, it respects our players' devices and privacy, and it produces competitive environments that are fair for the overwhelming majority of players.

That's the best we can do, and we think it's enough.
