## The Core Problem

The entire soul of Project Vega lives or dies on one system: **Ghost Protocols**. When a player places a tower in Loop 1, it needs to reappear in Loop 2 as a spectral echo — still functioning, still firing, still part of the defense grid. And then Loop 3 needs to replay *both* Loop 1 and Loop 2's towers simultaneously.

This sounds straightforward until you start thinking about the edge cases. What if a tower was upgraded mid-loop? What if it was sold and replaced? What if a tower was placed at the exact same grid position in two different loops? These questions kept me up at night, scribbling state diagrams on sticky notes.

Today I want to walk you through the TimeManager — the system that records, serializes, and replays player actions across time loops.

## The TimeManager Architecture

At its heart, the TimeManager is an **action log**. Every meaningful player action during a loop is captured as a timestamped event:

```typescript
type ActionType = 'place_tower' | 'upgrade_tower' | 'sell_tower' | 'activate_ability';

interface LoopAction {
  id: string;
  timestamp: number;        // milliseconds into the 5-minute loop
  type: ActionType;
  towerType?: TowerType;
  position: GridPosition;
  upgradeLevel?: number;
  metadata?: Record<string, unknown>;
}

interface LoopRecord {
  loopNumber: number;
  actions: LoopAction[];
  finalState: TowerSnapshot[];
}
```

The `LoopRecord` captures the complete history of a loop. The `finalState` is a snapshot of every tower on the grid when the loop ends — this serves as both a validation checkpoint and a fast-load mechanism.

### Recording Actions

During active gameplay, the TimeManager listens for state changes in the Zustand store. Whenever the player places, upgrades, or sells a tower, the action is pushed to the current loop's action log:

```typescript
const recordAction = (action: Omit<LoopAction, 'id' | 'timestamp'>) => {
  const elapsed = Date.now() - loopStartTime;
  currentLoop.actions.push({
    ...action,
    id: crypto.randomUUID(),
    timestamp: elapsed,
  });
};
```

I'm using the browser's high-resolution timer rather than frame counts for timestamps. Frame-based timing introduces drift — if the framerate dips during Loop 1 but stays stable during Loop 2, the replay would desynchronize. Wall-clock time is the safer bet, and I quantize to the nearest 16ms (roughly one frame at 60fps) to avoid sub-frame jitter.

### Replaying as Ghost Protocols

When a new loop begins, the TimeManager initializes a **replay queue** for each previous loop:

```typescript
class GhostReplayManager {
  private queues: Map<number, LoopAction[]>;
  private cursors: Map<number, number>;

  tick(currentTime: number) {
    for (const [loopNum, actions] of this.queues) {
      const cursor = this.cursors.get(loopNum) ?? 0;
      while (cursor < actions.length && actions[cursor].timestamp <= currentTime) {
        this.executeGhostAction(loopNum, actions[cursor]);
        this.cursors.set(loopNum, cursor + 1);
      }
    }
  }
}
```

Each tick, the replay manager checks if any queued actions should fire based on the current loop time. When a ghost action triggers, it creates a **Ghost Tower** — visually distinct from player-placed towers (translucent, with a spectral glow) but mechanically functional.

## The Ghost Tower Entity

Ghost towers aren't just cosmetic copies. They're full tower entities with specific behavioral constraints:

- **They fire at enemies.** Ghost towers deal damage based on the stats they had when originally placed (including any upgrades applied during their original loop).
- **They cannot be sold or upgraded.** They exist as echoes of past decisions. What's done is done.
- **They occupy grid space.** This is critical for pathfinding — ghost towers block enemy paths just like regular towers, which is exactly the kind of emergent strategy I want.
- **They have a visual "loop signature."** Loop 1 ghosts glow green. Loop 2 ghosts glow amber. This lets players quickly identify which loop a ghost belongs to.

```typescript
interface GhostTower extends BaseTower {
  isGhost: true;
  sourceLoop: number;
  originalPlacementTime: number;
  canUpgrade: false;
  canSell: false;
  opacity: number;   // visual translucency
  glowColor: string; // loop-specific color
}
```

### Handling Conflicts

The trickiest problem was **positional conflicts**. What happens when a player tries to place a tower on a grid cell that a ghost tower will eventually occupy later in the timeline?

My solution: **ghost towers claim their cells from the moment the loop begins**, not when they "replay" into existence. The grid is pre-populated with reserved cells based on the action log. This means the player can see a faintly outlined reservation marker showing where a ghost will appear, but they can't build there.

This turned a potential frustration into a strategic element. Players need to plan their tower placements with awareness of where their past selves already committed. It reinforces the time-loop fantasy — you're literally working around your own history.

## Serialization and State Recovery

The complete loop history needs to survive page refreshes and potential crashes. I serialize the entire `LoopRecord` array to `localStorage` with a compact binary-like JSON format:

```typescript
const saveLoopHistory = (records: LoopRecord[]) => {
  const compressed = records.map(r => ({
    l: r.loopNumber,
    a: r.actions.map(a => [a.timestamp, a.type, a.position.x, a.position.y, a.towerType]),
    s: r.finalState,
  }));
  localStorage.setItem('vega_loops', JSON.stringify(compressed));
};
```

Each action compresses to a small tuple rather than a full object, keeping the storage footprint manageable even with hundreds of actions across three loops.

## The Compounding Effect

The real magic of Ghost Protocols isn't technical — it's emotional. Watching your Loop 3 unfold with the cumulative weight of all your previous decisions is *exhilarating*. Your Loop 1 Synapse towers are rattling away in the north corridor. Your Loop 2 Suppressors are slowing enemies in the chokepoint you identified after getting overwhelmed the first time. And now you're placing Loop 3 towers to fill the gaps you can see in real-time.

It feels like conducting an orchestra where each section was recorded at a different time. The music comes together in the final performance, and when it works — when your three-loop defense grid holds against the final wave — the satisfaction is unlike anything I've felt in other tower defense games.

## Lessons Learned

Building the TimeManager taught me to **separate recording from rendering**. Early prototypes tried to visually replay towers appearing in real-time, which created confusing visual noise. The current approach — pre-populating ghosts and showing them as already-present echoes — is cleaner and actually better for gameplay readability.

I also learned that deterministic replay is harder than it sounds. Floating-point precision, timing quantization, and state ordering all introduce subtle bugs. The quantization to 16ms boundaries was the breakthrough that made replays feel rock-solid.
