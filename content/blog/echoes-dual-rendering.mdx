## The Experiment That Became a Feature

Let me tell you how Echoes of the Spire ended up with two completely separate rendering modes. It wasn't planned. It wasn't on any roadmap. It started as a weekend experiment and turned into one of the game's defining features.

Echoes was originally built entirely in 2D using HTML5 Canvas. Top-down perspective, sprite-based, performant across basically every device with a browser. It worked. It looked good. I was happy with it.

Then one Saturday afternoon, I started messing around with 3D rendering. "What if the dungeon had depth?" I asked myself, fully expecting the answer to be "it would run at four frames per second and look terrible." I hacked together a basic first-person view of one of the dungeon rooms using a WebGL pipeline — walls with height, floor with perspective, enemies as billboarded sprites.

It ran at sixty frames per second. It looked *awesome*. And suddenly I had a problem: I loved both versions and couldn't bring myself to cut either one.

## The Architecture Challenge

Supporting two rendering modes sounds simple in theory — just swap the renderer, right? In practice, it required a fundamental rethinking of the game's architecture. The original codebase had game logic tangled up with rendering code. Movement calculations referenced pixel positions. Combat logic triggered Canvas draw calls directly. Everything was connected to everything.

The first step was **separating game state from presentation entirely**. This meant building a clean game state layer managed by Zustand that knew nothing about how it would be displayed. Positions became abstract coordinates. Animations became state transitions. Effects became events.

```typescript
// Game state knows NOTHING about rendering
interface GameState {
  player: {
    position: Vec2;
    facing: Direction;
    currentAnimation: AnimationState;
    abilities: AbilityState[];
  };
  enemies: EnemyState[];
  dungeon: DungeonState;
  // No references to canvas, WebGL, pixels, or draw calls
}
```

Then I built two completely independent renderers that both subscribe to this shared state:

- **Canvas2DRenderer** — Reads game state, translates abstract positions to screen coordinates, draws sprites and effects using the 2D Canvas API.
- **WebGL3DRenderer** — Reads the same game state, translates positions to 3D world coordinates, renders the scene with perspective, lighting, and depth.

Both renderers run at their own pace, pulling from the same source of truth. The game doesn't care which one is active. Switch mid-run if you want — the state is the state.

## 2D Mode: The Classic Experience

The 2D renderer is the more mature of the two, and in many ways it's the *tighter* experience. Top-down perspective gives you perfect spatial awareness. You can see the entire room, track all enemies simultaneously, and plan your movements with full information.

I spent months refining the 2D visual style. The dark purple aesthetic — deep violets, electric blues, magenta accents on a near-black background — creates a sci-fi dungeon atmosphere that feels distinct from the pixel-art roguelikes that dominate the genre. The "INITIALIZING..." loading screen with its purple monospace text sets the tone immediately: this isn't a retro throwback, it's something from the future.

Particle effects in 2D mode run on a custom lightweight particle system designed for Canvas performance. Hit sparks, ability effects, environmental particles — everything contributes to the visual density without tanking framerate. I keep a strict particle budget per frame and use object pooling to avoid garbage collection spikes.

## 3D Mode: Immersion at a Cost

The 3D renderer transforms the experience entirely. Suddenly you're *in* the dungeon, not above it. Corridors feel claustrophobic. Enemy encounters feel dangerous because you can't see what's behind you. The spatial awareness advantage of 2D becomes a tension-building limitation in 3D.

The 3D pipeline features a custom **DungeonGenerator** that extrudes the 2D floor plan into 3D geometry. Walls have height, floors have texture, and lighting casts real-time shadows that shift as you move. The **EnemyManager** handles 3D-specific behaviors like line-of-sight awareness and positional audio cues. The **PlayerController** manages first-person camera movement with mouse look and keyboard strafing. The **WeaponSystem** renders your equipped weapon in view space with procedural sway and attack animations.

Building all of this for the browser was an exercise in aggressive optimization. Every draw call counts. Geometry is instanced where possible. Textures are atlased. The level-of-detail system reduces polygon count for distant objects. Despite all of this, 3D mode is more demanding than 2D, and I'm transparent about that — the game detects device capability and suggests the appropriate mode.

### The Uncanny Valley of Web 3D

There's a psychological challenge specific to 3D games in the browser. Players expect browser games to be simple. When they see a 3D first-person view, they subconsciously compare it to native 3D games — Doom, Quake, modern indie 3D titles. The bar is suddenly much higher than it would be for a 2D game.

I addressed this by leaning into a **stylized aesthetic** rather than chasing realism. The 3D mode uses the same purple/blue/magenta color palette as 2D, with flat-shaded surfaces, emissive edges, and volumetric glow effects. It looks intentionally stylized, not accidentally low-poly. This framing matters enormously for player perception.

## Sharing State Between Renderers

The Zustand-based state management layer is the linchpin of the dual-rendering architecture. Here's what makes it work:

**Input abstraction.** Player input goes through an input manager that produces game-intent events (move north, use ability 1, dodge) regardless of whether those inputs came from WASD keys, a virtual D-pad, or mouse look. The game state processes intents identically.

**Animation as state transitions.** Instead of rendering-specific animation timelines, the game state tracks abstract animation phases: idle, attacking, using ability, taking damage, dying. Each renderer interprets these phases through its own animation system.

**Effect events.** When a combat hit lands, the game state emits a "hit event" with position, damage, type, and source. The 2D renderer translates this into screen shake and 2D particles. The 3D renderer translates it into a camera punch and 3D particle burst. Same event, different presentations.

## Was It Worth It?

Absolutely, unequivocally yes. Building dual rendering modes roughly doubled the rendering-side development effort, but it **didn't** double the total development effort because the game logic, balance, content, and systems are shared code. The architectural discipline it forced — clean state separation, input abstraction, render-agnostic design — made the entire codebase better.

And from a player perspective, the ability to toggle between 2D and 3D gives Echoes a unique identity. Some players prefer the strategic clarity of 2D. Others crave the immersion of 3D. A few switch between them depending on the class they're playing — Vanguard feels incredible in 3D, Spectre arguably plays better in 2D.

Building both was one of the best accidental decisions I've made on this project. Sometimes the best features are the ones you didn't plan.
