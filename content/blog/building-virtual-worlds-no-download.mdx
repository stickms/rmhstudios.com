## The Promise of Instant Worlds

There's a particular kind of magic in clicking a link and finding yourself inside a living, breathing virtual world. No installer, no progress bar, no "restart required." Just a URL, a browser tab, and a world waiting for you.

That's the experience we're building at RMH Studios, and it's made possible by a stack of web technologies that have matured dramatically over the past few years. In this post, we'll break down exactly how we build 3D virtual worlds that run entirely in the browser.

## Three.js: The Foundation

**Three.js** is the workhorse of browser-based 3D rendering. Created by Ricardo Cabello (Mr.doob) and first released in 2010, Three.js provides a high-level JavaScript API on top of WebGL (and increasingly WebGPU), abstracting away the complexity of shader programming, buffer management, and GPU communication.

For game development, Three.js gives us:

- **Scene graph management** — organizing objects, lights, cameras, and groups in a hierarchical structure
- **Material and lighting systems** — physically-based rendering (PBR), environment maps, shadow casting
- **Geometry and mesh handling** — from primitive shapes to complex imported models (glTF, FBX, OBJ)
- **Animation systems** — skeletal animation, morph targets, and keyframe interpolation
- **Post-processing** — bloom, depth of field, color grading, and custom shader effects

Three.js doesn't try to be a game engine in the way Unity or Unreal are. It's a rendering library — and that distinction is important. It gives us the building blocks without imposing an opinionated framework on top. For a studio that wants fine-grained control over every aspect of the player experience, that flexibility is invaluable.

## React Three Fiber: 3D Meets Declarative UI

Raw Three.js is powerful but imperative. You create objects, add them to scenes, manage their lifecycle manually, and handle cleanup yourself. When you're building complex interactive experiences — especially ones that need to integrate with a React-based UI layer — this imperative approach creates friction.

**React Three Fiber** (R3F) solves this elegantly. Built by the Poimandres collective, R3F is a React renderer for Three.js. It lets us describe 3D scenes declaratively using JSX, with all the benefits of React's component model: composability, state management, lifecycle hooks, and a rich ecosystem of supporting libraries.

Here's what that means in practice. Instead of manually creating a mesh, adding it to a scene, updating it on every frame, and disposing of it when it's no longer needed, we write a React component:

```jsx
function FloatingCrystal({ position }) {
  const meshRef = useRef();
  useFrame((state) => {
    meshRef.current.rotation.y = state.clock.elapsedTime * 0.5;
    meshRef.current.position.y = position[1] + Math.sin(state.clock.elapsedTime) * 0.3;
  });

  return (
    <mesh ref={meshRef} position={position}>
      <octahedronGeometry args={[0.5, 0]} />
      <meshStandardMaterial color="#8b5cf6" emissive="#4c1d95" emissiveIntensity={0.4} />
    </mesh>
  );
}
```

This component handles its own animation, cleans up automatically when unmounted, and composes naturally with other React components. The R3F ecosystem also includes libraries like **@react-three/drei** (a collection of useful abstractions) and **@react-three/postprocessing** (GPU post-processing effects), which accelerate development significantly.

Since our entire frontend stack is built on **Next.js and React**, R3F integrates seamlessly with our UI layer, routing, and state management. A 3D game scene and a 2D settings menu can live in the same component tree, sharing state naturally.

## Echoes of the Spire: A Case Study

Our game **Echoes of the Spire** is a narrative extraction thriller that takes place in atmospheric, procedurally-influenced 3D environments. Building it as a browser game meant solving several interesting technical challenges.

### Asset Loading and Streaming

A native game can assume its assets are on disk. A browser game has to download everything over the network. For Echoes of the Spire, we use aggressive asset optimization:

- **glTF with Draco compression** reduces 3D model sizes by 80-90% compared to uncompressed formats
- **Texture atlasing and compression** using basis/KTX2 formats keeps texture memory manageable
- **Progressive loading** renders the scene with low-detail placeholders while higher-quality assets stream in
- **Asset caching** via service workers ensures returning players don't re-download the same assets

### Atmospheric Rendering

The mood of Echoes of the Spire depends on atmospheric effects that are traditionally expensive: volumetric fog, dynamic lighting, particle systems. In the browser, we achieve these through a combination of:

- Custom **shader materials** written in GLSL for fog and lighting effects
- **Instanced rendering** for particle systems, keeping draw calls low even with thousands of particles
- **Baked lighting** for static environments, supplemented by a small number of dynamic lights for interactive elements

The result isn't photorealistic — and it doesn't need to be. It's atmospheric, evocative, and runs at 60fps on mid-range hardware in a browser tab.

### Performance Budgets

Browser 3D development requires disciplined performance budgeting. We set hard limits:

- **Draw calls**: under 100 per frame
- **Triangle count**: under 200K visible triangles
- **Texture memory**: under 128MB total
- **Frame time**: under 16.6ms (targeting 60fps)

Every asset, every effect, every system has to justify its cost against these budgets. This constraint actually improves our design — it forces us to find creative solutions that achieve visual impact without brute-force rendering.

## WebXR: The Future of Browser Worlds

The **WebXR Device API** is the W3C specification for accessing virtual reality and augmented reality devices through the browser. It's still early — headset adoption is growing but not yet mainstream, and the API surface is still evolving — but the implications for browser-based virtual worlds are enormous.

WebXR means that the same 3D world we render in a browser tab could, in principle, be experienced in full VR through a headset, all without a native application. The user clicks a "Enter VR" button, puts on their headset, and they're *inside* the world they were just viewing on their screen.

We're not shipping VR experiences today, but our architecture is WebXR-aware. Our camera systems, interaction models, and scene structures are designed with the possibility of VR in mind. When the headset install base reaches critical mass, we want to be ready.

## Why This Matters

Building 3D virtual worlds in the browser isn't just a technical curiosity — it's a philosophical statement about accessibility. Every virtual world that requires a download, an install, a specific operating system, or a particular piece of hardware is a world with a velvet rope.

**Browser-based virtual worlds have no velvet rope.** If you have a browser, you're in. That democratization of access is what drives us at RMH Studios. We're building worlds that anyone can visit, instantly, from anywhere.

The technology to make those worlds beautiful, immersive, and performant exists today. Three.js, React Three Fiber, WebGPU, and WebXR are the tools. The only thing left is the craft to wield them well. That's what we show up to do every day.
