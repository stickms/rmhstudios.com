## 60+ Accessibility Options in a AAA Game

When Naughty Dog shipped **The Last of Us Part II** in June 2020, it included over 60 accessibility options — a number that stunned the industry. Players could enable high-contrast mode, text-to-speech for all UI elements, audio cues for navigation, remappable controls, auto-aim with granular adjustment, and dozens of other features that made the game playable by people with a wide range of physical, visual, auditory, and cognitive abilities.

The game was praised not just by accessibility advocates but by mainstream reviewers and players. It won multiple accessibility awards and set a new benchmark that other studios have been measured against ever since.

The lesson was clear: **accessibility isn't a niche feature — it's a mark of quality.** A game that more people can play is a better game, period.

We took that lesson seriously. As a browser game studio, we have both unique challenges and unique advantages when it comes to accessibility. This post is about how we think about it and what we actually do.

## The Legal and Ethical Landscape

Accessibility in digital products isn't just a best practice — it's increasingly a legal requirement.

The **21st Century Communications and Video Accessibility Act (CVAA)**, signed into law in 2010 in the United States, requires that communications functionality in games and software be accessible to people with disabilities. While the CVAA's gaming provisions (which took effect in 2019 after an extended waiver) focus primarily on communication features — voice chat, text chat, game menus related to communication — the law signals a clear regulatory direction.

The **Web Content Accessibility Guidelines (WCAG) 2.1**, published by the W3C in 2018, provide the technical standard for web accessibility. WCAG defines four principles — perceivable, operable, understandable, and robust — and three conformance levels: A, AA, and AAA. For web-based applications, WCAG 2.1 AA is widely considered the minimum standard, and it's the standard that courts have referenced in accessibility lawsuits against web companies.

For us, this legal landscape isn't a threat — it's a framework. WCAG gives us concrete, testable criteria for accessibility. The CVAA gives us guidance on where the floor is. And our own values tell us to aim well above the floor.

## Browser Games: The Unique Advantages

Building games for the browser gives us accessibility advantages that native game developers don't have:

### The platform does a lot of the work

Browsers are the most accessible application runtime in existence. Screen readers (NVDA, JAWS, VoiceOver) are built to interact with web content. The HTML semantics system — headings, landmarks, ARIA roles, form labels — provides a rich vocabulary for communicating UI structure to assistive technology.

When we build a game menu as semantic HTML rather than a canvas-rendered custom UI, screen readers can navigate it automatically. When we use proper `<button>` elements instead of `<div onClick>` handlers, keyboard users get focus management for free. The web platform has spent decades solving these problems.

### No installation barrier

A browser game requires no app store download, no hardware driver installation, no compatibility check. If your device has a browser, you can play. This is itself an accessibility feature — it removes the financial and technical barriers that prevent many people from accessing native games.

### CSS handles responsive design

With CSS media queries, `prefers-reduced-motion`, `prefers-color-scheme`, `prefers-contrast`, and viewport-relative units, we can adapt our games to a huge range of display configurations without requiring the player to find and adjust an in-game settings menu.

## What We Actually Implement

Here are the specific accessibility features we build into our games:

### Keyboard Navigation

Every interactive element in every RMH game is reachable and operable via keyboard. Tab order follows logical reading order. Focus indicators are visible and high-contrast. Game actions can be triggered with Enter, Space, or custom keybindings.

For games with real-time input (rhythm games like Slice It!, reflex-based minigames in RMHbox), we provide keyboard alternatives to click/touch targets and configurable key bindings so players can use whatever layout works for their abilities.

### Color and Contrast

We follow WCAG 2.1 AA contrast ratios as a minimum: **4.5:1 for normal text, 3:1 for large text and UI components**. We test our color palettes with tools like the Colour Contrast Analyser and browser-based simulators for protanopia, deuteranopia, and tritanopia.

Critically, we **never rely on color alone to convey information**. In a game like Laundry Sort, where items need to be sorted by category, we use shape, pattern, and label in addition to color. In Signal Forge, card types are distinguished by icon and border style, not just color.

### Screen Reader Support

Our game menus, lobbies, leaderboards, and non-gameplay UI are built with semantic HTML and ARIA attributes. We use `aria-live` regions to announce state changes — "Round 3 of 5," "Your score: 1,200," "Timer: 30 seconds remaining" — so screen reader users have real-time awareness of game state.

Real-time gameplay is harder to make screen-reader accessible, and we're honest about the fact that some of our games — particularly visually intensive ones like Neon Driftway and Echoes of the Spire — have core mechanics that are fundamentally visual. We're researching sonification and haptic alternatives, but we won't pretend we've solved this for every game.

### Reduced Motion

We respect the `prefers-reduced-motion` CSS media query. When a player's operating system indicates a preference for reduced motion, we:

- Disable parallax scrolling and decorative animations
- Replace animated transitions with instant state changes
- Reduce particle effects and screen shake
- Provide static alternatives to animated UI elements

This isn't just about vestibular disorders — it also improves battery life, reduces distraction, and makes the experience more comfortable for anyone who finds excessive motion unpleasant.

### Text and Font Options

We support browser-level text scaling. Our UI layouts are built with relative units (`rem`, `em`, `%`) so that increasing the browser's default font size doesn't break layouts. We avoid text embedded in images, and we provide sufficient line spacing and paragraph width to support comfortable reading.

## The Ongoing Work

Accessibility is not a feature you ship once and check off a list. It's a **continuous practice** that requires testing, feedback, and iteration.

We maintain an accessibility checklist that we run against every new game and every major update. We test with keyboard-only navigation, screen readers (VoiceOver on macOS, NVDA on Windows), and simulated vision conditions. We actively seek feedback from players with disabilities, because automated testing catches conformance issues but misses usability problems.

We also recognize our limitations. We're a small studio. We don't have a dedicated accessibility team with the resources of Naughty Dog or Microsoft. Some of our older games have accessibility gaps that we're working to close. We'd rather be honest about the current state and committed to improvement than claim we've solved accessibility completely.

## Why This Matters for the Industry

The browser game space — and the indie game space more broadly — has historically treated accessibility as an afterthought. The assumption is that accessibility is expensive, time-consuming, and relevant only to a small percentage of players.

All three assumptions are wrong. **The World Health Organization estimates that 1.3 billion people — roughly 16% of the global population — experience significant disability.** When you build an inaccessible game, you're telling 16% of your potential audience that they don't matter.

The good news is that the web platform makes accessibility easier than any other game distribution platform. The tools exist. The standards exist. The only thing missing is the will.

We're committed to doing this work, and we'll keep sharing what we learn.
