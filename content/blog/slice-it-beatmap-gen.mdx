## The Magic Trick

If there's one feature that defines Slice It!, it's this: **drop in any song, and the game creates a beatmap for you.** No editor. No manual charting. No waiting for a community mapper to decide your favorite obscure track is worth their time.

This is the feature that excited me the most when I started the project, and it's the feature that nearly broke me. Automatic beatmap generation sits at the intersection of digital signal processing, music theory, and game design. Getting it *technically* working was hard. Getting it to feel *musically satisfying* was even harder.

Let me show you how it works.

## The Pipeline

The beatmap generation system is a multi-stage pipeline that transforms raw audio into a sequence of timed game notes. Here's the high-level flow:

1. **Audio Decoding** — The uploaded file is decoded into raw PCM sample data
2. **Mono Mixdown** — Stereo channels are combined for analysis
3. **Spectral Analysis** — The audio is processed through windowed FFT to extract frequency content over time
4. **Onset Detection** — Transient events (beats, hits, attacks) are identified
5. **Feature Extraction** — Additional musical features like energy, spectral flux, and frequency band ratios are computed
6. **Note Placement** — Detected events are mapped to game notes with positions, types, and difficulty ratings
7. **Post-Processing** — The raw note sequence is filtered, balanced, and polished for playability

Each stage feeds into the next, and each stage has its own set of challenges.

## Spectral Analysis with FFT

The foundation of everything is the **Fast Fourier Transform (FFT)**. We process the entire audio buffer in overlapping windows (typically 2048 samples with 75% overlap at 44.1kHz), converting each window from the time domain into the frequency domain.

```typescript
function computeSpectrogram(samples: Float32Array, fftSize: number): Float32Array[] {
  const hopSize = fftSize / 4;
  const frames: Float32Array[] = [];
  const fft = new FFT(fftSize);

  for (let i = 0; i + fftSize <= samples.length; i += hopSize) {
    const window = samples.slice(i, i + fftSize);
    applyHannWindow(window);
    const spectrum = fft.forward(window);
    frames.push(spectrum);
  }

  return frames;
}
```

The result is a **spectrogram** — a 2D representation of frequency content over time. Each frame tells us exactly what frequencies are present at that moment in the song. Bass-heavy moments light up the low frequency bins. Cymbal crashes activate the high frequencies. Vocal notes create distinctive patterns in the mid-range.

## Onset Detection: Finding the Beats

With the spectrogram in hand, we need to identify **onsets** — the moments where new musical events begin. A snare hit, a guitar chord, a vocal syllable, a synth note — these are all onsets.

The core technique is **spectral flux**, which measures how much the frequency content changes between consecutive frames:

```typescript
function computeSpectralFlux(spectrogram: Float32Array[]): number[] {
  const flux: number[] = [];
  
  for (let i = 1; i < spectrogram.length; i++) {
    let sum = 0;
    for (let bin = 0; bin < spectrogram[i].length; bin++) {
      const diff = spectrogram[i][bin] - spectrogram[i - 1][bin];
      sum += Math.max(0, diff); // only positive changes (new energy)
    }
    flux.push(sum);
  }
  
  return flux;
}
```

The key insight is that we only count **positive** changes in energy. We don't care about energy fading away — we care about new energy appearing. A sharp spike in spectral flux almost always corresponds to a musical onset.

But raw spectral flux is noisy. A single sustained guitar chord might produce minor fluctuations that trigger false onsets. We apply an **adaptive threshold** — a running average of the flux values, scaled by a sensitivity parameter. Only flux values that exceed the threshold are marked as onsets.

```typescript
function detectOnsets(flux: number[], sensitivity: number): number[] {
  const windowSize = 10;
  const onsets: number[] = [];
  
  for (let i = windowSize; i < flux.length; i++) {
    const windowAvg = flux.slice(i - windowSize, i)
      .reduce((a, b) => a + b, 0) / windowSize;
    
    if (flux[i] > windowAvg * sensitivity) {
      onsets.push(i);
    }
  }
  
  return onsets;
}
```

## Sub-Band Analysis

Not all onsets are equal. A kick drum and a hi-hat are both onsets, but they carry very different musical weight. To distinguish between them, we split the frequency spectrum into **sub-bands**:

- **Low band (20–300 Hz):** Bass, kick drums, bass guitar
- **Mid band (300–2000 Hz):** Vocals, guitars, snare body
- **High band (2000–16000 Hz):** Hi-hats, cymbals, sibilance, brightness

We run onset detection independently on each band and use the results to inform note placement. Low-band onsets tend to generate notes in the center of the playfield (they feel "heavy"). High-band onsets generate notes at the edges (they feel "light" and "quick"). Mid-band onsets fill the space between.

This sub-band approach is what makes generated beatmaps feel **musical** rather than random. The spatial layout of notes on screen reflects the spectral content of the music. Players start to intuitively associate positions with sounds, creating a deeper connection between gameplay and audio.

## Making It Feel Right

The hardest part of beatmap generation isn't the signal processing — it's the game design. A technically accurate onset detection system will happily generate 15 notes per second during a busy passage, creating an unplayable wall of notes. The system needs to understand not just *what's happening* in the music, but *what's fun to play*.

We apply several post-processing steps to transform raw onsets into enjoyable gameplay:

### Minimum Note Spacing

Notes that are too close together are physically impossible to hit. We enforce a minimum gap between consecutive notes, merging nearby onsets into single notes.

### Difficulty Curves

Rather than maintaining a constant density, the system analyzes the overall energy profile of the song and creates natural difficulty curves. Quiet passages have fewer notes, building tension. Intense sections explode with activity. This mirrors how human mappers create charts — they follow the emotional arc of the music.

### Pattern Variety

Raw onset data tends to produce monotonous patterns — notes in the same lane over and over because the spectral content doesn't change much. We inject controlled randomness into lane assignment, weighted by the frequency band analysis, to ensure visual variety while maintaining musical coherence.

### Anchor Points

Certain musical events are too important to skip — a massive bass drop, a dramatic silence before a chorus, a signature riff. We identify these **anchor points** using energy analysis and ensure they always produce memorable gameplay moments, like simultaneous multi-note hits or distinctive patterns.

## The Result

After all of this processing, we produce a JSON structure that the game engine consumes:

```typescript
interface BeatmapNote {
  time: number;       // seconds from start
  lane: number;       // horizontal position (0-3)
  type: 'tap' | 'hold' | 'slide';
  duration?: number;  // for hold notes
  difficulty: number; // contribution to difficulty rating
}
```

The entire pipeline runs in a **Web Worker** to avoid blocking the main thread. For a typical 4-minute song, generation takes 2–5 seconds on modern hardware — fast enough that players barely notice the wait.

## Ongoing Evolution

Beatmap generation is never "done." Every song genre has its quirks, and what works brilliantly for electronic music might produce mediocre results for acoustic jazz. We're continuously tuning the parameters, adding genre-aware heuristics, and incorporating player feedback to make the system smarter.

The dream is a system that generates maps indistinguishable from hand-crafted ones. We're not there yet. But every iteration gets closer, and every time I generate a map for a song I love and find myself genuinely enjoying the gameplay, I know we're on the right track.
