## 13 Years of Three.js

In April 2010, **Ricardo Cabello** (known online as Mr.doob) published the first version of Three.js — a JavaScript library that made WebGL accessible to developers who didn't want to write raw shader code. At the time, WebGL itself was barely supported: Chrome and Firefox had experimental implementations, Safari wouldn't add support until 2014, and the idea of running 3D graphics in a browser tab was considered a novelty.

Thirteen years later, Three.js is the most widely used 3D library on the web. It powers everything from data visualizations and architectural walkthroughs to interactive art installations and, increasingly, games. The library has been downloaded over a billion times on npm. It has over 100,000 GitHub stars. And it remains actively developed, with Cabello and a community of contributors pushing regular updates.

Three.js succeeded because it solved the right problem at the right level of abstraction. WebGL is a low-level API — think OpenGL ES 2.0 exposed through JavaScript. Writing a WebGL application from scratch means managing shader compilation, buffer allocation, texture binding, matrix math, and render state manually. Three.js wraps all of that in a scene graph model that feels natural to anyone who's worked with a game engine: scenes, cameras, meshes, materials, lights.

For a browser game studio like ours, Three.js is foundational infrastructure.

## React Three Fiber: The Declarative Revolution

In 2019, **Paul Henschel** (Poimandres collective) released **React Three Fiber (R3F)** — a React renderer for Three.js. Instead of imperatively creating and managing Three.js objects, R3F lets you declare your 3D scene as JSX:

```jsx
<Canvas>
  <ambientLight intensity={0.5} />
  <mesh position={[0, 1, 0]}>
    <boxGeometry args={[1, 1, 1]} />
    <meshStandardMaterial color="orange" />
  </mesh>
</Canvas>
```

This might look like a trivial syntactic convenience, but the implications are profound:

- **Component composition.** 3D objects become React components. A dungeon room can be a `<DungeonRoom>` component that accepts props for layout, enemies, and lighting. An inventory panel can be a reusable `<InventoryUI>` that renders in 3D space.

- **State management integration.** Three.js objects participate in React's state system. When game state changes (a door opens, an enemy appears, a light flickers), the 3D scene updates through the same reactive data flow that drives the rest of the application.

- **Ecosystem access.** The entire React ecosystem — hooks, context, suspense, dev tools — applies to 3D rendering. `useState` for animation state. `useRef` for direct Three.js object access. `Suspense` for async asset loading with fallbacks.

- **Ecosystem of R3F libraries.** The Poimandres community has built an extraordinary ecosystem around R3F: `@react-three/drei` (helper components for cameras, controls, loaders, effects), `@react-three/postprocessing` (screen-space effects), `@react-three/rapier` (physics integration), and more.

For a studio that's already building its entire platform in React and Next.js, R3F means our 3D games share tooling, patterns, and mental models with the rest of our codebase.

## Echoes of the Spire: 3D in Production

**Echoes of the Spire**, our narrative extraction thriller, is the most 3D-intensive game in our catalog. The player explores a procedurally generated spire — a vertical dungeon of interconnected rooms, corridors, and chambers — viewed from a third-person perspective.

Here's how we use Three.js and R3F in practice:

### Procedural Dungeon Rendering

Each floor of the spire is generated from a room graph. Rooms have types (combat, puzzle, narrative, rest) that determine their visual style. We convert the room graph into Three.js geometry:

- **Walls and floors** are instanced meshes with shared geometry and per-instance materials. Instancing means the GPU draws hundreds of wall segments in a single draw call.
- **Lighting** is per-room. Each room has a primary light source that reflects its mood: warm amber for rest areas, cold blue for frozen chambers, flickering red for danger zones. We use Three.js `PointLight` with custom attenuation to keep light contained within rooms.
- **Props** (furniture, debris, artifacts) are loaded as GLTF models via `@react-three/drei`'s `useGLTF` hook, which handles async loading, caching, and Suspense integration.

### Performance on the Web

Running 3D graphics at 60fps in a browser requires aggressive optimization. Our approaches:

- **Level of Detail (LOD).** Objects farther from the camera use simpler geometry. Three.js's `LOD` class switches between detail levels automatically based on distance.
- **Frustum culling.** Three.js culls objects outside the camera's view frustum by default, but we additionally cull entire rooms that aren't visible based on the room graph connectivity.
- **Texture atlasing.** Rather than loading dozens of individual textures, we pack related textures into atlases and use UV offsets to select the right region. This reduces texture binding state changes.
- **Deferred rendering considerations.** For scenes with many lights, we evaluate whether forward or deferred rendering is more efficient. Three.js uses forward rendering by default, which struggles with many dynamic lights. For complex scenes, we limit active lights and use baked lighting where possible.

### The R3F Advantage

The declarative model pays off in scene composition. A room component looks like:

```jsx
function DungeonRoom({ layout, enemies, mood }) {
  return (
    <group>
      <RoomGeometry layout={layout} />
      <RoomLighting mood={mood} />
      {enemies.map(e => <Enemy key={e.id} {...e} />)}
      <AmbientParticles mood={mood} />
    </group>
  );
}
```

When the game state changes — an enemy is defeated, the mood shifts, a door opens — React's reconciliation updates only the components that changed. We don't manually traverse the scene graph to find and modify objects. The declarative model handles it.

## WebGPU: The Future

In **April 2023**, Chrome 113 shipped with **WebGPU enabled by default** — the first browser to make the next-generation graphics API generally available. WebGPU is to WebGL what Vulkan is to OpenGL: a lower-level, more explicit API that gives developers greater control over GPU resources.

The implications for browser 3D are significant:

### Compute Shaders

WebGL has no compute shader support. If you want to run general-purpose computation on the GPU — particle systems, cloth simulation, AI pathfinding — you either do it on the CPU or hack it through render-to-texture tricks. WebGPU provides proper compute shaders, enabling GPU-accelerated physics, procedural generation, and post-processing that simply wasn't possible before.

### Reduced CPU Overhead

WebGL's driver model requires the CPU to validate every API call, creating overhead that limits draw call counts. WebGPU uses a command buffer model where commands are recorded, validated once, and submitted in batches. This means more draw calls per frame, which means more complex scenes.

### Better Resource Management

WebGPU gives explicit control over memory allocation, buffer binding, and pipeline state. For experienced graphics programmers, this means the ability to optimize rendering paths in ways that WebGL's abstraction prevented.

### Three.js and WebGPU

Three.js has been developing a **WebGPU renderer** (`WebGPURenderer`) alongside its existing WebGL renderer. The Three.js team, led by sunag (Fernando Serrano), has been building a node-based material system (`TSL` — Three.js Shading Language) that abstracts over both WebGL and WebGPU backends.

For us, this means we can adopt WebGPU progressively — using it on browsers that support it while falling back to WebGL on others — without rewriting our rendering code.

## Where We're Headed

Our investment in Three.js and R3F is long-term. The combination gives us:

- **A mature, battle-tested 3D library** with 13 years of development and a massive community
- **A React-native integration** that fits our existing architecture
- **A path to WebGPU** through Three.js's abstraction layer
- **An ecosystem of tools and libraries** that accelerate development

We're exploring expanded use of 3D in our catalog — environmental storytelling in narrative games, spatial UI for party games, 3D visualizations for competitive games. The browser's 3D capabilities have crossed a threshold where the question is no longer "can we do 3D in the browser?" but "what should we build with it?"

The answer, as always, is games that run in a tab, load in seconds, and require nothing but a URL. Now in three dimensions.
