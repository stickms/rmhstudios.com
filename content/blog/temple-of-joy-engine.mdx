## The Problem with Mutable State

Let me tell you about the worst bug I've ever shipped.

Early in Temple of Joy's development, the game loop directly mutated the Zustand store. Every tick — sixty times per second — the loop would reach into state, increment happiness, apply multipliers, check achievements, update buffs, and write everything back. It worked. For about three hours. Then I noticed that occasionally, after a tab went to background and came back, a player's happiness would jump backward. Not by a lot. Just enough to feel deeply wrong. Sometimes buffs would silently expire twice. Sometimes a purchase would deduct cost but not grant the source.

The problem was race conditions. React re-renders, auto-save callbacks, and the game tick were all competing to read and write the same mutable state object. In an idle game — a genre where the *entire point* is that numbers accumulate reliably — data corruption is existential. A player who suspects their numbers might be wrong will never trust the game again.

So I burned it down and rebuilt the engine around one principle: **pure functions and immutable state**. Every computation takes a state object in and returns a new state object out. Nothing is ever mutated. The tick function, the action handlers, the achievement audit — they all follow the same contract: `(state: GameState, ...params) => GameState`. The Zustand store is the single source of truth, and only the store's `setState` call can update it, using the new object returned by those pure functions.

It sounds strict. It is strict. It also means I haven't shipped a data corruption bug since.

## The Tick Loop

The heartbeat of every idle game is the tick loop — the function that runs continuously and makes the numbers go up. In Temple of Joy, this is `applyTick`, and it runs on `requestAnimationFrame`.

Here's the design I landed on: the tick function receives the current game state and a `deltaSeconds` value — the real wall-clock time elapsed since the last tick. This is critical for idle games. If the player switches to another tab, the browser throttles `requestAnimationFrame` to roughly once per second. Some browsers throttle even harder. Using real elapsed time instead of assuming a fixed 16ms frame means the player's income is always correct, regardless of what the browser does.

The tick function processes systems in a specific order, and the order matters:

1. **Delta calculation and clamping.** The raw delta is capped at 300 seconds to prevent absurd bursts from system clock changes or very long sleep states. Offline progress uses a separate code path with its own rules (diminishing returns, efficiency caps).

2. **Engine computation.** Before any state changes, we compute the current HPS, HPC, karma rate, and all multipliers by calling pure engine functions against the current state. This gives us a complete snapshot of the player's economy — base HPS, source contributions, upgrade multipliers, buff multipliers, synergy multipliers, the global multiplier.

3. **Income application.** New happiness = old happiness + (totalHPS × deltaSeconds). New karma = old karma + (karmaRate × deltaSeconds). Lifetime and run accumulators are updated. Peak values are checked and stored.

4. **Buff tick-down.** Every active buff has its `remainingSeconds` decremented by delta. Expired buffs are removed from the array. This includes event effects, vibe check bonuses, prestige momentum, offering effects, and pilgrimage bursts.

5. **Pilgrimage timer.** If a pilgrimage is active, its timer advances. If it completes, the burst income is calculated and applied.

6. **Milestone checks.** The player's `runHappiness` is compared against milestone thresholds. Newly crossed milestones are marked as earned. This is O(n) against the milestone list each tick, but with only 56 milestones it's trivially fast.

7. **Achievement grants.** Certain achievements are best checked in real-time rather than on audits — particularly ones gated by count thresholds that increment during play. The tick checks a curated subset; the full audit runs on load.

8. **Auto-buy.** If the player has unlocked any auto-buy tier from the Wheel of Samsara, the auto-buy timer advances. When it fires, it automatically purchases sources based on the player's highest affordable option.

9. **Hedonic treadmill.** The baseline happiness drifts toward current happiness at a rate of roughly one unit per 50,000 seconds of play (approximately 14 hours). This is a cosmetic/thematic system — it doesn't reduce income.

10. **State assembly.** All the computed values are assembled into a new state object and returned. The tick never mutates its input.

The entire tick runs in under a millisecond on modern hardware. It needs to — at 60fps, we have about 16ms of frame budget, and the tick shares that with React rendering.

## The Engine Layer

The engine module is where the math lives, and it's the most important file in the project. Every number the player sees — their HPS, their click damage, their transcendence threshold, their karma rate — is computed by deterministic pure functions in the engine.

### HPS Calculation

The full HPS calculation is a cascade of multipliers:

```
totalHPS = rawBaseHPS × globalMult
```

Where `rawBaseHPS` is the sum of all source contributions (each source's base HPS × owned count × source-specific multipliers from upgrades, offerings, and synergies), and `globalMult` is the product of all global multipliers (upgrade multipliers, wheel bonuses, milestone multipliers, buff multipliers, Samsara's Gift stacks, relic effects, and active event effects).

The tricky part is **source-specific multipliers**. Each source can be affected by:
- **Upgrade multipliers** from the eight upgrade paths (e.g., "Nap Mastery Vol. I" doubles Nap Pod output)
- **Offering tiers** (×1.5, ×2, ×2.5) purchased with karma for specific sources
- **Synergy multipliers** (e.g., "The Drowsy Economy" gives ×2 when you own 10+ Nap Pods AND 10+ Snack Bars)
- **Relic bonuses** (e.g., "The Stuffed Pillow" gives ×3 to Nap Pods specifically)

These all stack multiplicatively. A Nap Pod with all applicable bonuses can produce thousands of times more HPS than its base 0.75. The engine computes each source's effective HPS independently, sums them, and then applies global multipliers on top. This means the display can break down exactly where your income comes from — valuable for players optimizing their strategy.

### Cost Scaling

Source costs follow a classic exponential curve: `baseCost × 1.15^owned`. This is the standard idle game formula, and it's standard for a reason — it creates a logarithmic progression curve where each new source of a given type costs about 15% more than the last, making the decision between buying more of one type vs. trying a new type continuously interesting.

But 1.15 stops working at extreme scales. When a source costs 1e77 happiness and generates 1e77 HPS, a 15% cost increase per unit means you can buy dozens per second once you hit breakpoints. The endgame Patch 2 sources use lower multipliers — 1.07 to 1.14 — to create a tighter, more deliberate economy where every purchase feels meaningful.

I also have per-prestige cost reductions from the Wheel of Samsara. Early Warmth gives -5% costs, Heaven's Infrastructure gives -10%, Holy Infrastructure gives -25%, and Celestial Architect gives -50%. These stack as multipliers on the cost formula, not on the exponent — an important distinction. Reducing the exponent base would break the game's fundamental progression curve. Reducing the final price preserves the shape but shifts the whole curve downward.

### Karma Rate

Karma generation is intentionally slow compared to happiness. Base karma comes from per-source generation (each source type contributes a fraction of karma per second, scaled by quantity) plus click-based karma (The Living Temple: +1 karma per click). The slow rate creates real scarcity — karma buys relics, and relics are permanent power. Making karma easy to earn would trivialize relic acquisition and remove a major strategic axis.

## State Shape and Persistence

The GameState type is the spine of the entire game. It's a flat-ish object — no deeply nested structures — carrying everything: source counts, owned upgrades (as string arrays), earned achievements (string arrays), equipped relics (string arrays), active buffs (an array of buff objects), all currency values, prestige count, wheel purchases, event history, pilgrimage state, settings, and a growing collection of lifetime statistics.

I initially organized state into nested objects: `state.currencies.happiness`, `state.progress.achievements`, etc. It was clean. It was also a nightmare for the persistence layer. Saving and loading state requires flattening to JSON, and version migrations across updates (adding new fields, renaming old ones, changing types) are exponentially harder with nested structures. Flat state with explicit field names is uglier in the type definition but dramatically simpler to serialize, migrate, and debug.

Persistence itself runs on two tracks: localStorage for immediate writes (every 30 seconds) and a server-side save endpoint for cross-device sync. The save format is a SaveData type that mirrors GameState but strips computed values and function references. Loading reverses the process and runs a backwards-compatibility pass that handles field renames (`buildings` → `sources` from the pre-alpha), missing fields (default to 0 or empty array), and type mismatches.

The import/export system serializes SaveData to a Base64 string. Players can copy their save, paste it on another device, and resume instantly. It's dead simple and exactly as robust as it needs to be.

## Offline Progress

Offline progress is the idle game's promise: the game respects your time away. When a player returns after being gone for an hour, they should see meaningful progress.

Temple of Joy calculates offline income as `totalHPS × offlineSeconds × efficiency`. The base efficiency is 50% — you earn half your normal rate while away. This can be boosted: Saint's Patience doubles the offline cap, the Eternal Nap relic gives 100% efficiency, and The Long View wheel upgrade removes the time cap entirely (using a square-root diminishing curve for very long absences to prevent astronomical offline bursts).

The offline modal shows the player exactly what they earned and how long they were gone. Pilgrimages that would have completed during the offline period are processed and their burst income added. It's a small touch, but a player who comes back to see that their pilgrimage completed while they slept feels like the game was *doing something* in their absence. That's the fantasy of idle games — a world that continues without you.

## Lessons from the Architecture

Three things I'd tell any developer building an idle game:

**Pure functions saved the project.** The initial mutable approach was faster to write and impossible to debug. When every state transition is a pure function, you can log the input and output of every tick, replay bugs exactly, and unit test the engine without touching the UI. The upfront cost of immutability pays for itself within the first week of development.

**Flat state beats clean state.** Nested state objects look great in the type definition and create hell in the persistence layer. Every migration, every backwards-compatibility shim, every save/load round-trip is simpler with flat state. Resist the urge to organize.

**Wall-clock delta is non-negotiable.** If your idle game assumes fixed frame intervals, it's broken. Browsers throttle background tabs. Devices sleep. Players close laptops. The only reliable time source is `Date.now()`, and the only correct delta is the wall-clock difference between frames. Everything else is a lie that will cost you player trust.

Temple of Joy's engine processes about 40-60 ticks per second, evaluating hundreds of multipliers across fifty-six sources, checking dozens of milestones and achievements, managing arbitrary numbers of active buffs, and saving gracefully every half-minute. The player sees a number go up. Behind that number is a machine built on the principle that every calculation must be reproducible, every transition must be traceable, and every frame must account for real time.

The happiness engine doesn't feel emotions. But it counts them honestly.
