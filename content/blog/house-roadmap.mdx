## Building in Public

Game development is a long road, especially when you're walking it mostly alone. **House Always Wins** has been in active development at RMH Studios for most of this year, and I want to be transparent about where things stand. What's built, what works, what doesn't work yet, and where it's all heading.

I believe in building in public. Not just because it keeps me accountable — although it absolutely does — but because the journey of making a game is part of the game's story. The abandoned casino exists because I decided to build it, and every design decision, every late-night debugging session, every moment of doubt and breakthrough is woven into what the final experience will be.

So here's the roadmap. Honest, detailed, and subject to change — because that's how indie dev works.

## What's Been Built: Milestone 0

**Milestone 0** was about foundation. Before the casino could have atmosphere, NPCs, Debt dynamics, and Event Rooms, it needed to have *infrastructure*. The kind of boring, invisible work that makes everything else possible.

### Authentication System ✅

The auth system is complete and functional. **Better Auth** handles email/password and Discord OAuth login. Users can create accounts, log in, and their sessions persist. The login page is styled to fit the game's aesthetic — dark, minimal, with gold accents. The auth flow works on both desktop and mobile browsers, though the game itself is designed for PC/laptop first.

### Database and State Persistence ✅

**Prisma with Neon Postgres** is up and running. The database schema supports user accounts, game state records, NPC relationship data, gambling history, and exploration flags. State writes and reads are working. The foundation for narrative persistence — the game remembering choices across sessions and resets — is in place at the data layer.

### Game Shell ✅

The basic game shell exists: a Canvas-based rendering context, a game loop running at 60fps, input handling for keyboard, and the structural framework for room loading and transitions. You can "enter" the casino, move through a prototype environment, and interact with placeholder elements. It's not pretty yet, but it runs.

### Coin Flip Ritual ✅

The opening coin flip is implemented. Every session begins with the ritual: the coin spins in darkness, the result displays, and the game's opening line appears. The sound design for this moment is complete. It's the first thing that feels *real* — the first piece that feels like the final game rather than a prototype.

## In Progress: Milestone 1

**Milestone 1** is where the casino starts to come alive. This is the current phase, and it's the hardest one — transforming infrastructure into experience.

### Tile Map System 🔧

The tile map rendering system is partially built. Individual rooms render correctly with multi-layer support (background, tile, foreground). What's still in progress: room transitions, the camera system that follows the player through connected spaces, and the dynamic tile-swapping for Debt variants. The architecture is designed, but implementation is ongoing.

### Player Character 🔧

The player character has basic movement: walk, run, jump, and fall. Collision detection against tile geometry is working but needs refinement — edge cases with slopes, one-way platforms, and coyote time are still being tuned. Animation states exist but use placeholder sprites. Final pixel art for the character is in progress.

### Debt System Core 🔧

The Debt value increments and decrements correctly. It persists to the database. The basic subscription system that broadcasts Debt changes to other systems is in place. What's not yet connected: the actual responding systems. Lighting, music, NPC behavior — they all need to subscribe to Debt changes and react. The pipes are laid but the water isn't flowing yet.

### First NPC: Mara 🔧

Mara, the Bartender, is the first NPC being fully implemented. Her sprite sheet is in progress. Her base dialogue tree is written. The Debt-tiered behavior variations are designed and partially scripted. She'll be the test case for the entire NPC dialogue system — once Mara works, the pattern scales to the rest of the cast.

## Planned: Milestone 2

**Milestone 2** is about systems integration — making everything talk to everything else.

### Dynamic Lighting System 📋

The lighting overlay that modulates based on Debt. This system takes a functioning Debt core and a functioning tile renderer and merges them into something atmospheric. Technically, it's a separate canvas layer that composites over the game world with a blend mode that allows masking and color manipulation.

### Music and Audio Engine 📋

The Web Audio API-based system that handles ambient layers, dynamic music composition, and Debt-driven distortion. The audio design document is complete. Implementation awaits the Debt system being fully functional so there's something to modulate against.

### Gambling Minigames 📋

The three core gambling experiences: Coin Flip tables, Card Draw, and the Wheel. The designs are finalized, the math is verified, and the UI mockups exist. Implementation is planned as self-contained components that integrate with the Debt system and the broader game state.

### First Event Room 📋

Mara's Event Room will be the first complete challenge space. The design is on paper — a precision platforming sequence through a tilted bar environment. Building it requires a mature tile map system, polished player movement, and the ability to load isolated room contexts. It depends on everything in Milestone 1 being solid.

## Future: Milestone 3 and Beyond

### Full NPC Cast 📋

Harlan, Lux, and Rem — the remaining primary NPCs — get the same treatment Mara received: full sprite sheets, dialogue trees, Debt-tiered behavior, Event Rooms, and integration with the persistence system. Each NPC adds approximately two to three weeks of work.

### Casino Map Expansion 📋

The casino grows from a handful of prototype rooms to its full layout: the Lobby, the Floor, the Back Halls, the VIP area, the Maintenance corridors, and the deeper spaces that I'm not ready to talk about yet. The interconnected, Hollow Knight-inspired map with shortcuts, hidden paths, and progressive discovery.

### Persistence Features 📋

Reset-awareness gets implemented: the game reacting to new playthroughs with altered text, NPC bleed-through memories, and escalating environmental changes. This is the feature that makes House Always Wins unique, and it's the one I'm most protective of. It has to feel right or it shouldn't ship.

### Polish Pass 📋

Sound design integration, particle effects, environmental animations, screen transitions, UI refinement, performance optimization. The work that separates a prototype from a product. This phase will take as long as it takes.

## The Timeline Reality

I'm not going to give specific dates for milestones beyond the current one. Here's why: I've been making things long enough to know that estimates in creative work are fiction. I estimated Milestone 0 at three weeks. It took six. I'm currently estimating Milestone 1 at two months. It might take four.

What I *can* commit to is consistent progress and transparent communication. These devlogs are part of that commitment. Every few weeks, I'll share what's new, what's blocked, and what I've learned. The cadence is more important than the speed.

### The Solo Dev Reality

House Always Wins is primarily a solo project. I'm handling design, programming, pixel art, sound direction, and project management. That's a lot of hats, and some days I wear them better than others.

The upside of solo development: creative purity. Every decision is mine. The vision is coherent because it comes from one brain. There's no design-by-committee dilution, no stakeholder compromises, no feature requests from marketing.

The downside: everything takes longer. There's no one to hand tasks to. When I'm debugging a rendering issue, the pixel art isn't getting drawn. When I'm composing music, the code isn't being written. Progress is real but *linear*, not parallel.

I've accepted this trade-off. House Always Wins is a game about systems that are bigger than any one person — a casino that outlasts its visitors, a Debt that outlasts your intentions. Making it alone, slowly, with patience and stubbornness, feels appropriate. The casino has time. So do I.

## The Vision

When I close my eyes and imagine the finished game, I see this:

A player logs in. The screen darkens. A coin spins in the void. *"You've been here before."* The casino assembles around them — dark hallways, flickering lights, the distant sound of a slot machine paying out for no one. They explore. They find Mara, who smiles and offers a drink. They gamble at a Coin Flip table and lose. They don't notice the lights dim slightly. They find their first Event Room and die fourteen times before clearing it. They talk to Harlan, who is watching them with unblinking precision.

Hours later, their Debt is high. The music is wrong. Mara won't make eye contact. The hallways are darker and the sounds are louder. They feel uneasy — not because the game is hard, not because they're stuck, but because the world has *changed around them* and they can feel it in their chest.

They close the browser. Next day, they come back. Log in. The coin spins. *"You came back."*

That's the game. That's the roadmap's destination. Not a feature checklist — a *feeling*. And every milestone, every system, every line of code and pixel of art is in service of that feeling.

The casino is waiting. It has all the time in the world. And so does the development.

Stay tuned. The next devlog will arrive when there's something worth showing. In the meantime — the House is under construction, and it's going to be worth the wait.
