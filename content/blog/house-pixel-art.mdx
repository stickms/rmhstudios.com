## Pixels in a Browser

There's something beautifully absurd about building a pixel art game in a web browser. The entire web platform was designed for documents — text, images, links. And here I am, trying to render a crumbling casino at 16x16 tile resolution inside a `<canvas>` element, fighting against everything the browser was built to do.

But that's the joy of web game development. Every limitation is a puzzle. Every browser quirk is a challenge to solve elegantly. And when it works — when the pixels line up, the animations flow, and the abandoned casino feels *real* despite being rendered in a technology designed for tax forms — there's nothing quite like it.

## The Rendering Pipeline

House Always Wins uses an HTML5 Canvas for rendering, managed through a custom game loop running in a `requestAnimationFrame` cycle. The choice of Canvas over WebGL was deliberate — for a 2D pixel art game, Canvas provides everything I need without the complexity overhead of a full 3D rendering pipeline. Performance is more than adequate for the game's scope.

The rendering pipeline processes layers in order:

1. **Background layer** — The deepest environmental details: distant walls, ceiling elements, far decorations
2. **Tile layer** — The primary world geometry: floors, walls, platforms, interactive surfaces
3. **Entity layer** — NPCs, interactive objects, particle effects
4. **Player layer** — The player character and directly attached effects
5. **Foreground layer** — Elements that render over the player: hanging signs, curtain edges, atmospheric fog
6. **Lighting layer** — A composited overlay that handles the Debt-driven dynamic lighting
7. **UI layer** — HUD elements, dialogue boxes, interaction prompts

```typescript
// Core render loop (simplified)
function render(ctx: CanvasRenderingContext2D, state: GameState) {
  ctx.clearRect(0, 0, canvas.width, canvas.height);
  
  // Apply camera transform
  ctx.save();
  ctx.translate(-camera.x, -camera.y);
  
  renderBackground(ctx, state);
  renderTiles(ctx, state);
  renderEntities(ctx, state);
  renderPlayer(ctx, state);
  renderForeground(ctx, state);
  renderLighting(ctx, state.debt);
  
  ctx.restore();
  
  renderUI(ctx, state);
}
```

## Tile Map Architecture

The casino is built from tile maps — grids of 16x16 pixel tiles that form the environment. Each room in the casino is a separate tile map, loaded dynamically as the player moves through the world.

### Tile Set Design

The casino's tile set is organized into functional categories:

- **Structural** — Floor tiles, wall tiles, ceiling tiles. These come in multiple variants that are randomly selected during room generation to prevent visual repetition. A hallway floor might use five different tile variants, subtly varying in crack patterns and stain placement.
- **Decorative** — Carpet patterns, wallpaper, baseboards, trim. These tiles carry the casino's aesthetic identity: deep reds faded to dusty rose, gold leaf flaking to reveal gray underneath, ornate patterns disrupted by water damage.
- **Interactive** — Tiles that respond to the player: pressure plates, crumbling floors, one-way platforms. These are visually distinct enough to be identifiable but subtle enough not to break the aesthetic.
- **Atmospheric** — Tiles specifically designed for mood: flickering light sources, dripping water, drifting dust particles. These are animated tiles with frames that cycle based on timers.

```typescript
interface TileDefinition {
  id: number;
  spriteSheetPosition: { x: number; y: number };
  collision: 'solid' | 'platform' | 'none';
  animated: boolean;
  frames?: number;
  frameRate?: number;
  debtVariant?: { threshold: number; alternateId: number };
}
```

### Debt-Responsive Tiles

One of the more technically interesting features: tiles can have **Debt variants**. When the player's Debt crosses certain thresholds, specific tiles in the environment swap to alternate versions. A golden chandelier becomes a broken fixture. Intact carpet becomes torn. Clean walls develop cracks.

These swaps aren't instant — they happen when rooms reload, so the player never sees a tile pop into a different state. They leave a room, explore elsewhere, come back, and things are... different. Slightly worse. The cumulative effect across dozens of tiles is an environment that *degrades* as Debt accumulates, without any single change being dramatic enough to break immersion.

## Animation Systems

Pixel art animation in House Always Wins falls into three categories, each with its own technical approach.

### Character Animation

The player character and NPCs use **sprite sheet animation** — pre-drawn frames arranged in strips, indexed by animation state. The player has animation states for idle, walking, running, jumping, falling, landing, and several narrative states (gambling, examining, reacting).

Each animation state has a defined frame count and frame rate:

```typescript
const playerAnimations: Record<string, AnimationConfig> = {
  idle: { frames: 4, frameRate: 6, loop: true },
  walk: { frames: 6, frameRate: 10, loop: true },
  run: { frames: 8, frameRate: 14, loop: true },
  jump: { frames: 3, frameRate: 12, loop: false },
  fall: { frames: 2, frameRate: 8, loop: true },
  land: { frames: 3, frameRate: 10, loop: false },
  coinFlip: { frames: 12, frameRate: 8, loop: false },
};
```

The animation system handles transitions between states with configurable blending — some transitions are instant (idle to walk), while others have brief transition frames (landing to idle includes a recovery frame). This small detail adds significant polish to character movement.

### Environmental Animation

Slot machines, flickering lights, dripping water, swaying signs — the casino is full of environmental animations that bring the abandoned space to life. These run on independent timers, staggered so not everything animates in sync. The desynchronization is crucial: perfectly synced environmental animations look artificial. Staggered timers create visual rhythm that feels organic.

I use a simple but effective technique: each environmental animation is assigned a random time offset at room load. A flickering light might update every 200ms, but one light starts at offset 0, another at offset 73ms, another at offset 141ms. Same animation, same frame rate, but they phase in and out of alignment naturally.

### Particle Effects

Dust motes, sparks, dripping liquid, drifting smoke — these are handled by a lightweight **particle system** that manages pools of particles with basic physics (gravity, drag, lifetime). Particles are rendered as small clusters of pixels, often just 1x1 or 2x2, to maintain the pixel art aesthetic without breaking the resolution.

The particle system is one of the areas most affected by Debt. At low Debt, particles are sparse and gentle — occasional dust, a rare spark. At high Debt, the air is thick with particles: ash drifts, smoke curls, sparks shower from damaged wiring. The casino is literally falling apart around you.

## Creating the Abandoned Casino Feel

The biggest challenge isn't technical — it's artistic. How do you make 16x16 tile grids feel like a *place*? How do you create atmosphere with squares?

### The Three Principles

Through months of iteration, I've arrived at three principles for making the casino feel real:

**1. Imperfection is everything.** No straight lines. No clean surfaces. Every wall has a crack, every floor has a stain, every light flickers at a slightly different rate. The grid structure of tile maps naturally creates regularity, so every artistic choice needs to fight against that regularity.

**2. Light tells the story.** In a dark environment, light is the most powerful narrative tool. Where light falls, the player looks. Where shadows pool, mystery lives. The dynamic lighting system — modulated by Debt — is the single most impactful visual system in the game. Without it, the casino is a grid of tiles. With it, it's a haunted place.

**3. Sound completes the picture.** This deserves its own devlog (and it'll get one), but pixel art environments *need* sound to feel real. The visual abstraction of pixel art means the player's brain is already working to fill in details. Audio provides the details that pixels can't: the hum of a dying ventilation system, the distant crash of something falling, the creak of a floor that remembers your weight.

## Performance in the Browser

Running a multi-layer rendering pipeline with dynamic lighting, particles, and dozens of animated tiles in a browser requires careful optimization. A few techniques that have made a significant difference:

- **Off-screen canvas pre-rendering** — Static tile layers are rendered once to off-screen canvases and composited as images, avoiding per-frame tile-by-tile rendering.
- **Dirty rectangle tracking** — Only regions where something changed are re-rendered. If the player is standing still and no animations are in view, the renderer does almost nothing.
- **Object pooling** — Particles, projectiles, and temporary effects are pooled and recycled rather than created and garbage-collected. GC pauses are the enemy of smooth animation.
- **Resolution independence** — The game renders at a low internal resolution (e.g., 320x180) and scales up with CSS `image-rendering: pixelated`. The Canvas does less work, and the scaling is free.

The result is a game that runs at a locked 60fps on modern hardware and a stable 30fps on lower-end laptops. For a pixel art game, that's more than sufficient.

## The Joy of Limitations

Building pixel art in a browser is an exercise in constraint. Every tool, every technique, every optimization is a response to a limitation. And I love it. Constraints force creativity. They force you to find elegant solutions instead of brute-forcing your way to visual fidelity.

The abandoned casino is taking shape, sixteen pixels at a time. It's slow, it's meticulous, and it's deeply satisfying. Every tile placed is a story told. Every animation frame is a moment captured. And every time I load the game in a browser and see the casino flicker to life, I'm reminded why I chose this medium.

The browser wasn't built for this. But the casino doesn't care what you were built for. It just needs you to play.
