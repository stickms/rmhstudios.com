## The Performance Ceiling We Keep Pushing

Browser games have always lived under a performance ceiling. JavaScript is remarkably fast for a dynamic, interpreted language — V8 and SpiderMonkey have spent over a decade optimizing JIT compilation — but there are classes of computation where it simply cannot compete with compiled languages. Physics simulations, pathfinding over large grids, procedural generation, real-time audio synthesis: these are the workloads that make JS sweat.

**WebAssembly (WASM)** is the most credible answer the web platform has ever produced for this problem. And we think it's going to reshape browser gaming over the next five years.

## A Brief History of WASM

WebAssembly's origins trace back to Mozilla Research in 2015. Engineers including Luke Wagner, working on the asm.js project (a strict subset of JavaScript designed for ahead-of-time optimization), recognized that the web needed a proper binary instruction format — something that could serve as a compilation target for C, C++, Rust, and other systems languages.

The WebAssembly specification was developed as a joint effort between Mozilla, Google, Microsoft, and Apple — a rare moment of alignment among browser vendors. The MVP shipped in all four major browsers in 2017. Since then, the spec has steadily expanded: threads (SharedArrayBuffer), SIMD instructions, reference types, and the ongoing work on the garbage collection proposal (WasmGC) and the component model.

The key properties of WASM that matter for game development:

- **Near-native speed.** WASM modules execute at roughly 80-95% of native performance for compute-bound workloads, depending on the benchmark and the browser's implementation.
- **Deterministic compilation.** Unlike JavaScript's JIT, which can produce different optimization tiers depending on runtime heuristics, WASM compiles ahead of time to a predictable baseline.
- **Language agnosticism.** You can write WASM modules in C, C++, Rust, Go, AssemblyScript, Zig, and increasingly other languages. This opens the browser to codebases and expertise that were previously locked out.
- **Sandboxed execution.** WASM runs in the same security sandbox as JavaScript. No native code escapes, no filesystem access, no system calls outside what the host provides.

## Figma Proved the Model

Before anyone took WASM seriously for complex applications, **Figma** proved the model. Figma's rendering engine is written in C++ and compiled to WebAssembly, enabling a professional-grade design tool to run entirely in the browser at performance levels competitive with native applications. When Figma filed its S-1 in 2022, it reported over 4 million weekly active users — all running a WASM-powered application in a browser tab.

Figma's success was a watershed moment. It demonstrated that WASM wasn't a toy, a demo, or a proof of concept. It was production infrastructure for performance-critical applications used by millions.

## Game Engines and WASM

The game engine ecosystem has been moving toward WASM with varying degrees of commitment.

**Unity** has supported WebGL builds for years, and its more recent WASM export pipeline has improved significantly. Unity 6 (released 2024) brought better memory management and faster startup times for web builds, though the output size remains substantial — a minimal Unity WASM build can easily exceed 20MB compressed.

**Unreal Engine 5** has experimental web export capabilities, though Epic has historically treated the browser as a second-class target. The sheer size of UE5's runtime makes lean web builds challenging, but the technical foundation exists.

**Godot 4** has emerged as a particularly web-friendly engine, with relatively compact WASM exports and a community that actively targets the browser.

Then there's the **Rust + WASM** ecosystem, which is where things get especially interesting for studios like ours. Crates like `wasm-bindgen`, `web-sys`, and `wgpu` provide direct, low-overhead access to browser APIs from Rust. The Bevy game engine, written in Rust, can compile to WASM, and the Rapier physics engine (also Rust) offers a WASM build that dramatically outperforms JavaScript physics libraries.

## How RMH Evaluates WASM

We don't use WebAssembly in all of our games. For many of our titles — party games in RMHbox, the typing challenges in RMH Type, the idle mechanics in Temple of Joy — JavaScript is more than sufficient. The overhead of maintaining a WASM compilation pipeline isn't justified when your bottleneck is network latency, not CPU throughput.

But we actively evaluate WASM for specific use cases:

### Physics Simulation

Our game **Laundry Sort** requires real-time rigid body simulation with dozens of objects interacting simultaneously. We explored Matter.js (pure JavaScript) and Rapier (Rust compiled to WASM). The performance difference was stark — Rapier handled complex scenes at 60fps where Matter.js dropped to 30-40fps on mid-range devices. For a game where the core mechanic *is* physics, that difference matters.

### Procedural Generation

**Signal Forge**, our roguelike deckbuilder, generates dungeon layouts, enemy configurations, and card synergies procedurally. The generation algorithms involve significant computation — constraint satisfaction, graph traversal, weighted randomization over large possibility spaces. Moving these computations to a WASM module could reduce generation times, which directly impacts perceived load times between runs.

### Audio Processing

The **Web Audio API** is powerful but has limitations for real-time audio synthesis and effects processing. WASM AudioWorklets allow us to run custom DSP code at audio-thread priority, which opens up possibilities for dynamic music systems and procedural sound effects that would be impractical in pure JavaScript.

## The Tradeoffs We're Honest About

WASM is not a silver bullet, and we think the discourse around it sometimes glosses over real costs:

- **Bundle size.** A WASM module adds to your initial download. For a studio that values instant page loads (we use Next.js SSR specifically for this), adding a 2-5MB WASM binary is a meaningful tradeoff.
- **Debugging.** WASM debugging tools have improved — Chrome DevTools can now set breakpoints in WASM source maps — but the experience is still significantly worse than debugging JavaScript. When something goes wrong in a WASM module, the investigation is harder and slower.
- **Interop overhead.** Calling between JavaScript and WASM has a cost. If you're crossing the boundary thousands of times per frame (say, querying individual physics bodies from JS), the overhead can eat into your performance gains. Good WASM architecture minimizes boundary crossings by batching operations.
- **Memory management.** WASM uses a linear memory model. If you're coming from JavaScript's garbage-collected world, manual memory management (or Rust's ownership model) is a significant shift in how you think about your code.

## Where This Is Heading

We're watching several developments closely:

**WasmGC** will allow garbage-collected languages (Kotlin, Dart, potentially future versions of JavaScript toolchains) to compile to WASM without shipping their own GC. This could dramatically expand the ecosystem of languages targeting the browser.

**WASI (WebAssembly System Interface)** is extending WASM beyond the browser into server-side and edge computing. For a studio that runs game servers on cloud infrastructure, the possibility of sharing compiled game logic between client and server is tantalizing.

**WebGPU + WASM** is the combination we're most excited about. WebGPU provides modern GPU access in the browser; WASM provides the CPU-side performance to feed it. Together, they could enable browser games that are visually and computationally competitive with native titles.

## Our Position

We believe WASM is the most important evolution in browser capabilities since the introduction of the `<canvas>` element. It doesn't replace JavaScript — it complements it, handling the workloads that JavaScript was never designed for.

At RMH Studios, we're committed to building browser-native games. That means we go where the platform goes. As WASM matures, as tooling improves, as bundle sizes shrink and debugging gets better, we'll integrate it more deeply into our stack. Not because it's trendy, but because it lets us build better games for our players — games that run in a browser tab, require no installation, and perform like they were built for your machine.

Because, increasingly, they are.
