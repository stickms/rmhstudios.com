## The Core Interaction

Drag and drop is the *entire* game in Laundry Sort. Every action a player takes — every decision, every moment of skill expression — flows through the drag-and-drop system. If it doesn't feel perfect, nothing else matters. The physics can be beautiful, the art can be stunning, but if grabbing and releasing a shirt feels even slightly wrong, the game is dead.

This is the story of how we built a drag-and-drop system that feels good with a mouse, feels good with one finger, and feels good with *two fingers at once* — all while interacting seamlessly with our custom physics engine.

## Mouse Events: The Starting Point

We started where every web developer starts: mouse events. The basic structure is familiar:

```typescript
canvas.addEventListener("mousedown", onGrab);
canvas.addEventListener("mousemove", onDrag);
canvas.addEventListener("mouseup", onRelease);
```

On mousedown, we check if the cursor position overlaps any clothing item. If it does, we "grab" that item — removing it from the physics simulation and attaching it to the cursor. On mousemove, we update the item's position to follow the cursor. On mouseup, we release it back into the physics world with velocity based on the cursor's recent movement.

Simple enough. But the devil, as always, is in the details.

### Hit Testing with Rotation

Our clothing items rotate. That means you can't just check if the cursor is within a simple bounding box — you need to account for the item's current rotation angle. We do this by applying the *inverse rotation* to the cursor position relative to the item's center, then checking against the un-rotated bounding box:

```typescript
function hitTest(item: ClothingItem, px: number, py: number): boolean {
  const cx = item.x + item.width / 2;
  const cy = item.y + item.height / 2;

  // Translate point relative to item center
  const dx = px - cx;
  const dy = py - cy;

  // Apply inverse rotation
  const cos = Math.cos(-item.rotation);
  const sin = Math.sin(-item.rotation);
  const localX = dx * cos - dy * sin;
  const localY = dx * sin + dy * cos;

  // Check against un-rotated bounds
  return (
    Math.abs(localX) <= item.width / 2 &&
    Math.abs(localY) <= item.height / 2
  );
}
```

This was one of those moments where the custom physics engine really paid off. Because we own the rotation math, we can reuse the exact same transforms for hit testing. No coordinate system mismatches, no conversion layers, no surprises.

## Touch Events: A Different Beast

Adding touch support sounds simple — just swap `mousedown` for `touchstart`, right? Wrong. Touch events are fundamentally different from mouse events in ways that will bite you if you're not careful.

### Multiple Simultaneous Touches

A mouse has one cursor. A touchscreen can have ten. When a player touches the screen with two fingers, you get a `touchstart` event with *two* `Touch` objects in the `changedTouches` list. Each touch has a unique `identifier` that persists across `touchmove` and `touchend` events.

This means our grab system needs to track multiple grabbed items simultaneously:

```typescript
const activeGrabs = new Map<number, GrabState>();

interface GrabState {
  item: ClothingItem;
  offsetX: number;
  offsetY: number;
  velocityHistory: Array<{ x: number; y: number; time: number }>;
}

function onTouchStart(e: TouchEvent) {
  e.preventDefault(); // Prevent scrolling!

  for (const touch of Array.from(e.changedTouches)) {
    const pos = getTouchPosition(touch);
    const item = findItemAtPosition(pos.x, pos.y);

    if (item) {
      activeGrabs.set(touch.identifier, {
        item,
        offsetX: pos.x - item.x,
        offsetY: pos.y - item.y,
        velocityHistory: [],
      });
      item.isGrabbed = true;
    }
  }
}
```

The `Map<number, GrabState>` keyed by touch identifier is the critical data structure. It lets us track each finger independently, so a player can grab a red shirt with their left thumb and a blue shirt with their right index finger simultaneously.

### Touch Position Coordinates

Touch events give you coordinates in `clientX/clientY`, which are relative to the viewport. To get canvas-relative coordinates, you need to subtract the canvas's position and account for any CSS scaling:

```typescript
function getTouchPosition(touch: Touch): { x: number; y: number } {
  const rect = canvas.getBoundingClientRect();
  return {
    x: (touch.clientX - rect.left) * (canvas.width / rect.width),
    y: (touch.clientY - rect.top) * (canvas.height / rect.height),
  };
}
```

That scaling factor (`canvas.width / rect.width`) is essential when the canvas's internal resolution differs from its CSS display size — which it always does on high-DPI screens.

### Preventing Default Behavior

The single most important line in our touch handler is `e.preventDefault()`. Without it, the browser will interpret touch movements as scroll gestures, pan gestures, or zoom gestures. Your player will try to drag a shirt and instead scroll the entire page. It's a disaster.

We call `preventDefault()` on `touchstart`, `touchmove`, and `touchend`. We also set `touch-action: none` in CSS on the canvas element as a belt-and-suspenders approach.

## The Velocity Flick

When a player releases a clothing item, it shouldn't just drop straight down. It should continue moving in the direction the player was dragging, as if thrown. This flick mechanic is essential for making the game feel responsive and skillful.

We track the cursor or touch position over the last few frames in a velocity history buffer. On release, we calculate the average velocity and apply it as the item's physics velocity:

```typescript
function onRelease(grabState: GrabState) {
  const { item, velocityHistory } = grabState;
  item.isGrabbed = false;

  if (velocityHistory.length >= 2) {
    const recent = velocityHistory.slice(-3);
    const oldest = recent[0];
    const newest = recent[recent.length - 1];
    const dt = newest.time - oldest.time;

    if (dt > 0) {
      item.vx = ((newest.x - oldest.x) / dt) * 16; // Scale to frame time
      item.vy = ((newest.y - oldest.y) / dt) * 16;
      item.angularVelocity = item.vx * 0.02; // Impart spin
    }
  }
}
```

That `* 0.02` angular velocity factor is a tiny detail that makes the flick feel *so much better*. When you throw a shirt to the right, it spins clockwise slightly — just like a real object would. Players don't consciously notice it, but they'd notice if it were missing.

## Conflict Resolution

What happens when two items are overlapping and the player taps on the overlap? We need to decide which item gets grabbed. Our rule is simple: **the topmost item wins**. Items are sorted by their vertical position (lower items are drawn on top), and we iterate from top to bottom when hit testing.

What if a player tries to grab an item that another finger is already holding? We skip it. Each item can only be grabbed by one touch at a time. This prevents weird tearing behavior where two fingers fight over the same shirt.

## Lessons from the Touch Trenches

Building multi-touch drag and drop for a physics game taught me several painful lessons:

1. **Test on real devices early and often.** The Chrome DevTools touch simulator lies to you. It doesn't capture the timing oddities, the coordinate imprecision, or the gesture conflicts you'll encounter on a real phone.

2. **Touch event handling is not mouse event handling with different names.** The multi-touch model is fundamentally different, and trying to abstract them behind a single interface leads to bugs.

3. **The flick velocity needs smoothing.** Raw instantaneous velocity is too noisy — a player's finger jitters during release, producing erratic velocities. Averaging over the last 3-5 samples produces much better results.

Building this system was challenging, but the payoff is immense. When a player grabs two items and sorts them simultaneously into different bins, there's a moment of *flow* that makes the whole effort worthwhile.
