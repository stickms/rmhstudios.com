Learning Objectives

  • Define server-side rendering, hydration, and file-based routing, and explain how the three combine in a TanStack Start application.
  • Trace a single page request through the rmhstudios.com web tier — from the browser, through the Nitro server runtime, to a rendered HTML document and its subsequent hydration.
  • Distinguish a page route (renders UI) from a server route (returns data), and read both kinds out of real route files under app/routes/.
  • Explain the role of Nitro as the production server runtime and how the Vite build emits the .output/server/index.mjs entry that Docker actually runs.
  • Justify why a content-and-games site chooses SSR with selective client-only escape hatches, rather than a pure single-page app or a fully static build.

A web framework's job is to answer one question well: when a browser asks for a URL, what HTML comes back, and how does that static HTML become a living React application? rmhstudios.com answers it with three cooperating layers — TanStack Start for routing and server functions, Nitro for the server runtime, and Vite for the build. This chapter reads those layers as they actually appear in the repository: the root route in app/routes/__root.tsx, the build configuration in vite.config.ts, and the production entry the container launches, node .output/server/index.mjs. By the end you should be able to follow a request end to end and explain every hop it takes.

What server-side rendering means here

There are three broad ways to put a React page in front of a user. A single-page application (SPA) ships an empty HTML shell plus a JavaScript bundle; the browser downloads the bundle, runs it, and only then paints content. Static site generation renders every page to HTML once at build time and serves the files. Server-side rendering sits between them: for each request, a server runs the React component tree to a finished HTML string, sends that to the browser so the user sees content immediately, and then ships the same component code to the browser so React can take over the already-painted DOM. rmhstudios.com is an SSR app.

Server-side rendering (SSR) — rendering a UI component tree to an HTML string on the server, per request, so the browser receives a complete, content-filled document instead of an empty shell. The same components later run in the browser to make the page interactive.
Hydration — the browser-side phase in which React attaches event listeners and internal state to server-rendered DOM that already exists, rather than recreating it. After hydration the page is a normal interactive React app; before it, the page is visible but not yet clickable.

The root route makes the tradeoff concrete. app/routes/__root.tsx defines a server function getInitialUser that reads the session cookie on the server and threads the signed-in user into the first render. The comment in the file states the motivation precisely: it runs "so the shell can render signed-in on the first paint instead of flashing 'signed out' while better-auth's client session loads after hydration." That single sentence is the whole argument for SSR on a social site — the server already holds the cookie, so it can render the correct, personalized shell before a byte of client JavaScript executes.

// app/routes/__root.tsx (excerpt)
const getInitialUser = createServerFn({ method: "GET" }).handler(async () => {
  const request = getRequest();
  const session = await auth.api.getSession({ headers: request.headers });
  const u = session?.user;
  if (!u) return null;
  return { id: u.id, name: u.name ?? null, /* ... */ isAdmin: ... };
});

export const Route = createRootRoute({
  loader: () => getInitialUser(),
  component: RootComponent,
  shellComponent: RootDocument,
});
Server function — in TanStack Start, a function created with createServerFn that only ever executes on the server, even when called from component code. It compiles to an RPC: the client calls it like a function, but the body runs on Nitro, giving safe access to the database, cookies, and secrets.

File-based routing

rmhstudios.com has no central route table that a developer edits by hand. Instead, the file system is the route table. Every file under app/routes/ becomes a route whose URL is derived from its path. The README states this directly: app/routes/ holds "All page and API routes," organized into folders such as _site/ (public pages), api/ (server endpoints), and one folder per game.

File-based routing — a routing scheme in which the location and name of a file on disk determine the URL it serves. Adding a page means adding a file; no separate registry is edited. A build-time generator turns the file tree into a typed route tree.

The naming conventions are worth learning because they encode real behavior:

File on diskURL / meaning
app/routes/__root.tsxThe root layout wrapping every route — the HTML document shell.
app/routes/_site.tsxA pathless layout (leading _): adds the sidebar but contributes no URL segment.
app/routes/_site/index.tsxThe / home feed, nested inside the _site layout.
app/routes/blog.$slug.tsx/blog/<slug> — the $slug segment is a dynamic parameter.
app/routes/api/search.ts/api/search — a server route (note: .ts, not .tsx).
app/routes/sitemap[.]xml.ts/sitemap.xml — brackets escape a literal dot in the URL.

The pathless layout is the most pedagogically useful of these. app/routes/_site.tsx declares createFileRoute('/_site') and renders a LeftSidebar, mobile navigation, and an <Outlet /> where its children mount. Its own file comment explains the design: "Game routes, /secret, and /login are NOT nested under this layout, so they render full-screen without any sidebar." Because the home feed lives at app/routes/_site/index.tsx, it inherits the sidebar; because app/routes/altair.tsx lives at the top level, it does not. Layout is expressed purely by where a file sits in the tree.

That tree is not interpreted at runtime. The TanStack Start plugin generates a single typed file, app/routeTree.gen.ts (over 9,000 lines in this repository), whose banner warns: "This file was automatically generated by TanStack Router. You should NOT make any changes in this file." The router entry, app/router.tsx, simply imports that generated tree and wires it into a router instance:

// app/router.tsx
import { createRouter as createTanStackRouter } from "@tanstack/react-router";
import { routeTree } from "./routeTree.gen";

export function getRouter() {
  return createTanStackRouter({
    routeTree,
    scrollRestoration: true,
    defaultPreload: "intent",
  });
}
What defaultPreload: "intent" buys you. "Intent" means the router begins loading a route's code and data as soon as the user signals they might go there — typically on hover or focus of a link — so the navigation feels instant when the click actually lands. It is a small line with a large UX effect on a link-dense site.

Page routes versus server routes

Two route files in the same directory tree can do completely different jobs. A page route exports a component and renders UI; a server route exports HTTP method handlers and returns data. Compare the home page with the search endpoint.

The home page, app/routes/_site/index.tsx, is a page route. It declares a loader that calls a server function to fetch sidebar data, a head for metadata, and a component that renders the feed:

// app/routes/_site/index.tsx (excerpt)
const fetchSidebarData = createServerFn({ method: 'GET' }).handler(async () => {
  return getSidebarData();
});

export const Route = createFileRoute('/_site/')({
  validateSearch: (search) => { /* parse ?q= */ },
  loader: () => fetchSidebarData(),
  head: () => ({ meta: [{ title: 'RMH Studios' }, /* ... */] }),
  component: Home,
});

The search endpoint, app/routes/api/search.ts, is a server route. It exports no component at all — instead it provides a server.handlers.GET function that rate-limits the caller, reads the query string, runs four Prisma queries in parallel, and returns JSON:

// app/routes/api/search.ts (excerpt)
export const Route = createFileRoute('/api/search')({
  server: {
    handlers: {
      GET: async ({ request }) => {
        const ip = getClientIp(request);
        const { allowed } = rateLimit(ip, { limit: 60, windowMs: 60_000, prefix: 'search' });
        if (!allowed) return Response.json({ error: 'Too many requests' }, { status: 429 });
        // ... Prisma queries across users, posts, builds, blog ...
        return Response.json({ people, posts, builds, blog });
      },
    },
  },
});

The same pattern appears across app/routes/api/ — explore.ts assembles trending tags and suggested users; sitemap[.]xml.ts returns an XML document for crawlers. These handlers receive a standard Web Request and return a standard Web Response, the same primitives the page-render path uses. There is no second framework for the API; it is the same file-based router, just exporting handlers instead of a component.

Backend-for-frontend (BFF) — a server layer that exists specifically to serve one frontend, shaping data and aggregating sources exactly as that UI needs. The app/routes/api/ handlers and the createServerFn loaders together form rmhstudios.com's BFF: they sit between the browser and Postgres, applying auth, rate limits, moderation filters, and per-page shaping before any data crosses the network.
Scope: not all server work lives in the web tier. The BFF described here handles request-shaped reads and writes. Long-lived realtime traffic does not go through Nitro at all — it is served by standalone Socket.io processes (server/socket-server, server/rmhbox, server/rmhtube) started separately by the start script and covered in a later chapter. The web tier is one process in a fleet, not the whole backend.

Nitro: the server runtime

TanStack Start defines what to render; it does not, by itself, listen on a TCP port. That job belongs to Nitro, the server runtime configured in vite.config.ts. The README's tech-stack section names it explicitly under "SSR / Server: Nitro (server runtime)."

Nitro — a server runtime and build target that takes a framework's render logic and packages it into a deployable server. It produces a portable .output/ directory containing a Node entry point, the SSR bundle, and static assets, and it abstracts over deployment targets (Node, edge, serverless) behind one build.

In the Vite config, Nitro is added as a plugin alongside the TanStack Start plugin. The interesting detail is dependency handling: heavy, browser- or native-only packages must be kept out of the server bundle, and Nitro's bundler ignores Vite's normal ssr.external list. The config's own comment makes the rule explicit — traceDeps is "the only way to externalize from the production server bundle":

// vite.config.ts (excerpt)
plugins: [
  stubServerFiles(),
  tailwindcss(),
  tanstackStart({ srcDirectory: "app" }),
  react(),
  nitro({
    // externalize from Nitro's Rolldown server bundle, then trace into
    // .output/node_modules for runtime resolution
    traceDeps: ["@prisma/client", ".prisma", "@resvg/resvg-js", "satori", "esbuild"],
    rollupConfig: { external: heavyExternals.map(/* three, monaco, pixi, ... */) },
  }),
],

The build:frontend script — vite build with an 8 GB Node heap — drives this build, and the start script runs its output. The Dockerfile confirms exactly what the container executes: after copying .output into the image it declares CMD ["node", ".output/server/index.mjs"] and EXPOSE 7005. A build-time check in the Dockerfile even asserts the entry exists (test -f /app/build-output/server/index.mjs) before the image is allowed to proceed. The web tier's whole runtime, in one line, is that file.

Ports, from the deploy script. The web process listens on 7005 in production (with a blue/green sibling on 7015 for zero-downtime swaps, per deploy.sh). The socket fleet uses 7001, RMHBox 7676, and RMHTube 7003 — the same ports the Dockerfile exposes.

The request lifecycle, end to end

We can now follow a real request for the home feed, GET /, from browser to interactive page. Each numbered hop corresponds to code we have already read.

Step by step: (1) The browser sends GET / with its session cookie. (2) Nitro matches the URL against the generated route tree, finding __root → _site → _site/index. (3) The loaders run on the server: the root's getInitialUser reads the cookie via getRequest() and queries Better Auth; the index's fetchSidebarData pulls official builds, user builds, recommended users, and blog posts. (4) React renders the matched component tree — sidebar, feed, the correct signed-in shell — to an HTML string. (5) Nitro returns that HTML plus the serialized loader data embedded in the document. (6) The browser paints content immediately, then downloads the client bundle. (7) Hydration: React re-runs the components in the browser, finds the matching DOM already present, and attaches event handlers and state without rebuilding it. The page becomes interactive.

A subtlety visible in __root.tsx is how SSR avoids a flash of unstyled content. The root injects a small inline themeScript that applies the persisted theme class to <html> before React hydrates, and the document shell is marked suppressHydrationWarning because the server cannot know the user's chosen theme. This is the recurring SSR engineering problem in miniature: the server renders a generic shell, and a tiny pre-hydration script reconciles it with browser-only state (theme, in this case) before React takes over.

Why SSR for a content-and-games site

The choice of SSR is not aesthetic; it follows from what rmhstudios.com is. It is two sites in one body: a content platform (feed, blog, news, research, user profiles, builds) and a games library (3D, audio, multiplayer). Those halves want opposite things, and SSR with escape hatches serves both.

The content half benefits from SSR for three concrete reasons. Crawlability: blog posts, news, and public builds must be indexable, which is exactly why sitemap[.]xml.ts exists and emits real URLs. A pure SPA serves crawlers an empty shell. Personalized first paint: the signed-in shell is rendered on the server from the cookie, the precise problem getInitialUser solves. Time-to-content: a content document should show text on the first response, not after a megabyte of JavaScript executes.

The games half does not want to be rendered on the server at all — three, pixi.js, monaco-editor, and tone are browser-only and enormous. The web tier accommodates them with two mechanisms already in vite.config.ts: these packages are listed in heavyExternals so they never enter the server bundle, and they are split into named manualChunks (vendor-three, vendor-pixi, vendor-monaco-*) so a content visitor never downloads game code. The stubServerFiles plugin even replaces *.server.ts imports with empty stubs on the client so Node-only dependencies like pg never leak into the browser bundle.

The design in one sentence. SSR by default gives the content site fast, crawlable, personalized pages; per-package externalization and chunking carve out a client-only lane for the games, so one framework serves a portfolio blog and a 3D physics game without compromising either.

Summary

  • SSR renders the React tree to HTML on the server per request, then hydrates it in the browser; rmhstudios.com uses it so the personalized shell renders correctly on the first paint (getInitialUser in app/routes/__root.tsx).
  • File-based routing turns the app/routes/ tree into URLs. A build step generates app/routeTree.gen.ts, which app/router.tsx imports; pathless layouts like _site.tsx add structure without URL segments, and $slug marks dynamic parameters.
  • Page routes export a component and loader; server routes (e.g. app/routes/api/search.ts) export HTTP handlers returning Web Responses. Together with createServerFn they form the app's BFF.
  • Nitro is the runtime: the Vite build emits .output/server/index.mjs, which the Dockerfile runs as node .output/server/index.mjs on port 7005. traceDeps controls what gets externalized from the server bundle.
  • SSR fits a mixed site: it makes content crawlable and fast while heavyExternals, manualChunks, and stubServerFiles keep browser-only game code out of both the server bundle and content visitors' downloads.
