B1 Answer: (b). A rhythm game must stay locked to the music, so GameEngine.update() reads the playhead from the audio subsystem (this.audioManager.getCurrentTime()) and subtracts the user's calibration offset, rather than advancing one tick per animation frame. Because the loop is a pure consumer of an external clock, a dropped or slow frame cannot desynchronize the simulation — the next frame simply reads the true audio time. (a), (c), and (d) are unrelated to the audio-clock decision; in fact the per-frame path deliberately performs no network I/O except a throttled multiplayer sync.
B2 Answer: (c). A forged-but-plausible value passes every gate: it is a well-formed number inside 0 … 1_000_000_000, played at speed ≥ 1.0, from an authenticated session, under the 5/min rate limit. The server cannot distinguish it from an honest play because it never received the evidence of the play — the per-note hit timings against the beatmap. The range check (b) only rejects obviously impossible values; the rate limiter (a) and auth (d) gate abuse rate and ownership, not honesty. To truly re-validate, the server would need the input trace plus seed/beatmap id and would re-simulate against the same HIT_WINDOWS — which is why bounding "limits the blast radius, it does not prove honesty."
B3 Answer: (b). RmhTube embeds the player itself, so the server holds the canonical videoState and must extrapolate the playhead forward (extrapolate/reanchor in sync-math.ts), measure clock offsets (the SYNC_PING/SYNC_PONG NTP-lite handshake), and correct drift (±5% rate nudge, then hard seek). RmhMusic is Spotify-powered: each client's Web Playback SDK owns the real timeline, so the server only rebroadcasts an authoritative position with one inline expression (positionMs + (Date.now() - updatedAt)) on a 5-second heartbeat. The general lesson: how much sync machinery you build depends on how much the underlying player gives you for free. (a) is true but not the cause; (c) and (d) confuse unrelated subsystems.
B4 Answer: (b). The clause userId: { in: followingIds } is fan-out-on-read: no per-follower timeline is ever written; every home feed is computed from scratch by querying the authors a viewer follows. Its characteristic cost is the read side — each timeline is a fresh query over a potentially huge follow list — while writes are cheap (one row inserted once). (a) describes the opposite strategy; (c) and (d) describe the hybrid in docs/feed/plan.md, which is designed but unbuilt — there is no FeedEntry model and no isHighFanout flag in the live schema.
B5 Answer: (b). B6 Answer: (c). The key insight, stated atop pkg/auth/auth.go, is that the Node services already authenticate without the Better Auth SDK — by running one SQL lookup against the shared session table (SELECT s."userId", s."expiresAt", u."name", u."image", u."isAdmin" FROM "session" s JOIN "user" u … WHERE s."token" = $1). Go reproduces that query verbatim and checks expiry. This separates the hard, framework-bound work of issuing sessions (kept in JS) from the trivially portable work of validating them (one query). (d) is the opposite of what the gateway does — it strips client-supplied X-Rmh-* headers at the trust boundary so they cannot be forged.

B9 Answer: (c). So if it never becomes healthy, the reverse proxy is never flipped and the old container keeps serving — a bad build degrades to "old version stays up," never to an outage. B10 Answer: (b). socket, rmhbox, and rmhtube hold per-lobby game state in the memory of the process that owns the connection. If two pods ran behind a plain load balancer, two players in one lobby could land on different pods — pod A holds the lobby, pod B has never heard of it — so they could not see each other. Scaling past one replica therefore needs sticky sessions and a shared backplane first; until then the chart correctly holds these at one while marking web "the scale target." (a), (c), and (d) misstate the cause — the constraint is in-memory room state, not a database limit, a cache, or a Kubernetes rule.
B11 Answer: Reading app/routes/api/slice-it/score.ts in order, the gates are: (1) rate-limit by IP — rateLimit(ip, { limit: 5, windowMs: 60_000, prefix: 'slice-score' }), returning 429 with Retry-After; (2) shape and bound the payload — username a string and score a number, the username sanitized to an allow-list and capped at 24, the score bounded to 0 … 1_000_000_000, and sub-1.0× plays rejected as unranked, all 400; (3) authenticate the session — auth.api.getSession(), returning 401 if there is no session.user; then (4) persist only what survived. The IP rate limit runs first because it is the cheapest check and guards the most abusable property: it costs almost nothing (an in-memory Map lookup) and caps how fast a hostile client can even attempt the more expensive downstream work — JSON parsing, sanitization, a database round-trip in getSession — so a flood is rejected before it can consume those resources.
B12 Answer: The blob (savedRunState Json?) carries the full state needed to resume an in-progress roguelike run — deck, relics, floor, enemy state — which is too rich and too client-evolving to model relationally; storing it whole lets the client change its shape without a migration. But Postgres cannot efficiently ORDER BY a field buried inside a JSON document, so the blob cannot rank. The derived columns highScore and floorReached exist precisely for ranking: they are indexed (@@index([highScore(sort: Desc)])) so the leaderboard's findMany … orderBy desc … take is a fast index scan. Reading the blob to rank would mean deserializing every player's full run and sorting in application code — O(players) JSON parses per page load — which is why the leaderboard reads the columns. Each does what the other cannot: the blob resumes, the columns rank.
B13 Answer: The .mdx source files were authored under content/blog and content/news and parsed offline with gray-matter by scripts such as scripts/generate-blog-data.ts, which upsert them into Postgres (BlogPost / NewsArticle, tables blog_post / news_article). B14 Answer: Ported = a faithful, tested re-implementation of the Node behavior, safe to treat as equivalent (modulo the wire-protocol gap). Scaffolded = the structure, interfaces, and lifecycle are real and tested, with a representative slice implemented while the long tail is not — a framework with the first example filled in. Stubbed = a placeholder that compiles behind a real interface and runs but does nothing meaningful (marked // TODO(migration):). Classifications: (a) doctrine-worker is ported (bit-exact, mulberry32 golden-tested, all three jobs); (b) the eight non-Rhyme-Time rmhbox minigames are stubbed (each a stubGame that auto-completes after one second with everyone ranked zero); (c) the gateway is ported + new (reverse proxy plus the auth trust boundary). A stub like stubGame.Start() still compiles because it implements the real game interface — it just calls OnComplete(ComputeResults()) on a timer instead of running real logic. That satisfies the FOUNDATION rule (go build ./... must succeed; never leave broken code), which matters because it lets the surrounding system be exercised today and makes finishing each stub additive (drop in the real handler) rather than structural (no rewiring).
B15 Answer: The Go core is built by Bazel and is hermetic: a concrete mechanism is go_sdk.download(version = "1.23.0") in MODULE.bazel (with bazelisk pinning Bazel 7.4.1 and --incompatible_strict_action_env scrubbing the environment), so the build ignores the host's tools and yields identical artifacts everywhere. The //:frontend leaf is deliberately non-hermetic: scripts/frontend-build.sh runs cd into the source tree and reuses whatever node_modules the developer already has (installing only if absent), so the output can vary machine to machine and Bazel cannot guarantee byte-for-byte reproducibility. The payoff the team accepts is ergonomic: because the leaf is a single coarse target wrapping the whole Vite/esbuild build, "new frontend files need no BUILD changes" — a developer can add a hundred React components without ever editing a BUILD.bazel. The trade is justified because the JS native-dependency reality makes a fully hermetic frontend expensive and the leaf has no Bazel consumers downstream.
B16 Answer: Constraint A — a single Postgres throttled by connection count. Every new replica opens its own pool to the one database, and Postgres runs out of connection slots (each idle connection costs memory and a backend process) long before it runs out of CPU. Stage 1 relieves it with PgBouncer, which multiplexes many mostly-idle application connections onto a small fixed pool of real backend connections (transaction pooling), so the same database tolerates a far larger fleet. Constraint B — un-CDN-able, single-replica websockets. The realtime services hold per-lobby state in process memory, so they cannot be cached at an edge and two replicas are not interchangeable. Stage 1 relieves it with a Redis backplane (Socket.io adapter) plus sticky sessions, letting replicas share room events while each connection stays pinned. A Go rewrite relieves neither: Constraint A is connection count against one DB (fixed by PgBouncer/replicas, language-agnostic), and Constraint B is shared room state across replicas (fixed by the backplane and stickiness, which a Go service needs in exactly the same form). Go lowers per-connection cost, but neither binding constraint is a per-connection-cost problem — so it is a per-service optimization gated on metrics, not a scaling plan.
B17 Answer: A correct state diagram, with blue (7005) initially live: 
stateDiagram-v2
  [*] --> BlueLive
  BlueLive --> GreenStarting: docker compose run web on 7015
  GreenStarting --> GreenHealthy: GET / on 7015 returns 200 within 60s
  GreenStarting --> BlueLive: green exits / 60s timeout (traffic never moved)
  GreenHealthy --> Flipped: write Define WEB_UPSTREAM_PORT 7015; Apache graceful reload
  Flipped --> Verified: drain 8s; stop old; probe :80 with Host header = 200
  Flipped --> RolledBack: probe :80 fails (restart old, revert port file, abort)
  Verified --> [*]: remove old container, swap complete
Live user traffic moves at the GreenHealthy → Flipped transition — the moment the active-port file is rewritten and Apache gracefully reloads onto 7015 (the graceful reload finishes in-flight requests on the old worker set first). (a) If green never becomes healthy, the system stays in / returns to BlueLive — the flip is never written, so the old container keeps serving. (b) If the verification probe fails, the system lands in RolledBack — the old container is restarted, the port file reverted, and the deploy aborts. The old container is removed only after the probe because the probe is the proof that Apache is genuinely routing to the new port (it guards against, e.g., the IncludeOptional path diverging from ACTIVE_CONF); deleting it earlier would turn a bounded, self-healing blip into a sustained outage with no rollback target.
B18 Answer: A sound plan:
  • (1) Gateway route & rollback. Adopting one service is one prefix in internal/gateway/proxy.go: point /rmhmusic-ws/ at the Go rmhmusic binary (port 7002) instead of the Node upstream; everything else still falls through the catch-all to the unchanged Node web tier. Rollback is removing that one prefix (or repointing it back) and restarting the gateway — the catch-all carries the traffic. Per-prefix, reversible.
  • (2) Wire-protocol gap. Browsers speak Socket.IO v4; the Go realtime services speak JSON-over-raw-WebSocket with the same semantics but a different wire format — they are not interoperable. Before any real browser can connect, the client-side adapter (designed but unbuilt, per 24-risks.html) must exist and ship to the frontend so the browser emits the Go protocol over /rmhmusic-ws/.
  • (3) Validate equivalence. Exercise the Go service with new-protocol end-to-end clients first: replay representative room/sync/queue/chat sessions against both implementations and diff outcomes (snapshot+delta sync, drift, host reassignment), and confirm session validation matches (same shared session-table query). Only once behaviour matches and the adapter is in place do you route real traffic, ideally to a small cohort, watching metrics.
  • (4) Why this honours Stage 4. PR #121 set Go as a Stage-4 option to be adopted per service, only if metrics prove a bottleneck. Here the trigger is exactly that — measured CPU saturation on Node rmhmusic — and the change is one isolated, reversible service, not a wholesale cutover. The React SSR tier and Better Auth issuance stay in JavaScript (proxied and validated, never re-implemented); the eight other Node services are untouched. That is the strangler-fig discipline, the opposite of a big-bang rewrite, which would have endangered the whole platform with a single switch and no rollback.
B19 Answer: (a) You cannot scale rmhbox to three plain replicas because it holds per-lobby state in the memory of the process owning the connection. With an ordinary load balancer, two lobby-mates can land on different pods — pod A holds the lobby, pod B has never heard of it — so the two players simply cannot see each other (X's moves never reach Y, and vice-versa). (b) Stage 1 adds two complementary mechanisms: sticky sessions (session affinity at the ingress) guarantee that a given client's many websocket frames all hit the one replica that already holds its connection — connection coherence; and a Redis backplane (the Socket.io adapter) is a shared pub/sub bus every replica subscribes to, so an event for a room published by one replica reaches members held by the others — lobby coherence across replicas. (c) Enable exactly one and it breaks: with stickiness but no backplane, X (replica 1) and Y (replica 2) each have a coherent connection but their replicas never relay room events, so they still cannot see each other; with the backplane but no stickiness, a single client's frames scatter across replicas mid-conversation, so the replica handling a given frame may not hold that client's connection/auth state and the conversation becomes incoherent. You need both. (d) Post-Stage-1 path of an event from X to Y: 
X's browser --(websocket, sticky)--> LB --> rmhbox replica 1
  replica 1 --(publish room event)--> Redis backplane (pub/sub)
  Redis --> rmhbox replica 2
  replica 2 --(emit to its lobby members)--> LB --> Y's browser
This same machinery would still be required if rmhbox were rewritten in Go: Constraint B is shared room state across replicas, not per-connection cost, so a Go service still needs sticky-session ingress for connection integrity and the Redis backplane for cross-replica room coherence — the backplane is the load-bearing mechanism in either language. (Static assets, by contrast, are CDN-cached at the edge; the websocket cannot be, because it is a unique, long-lived, bidirectional conversation with nothing to cache and replay.)
