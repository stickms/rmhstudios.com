8.1 Answer: Server-authoritative scoring means the server, not the browser, is the final authority over what score is allowed to affect persistent state. The client may compute and display a score for responsiveness, but every submission is re-authenticated, re-validated, and bounded by the server before it can touch a leaderboard. The browser's number cannot be trusted because the player owns the machine running the JavaScript: handleHit() runs in their process, they can edit it, replay it, or skip it entirely and call POST /api/slice-it/score directly with any number. The arithmetic being correct on screen is irrelevant — the server never observed the keystrokes, so the submitted number is a claim, and treating a claim as truth is the whole vulnerability.
8.2 Answer: In order, the gates in app/routes/api/slice-it/score.ts are: (1) Rate limit by IP — rateLimit(ip, { limit: 5, windowMs: 60_000, prefix: 'slice-score' }); fails with 429 plus a Retry-After header. (2) Payload shape — username must be a string and score a number; fails with 400 ("Invalid score data."). (3) Username sanitization — stripped to an allow-list and capped at 24; if under 2 chars, 400 ("Invalid username"). (4) Score bound — must be 0 … 1_000_000_000, else 400 ("Invalid score"). (5) Speed check — playSpeed < 1.0 is rejected as unranked with 400. (6) Authentication — auth.api.getSession(); no session.user yields 401 ("Unauthorized."). Only after all six does it write: increment Player.totalScore/gamesPlayed (or create the profile, with a 409 if the username is taken), and if songId matches a real Song and the score beats the existing personal best, upsert a SongLeaderboard row. A thrown error returns 500.
8.3 Answer: Writes mutate persistent, publicly visible state (a leaderboard) and have side effects — incrementing totalScore, creating rows. A flood of forged writes corrupts the rankings everyone sees and grows the database, so the cost of abuse is high and permanent; capping at 5/min sharply limits how fast a single IP can attempt forgeries. Reads (leaderboard GET) only return existing data, are cacheable, and leave no residue; the worst a flood does is consume CPU and bandwidth, which is a denial-of-service concern handled more cheaply elsewhere. So the generous 20/min read limit trades a little DoS exposure for a smooth UI (refreshing the board, paging songs), while the tight 5/min write limit protects the integrity of the data itself — the asymmetry mirrors the asymmetry in consequences.
8.4 Answer: The range check only rejects values that are obviously impossible (negative, or above a billion). It does nothing to a cheater who submits a forged-but-plausible number — say a legitimate-looking 480,000 for a song where a strong human might score 250,000. Such a submission passes every gate: it is a well-formed number in range, played at speed ≥ 1.0, from an authenticated session, under the rate limit. The server cannot tell it apart from an honest play because it never received the evidence of the play — the per-note hit timings against the beatmap. To actually re-validate, the server would need to receive (and re-simulate) the input trace: the sequence of (lane, timestamp) presses plus the seed/beatmap id, then recompute the score with the same HIT_WINDOWS and calculateScoreMultiplier() logic server-side and confirm it matches. That is the deterministic-replay design the deterministic RNG (8.6) is a precondition for; bounding only limits the blast radius.
8.5 Answer: The blob (savedRunState Json?) carries the full state needed to resume an in-progress roguelike run — deck, relics, floor layout, enemy state — which is too rich and too client-evolving to model relationally; storing it whole lets the client change its state shape without a database migration. But Postgres cannot efficiently ORDER BY a field buried inside a JSON document, so the blob is useless for ranking. The derived scalar columns highScore and floorReached exist precisely for that: they are indexed (@@index([highScore(sort: Desc)])) so the leaderboard's findMany … orderBy desc is a fast index scan. The leaderboard query cannot read the blob because doing so would mean deserializing every player's full run and sorting in application code — O(players) JSON parses on every page load. So each representation does what the other cannot: the blob resumes, the columns rank.
8.6 Answer: createSeededRandom(seed) is deterministic because its entire internal state is initialized from the seed string (hashed into h) and then advanced by a pure integer recurrence (the mulberry32 step) with no external entropy — no Date.now(), no Math.random(), no I/O. Given the same seed, the closure produces the identical sequence of numbers every time, on any machine. The feature this enables, named in the chapter, is a seeded-run leaderboard (and the closely related deterministic replay): everyone playing seed S faces the exact same generated content, so their scores are comparable, and a stored seed lets the server reconstruct the run for validation (see 8.4). Neon Driftway's spec records seed = timestamp XOR random in run stats for exactly this reproducibility.
8.7 Answer: For a window where the left channel is much louder, BeatMapGenerator.generateFromBuffer() hits the branch if (maxL > maxR * 1.2) lane = 0, so the Slice is assigned lane 0 (top) — the engine reads a left-dominant transient as a "top" note. (Symmetrically, maxR > maxL * 1.2 gives lane 1.) The slice is only emitted if its peak exceeds the threshold and at least minDistance (0.2s) has passed since the last beat, preventing spam. When the channels are balanced (neither exceeds the other by the 1.2× factor), the lane is chosen by the seeded generator — lane = rng() > 0.5 ? 1 : 0 — so balanced beats still get a lane, but a reproducible one tied to the track id, not a fresh Math.random().
8.8 Answer: A correct flowchart keeps the validation spine of Figure 8.2 and adds a parallel, non-persistent branch during play. Example: 
flowchart TD
  P[Play: GameEngine.update] -->|throttled ~3x/sec| MP[mp.updateScore lobbyId]
  MP --> FLEET[Socket fleet: opponent HUD]
  P -->|on FINISHED| SUB[POST score.ts]
  SUB --> RL{rate limit 5/60s}
  RL -->|429| STOP1[reject]
  RL -->|ok| VAL{sanitize + bound + speed}
  VAL -->|400| STOP2[reject]
  VAL -->|ok| AUTH{getSession}
  AUTH -->|401| STOP3[reject]
  AUTH -->|ok| DB[(write Player + SongLeaderboard)]
The mp.updateScore branch leaves update() mid-play and goes only to the socket fleet to drive the live opponent HUD; it carries { score, combo } for presence, is throttled (Math.random() < 0.05), and is never authoritative. It does not reach the leaderboard write because the database is the record of truth and only the final, authenticated, validated score.ts submission may mutate it — realtime sync is for presence, not truth.
8.9 Answer: (1) Infrastructure layer: put a trusted reverse proxy / CDN (nginx or Cloudflare) in front of the app and have it overwrite — not append — the x-forwarded-for header with the real connecting IP, and drop any client-supplied value. This restores the assumption getClientIp() documents, so the spoof in the warning callout no longer works; the proxy can also enforce its own edge rate limiting and bot challenges before requests reach the Node process. (2) Application layer: rate-limit and account on a server-trusted identity rather than IP — key the limiter (or a second limiter) on the authenticated session.user.id, which the client cannot forge because it comes from auth.api.getSession(). Per-account limits (e.g. N ranked submissions per minute, plus anomaly checks like rejecting a new personal best that exceeds the previous one by an implausible margin) bind abuse to a real, revocable account regardless of network path. Either measure is independent of IP rate limiting and survives its bypass.
8.10 Answer: Altair is a survivors-like — it can have hundreds of enemies, projectiles, and particles on screen at once, i.e. a very high draw count ("the heaviest draw load," per the chapter). Canvas 2D issues one immediate-mode draw call per object on the CPU and stalls at that volume, so Altair pays for a hand-written WebGL2 SpriteBatch that packs many sprites into one GPU draw call. Slice-It, by contrast, draws a handful of lanes and notes, so Canvas 2D is plenty. What would not change if Slice-It were ported to WebGL is everything above the renderer: the loop body GameEngine.update() still reads the audio clock and resolves notes against HIT_WINDOWS; submitInput(), the seeded RNG, the beatmap, the score pipeline, and the MultiplayerFactory all stay identical. The loop-update-render pattern is renderer-agnostic — only the final "draw the current state" step is swapped.
