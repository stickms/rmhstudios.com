C15 Answer: Authentication = who you are; authorization = what you may do. On an expired cookie, auth.api.getSession({ headers }) returns null (no session.user), so the route sends 401 Unauthorized — e.g. Response.json({ error: 'Unauthorized.' }, { status: 401 }). 401 (not 403) is correct because the failure is one of identity: the request has no valid established identity at all. 403 would mean “we know who you are, but you are not allowed to do this” — an authorization failure — which is not the situation here. (Full credit: the authn/authz distinction, getSession returning null, the 401 status, and the identity-vs-permission justification.)
C16 Answer: Server-authoritative means the server holds canonical state and recomputes anything that affects the outcome. In rmhtype:game:progress the client sends only raw position (characters typed) and errors — not a score. The server refuses the event unless the room is in state TYPING with room.roundStartTime set, then computes WPM itself as (position / 5) / (elapsedMs / 60000) and accuracy as ((position - errors) / position) * 100, where elapsedMs is measured against the server’s clock (Date.now() - room.roundStartTime). The one lever a cheating client has is how many characters it claims to have typed, and that is bounded by the round timer the client does not control — it cannot simply announce “400 WPM.” (Full credit: client sends only position/errors, server computes WPM from its own clock, room must be TYPING, and the bounded-lever sentence.)
C17 Answer: A backplane is a shared pub/sub bus (Redis) that lets multiple replicas of a service forward events to one another, so a broadcast originating on one replica reaches sockets connected to a different replica. With only an in-process bus, each replica is an island. The same primitive recurs because the underlying problem is identical everywhere state is held per-process: the feed-SSE bus (Chapter 9) needs it to deliver rmhark.created across web replicas; the Go pkg/events bus (Chapter 10) needs it to fan realtime events across Go instances; and the Stage-1 scaling plan (Chapter 13) needs the Socket.io Redis adapter to make io.to(room).emit cross replicas. All three default to in-process and “light up” only when REDIS_URL is set — the backplane is a language-agnostic scaling primitive. (Full credit: correct definition of a backplane and the “same per-process-state problem ⇒ same primitive” argument across the three appearances.)

C20 Answer: A complete trace covers three legs. (a) Live updates: while playing, Neon Driftway runs over the main socket-server (socket, port 7001, path /socket/), reached client-side through the MultiplayerFactory against VITE_SOCKET_URL. The updateScore stream is throttled (~3×/sec) precisely because it is cosmetic — it only drives the opponent HUD, not the ranked record — so flooding the room with every tick is wasteful. (b) Game end: the final, authoritative score is sent as an HTTP POST to the app/routes/api/<game>/score.ts family (e.g. the Neon Driftway score handler), which passes gates in order: (1) IP rate-limit, (2) bound/shape the payload (score clamped to its max, level clamped to its range), (3) authenticate via auth.api.getSession({ headers }) → 401 if no session, and (4) persist. Crucially, the server keys the write on session.user.id from the validated session — never a user id from the request body — so the client cannot write someone else’s row. High-water-mark fields like highScore/bestDistance are written with Math.max(existing, incoming). (c) Rest + read: the row lands in the single shared PostgreSQL instance; the leaderboard read later is made cheap by a descending score index (e.g. @@index([highScore(sort: Desc)])) so a findMany … orderBy score desc take 10 is an index scan, not a full sort. Why two paths: the socket path optimizes for low-latency, lossy, cosmetic liveness, while the HTTP path optimizes for a single trusted, rate-limited, authenticated, durable write — conflating them would either make the durable write as lossy as a HUD tick or make every HUD tick a heavyweight authenticated DB transaction. (Full credit: socket-server 7001//socket/ + throttled HUD reasoning; the score.ts gate order; identity keyed on the session not the body; single Postgres + a descending leaderboard index; and the “different jobs ⇒ different paths” sentence.)
C21 Answer: (a) The two constraints. Constraint A — a single Postgres instance (data layer): each replica opens its own pool, and Postgres runs out of connection slots before CPU, so it bites “as soon as you scale anything out.” Constraint B — un-CDN-able, single-replica websockets (realtime fleet): each lobby’s state lives only in the memory of the one process holding the connection. (b) Stateless vs stateful (interchangeability test): “If I kill this replica and the client’s next packet lands elsewhere, does anything break?” web (SSR) and the workers (doctrine-worker, vibe-worker, bot-worker, recap, discord-bot) hold no per-client memory — any replica can serve any packet — so they scale out cleanly today. socket, rmhbox, rmhtube hold per-lobby state in process memory, so a packet landing on a different replica finds no lobby — they are pinned to one replica. (c) Stage-1 relief: Constraint A → a PgBouncer connection pooler in transaction-pooling mode (infrastructure/data tier), later read replicas; Constraint B → a Socket.io Redis backplane + sticky sessions (realtime tier + ingress). (d) Why a CDN cannot help websockets: a CDN caches content that is identical for everyone and slowly changing, serving copies from the edge. A websocket is a unique, long-lived, bidirectional conversation — there is nothing identical-for-everyone to cache and nothing to replay — so a CDN has no cache to offer it. (The static surface, by contrast, is exactly that cacheable content, which is why Stage 1 puts a CDN only in front of it and leaves the socket./box./tube. hosts un-proxied.) (Full credit: both constraints with what bites + which subsystem; the interchangeability test correctly applied to stateless/stateful; PgBouncer and backplane+sticky as the two reliefs; and a correct “a CDN caches identical content” argument.)

C23 Answer: (a) Naive failure. Lobby state lives in each rmhbox process’s in-memory Map (the LobbyManager), which is not shared. Scaling to three replicas behind an ordinary LB routes player X to replica 1 and player Y to replica 2; each replica creates its own lobby object for the same six-character code, and io.to(room).emit(...) on replica 1 reaches only sockets on replica 1. The two players “join the same code” but never see each other — separate, silent lobbies. (b) The two mechanisms. A sticky session (session affinity at the ingress) pins one client to one replica for the life of its WebSocket connection — it keeps the connection coherent. A backplane (Socket.io Redis adapter / pub-sub bus) forwards room broadcasts between replicas so an event reaches everyone in the lobby regardless of which replica holds each connection — it keeps the lobby coherent across replicas. You need both. (c) One-without-the-other. Backplane but no sticky session: a client’s long-lived WebSocket can be bounced between replicas mid-session (or its handshake polling lands on different replicas), breaking the connection’s continuity even though broadcasts would propagate. Sticky but no backplane: each client stays put, but a broadcast emitted on replica 1 never crosses to replica 2, so X and Y on different replicas still never hear each other. Either alone leaves the lobby broken. (d) Post-Stage-1 path. Player X → ingress (sticky, routes X to replica 1) → rmhbox replica 1 receives the event → replica 1 publishes it to the Redis backplane → replica 2 (subscribed) receives it → replica 2 emits to its local sockets → over the sticky connection to player Y’s browser → Y’s screen. (Both replicas still validate sessions and persist against the one Postgres, fronted by PgBouncer.) (e) Why a Go rewrite does not replace this. Lobby state is still per-process in Go, so cross-replica coherence still demands the same backplane + sticky sessions, and the shared Postgres still needs a connection pooler — the rewrite lowers per-connection cost but is a per-service optimization, not a scaling plan. (Full credit: the per-process-Map failure; sticky=connection vs backplane=lobby with “need both”; a correct one-without-the-other scenario; a path that names ingress + both replicas + the backplane; and the Go-doesn’t-change-the-machinery sentence.)
