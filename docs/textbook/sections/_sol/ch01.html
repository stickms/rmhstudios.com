1.1 Answer: A process-split architecture runs one logical application as several independent operating-system processes, each with a single responsibility, instead of one monolithic server. In rmhstudios.com this means one web/SSR tier (Nitro) plus a fleet of standalone processes — three Socket.io realtime servers and several background workers. Two distinct benefits (any two): (1) fault isolation — if one process (e.g. the vibe-worker rendering thumbnails) crashes, the web tier and the rest keep serving, whereas a monolith would fail as a unit; (2) independent scaling — you can run extra copies of just the part under load (e.g. more realtime servers during a tournament) without duplicating everything; (3) independent deployment/runtime — long-lived WebSocket servers and stateless request renderers can be restarted, tuned, or even rewritten in another language separately.

1.4 Answer: The browser holds two connections at once (Figure 1.3):
  1. HTTP to the web tier (Nitro SSR) — fetches the rendered game page and its JS, and serves API routes. In production this is the web service (port 7005).
  2. WebSocket to a realtime server — for joining the lobby and receiving live game events. For most games this is socket-server (port 7001); for the minigame lobby it is rmhbox (7676).
What they have in common is the shared PostgreSQL database: both the web tier and the realtime server read/write it, and that is the only thing connecting the two halves — they do not call each other directly.
1.5 Answer: Publishing a host port means a service accepts inbound network connections from users (via the reverse proxy). socket, rmhbox, and rmhtube publish loopback ports (7001 / 7676 / 7003) because they are realtime servers that clients connect to over WebSockets. The four bot/worker services publish no port because they are background workers: they take no inbound user traffic and instead reach the rest of the system through the shared database and outbound calls (e.g. Discord, DeepSeek). The presence or absence of a published port is therefore a reliable signal of "server vs. worker."

1.7 Answer: A strangler-fig migration replaces a system incrementally: new components grow alongside the old ones and take over responsibilities one at a time until the original can be removed (named after the strangler fig that grows around and eventually replaces its host tree). It is feasible here because the realtime tier is already split into independent processes that coordinate only through the shared database — they do not call each other in-process. The load-bearing property is the loose coupling between tiers (communication via the database rather than direct in-process calls). Because a Go binary under go-services/cmd/<service>/ can speak to the same PostgreSQL, it can replace one Node service at a time, with the web tier and the remaining Node services none the wiser. A tightly coupled monolith would have to be cut over all at once.
1.8 Answer: Because RMHBox is its own process (rmhbox on port 7676), the natural response is to run multiple instances of just the rmhbox service during tournaments and load-balance new connections across them, leaving the single web tier untouched. A single-process monolith cannot do this: every responsibility shares one process, so scaling RMHBox would mean duplicating the home page, blog rendering, and every other server too — wasteful and possibly impossible if the monolith holds shared in-process state. The new problem the split introduces is stateful WebSocket connections: a lobby's players must reach the same instance that holds their room. Naïve round-robin would scatter a lobby across instances. A strong answer proposes sticky sessions / room-aware routing, or sharing room state out of the process (e.g. via Redis or the database) so any instance can serve any room.
1.9 Answer: One image, many commands: Advantage — the image is built once and shared by every service (no redundant rebuilds, identical dependencies and code across the fleet, and rollback is a single tagged image). Drawback — every service carries the full payload even if it only needs a slice of it (e.g. workers ship the whole web build; the vibe-worker drags in everything although only it needs headless Chromium), so images are larger and a vulnerability anywhere in the shared image affects all services. Per-service minimal images would be leaner and have smaller attack surfaces, at the cost of more build complexity and more artifacts to manage.
