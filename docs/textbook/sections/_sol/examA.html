A4 Answer: (A) A page route exports a component (and usually a loader and head) and renders UI; a server route exports server.handlers for HTTP methods (e.g. GET) that receive a Web Request and return a Web Response, with no component at all — as in app/routes/api/search.ts. The distinction is about what is exported, not the file extension per se (though server routes are conventionally .ts, not .tsx). It is "the same file-based router, just exporting handlers instead of a component."
A5 Answer: (B) A hydration mismatch. The server has no localStorage global, so it cannot produce the same markup the client's first render would; React detects the divergence at the hydration boundary and must repair the subtree, producing a visible flash or a console warning. The correct pattern is to render a server-safe default first and reconcile to the real value inside a useEffect (which runs only in the browser), or to seed the real value from a server loader as the session does.
A6 Answer: (C) the puzzle leaderboard / reputation score. Its source of truth is the database behind an API; the client only holds a possibly-stale cache of it, which is why it belongs in React Query with a staleTime and refetch policy. A dropdown's open flag is local state; the chosen theme and the live game-run HP are client state (owned by the browser session, with no authoritative server copy). The deciding test: if unplugging the network can never make the value "wrong," it is client state; if it can drift because someone else changed the database, it is server state.
A7 Answer: (B) The integer columns are a denormalized cache. Aggregating RMHarkLike/comment/repost rows on every feed read is expensive at feed scale, so the counts are stored on RMHark and updated transactionally (atomic increment/decrement) by the like/comment/repost/view routes, keeping the read path off _count. A9 Answer: (C) Because a session is a server-stored row (a session row with a unique token), the token in the cookie is meaningless on its own — deleting or expiring the row instantly invalidates the cookie everywhere, so sessions are revocable. Self-contained signed tokens, by contrast, remain valid until they expire because the server keeps no record to revoke. This revocability is precisely what makes a "log out of all other devices" feature straightforward. (A) is false — sessions have an expiresAt; (B) describes self-contained tokens, the opposite choice; (D) is unrelated — HttpOnly/Secure are still used.
A10 Answer: (A) socket.to(room).emit(...) broadcasts to every socket in the room except the sender — correct for relaying a player's own position update, since the sender already knows where it is. io.to(room).emit(...) broadcasts to everyone in the room including the sender — correct for the countdown, which every client (the host included) must see start together. They are not aliases (C), and neither touches the database (D).
A11 Answer: A process-split architecture runs one logical application as several independent operating-system processes, each with a single responsibility, instead of one monolithic server doing HTTP, WebSockets, rendering, and background jobs in one event loop. rmhstudios.com runs one web/SSR tier plus a fleet of standalone processes. Two distinct benefits (any two): fault isolation — if the vibe-worker crashes mid-render the home page keeps serving, where a monolith would fail as a unit; independent scaling — you can run extra copies of just the part under load (e.g. more realtime servers during a tournament) without duplicating everything; independent deployment/runtime — a long-lived WebSocket server and a stateless renderer can be restarted, tuned, or even rewritten in another language separately. A realtime server example: socket-server (:7001), rmhbox (:7676), or rmhtube (:7003). A worker example: discord-bot, doctrine-worker, vibe-worker, or bot-worker.
A12 Answer: The server function is getInitialUser in app/routes/__root.tsx, which reads the session cookie via getRequest() and queries Better Auth on the server. Ordered hops: (1) browser sends GET / with its session cookie; (2) Nitro matches the URL against the generated route tree (__root → _site → _site/index); (3) the loaders run on the server — root's getInitialUser resolves the user, the index's fetchSidebarData pulls feed/sidebar data; (4) React renders the matched component tree to an HTML string; (5) Nitro returns that HTML plus the serialized loader data; (6) the browser paints content immediately and downloads the client JS bundle; (7) hydration — React re-runs the components in the browser, finds the matching DOM, and attaches handlers/state, making the page interactive. Between first paint and the completion of hydration the page is visible but not yet interactive: content (and the correct signed-in shell) is on screen, but event handlers are not yet attached, so clicks do nothing until hydration finishes.
A13 Answer: One network request fires. Both calls share the query key ['doctrine', 'reputation'] against the one QueryClient, so React Query deduplicates the concurrent requests — identical keys mounted at the same time collapse into a single in-flight fetch — and the second component receives the cached/shared result rather than issuing its own request. Ninety seconds later the cached value is older than staleTime (60 s), so it is stale: the next mount or trigger refetches in the background while still showing the cached value until the fresh data arrives. (Deduplication: collapsing multiple simultaneous requests for the same key into one network call whose result is shared by all subscribers.)
A14 Answer: @@unique([userId, gameMode, dateKey]) enforces the invariant "at most one score row per user, per game mode, per day" — the database itself refuses a duplicate, and the constraint doubles as the index an upsert keys on. @@index([gameMode, dateKey, score(sort: Desc)]) backs the leaderboard read: "for this mode and date, the top scores descending" becomes a single index range scan with no full-table sort. A @@unique is "both an index and a correctness rule" because Postgres implements uniqueness with an index (so it accelerates lookups on those columns) and rejects any insert that would violate the constraint — so a uniqueness bug cannot slip through even under a race condition. Column order matters: the leaderboard index leads with the equality-filtered columns (gameMode, dateKey) and ends with the sort column (score desc) so the index can both filter and order in one pass.
A15 Answer: Authentication (authn) establishes who a request is from ("this is user u_abc"); authorization (authz) decides what that already-identified user may do ("they may write only their own leaderboard row"). At /api/slice-it/score the route calls auth.api.getSession({ headers: request.headers }); with an expired cookie the library finds no matching unexpired Session row and returns null (no session.user). The route then returns Response.json({ error: 'Unauthorized.' }, { status: 401 }). 401 is correct because a missing/expired session means no identity at all — an authentication failure ("we don't know who you are") — whereas 403 Forbidden would mean "we know who you are, but you may not do this," a pure authorization failure that does not apply when there is no user.
A16 Answer: Each lobby is a plain object in a module-level Map in one process's memory, and io.to(room).emit(...) reaches only sockets connected to that process. So two replicas of the same service do not share lobbies: if a load balancer sent player A to replica 1 and player B to replica 2, they could "join" the same room code yet never see each other — each replica has its own map and its broadcasts stay local. Horizontal scaling therefore requires two additions: a backplane (adapter) — a shared message bus (commonly a Redis pub/sub adapter for Socket.io) that forwards room broadcasts between instances so an emit on any replica reaches the whole room; and sticky sessions — a load-balancer policy that pins each client to the same backend for the life of its WebSocket connection so the long-lived handshake is not bounced between replicas. (Even then, authoritative lobby state would need a shared home — Redis or the database — or lobbies must be sharded so all members land on one instance.)

A18 Answer: Arguments against a pure SPA (need at least three): (1) Crawlability — blog posts, news, and public builds must be indexable; sitemap[.]xml.ts emits real URLs for crawlers, and a pure SPA would serve them an empty shell with no content. (2) Personalized first paint — getInitialUser in __root.tsx reads the session cookie on the server and renders the signed-in shell on the very first frame; an SPA would flash "signed out" until client JS resolves the session after hydration. (3) Time-to-content — a content document should show text on the first response, not after a megabyte of JavaScript downloads and executes; SSR returns content-filled HTML immediately. (Also acceptable: the BFF server routes/createServerFn already shape data server-side with auth and rate limits.) The legitimate concession: the games library genuinely should not be server-rendered — three, pixi.js, monaco-editor, tone are browser-only and enormous. The app already supports this client-only lane in vite.config.ts: heavyExternals/traceDeps keep these packages out of the server bundle, manualChunks (e.g. vendor-three, vendor-pixi, vendor-monaco-*) split them so a content visitor never downloads game code, and stubServerFiles replaces *.server.ts imports with empty stubs on the client so Node-only deps like pg never leak into the browser bundle. One-sentence design: SSR by default gives the content site fast, crawlable, personalized pages, while per-package externalization and chunking carve out a client-only lane for the games — so one framework serves a portfolio blog and a 3D physics game without compromising either.
A19 Answer: State system: the unread count's source of truth is the database (other users' actions change it), so by the ownership/freshness test it is server state — use a TanStack React Query cache (e.g. useQuery(['notifications','unread'])), not a hand-rolled variable, since a hand-rolled value would go stale when someone else acts. (A defensible combination: React Query for the count, plus an SSE/socket push to invalidate or bump it between polls for near-realtime freshness.) Seeding the first render: to avoid a hydration mismatch the first count rendered on the client must match the server's HTML. Mirror how the session is handled — compute the count in a server loader (it already holds the cookie/identity) and seed it as the query's initialData/initial state, so the very first painted HTML and the first client render show the same number; the live query then reconciles and "always wins" if the seed was stale. Reading the count only in a client effect after hydration would flash a wrong/empty number, the classic mismatch. Refetch cadence: a modest refetchInterval (on the order of 30–120 s, mirroring the reputation hook's 120 s) keeps it "roughly current" without hammering the one Postgres instance; if a realtime push exists, lengthen the poll and rely on push for immediacy. Consistency hazard + mitigation: the seeded count can briefly disagree with reality (you opened a notification on another device, or a new one arrived between seed and refetch) — mitigate by letting the live query authoritatively overwrite the seed on resolve, invalidating the query key on the user's own read/mark-as-read actions, and (optionally) deduplicating against a realtime echo so an optimistic decrement and the server confirmation do not double-count.
