10.1 Answer: A microservice owns one slice of behavior, runs as its own process/container, and communicates only over well-defined transports (HTTP, WebSocket, a database, a message bus) rather than by sharing in-process memory. A monolith puts all behavior in one process where modules call each other by direct function calls. In the Go tier the rule from FOUNDATION.md is “every service is a binary under cmd/<service>/main.go” — e.g. cmd/rmhbox/main.go compiles to one binary, packaged as one container (rmhstudios-go-rmhbox:dev), deployed as one unit. A service author may write only under cmd/<svc>/ and internal/<svc>/; they must not edit pkg/ or go.mod.
10.2 Answer: Strangler-fig migration replaces a system incrementally: a stable interface is placed in front of the old system, the replacement is built behind it one piece at a time, and traffic is routed to each new piece only once it is proven; old and new run side by side, so you can always route back. A big-bang rewrite switches everything at once — no incremental validation and no partial rollback, so a failed switch loses both systems. The enabling component is the gateway (port 7005). Because it is the single ingress and selects an upstream by longest-matching path prefix — falling back to the Node web tier for everything Go doesn't own — cutting over one more service is just adding one prefix to the routes slice in internal/gateway/proxy.go, and rolling back is removing it. Routing decisions live at exactly one boundary, so services migrate one at a time.
10.3 Answer: Session issuance — creating a session: OAuth callbacks, cookie signing, writing the session row — is framework-bound work that stays in Better Auth in the React tier, because there is no Go equivalent and porting it would mean rewriting the frontend's auth stack. Session validation — checking that a presented token corresponds to a live session — is just a database read. The Node services already did validation without the Better Auth SDK, by one SQL lookup. pkg/auth's ValidateSession reproduces that query verbatim: SELECT s."userId", s."expiresAt", u."name", u."image", u."isAdmin" FROM "session" s JOIN "user" u ON u."id" = s."userId" WHERE s."token" = $1 LIMIT 1, returning ErrUnauthenticated on no row or an expired expiresAt. So the Go fleet only validates against the shared session table; it never issues.
10.4 Answer: The request reaches the gateway's catch-all handler, which delegates to Router.ServeHTTP → Router.pick(path). pick walks the prefix list (sorted longest-prefix-first so “most specific wins”) and matches /rmhbox-ws/, returning the rmhbox upstream. The path lands on the Go rmhbox service at port 7676, which serves that exact prefix via hub.ServeWS — the full path is proxied unchanged, no prefix stripping. The gateway does not strip Connection/Upgrade because Go 1.23's httputil.ReverseProxy detects the WebSocket upgrade, switches to bidirectional-copy mode on a 101 Switching Protocols response, and manages those hop-by-hop headers itself; stripping them manually would break the upgrade. The one reason this does not work with a real browser today: the browser client speaks Socket.IO v4, while the Go service speaks JSON-over-raw-WebSocket — same semantics, different wire format, not interoperable — and the client adapter is designed but not built.

10.6 Answer: The redrawn figure keeps every existing arrow and adds one route from the gateway to recap (port 7004), e.g. labeled with a new prefix such as /recap/; recap moves from “hit directly” to behind the front door, exactly as adding one entry to the gateway's routes slice would do. 10.7 Answer: A sound plan. Gateway route: point the /rmhmusic-ws/ prefix's upstream (RMHMUSIC_UPSTREAM) at the Go rmhmusic service (port 7002) and leave the other prefixes on the Node services; this is a one-line change in the routes slice / one env override, with the catch-all still serving the unchanged SSR app. Protocol gap: the live browsers speak Socket.IO v4 but Go rmhmusic speaks JSON-over-raw-WebSocket, so before any real traffic you must build the designed-but-unbuilt client-side adapter that maps the existing event names onto the new transport — without it, no production browser can connect. Validate equivalence: run the new-protocol e2e clients against the Go service, compare against the Node service for the same scenarios (snapshot+delta, playback drift sync, host reassignment), and shadow/dark-launch a small fraction of sessions before full cutover; remember rmhmusic requires auth, so verify session validation against the shared session table too. Rollback: repoint RMHMUSIC_UPSTREAM back at the Node service (or remove the prefix so the catch-all/other routing takes over) — a config flip, since the change is additive and the Node service was never removed. This honors PR #121's Stage-4 condition because the decision is metrics-driven (CPU-bound evidence), per-service (only rmhmusic moves), and reversible; the rest of the fleet and the React/Better Auth/Prisma core are untouched.
10.8 Answer: The foundation is the shared contract that every service depends on; if it were incomplete or buggy, no service could be trusted and a bug would have to be fixed in nine places. Investing to make pkg/ faithful and race-tested first means every service inherits correct config parsing, logging, db access, auth, http lifecycle, metrics, the events backplane, and the realtime framework — so the remaining work is concentrated in thin, isolated internal/<svc> logic. It is good engineering to make the dependency more complete than its dependents: stability flows upward. Stubs make finishing “additive rather than structural” because each stub already implements a real interface and compiles (go build ./... passes); the surrounding lifecycle, registry, and transports are wired and exercised against it. Completing a service therefore means filling one handler behind an existing contract — no interfaces to redesign, no plumbing to add — which is additive work, not a structural change.
10.9 Answer: The rebuttal: the branch delivers value whether or not it ships. (1) A complete, tested reference implementation of what a Go target looks like for this fleet — nine services on a faithful, race-tested foundation — so a future per-service migration starts from working code, not a blank page. (2) Extracted good ideas that are language-agnostic and adoptable now: the single auth-validation SQL query, the events.Bus Redis backplane (which lets even the existing Node realtime services scale past one replica), and the atomic stat-merge fix. (3) An incremental option: because every Go service validates against the shared session table, any one service can be migrated and measured in isolation when metrics justify it — exactly PR #121's Stage-4 discipline. The risk a rewrite normally carries is neutralized by design: the work is additive (the web image is unchanged), branch-isolated (no production traffic ever routed to it), interface-honest (every stub compiles behind a real contract), and gateway-fronted (any cutover is per-prefix and reversible). The failure mode would have been shipping a big-bang rewrite; building the target to learn from it, with no way for it to endanger production, is the opposite of waste.
10.10 Answer: A hermetic Go SDK is a toolchain the build system downloads and pins itself, so every build uses the exact same compiler version regardless of what is installed on the machine — here, rules_go 0.50.1 ships a hermetic Go 1.23.0 SDK under Bazel, and the no-Bazel Makefile path runs the official golang:1.23-alpine image in Docker. The build is arranged so “Go is not required locally” to make it reproducible and low-friction: a contributor needs only Docker (or Bazel), not a correctly-versioned local Go, which removes “works on my machine” toolchain drift across a nine-service fleet. The two paths: (1) Makefile-via-Docker — make build runs go build ./... inside the pinned Go image, producing the service binary; (2) Bazel — make build/make images uses the hermetic SDK and the go_service_image macro to produce, per service, an oci_image loaded locally as rmhstudios-go-<svc>:dev.
