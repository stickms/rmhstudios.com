11.1 Answer: A hermetic build is one whose every input — compiler, libraries, sources — is declared and version-pinned, so the build ignores whatever tools happen to be installed on the host and produces the same result on any machine. Hermetic for the Go side (any one): Bazel is pinned to 7.4.1 via .bazelversion/.bazeliskrc; rules_go downloads a pinned Go 1.23 SDK (go_sdk.download(version = "1.23.0")) so no system Go is used; --incompatible_strict_action_env scrubs the host environment from build actions. Non-hermetic on the frontend (any one): scripts/frontend-build.sh runs in the source tree (cd $BUILD_WORKSPACE_DIRECTORY) and reuses the repo's existing node_modules — including native packages like @napi-rs/canvas/@resvg/resvg-js — rather than building in a sealed sandbox.
11.2 Answer: It is true because rules_go ships a hermetic Go toolchain that Bazel downloads, instead of using a Go on the host's PATH. The pinning lives in MODULE.bazel: the extension go_sdk = use_extension("@rules_go//go:extensions.bzl", "go_sdk") followed by go_sdk.download(version = "1.23.0"). Because Bazel fetches and uses that exact SDK, a contributor can build all nine Go services with no local Go install.
11.3 Answer:
  1. make gazelle (i.e. bazel run //:gazelle) — Gazelle re-reads the Go imports and generates a BUILD.bazel for the new go-services/pkg/ratelimit package and adds //go-services/pkg/ratelimit to the deps of cmd/gateway:gateway_lib. (If the new package pulled in an external module, you would also run bazelisk mod tidy to update MODULE.bazel's use_repo list.)
  2. make build or bazel build //go-services/cmd/... — recompiles the gateway binary against the new library using the hermetic Go SDK.
  3. make images (or bazel run --config=linux_amd64 //go-services/images:gateway_load) — re-tars the rebuilt binary (pkg_tar), rebuilds the oci_image on the distroless base, and loads it into Docker as rmhstudios-go-gateway:dev.
The key point: BUILD files are generated by step 1, not edited by hand.
11.4 Answer: For a service named X, go_service_image generates: X_image — the oci_image itself (binary tar layered on the base, entrypoint /usr/local/bin/X); X_load — an oci_load that loads the image into the local Docker daemon with a :dev repo tag; and X_push — an oci_push that pushes to a registry. For rmhbox specifically: local tag rmhstudios-go-rmhbox:dev, default push repository ghcr.io/rmhstudios/rmhstudios-go-rmhbox (overridable at runtime with -- --repository=… --tag=…).
11.5 Answer: @distroless_base (gcr.io/distroless/static-debian12) contains no shell, package manager, or general userland, so it cannot supply the extra runtime these two need. vibe-worker needs headless Chromium and uses @base_chromium (the macro also sets PLAYWRIGHT_CHROMIUM_EXECUTABLE_PATH=/usr/bin/chromium-browser in the image env); discord-bot needs git/ssh and uses @base_git. Unlike distroless_base (pinned by digest), those bases are pulled by tag = "latest" from GHCR and are built/pushed separately by make base-images. So make base-images must run first, otherwise make images for those two services has no base to pull.
11.6 Answer: Benefits (any two): (1) no BUILD maintenance — adding frontend files needs no edits to any BUILD.bazel; (2) matches the JS toolchain — Vite/pnpm/esbuild handle their own module graph and native deps, which Bazel would model awkwardly; (3) simplicity — one sh_binary target instead of thousands. Costs (any two): (1) no hermeticity/reproducibility — it reuses host node_modules and runs in the source tree; (2) coarse incrementality — Bazel sees the whole frontend as one opaque action and cannot cache sub-parts; (3) weaker dependency tracking — Bazel cannot reason about what changed inside the leaf. Decision: a defensible answer keeps the leaf because the frontend is a terminal node with no Bazel consumers and the JS native-dependency reality makes full hermeticity expensive; if reproducible frontend builds became mandatory, you would move toward a Bazel JS ruleset (or build the frontend inside a pinned container) — accept the extra BUILD maintenance in exchange for byte-stable output. Either conclusion is acceptable if the trade-offs are argued.
11.7 Answer: (a) The SSR web app is built by Vite (with the TanStack Start and Nitro plugins), output to .output/, invoked by the build:frontend script (build-vibe-packages then vite build) — which is what scripts/frontend-build.sh runs. (b) server/rmhbox/index.ts is built by esbuild, output to dist-server/ (as dist-server/server/rmhbox/index.cjs), invoked by the build script's explicit pnpm exec esbuild … stage (--bundle --platform=node --format=cjs). The standalone WebSocket/worker servers are deliberately not built by Vite.
11.8 Answer: A correct graph (linear with the base attaching at the image node): 
main.go
  --> go_library (gateway_lib)
  --> go_binary  (gateway)
  --> pkg_tar    (gateway_layer, into /usr/local/bin)
  --> oci_image  (gateway_image)  <-- @distroless_base
  --> oci_push   (gateway_push -> ghcr.io/rmhstudios/rmhstudios-go-gateway)
The distroless base is an input to oci_image only (alongside the tar layer); go_binary embeds the go_library; an oci_load branch off oci_image is the local equivalent of oci_push. This matches Figure 11.2.
11.9 Answer: --incompatible_strict_action_env scrubs/normalizes the environment passed to build actions so a build does not inherit arbitrary host variables; this removes a hidden, machine-specific input and so improves reproducibility. The two --platforms config groups (build:linux_amd64, build:linux_arm64) let the same Go source be deterministically cross-compiled for a chosen target architecture independent of the developer's CPU, keeping the binary, base selection, and image arch consistent. A plausible bug if strict action env were removed: a tool reading something like GOFLAGS, locale, or a stray PATH entry from the developer's shell could change generated output, so two laptops produce different binaries from identical source — the classic non-reproducible "works on my machine" divergence, and the disk cache could even serve mismatched results.
11.10 Answer: The dev inner loop optimizes for fast feedback, not reproducibility: pnpm run dev launches Vite's HMR dev server plus the watch-mode Node servers via concurrently, giving sub-second reloads. Routing that through Bazel's hermetic sandbox would add overhead and lose HMR, so the team keeps dev off the build path on purpose. The risk: dev and production build through different paths (Vite dev server vs. vite build + esbuild + the Bazel/Docker image flow), so a bug that only manifests in the production bundle/image — externalization, chunking, or a native-dep resolution difference — will not show up until make build/make images or CI. Mitigation is to run the production build path regularly, not only at release time.
