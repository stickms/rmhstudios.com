12.1 Answer: A zero-downtime deployment releases a new version with no interval in which the service is unavailable: the old version keeps answering requests until the new one is proven able to take over, then traffic is switched. The guarantee hotswap-web.sh provides is that if the new web container never becomes healthy, traffic is never moved and the old container keeps serving. The new color is brought up on a spare port and polled with curl for up to HEALTH_TIMEOUT (60s); if it exits or never returns a success, the script removes the new container (or leaves it for diagnosis), never writes the active-port file, and the old color stays live. A bad build therefore degrades to "old version stays up," never to an outage.
12.2 Answer: The "process manager" role is filled by Docker Compose (not PM2). The two docker-compose.yml settings that express it are (1) restart: unless-stopped — Compose/Docker restarts a crashed service automatically and brings it back after a reboot, and (2) the logging driver json-file with options: max-size: "10m", max-file: "3" — capping each service's logs the way a process manager would. In the migration target the same role is played by Kubernetes Deployments.
12.3 Answer: In order:
  1. GitHub sends a push event to /webhook, which Apache proxies to webhook-server.cjs on 127.0.0.1:7002.
  2. webhook-server.cjs verifies the X-Hub-Signature-256 HMAC against WEBHOOK_SECRET, maps refs/heads/main → production, answers GitHub immediately, and spawns deploy.sh production detached (the script path comes from DEPLOY_SCRIPT, defaulting to deploy.sh).
  3. deploy.sh takes the flock lock, does git fetch + reset --hard, (re-execs itself if it changed), re-asserts public/ permissions, runs dc build web, SHA-tags the image, runs the Prisma migration, and brings up all services except web with --scale web=0.
  4. It then calls deploy/hotswap-web.sh, which starts the new web container on the spare blue/green port, health-checks it, and writes the active port into /home/rmhstudios/rmhstudios-web-active.conf.
  5. A root-owned systemd path-unit watching that file (or apachectl graceful directly) reloads Apache, whose ProxyPass / http://localhost:${WEB_UPSTREAM_PORT}/ now points at the new port. Apache serves the new version.
12.4 Answer: Blue (7005) is live, so the target is green on 7015.
  1. Start green with docker compose run on 127.0.0.1:7015, overriding PORT=7015.
  2. Health check: curl http://127.0.0.1:7015/ in a loop up to 60s; bail early if the container exits.
  3. Once healthy, promote its restart policy to unless-stopped and write Define WEB_UPSTREAM_PORT 7015 into rmhstudios-web-active.conf; Apache reloads graceful (finishing in-flight requests on the old workers).
  4. Drain DRAIN_SECONDS (8s).
  5. Flip-verification: stop (not remove) the old blue container, then probe Apache on :80 with Host: rmhstudios.com. If it serves, the flip landed — remove blue. If not, restart blue, revert the port file, and abort.
Live user traffic moves at step 3, the instant Apache completes its graceful reload pointing at port 7015 (existing connections drain on blue; new requests go to green).
12.5 Answer: Docker Compose recreates a container in place: it stops the old one, then starts the new one, leaving a multi-second gap where nothing answers on the web port. For the realtime/worker services that gap is invisible (a websocket/worker just reconnects), but for the user-facing web port it produced a Cloudflare 520 (origin unreachable) on every request during the gap. By bringing the rest of the stack up with --scale web=0 and handing web to hotswap-web.sh, the old web container keeps serving until the new one is healthy and Apache has flipped — eliminating the gap and the 520.
12.6 Answer: A reversible fallback means the previous, working deploy mechanism (Compose + deploy.sh + docker-compose.yml) is kept fully in place during the k3s migration, so switching back is a tiny, well-understood operation. Cutover (per deploy/README.md): set DEPLOY_SCRIPT=/home/rmhstudios/rmhstudios.com/deploy/deploy-k8s.sh in the webhook's environment and restart the webhook — new pushes now run the k8s deploy. Rollback: unset DEPLOY_SCRIPT (it defaults back to deploy.sh), restart the webhook, and run docker compose -p rmhstudios-prod --env-file .env.production up -d to bring the Compose stack back. Both directions hinge on the single DEPLOY_SCRIPT env var that webhook-server.cjs reads, which is why the script path is an env var rather than hard-coded.
12.7 Answer: The deploy would write Define WEB_UPSTREAM_PORT <new> into the home file (ACTIVE_CONF), but Apache now includes only the /etc/apache2/ copy, which still holds the old port. So the graceful reload never changes the upstream — the flip never lands. The hotswap then drains and runs its flip-verification: it stops the old container and probes Apache on :80. Apache is still routing to the now-stopped old port, so the probe fails. The safeguard is exactly this flip-verification + automatic rollback: on a failed probe the script starts the old container again, reverts the active-port file, reloads, and aborts the deploy. That converts "503 on every route until a human notices" into a bounded blip that self-heals back to the old version. (The deploy fails loudly, but the site stays up.)
12.8 Answer: resolve --applied 0_baseline marks the baseline migration as already applied, telling Prisma the existing tables are accounted for; without it a re-deploy onto a populated database would try to recreate tables and error. It is wrapped in || true, so it is idempotent — harmless on every run after the first. resolve --rolled-back <name> handles a migration that previously failed mid-apply: Prisma refuses to proceed while a failed migration is recorded, so marking it rolled-back lets migrate deploy retry it cleanly instead of leaving the DB stuck. Together they make the step safe to re-run and recoverable from a partial failure. The same logic appears in the k3s layer as the Helm pre-install,pre-upgrade migration hook in deploy/helm/rmhstudios/templates/migrate-job.yaml, which runs the identical reachability check, baseline-resolve, rolled-back-resolution, and migrate deploy.
12.9 Answer: Building on the VPS with no CI/registry is acceptable on a single node because (any two): (1) there is only one node, so the freshly built image is already on the machine that will run it — importing it into k3s's containerd with k3s ctr is enough; no other node needs to pull it, so a registry adds no value. (2) BuildKit cache mounts (pnpm store, Vinxi/TanStack cache) live on that same host, so incremental builds are fast and a separate build farm would lose that warm cache. (3) Fewer moving parts and no external CI service to secure, pay for, or depend on. Setting REGISTRY (e.g. registry.example.com or ghcr.io/owner) changes three things in deploy-k8s.sh: the image path becomes ${REGISTRY}/rmhstudios instead of the bare rmhstudios; the script docker pushes the image instead of importing it with k3s ctr; and the chart's pullPolicy flips from Never to IfNotPresent (with an optional REGISTRY_PULL_SECRET for a private registry) so every node pulls the same image.
12.10 Answer: A correct state diagram: 
[*] --> BlueLive
BlueLive    --> GreenStarting : deploy starts (run green on 7015)
GreenStarting --> GreenHealthy : health check passes (GET / 200)
GreenStarting --> BlueLive     : never healthy / container exits  (remove green)
GreenHealthy --> Flipped       : write active port + Apache graceful reload
Flipped      --> Verified      : stop old, probe :80 succeeds  --> remove old
Flipped      --> RolledBack    : verify probe fails (restart old, revert port)
RolledBack   --> BlueLive      : old container serving again
Verified     --> [*]           : zero-downtime swap complete (green is new live)
If the new container never becomes healthy, the system lands back in BlueLive from GreenStarting — traffic was never moved, so blue is still live and no rollback of the proxy is even needed. The only path that touches the active-port file is GreenHealthy → Flipped; a failure after that (verify probe fails) goes to RolledBack and then back to BlueLive.
