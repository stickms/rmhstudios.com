This is the Final, covering Chapters 8 through 13: the games platform, the apps and content systems, the Go microservices tier, the build toolchain, deployment and operations, and scaling. The exam has three parts — multiple choice, short answer, and diagram/design — for 100 points total over 90 minutes. Answer in your own words; where a question names a real file, port, script, or model, cite it. Throughout, favour the book's honest-scope reasoning: say plainly what is ported versus scaffolded, what is built versus planned, and which constraint a given mechanism actually relieves. Budget your time roughly in proportion to the points.

Part I — Multiple Choice (30 points)

Ten questions, 3 points each. Choose the single best answer.

B13 pts In the Slice-It engine, GameEngine.update() reads this.audioManager.getCurrentTime() rather than counting animation frames. What property does sourcing time from audio playback give the simulation?
  1. It lets the game run without a renderer.
  2. It keeps the simulation synchronized to the music so it cannot drift even if frames are dropped.
  3. It removes the need for a deterministic RNG.
  4. It makes the per-frame path perform network I/O.
B23 pts A determined player edits the browser JavaScript and POSTs a score of 480,000 — in range, at speed 1.0, from a real logged-in session — directly to app/routes/api/slice-it/score.ts. What does the chapter say happens?
  1. The rate limiter blocks it because forged scores are detected by IP.
  2. The 0 … 1_000_000_000 range check rejects it as impossible.
  3. It passes every gate, because the server never received the keystrokes and cannot re-validate a plausible value.
  4. The auth.api.getSession() call rejects it as unauthorized.
B33 pts RmhTube ships extrapolate/reanchor sync math, an NTP-lite clock handshake, and two-tier drift correction; RmhMusic has none of these and syncs with one inline expression. What single architectural fact explains the difference?
  1. RmhMusic runs on its own process and port, so it needs less code.
  2. RmhTube embeds the player itself and must correct drift, while RmhMusic lets an external Spotify player own the playhead.
  3. RmhMusic uses fan-out-on-write and RmhTube uses fan-out-on-read.
  4. RmhTube stores its media in MinIO and RmhMusic does not.
B43 pts The live RMHark feed assembles each home timeline by querying prisma.rMHark.findMany({ where: { userId: { in: followingIds }, ... } }). Which fan-out strategy is this, and what is its characteristic cost?
  1. Fan-out-on-write: cheap reads, expensive writes (the "whale" problem).
  2. Fan-out-on-read: cheap writes, expensive reads (a fresh query over the follow list per timeline).
  3. A hybrid that flags high-follower authors isHighFanout.
  4. Neither — timelines are materialized into a FeedEntry table on post.
B53 pts What makes the Go gateway the strangler-fig boundary of the migration?
  1. It re-implements the React SSR renderer in Go so the Node app can be deleted.
  2. It routes by longest-matching path prefix to Go services and sends everything else to the unchanged Node web tier, so services cut over one prefix at a time.
  3. It issues Better Auth sessions for every upstream service.
  4. It compiles all nine binaries into one container.
B63 pts The Go pkg/auth package authenticates users without the Better Auth SDK. How?
  1. It re-issues sessions by signing new cookies in Go.
  2. It calls back into the Node tier over HTTP for every request.
  3. It runs one SQL lookup against the shared session table — a verbatim port of the Node query — and only validates, never issues.
  4. It trusts the client-supplied X-Rmh-* identity headers.
B73 pts The runbook claims "Go is NOT required locally." Which mechanism makes that true?
  1. rules_go ships a hermetic Go 1.23 SDK that Bazel downloads and pins, so the build never touches a system go.
  2. The frontend leaf shells out to pnpm instead of Go.
  3. Gazelle generates the binaries directly from imports without compiling.
  4. The distroless base image contains a Go compiler.
B83 pts During a deploy, deploy.sh brings up every service with --scale web=0 and hands web to hotswap-web.sh. Why is web treated specially?
  1. The web container is the only one that needs database migrations.
  2. Compose recreates a container by stopping the old then starting the new, and that multi-second gap on the web port once produced a user-facing Cloudflare 520; the page must never go dark.
  3. The web container cannot be SHA-tagged for rollback.
  4. Apache cannot proxy to a Compose-managed container.
B93 pts If a newly built web container never passes its health check in hotswap-web.sh, what happens?
  1. Traffic is moved anyway and the site serves 503s until a human intervenes.
  2. Apache is flipped to the new port and then rolled back after draining.
  3. Traffic is never moved; the old container keeps serving, so a bad build degrades to "old version stays up."
  4. The deploy proceeds and Helm performs an --atomic rollback.
B103 pts Why does the production Helm chart pin every realtime service (socket, rmhbox, rmhtube) to replicas: 1?
  1. Postgres can only accept connections from one realtime replica at a time.
  2. They hold per-lobby state in memory, so without a shared backplane and sticky sessions, clients on different replicas cannot see each other.
  3. A CDN already caches their websocket responses, so more replicas are redundant.
  4. Kubernetes forbids running a stateful service on more than one node.
