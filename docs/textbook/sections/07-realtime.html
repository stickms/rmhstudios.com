  • Trace a player through the lobby lifecycle — join, ready-up, start, play, finish — and follow real events such as ndw:joinLobby and ndw:scoreUpdate.
  • Reason about in-memory per-lobby state, the host-reassignment rule, and what happens to that state when a process restarts.
  • Justify why the realtime layer is split into many standalone processes (isolation, independent scaling, blast-radius) and read the fleet's port table.
  • Identify the server-authoritative and anti-cheat patterns in the code: cooldowns, per-socket rate limits, and server-computed scores.
  • State the honest scaling limit: in-memory state caps each service at one replica without a backplane and sticky sessions (forward reference to Chapter 13).

A portfolio page can be served, cached, and forgotten — the server answers one request and moves on. A multiplayer game cannot work that way. When one player drifts into a corner on the Neon Driftway track, every other player must see it within a frame or two, and the server has to remember who is in the race even though nobody is currently asking it anything. This chapter is about the part of rmhstudios.com that holds that live, shared state: a fleet of standalone WebSocket processes, each owning one slice of the realtime surface. We read the real socket server, the RMHbox lobby service, and the shared helpers they all lean on, and we end on the single honest limitation that shapes the platform's scaling story.

Why request/response is not enough

The web's default conversation is HTTP request/response: the client asks, the server answers, the connection closes. That model is a superb fit for fetching a blog post. It is a poor fit for a game for two structural reasons.

First, it is client-initiated. The server can only speak when spoken to. If another player moves, the server has new information for you — but with plain HTTP it has no way to push it. The usual workaround is polling: the client re-asks every few hundred milliseconds. Polling wastes bandwidth, adds latency equal to the poll interval, and still cannot deliver a 60-updates-per-second racing feed without melting the network.

Second, it is stateless. Each HTTP request is meant to stand alone; the server is encouraged to forget you the moment it replies. But a lobby is the opposite of stateless — it is a small, evolving world (who is here, who is ready, what level was chosen) that must persist between messages and be shared by everyone in it.

Realtime communication solves both problems by keeping the connection open and making it two-way.

WebSocket — a protocol that upgrades a single HTTP connection into a persistent, full-duplex channel. After the upgrade handshake, either side may send a message at any time over the same socket, with almost no per-message overhead. This is what lets a server push to a client.
Realtime / bidirectional communication — a model in which client and server hold a long-lived connection and exchange small messages in both directions as events happen, rather than the client driving every exchange. Latency is bounded by the network round trip, not by a polling interval.

Socket.io: rooms, namespaces, and the emit/on protocol

Raw WebSockets give you a byte pipe and little else — no reconnection, no fallback when a corporate proxy blocks the upgrade, no notion of "everyone in lobby ABC123." rmhstudios.com uses Socket.io, a library that wraps WebSockets (falling back to HTTP long-polling when needed) and adds the structure a multiplayer server actually needs. Every realtime process in the fleet constructs a Socket.io Server over a bare Node http server:

const httpServer = createServer(requestHandler);

const io = new Server(httpServer, {
  path: config.SOCKET_PATH,            // '/socket/'
  cors: {
    origin: config.CORS_ORIGIN.split(','),
    methods: ['GET', 'POST'],
    credentials: true,
  },
  maxHttpBufferSize: config.MAX_HTTP_BUFFER_SIZE,
  pingInterval: config.PING_INTERVAL_MS,   // 25_000
  pingTimeout: config.PING_TIMEOUT_MS,     // 20_000
});

Two ideas in that snippet recur throughout the chapter. The path (/socket/ for the main server, /rmhbox-ws/ for RMHbox) is the URL prefix the reverse proxy uses to route a WebSocket upgrade to the right process. The cors.origin comes from SOCKET_CORS_ORIGIN — in development it is the web tier at http://localhost:7005 — and the client connects using VITE_SOCKET_URL (http://localhost:7001). Those two environment variables are the contract between the browser and the fleet.

Event-driven protocol (emit / on) — instead of calling URLs, a Socket.io client and server exchange named events with JSON payloads. One side emits an event; the other registers a handler with on. The event name (e.g. 'ndw:joinLobby') is the verb; the payload is the data. The whole multiplayer API is a vocabulary of such events.
Room — a server-side label that groups a set of connected sockets. A socket joins a room; the server can then io.to(room).emit(...) to broadcast to exactly that group. Rooms are how one process hosts thousands of independent lobbies without messages leaking between them. In the code, a Neon Driftway lobby roomId becomes the room ndw:${roomId}.
Namespace — a higher-level Socket.io partition of a single connection (e.g. /admin). rmhstudios.com generally does not split by namespace; RMHbox's auth middleware comment notes it is "applied server-wide — no namespace." Isolation between games comes instead from event-name prefixes (ndw:, rmhtype:, bj:) and from rooms.

When the platform splits its many games inside one process, it does so by prefixing every event. The main socket server's index.ts simply registers each game's handlers against the same socket on connection:

io.on('connection', (socket) => {
  // Register all game handlers
  registerSliceItHandlers(io, socket);
  registerNeonDriftwayHandlers(io, socket);
  registerSynapseStormHandlers(io, socket);
  registerRmhTypeHandlers(io, socket);
  registerRmhStudyHandlers(io, socket);
  registerAltairHandlers(io, socket);
  registerKowloonKnockoutHandlers(io, socket);
  registerRmhMusicHandlers(io, socket);
  registerBlackjackHandlers(io, socket);
  registerHoldemHandlers(io, socket);
  registerBaccaratHandlers(io, socket);
  registerRouletteHandlers(io, socket);
  registerLightsOutHandlers(io, socket);
  registerDoctrineHandlers(io, socket);
  // ...
});

Each register*Handlers call attaches that game's socket.on('prefix:...') listeners. Because the prefixes are disjoint, a blackjack event never reaches the neon-driftway handler even though both live in the same Node process and the same connection.

The fleet: eleven standalone processes

The single most important architectural fact about the realtime layer is that it is not one server. The server/ directory holds eleven subdirectories, and the package.json scripts and docker-compose.yml services confirm that most of them run as separate operating-system processes. Each compose service runs one command of the form node dist-server/server/<name>/index.cjs; in development each has its own tsx watch script.

Worker process — a long-lived process with no client-facing socket. It wakes on a timer (or a queue) to do background work — generating puzzles, rendering thumbnails, posting as bots — and shares the database with the rest of the platform but serves no requests.

Why split, and not one big server?

Running eight-plus processes is more moving parts than one. The platform accepts that cost for three concrete payoffs.

  • Isolation / blast-radius. If the RMHbox party server hits an unhandled exception and crashes, the racing game on 7001 and the video rooms on 7003 keep running. A bug's blast radius is one service, not the whole realtime surface. Docker's restart: unless-stopped then brings just that container back.
  • Independent scaling. A typing-race surge and a watch-party surge are different load shapes. Separate processes can (in principle) be given different CPU, memory, and replica counts. A worker that renders Chromium thumbnails should never compete for the event loop that is broadcasting a 60 Hz race.
  • Independent deployment and lifecycle. The Discord bot needs a 90-second stop grace so in-flight AI edits finish; the vibe worker needs 30 seconds for screenshot captures; a socket server can drain in 10. Separate services let each declare its own shutdown contract, which we see directly in their shutdown() handlers.

This is the same process-split philosophy introduced in Chapter 1, applied to the realtime tier specifically. The realtime fleet is the place where it pays off most, because realtime state is the hardest to move and the most sensitive to a crash.

Anatomy of an event-driven handler

Let us read one real game end-to-end. Neon Driftway (event prefix ndw:) is a six-player racing game. Its entire shared state lives in a single module-level map, and a lobby is just a struct:

interface NDWLobby {
  id: string; hostId: string; players: Map<string, NDWPlayer>;
  status: 'WAITING' | 'COUNTDOWN' | 'PLAYING' | 'FINISHED'; levelId: number;
}

const ndwLobbies = new Map<string, NDWLobby>();
const ABILITY_COOLDOWN_MS = 5000;
const MAX_NDW_PLAYERS = 6;

Joining a lobby creates it on first arrival, enforces capacity, records the player, and — crucially — calls socket.join so future broadcasts reach this client:

socket.on('ndw:joinLobby', (payload: { roomId?: string; playerName?: string }) => {
  const roomId = sanitizeLobbyId(payload?.roomId);
  const name = sanitizeUserName(payload?.playerName);
  let lobby = ndwLobbies.get(roomId);
  if (!lobby) {
    lobby = { id: roomId, hostId: socket.id, players: new Map(), status: 'WAITING', levelId: 1 };
    ndwLobbies.set(roomId, lobby);            // first joiner becomes host
  }
  if (lobby.players.size >= MAX_NDW_PLAYERS || lobby.status !== 'WAITING') return;
  lobby.players.set(socket.id, { id: socket.id, name, /* ...zeroed state... */ });
  socket.join(`ndw:${roomId}`);               // subscribe to this room's broadcasts
  broadcastLobby(io, roomId);
});

Notice what is not trusted. The client sends a playerName, which is run through sanitizeUserName; the roomId is run through sanitizeLobbyId. The first connection to a fresh room silently becomes the host (hostId: socket.id). And only the host can start the game — the start handler checks lobby.hostId !== socket.id and refuses otherwise. That check is the seed of server authority.

During play, each client reports its own position. The server stores it and fans it out to everyone except the sender using socket.to(room) (as opposed to io.to(room), which would echo back to the sender too):

socket.on('ndw:playerUpdate', (payload) => {
  const lobby = ndwLobbies.get(sanitizeLobbyId(payload?.roomId));
  if (!lobby || lobby.status !== 'PLAYING') return;   // only relay mid-race
  const player = lobby.players.get(socket.id);
  if (!player) return;
  if (typeof payload?.x === 'number') player.x = payload.x;
  // ...copy the other numeric fields, type-checking each...
  socket.to(`ndw:${roomId}`).emit('ndw:playerUpdate', {
    id: socket.id, name: player.name, x: player.x, /* ... */
  });
});
In-memory state — application state held in ordinary process memory (here, JavaScript Maps and objects) rather than in a database. It is fast and simple but volatile: it lives and dies with the process and is invisible to any other process.

In-memory state, the host, and reassignment

Because a lobby is a plain object in a Map, the lifecycle of that object is the lifecycle of the game session. The status field walks through WAITING → COUNTDOWN → PLAYING → FINISHED, driven entirely by events and timers inside the one process.

The host is the player whose socket.id equals lobby.hostId — the privileged client that may start the game. But the host can leave or disconnect mid-session, and the lobby must survive. The platform's answer is host reassignment: when the host departs, the role passes to whoever is next in the player map.

export function handleNeonDriftwayDisconnect(io: Server, socket: Socket): void {
  for (const [roomId, lobby] of ndwLobbies.entries()) {
    if (lobby.players.has(socket.id)) {
      lobby.players.delete(socket.id);
      socket.leave(`ndw:${roomId}`);
      if (lobby.players.size === 0) {
        ndwLobbies.delete(roomId);                       // empty lobby is destroyed
      } else {
        if (lobby.hostId === socket.id)
          lobby.hostId = lobby.players.keys().next().value || '';  // reassign host
        io.to(`ndw:${roomId}`).emit('ndw:playerDisconnected', { id: socket.id, reason: 'disconnect' });
        broadcastLobby(io, roomId);
      }
      break;
    }
  }
}

RMHbox implements the same idea with more ceremony, because it is a service-layer codebase rather than a single file. Its LobbyManager holds private readonly lobbies = new Map<string, RMHboxLobby>() plus index maps (userToLobby, channelToLobby) for O(1) lookup. On host departure it explicitly handles host succession — promoting the next-joined player, auto-readying them (the host "is always ready"), emitting a host-changed event, and posting a system chat line "Host transferred to …" — and if no players remain it can promote the earliest spectator instead. RMHbox also supports reconnection: its Socket.io server enables connectionStateRecovery with a grace period, and a dedicated ReconnectionHandler tries to re-seat a returning socket into its prior lobby.

The state is volatile. Every lobby in both services lives only in process memory. If the socket-server container is restarted — a deploy, a crash, an OOM kill — every ndwLobbies entry is gone, and so is every in-progress race. There is no database row for "lobby ABC123 is on lap 2." This is a deliberate trade (speed and simplicity for a transient game session) but it has a sharp consequence for scaling, which we reach below.

Server authority and anti-cheat

If clients are not trusted, the server must be the referee. rmhstudios.com applies several overlapping patterns, all visible in the code.

Server-authoritative — the principle that the server holds the canonical state and validates or recomputes anything that affects the outcome, treating client messages as requests rather than facts. Clients render; the server decides.

Server-computed scores. RMHType does not let a client report its words-per-minute. The client sends only raw progress — characters typed and error count — and the server computes the score itself from a timestamp it controls:

socket.on('rmhtype:game:progress', (payload?: { position?: number; errors?: number }) => {
  if (!checkRateLimit(socket.id, 'rmhtype:game:progress')) return;
  const room = rooms.get(socketRoomMap.get(socket.id));
  if (!room || room.state !== 'TYPING' || !room.roundStartTime) return;
  // ...
  const elapsedMs = Date.now() - room.roundStartTime;      // server's clock, not the client's
  if (elapsedMs > 0 && position > 0) {
    player.wpm = (position / 5) / (elapsedMs / 60000);
    player.accuracy = ((position - errors) / position) * 100;
  }
});

A cheater cannot simply announce "I typed at 400 WPM," because WPM is derived on the server from elapsed time the server measured. The same handler also refuses progress unless the room is genuinely in the TYPING state and the round has actually started.

Cooldowns. Abilities are time-gated on the server. Neon Driftway's slowdown power-up checks both a charge count and a wall-clock cooldown before it will fire, so spamming the event achieves nothing:

if (!player || player.abilityCharges <= 0) return;
const now = Date.now();
if (now - player.lastAbilityTime < ABILITY_COOLDOWN_MS) return;   // 5000ms gate
player.abilityCharges--;
player.lastAbilityTime = now;

Per-socket rate limits. Underneath everything is a shared sliding-window limiter (server/shared/rate-limit.ts) configured per event in config.SOCKET_RATE_LIMITS. Lobby creation is capped at 3/min, chat at 30/min, while high-frequency events get generous ceilings (Altair's altair:game:input allows 1200/min). The limiter keys on socketId:eventName, garbage-collects expired windows, and caps total entries at MAX_ENTRIES = 50_000 so an attacker cannot exhaust memory by flooding distinct events:

check(socketId: string, eventName: string): boolean {
  const limit = rules[eventName];
  if (!limit) return true;                       // unlisted events are unlimited
  const key = `${socketId}:${eventName}`;
  const entry = counters.get(key);
  if (!entry || now - entry.windowStart >= limit.windowMs) {
    counters.set(key, { count: 1, windowStart: now });
    return true;                                 // new window
  }
  if (entry.count < limit.max) { entry.count++; return true; }
  return false;                                  // over the limit — drop the event
}

Soft authentication. The main server's softAuthMiddleware runs on every connection. If the client supplies a session token in the handshake, the server validates it directly against the shared Postgres session table (joining to user) and, if unexpired, attaches userId/userName to socket.data. If there is no token, the connection is still allowed — anonymously — so legacy games keep working. RMHbox, by contrast, applies a stricter authMiddleware server-wide.

Why query the session table directly? A realtime process is not the web tier and does not run Better Auth in-process. Validating tokens with one SQL query against the same database lets every fleet member authenticate without importing the auth framework — the decoupling that Chapters 8–9 explore and that makes the Go migration of Chapter 13 tractable.

The honest limit: one replica without a backplane

Everything that makes this design fast also caps how far it scales. Because each lobby lives in one process's memory, two replicas of the same service do not share lobbies. If a load balancer sent player A to replica 1 and player B to replica 2, they could "join" the same room code and never see each other — each replica has its own ndwLobbies map, and io.to(room).emit(...) on replica 1 reaches only sockets connected to replica 1.

Backplane (adapter) — a shared message bus (commonly a Redis pub/sub adapter for Socket.io) that forwards room broadcasts between server instances, so io.to(room).emit(...) on any replica reaches every member of that room across all replicas. Without it, broadcasts are confined to a single process.
Sticky session — a load-balancer policy that pins a given client to the same backend instance for the life of its connection. WebSocket upgrades and Socket.io's handshake require this even with a backplane, so a client's long-lived connection is not bounced between replicas.

So the realtime fleet today is, by construction, a set of single-replica services. Scaling any one of them past one instance requires two additions that are not present in the code we read: a Socket.io backplane (e.g. the Redis adapter) so broadcasts cross process boundaries, and sticky sessions at the proxy so each client stays pinned to its instance. Even then, the lobby's authoritative state would need a shared home (Redis or the database) rather than a private Map, or lobbies would have to be sharded so that all members of one lobby land on one instance.

Scope: scaled-out realtime is not yet built. As of the code in this chapter, no Redis adapter or sticky-session configuration is wired into the socket processes; each runs as exactly one container with restart: unless-stopped. The platform scales the realtime tier vertically (a bigger box) and by splitting concerns across services, not by replicating a single service. Horizontal replication of a stateful socket server — backplane, sticky routing, externalized lobby state — is the unsolved piece, and it is exactly the problem Chapter 13 takes up when it discusses Kubernetes and horizontal scaling.

Summary

  • HTTP request/response is client-initiated and stateless; multiplayer needs server push and persistent shared state, which WebSockets provide and Socket.io structures.
  • The realtime API is an event-driven protocol: named events with JSON payloads, emit on one side and on the other, scoped to rooms like ndw:${roomId}.
  • The realtime layer is a fleet of standalone processes — socket-server (7001), rmhbox (7676), rmhtube (7003), recap (7004), status (7008), plus the Discord bot and three workers — split for isolation, independent scaling, and small blast radius.
  • Lobby state is held in memory as JavaScript maps; the host is just a stored socket.id, and on departure the role is reassigned to the next player. Restarting the process destroys all live sessions.
  • Server authority shows up as server-computed scores (RMHType WPM), server-enforced cooldowns, per-socket rate limits, and host-only actions.
  • In-memory state caps each service at one replica; horizontal scaling needs a backplane and sticky sessions — the forward reference to Chapter 13.
