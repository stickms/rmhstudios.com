Learning Objectives

  • Explain why media (video and audio) and content (articles) push the platform past the games-and-realtime model of earlier chapters, and what new infrastructure each demands.
  • Distinguish object storage from a relational database, and trace a media-image upload from the browser through the server to a bucket and back.
  • Compare two realtime apps built on the same socket-fleet pattern — RmhTube (video) and RmhMusic (listening rooms) — and explain why their sync engines differ.
  • Describe the social feed as a fan-out problem and identify which fan-out strategy the live code actually uses.
  • Explain the content-as-code idea and read honestly how the blog/news/research/library systems are really implemented.
  • Read a scope note that separates a built app from a designed-but-unshipped one.

The previous chapters built a platform out of games and the realtime fleet that drives them. This chapter follows the system as it grows past that mould. A video platform and a collaborative listening room need somewhere to put bytes that are too big for a row in a table, and a way to keep many viewers in step on the same timeline. A social feed needs to decide, every time anyone posts, whose screens should light up — a question with no single right answer at scale. And the site's prose — devlogs, news, research, a PDF library — needs a pipeline that treats writing the way the rest of the repository treats code. Four subsystems, four different shapes; we take them in that order, and we are honest throughout about which ones are finished and which are still drawings.

Beyond games: media and content as new pressures

Up to now the platform's data has been small and structured: a leaderboard score, a room's membership, a chat line. All of it fits comfortably in PostgreSQL rows, which is exactly what Chapter 8 described. Media and content break that assumption from two directions.

Media is large and opaque. A video file or an uploaded image is megabytes of bytes that the database has no reason to parse, index, or join against. Storing such blobs in Postgres is possible but wasteful — it bloats backups, slows replication, and competes for the same connection pool that serves your queries. The standard answer is object storage: a separate service whose entire job is to hold named blobs cheaply and serve them back by key.

Content is structured but authored. A blog post is not a runtime object the way a game room is; it is a document a human writes once and the system serves many times. That invites a different discipline — version it, review it, build it — which the industry calls content-as-code. As we will see, rmhstudios.com reaches for that idea at authoring time but, in its current code, serves the result from the database rather than from files.

Object storage — a storage service that holds arbitrary blobs ("objects") addressed by a string key inside a named bucket, exposed over an HTTP API rather than a filesystem or SQL. It optimizes for cheap, durable, flat storage of large immutable files, not for querying their contents. The de-facto interface is Amazon's S3 API; compatible implementations (MinIO, Cloudflare R2) speak the same protocol.

Object storage: MinIO, S3, and the media bucket

The repository runs an S3-compatible store as its own container. In docker-compose.yml the minio service uses the official image and a companion one-shot, minio-init, that creates the bucket and exits:

minio:
  image: minio/minio:latest
  command: ["server", "/data", "--console-address", ":9001"]
  environment:
    - MINIO_ROOT_USER=${S3_ACCESS_KEY_ID:-rmhminio}
    - MINIO_ROOT_PASSWORD=${S3_SECRET_ACCESS_KEY:-rmhminio-dev-secret}
  ports:
    - "127.0.0.1:9000:9000"   # S3 API
    - "127.0.0.1:9001:9001"   # web console

minio-init:
  image: minio/mc:latest
  # mc mb --ignore-existing local/${S3_BUCKET:-rmh-media}

The configuration lives in .env.example under a comment that states the migration intent plainly — "Object storage (S3-compatible: MinIO now, Cloudflare R2 later)":

S3_ENDPOINT=http://minio:9000
S3_REGION=us-east-1
S3_ACCESS_KEY_ID=rmhminio
S3_SECRET_ACCESS_KEY=rmhminio-dev-secret
S3_BUCKET=rmh-media
S3_FORCE_PATH_STYLE=true
S3-compatible — speaking the same HTTP request/response protocol as Amazon S3, so that a client written against the S3 SDK works unchanged against a different backend. This is why the same code path can target MinIO in development and Cloudflare R2 in production by changing only S3_ENDPOINT and the keys.

The application's gateway to the bucket is a small server-only module, lib/storage/s3.server.ts, built on the official @aws-sdk/client-s3. It exports exactly the primitives an object store needs and nothing more: putObject(key, body, contentType), getObject(key), deleteObject(key), objectExists(key), and getBucket(). Keys are minted by lib/storage/keys.ts, which defines a single prefix, FEED_IMAGE_PREFIX = "rmharks/", and helpers such as feedImageKey and contentTypeForFilename.

Scope: object storage is wired, but narrow. The store is real and running, yet its only consumers in the code are the social feed's image upload and serve routes (the rmharks/ prefix in bucket rmh-media). RmhTube, RmhMusic, RmhType, RmhStudy, and RmhCode do not use it — as we will see, their media comes from external providers. There are also no presigned URLs, no multipart upload, and no HTTP Range streaming anywhere: a repo-wide search for presigned, getSignedUrl, and multipart returns nothing. Uploads and downloads both proxy the whole file through the Node server.

An upload's round trip

The feed's image endpoints show the pattern end to end. The upload route app/routes/api/rmharks/image.ts handles a POST: it requires a Better Auth session, rate-limits to 10 requests per minute, caps the request at 4 images of 5 MB each, and — crucially — verifies each file is really an image by inspecting its leading bytes (validateImageBuffer) rather than trusting the filename. Only then does it write:

const filename = `${session.user.id}-${Date.now()}-${rand}${ext}`;
await putObject(
  feedImageKey(filename),                 // "rmharks/" + filename
  buffer,
  contentTypeForFilename(filename),       // png | jpg | jpeg | webp | gif
);

Serving is the mirror image. app/routes/api/feed/image/$filename.ts handles a GET, fetches the bytes with getObject(feedImageKey(filename)), and returns them with a long immutable cache header so the browser and any CDN never re-ask for the same object:

return new Response(buffer, {
  headers: {
    "Content-Type": contentType,
    "Cache-Control": "public, max-age=31536000, immutable",
  },
});
Streaming — delivering media as a sequence of byte ranges the player can begin consuming before the whole file arrives, typically via HTTP Range requests or a manifest format (HLS .m3u8, DASH .mpd). The feed's image route does not stream; it returns each object in one response. True range-streaming would be the next step if the platform ever self-hosted large video.

RmhTube: a watch-together video platform

RmhTube is the most fully built of the apps. It is a watch-together platform: a host opens a room, others join by a six-character code, and everyone's player stays locked to the host's timeline. Critically, RmhTube stores no media of its own. Its detectMediaType helper (in lib/rmhtube/utils.ts) recognizes three external sources and refuses anything else:

// host youtube.com | youtu.be | m.youtube.com  -> 'youtube'
// host twitch.tv | *.twitch.tv                 -> 'twitch'
// pathname matching .mp4|.webm|.ogg|.m3u8|.mpd -> 'direct'
// else null

So the bytes live on YouTube, Twitch, or a direct URL; RmhTube only synchronizes playback state. That is the whole engineering problem, and it reuses the socket-fleet pattern from Chapter 7 exactly. RmhTube runs as its own standalone Node process — header comment: "Runs as a separate Node.js process on port 7003" — listening on Socket.io path /rmhtube-ws/. Its index.ts bootstraps the same service-layer shape we have seen before: an auth middleware, then a RoomManager, SyncEngine, MediaQueue, ChatHandler, and ReconnectionHandler, all sharing one io instance.

Authentication is the cross-tier trick from Chapter 7 made concrete. RmhTube does not call into the Better Auth library; it validates the session token by querying the shared Postgres tables directly, through a small pg.Pool:

SELECT s."userId", s."expiresAt", u."name", u."image"
FROM "session" s
JOIN "user" u ON u."id" = s."userId"
WHERE s."token" = $1
LIMIT 1

An empty or unknown token is rejected with AUTH_REQUIRED / AUTH_FAILED; an expired one with SESSION_EXPIRED. On success the middleware attaches socket.data.userId, userName, and avatarUrl for every downstream handler.

Two sync engines, one pattern

The heart of RmhTube is its SyncEngine, and it is worth dwelling on because it is the textbook example of host-authoritative realtime. The host's player is the source of truth. The server keeps a canonical videoState — { playing, currentTime, playbackRate, updatedAt } — and broadcasts it to non-host members on a two-second heartbeat (SYNC_HEARTBEAT_INTERVAL_MS = 2_000), while explicit host actions (play, pause, seek) broadcast immediately.

The subtle part is time. Because currentTime is captured at a server-clock instant updatedAt, finding the playhead "now" means extrapolating forward by elapsed wall-clock multiplied by the playback rate. That math lives in lib/rmhtube/sync-math.ts and is shared identically by client and server:

export function extrapolate(vs, serverNow) {
  if (!vs.playing) return vs.currentTime;
  const elapsedMs = Math.max(0, serverNow - vs.updatedAt);
  return vs.currentTime + (elapsedMs / 1000) * (vs.playbackRate || 1);
}

export function reanchor(vs, serverNow) {
  return { ...vs, currentTime: extrapolate(vs, serverNow), updatedAt: serverNow };
}

Extrapolating on the server matters for a reason that is easy to miss: when the host's browser tab is throttled or frozen, its state reports stall, but the heartbeat keeps emitting an advancing timeline so the room does not freeze with it. To compare two clocks fairly, RmhTube also ships an NTP-lite handshake — SYNC_PING/SYNC_PONG in lib/rmhtube/clock.ts — that estimates each client's offset from the lowest-round-trip sample, and a two-tier drift correction (gentle ±5% playback-rate nudge inside two seconds of drift, a hard seek beyond it).

Host-authoritative sync — a model in which one participant (the host or "leader") owns the canonical state and the server treats their reports as truth, broadcasting a single agreed timeline to everyone else. It avoids the impossible problem of merging conflicting playback states from many clients by designating one of them as correct.

RmhTube even separates two roles: hostUserId (owns the room, can kick and ban) and leaderUserId (drives playback). Most playback handlers gate on the leader, returning NOT_LEADER otherwise. The MediaQueue layers on the social machinery — add, reorder, vote-to-skip (passes above 50% of connected members), queue voting, Fisher–Yates shuffle, loop, and a fifty-item history ring — persisting queue items to the RmhTubeQueueItem Prisma model while keeping the live room in memory for speed.

RmhMusic: listening rooms on the shared socket fleet

RmhMusic is a listening room: the audio analogue of RmhTube, where a group hears the same track in step. It is a perfect contrast case because it reuses the same room/sync/queue/chat decomposition but makes two telling architectural choices differently.

Listening room — a realtime space in which a host controls playback of an audio queue and all members' players are kept synchronized to a shared position, so everyone hears the same moment of the same song together.

First difference: where it runs. RmhTube earns its own process and port; RmhMusic does not. There is no server/rmhmusic/index.ts. Instead its managers are registered as handlers inside the unified server/socket-server (port 7001, path /socket/) alongside the games. The wrapper server/socket-server/handlers/rmhmusic.ts builds the four managers as shared singletons and wires them only for authenticated sockets:

export function registerRmhMusicHandlers(io, socket) {
  if (!socket.data.userId) return;        // RmhMusic requires auth
  const { roomManager, syncEngine, queueManager, chatHandler } = ensureInit(io);
  roomManager.handleConnection(socket);
  syncEngine.handleConnection(socket);
  queueManager.handleConnection(socket);
  chatHandler.handleConnection(socket);
}

Second difference: how it syncs. RmhMusic's SyncEngine is dramatically simpler than RmhTube's. There is no sync-math.ts, no clock handshake, no drift tolerance, no playback-rate nudging. The entire extrapolation is one inline expression in a five-second heartbeat:

io.to(room.id).emit(S2C.SYNC_HEARTBEAT, {
  trackUri: room.playback.trackUri,
  positionMs: room.playback.positionMs + (Date.now() - room.playback.updatedAt),
  isPlaying: room.playback.isPlaying,
});

Why so much thinner? Because the actual playhead is owned by an external player — RmhMusic is Spotify-powered (its design doc describes the Web Playback SDK and per-user Premium), so the server only needs to rebroadcast an authoritative position and let each Spotify client hold the timeline. RmhTube embeds the player itself and must therefore correct drift by hand. The lesson is general: how much sync machinery you build depends on how much the underlying player gives you for free.

The managers, events (lib/rmhmusic/events.ts with the rmhmusic: prefix), zod schemas, routes under app/routes/rmhmusic/, and four Prisma models (RmhMusicRoom, RmhMusicRoomMember, RmhMusicQueueItem, RmhMusicChatMessage) all exist. Treat RmhMusic as a working feature on the shared fleet, not an independently deployed one.

The other three apps are thinner still. RmhType (typing races) and RmhStudy (flashcards) are real features that run as handlers on the shared socket-server; RmhCode is the least built — its route app/routes/rmhcode.tsx is essentially an <Outlet/>, and its RmhCodeToken model reads as an auth bridge to an external editor rather than an in-app one.

The social feed as a fan-out problem

The feed (branded RMHark in code; the post entity is the RMHark Prisma model, mapped to table rmheet) is where the platform meets the hardest distributed-systems question in the chapter. When someone posts, which followers' timelines must change? That is the fan-out problem, and it has two classic answers.

Fan-out — the act of propagating one event (a new post) to the many places that must reflect it (every follower's home timeline). The two strategies are fan-out-on-write (push the post into each follower's stored timeline the moment it is created) and fan-out-on-read (store the post once and assemble each timeline on demand by querying the authors a viewer follows).

Fan-out-on-write (push) does the work at post time: it writes a copy, or a reference, into a materialized timeline per follower, so reads are a cheap scan of one pre-built list. It is fast to read and brutal to write when an author has millions of followers (the "whale" problem). Fan-out-on-read (pull) does the opposite: it stores the post once and, at read time, gathers posts from everyone the viewer follows. It is cheap to write and expensive to read, because each timeline is a fresh query over a potentially huge follow list.

The live RMHark code is pure fan-out-on-read. The unified entry point is getTimeline() in lib/feed/timeline.ts, which dispatches on a surface of "following" or "foryou". The following timeline loads the viewer's follow graph and then queries posts with an unbounded IN clause:

// getFollowingTimeline (paraphrased)
const follows = await prisma.follow.findMany({ where: { followerId: userId } });
const followingIds = follows.map(f => f.followingId);

const posts = await prisma.rMHark.findMany({
  where: { userId: { in: followingIds }, deletedAt: null,
           audience: { not: "PRIVATE" }, communityId: null, ...keysetWhere(cursor) },
  orderBy: [{ createdAt: "desc" }, { id: "desc" }],
  take: limit,
});

That userId: { in: followingIds } is fan-out-on-read: no per-follower timeline is ever written; every home feed is computed from scratch. The companion RMHarkRepost query runs in parallel and the two streams are merged, deduplicated, and sliced.

Two engineering details make this pull model workable. First, keyset pagination. Rather than offset-based paging (which drifts as new rows arrive), lib/feed/cursor.ts encodes a composite cursor of (createdAt, id) and emits a WHERE that walks strictly past the last row seen:

// keysetWhere(cursor) for (createdAt desc, id desc)
OR: [
  { createdAt: { lt: cursor.createdAt } },
  { createdAt: cursor.createdAt, id: { lt: cursor.id } },
]

This is backed by a real index on the model: @@index([createdAt(sort: Desc), id(sort: Desc)]). Second, denormalized counts. Engagement totals are stored as columns on RMHark — likeCount, commentCount, repostCount, viewCount — and maintained atomically. A like, for instance, toggles the join row and bumps the counter in one transaction, so the feed never has to COUNT(*) at read time:

await prisma.$transaction([
  prisma.rMHarkLike.create({ data: { rmheetId, userId } }),
  prisma.rMHark.update({ where: { id: rmheetId },
    data: { likeCount: { increment: 1 } } }),
]);

Drift in those counters is tolerated and repaired out of band by a reconciliation cron, scripts/reconcile-feed-counts.ts (run as pnpm feed:reconcile-counts), which re-derives all four columns with one set-based UPDATE. New posts reach open clients through a server-sent-events bus (feedEventBus in lib/feed-sse.ts over app/routes/api/feed/stream.ts): the create route publishes a rmhark.created event, the stream tags it per viewer as followed or own, and the client either prepends it or counts it behind an "N new posts" pill.

Scope: the push path is designed, not built. docs/feed/plan.md lays out a future hybrid — a materialized FeedEntry timeline written on post (fan-out-on-write) for normal authors, with high-follower "whale" authors flagged isHighFanout and merged at read time. None of that exists in the schema or code today: there is no FeedEntry model, no isHighFanout flag, and no write fan-out — only the SSE publish. The live system is entirely pull-based. (One thing the plan understates: the SSE bus already has a Redis backplane behind REDIS_URL, dormant only because that variable is unset.) A light ranking pass (lib/feed/ranking.ts) reorders the "For You" surface within a page without moving the chronological cursor.

Content as code: blog, news, research, library

The site's prose comes in four flavours — blog, news, research, and a PDF library — and the tidy story you might expect is "one MDX pipeline compiles Markdown-with-components into pages." The reality, read honestly from the code, is more interesting: it is four separate, heterogeneous systems, and MDX survives only as an authoring format at build time.

MDX — Markdown extended so that documents may embed and render components (interactive widgets, custom layouts) inline. It lets prose live as source files that compile to UI, blurring the line between a document and a program.
Content-as-code — treating editorial content as source artifacts that flow through the same version control, review, and build tooling as program code, rather than living only in a CMS database edited through a web form.

Blog and news are the closest to content-as-code, but with a twist. The .mdx source files were authored under content/blog and content/news and parsed offline with gray-matter by scripts such as scripts/generate-blog-data.ts, which upsert them into Postgres. The runtime, however, reads from the database, not from files: lib/blog.ts opens with import { prisma } from "./prisma.server" and exposes getPostSlugs(), getPostBySlug(), and getAllPosts() over the BlogPost model (table blog_post). Its Post interface is plain:

export interface Post {
  slug: string; title: string; date: string; description: string;
  image?: string; tags?: string[]; content: string;
}

The route app/routes/blog.$slug.tsx uses a TanStack Start server function to load a post by slug and renders the body with react-markdown — not an MDX runtime — through an animatedComponents map. news.$slug.tsx is the same shape over the NewsArticle model, with extra source-attribution fields.

Research and library abandon the pattern entirely. Research articles are a hardcoded TypeScript array in lib/research.ts (slugs like quantum-nash-equilibrium); research.$slug.tsx renders a PaperViewer of metadata and an abstract, with no Markdown body. The library is PDFs in public/library/ described by data/library-metadata.json; library.$slug.tsx loads a book synchronously and renders a BookReader. There is no shared content helper across the four — only blog and news happen to share the react-markdown dependency.

Scope: the chapter's tidy premise is wrong, on purpose. It is tempting to teach "one MDX content pipeline." The real code is four systems: blog and news are DB-backed and rendered with react-markdown (the .mdx files were even removed from the tree and are recovered from git history by the seed scripts); research is a static array; library is PDFs. No next-mdx-remote, @mdx-js, or mdx-bundler appears anywhere. The content-as-code idea is honored at authoring time and discarded at serving time.

Summary

  • Media and content push past the games model: large opaque blobs want object storage; authored documents invite content-as-code.
  • The platform runs an S3-compatible store (MinIO → future R2) in bucket rmh-media via lib/storage/s3.server.ts, but currently only for feed image uploads — no presigned URLs, no range streaming.
  • RmhTube is a fully built watch-together video app on its own process (port 7003) with a sophisticated host-authoritative SyncEngine (extrapolate/reanchor + NTP-lite + drift correction); it stores no media, only syncs external players.
  • RmhMusic reuses the same room/sync/queue/chat decomposition but runs as handlers on the shared socket-server and syncs far more simply, because Spotify owns the playhead. Its tables exist but only RmhMusicRoom is persisted.
  • The RMHark feed is a fan-out problem; the live code uses fan-out-on-read with keyset pagination, denormalized counters, and SSE for live arrivals. The push path is designed in docs/feed/plan.md but unbuilt.
  • The four content systems are heterogeneous: blog/news are DB-backed Markdown, research is a static array, library is PDFs. MDX is an authoring-time format only.
