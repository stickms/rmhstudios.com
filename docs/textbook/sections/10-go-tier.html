Learning Objectives

  • Define a microservice and explain how splitting a process tier into independently deployable binaries differs from a monolith.
  • Explain the strangler-fig migration pattern and contrast it with a big-bang rewrite, using the rmhstudios Go tier as the worked example.
  • Justify why some components — the React SSR web tier and Better Auth session issuance — are deliberately not ported, and are integrated rather than re-implemented.
  • Describe the role of a shared foundation (the pkg/ library) and what it buys a fleet of nine services.
  • Read a service inventory and tell apart what is genuinely ported, scaffolded, and stubbed — and why every stub still compiles.
  • Explain why this entire tier is a branch-isolated experiment rather than a production cutover.

This chapter studies a migration that was, as a matter of recorded architecture decision, declined. PR #121 — the infrastructure direction that moved rmhstudios.com onto k3s and Helm — explicitly chose not to rewrite the backend in Go. And yet the repository contains go-services/: nine Go binaries that mirror the Node realtime and worker tier. The two facts are not in tension. They are the whole lesson. The Go tier is a disciplined, branch-isolated experiment built so that Go could be adopted per service, later, only if metrics prove a bottleneck — never as a wholesale replacement. Read this chapter as a case study in how to explore a rewrite without betting the platform on it.

What a microservice tier is

The earlier chapters described rmhstudios.com as a process-split platform: one React SSR web server plus a fleet of small, single-purpose processes for realtime games, watch parties, music rooms, scheduled jobs, and a Discord bot. Each of those processes is a microservice — and the Go tier is a second, parallel implementation of that same fleet, one binary per service.

Microservice — a small service that owns one slice of a system's behavior, runs as its own process (here, its own container), and communicates with the rest of the system only over well-defined transports (HTTP, WebSocket, a database, a message bus) rather than by sharing in-process memory. The opposite is a monolith, where all behavior lives in one process and modules call each other by direct function calls.

The structural rule of the Go tier is stated in go-services/FOUNDATION.md and is worth memorizing because the whole layout follows from it: “Every service is a binary under cmd/<service>/main.go with its private code under internal/<service>/.” One binary equals one container equals one deployable unit. The module path is github.com/rmhstudios/rmh-go, pinned to Go 1.23.

This division — many small replaceable processes behind a stable set of transports — is precisely what makes the next idea, incremental migration, mechanically possible.

The strangler-fig pattern

There are two ways to replace a working system. The reckless way is the big-bang rewrite: build the new system in full, then on one terrifying day switch every user from old to new. Big-bang rewrites are notorious for shipping late, losing behavior nobody documented, and offering no safe rollback — when the switch fails, the old system is already gone. The disciplined way is the strangler fig.

Strangler-fig migration — a pattern named after a vine that grows around a host tree, gradually taking over until the original can be removed. In software: place a stable interface (often a gateway or proxy) in front of the old system, build the replacement behind it one piece at a time, and route traffic to each new piece only once it is proven. The old and new systems run side by side; you can always route back. The host is “strangled” incrementally, never amputated.
Big-bang rewrite — replacing a system all at once, with a single cutover from old to new. High risk: no incremental validation, no partial rollback, and a long period during which neither system is shippable.

The rmhstudios Go tier is built for the strangler-fig path and not the big-bang one. The mechanism that makes it strangler-fig is a new service, the gateway, which becomes the single front door. The gateway reverse-proxies a handful of WebSocket path prefixes to the Go services and proxies everything else straight through to the still-running Node web tier. Because routing happens at exactly one boundary, services can be cut over one prefix at a time — or not at all.

The routing decision itself is small and tested. In internal/gateway/proxy.go the router selects an upstream by longest-matching path prefix, falling back to the web upstream:

routes := []struct {
    prefix string
    target string
}{
    {"/socket/", cfg.GamehubUpstream},
    {"/rmhbox-ws/", cfg.RmhboxUpstream},
    {"/rmhtube-ws/", cfg.RmhtubeUpstream},
    {"/rmhmusic-ws/", cfg.RmhmusicUpstream},
}
// ... sorted longest-prefix-first so "most specific wins" ...

func (r *Router) pick(path string) *upstream {
    for _, u := range r.prefixed {
        if hasPrefix(path, u.prefix) {
            return u
        }
    }
    return r.web // catch-all: SSR pages and any API Go does not own
}

Note what is absent: there is no list of “migrated” routes baked into the clients, no feature flag, no rewrite of the Node app. To strangle one more service you add one prefix to this slice. To roll back, you remove it and the catch-all carries the traffic. This is the strangler fig in four lines of Go.

A subtle detail makes WebSocket pass-through work at all. The gateway uses Go 1.23's httputil.NewSingleHostReverseProxy with its default director. The comment in newProxy explains why: Go’s reverse proxy detects a WebSocket upgrade request, switches to bidirectional-copy mode on a 101 Switching Protocols response, and manages the hop-by-hop headers itself. Stripping Connection/Upgrade manually would break it. The realtime hubs serve their exact prefixes (e.g. hub.ServeWS on /rmhbox-ws/), so the gateway proxies the full path through unchanged — no prefix stripping.

What is deliberately not portable

A migration earns trust by being precise about its non-goals. Two large components are kept in JavaScript on purpose, and the honesty of the whole tier rests on admitting this plainly.

The React SSR web tier stays JavaScript. The TanStack Start renderer and its roughly 130 API routes are not ported. There is no faithful way to re-implement a React server-side renderer in Go; it would mean rewriting the entire frontend. So the Go gateway proxies to it rather than replacing it. The SSR app is integrated, not faked.

Better Auth session issuance stays JavaScript. OAuth callbacks, cookie signing, and session creation remain owned by Better Auth in the React tier. There is no Go equivalent for that stack. Crucially, though, the Go services do not need one. The single most important insight of the whole migration — stated verbatim atop pkg/auth/auth.go — is that the Node services already authenticate without the Better Auth SDK, by running one SQL lookup against the shared session table. Go reproduces that lookup exactly:

// ValidateSession is the verbatim port of the Node query.
const q = `SELECT s."userId", s."expiresAt", u."name", u."image", u."isAdmin"
    FROM "session" s JOIN "user" u ON u."id" = s."userId"
    WHERE s."token" = $1 LIMIT 1`

err := v.pool.QueryRow(ctx, q, token).Scan(&userID, &expiresAt, &name, &image, &isAdmin)
if errors.Is(err, pgx.ErrNoRows) {
    return Identity{}, ErrUnauthenticated
}
if time.Now().After(expiresAt) {
    return Identity{}, ErrUnauthenticated
}

This is the dividing line that makes the migration tractable. Better Auth issues sessions (hard, framework-bound, kept in JS). Any service merely validates them (one query, trivially portable). The Go fleet only validates. The gateway turns a validated session into trusted X-Rmh-* identity headers for upstreams, and — at the trust boundary — strips any client-supplied X-Rmh-* header first so it cannot be forged.

Validating is not the same as interoperating. The browsers speak the Socket.IO v4 wire protocol; the Go realtime services speak a clean JSON-over-raw-WebSocket protocol with the same semantics but a different wire format. They are not interoperable. A production cutover of any WebSocket prefix needs a client-side adapter that is, per 24-risks.html, designed but not built. Until then the Go realtime services are exercised by new-protocol end-to-end clients, never by live browsers. This is the single largest limitation of the tier.

The shared foundation

Nine services could easily become nine snowflakes, each with its own logging style, config parsing, and database boilerplate. The Go tier avoids that with a shared foundation: the packages under pkg/, imported by every service.

Shared foundation (shared library) — a set of common packages that every service in a fleet imports for cross-cutting concerns (config, logging, database access, auth, HTTP server lifecycle, metrics, realtime transport). It makes the services consistent and lets a fix or feature land once for all of them. In go-services/ the rule is strict: service authors import pkg/ but must never edit it or go.mod — they write only under their own cmd/ and internal/ directories.

The foundation is the part the project documents as faithful and fully tested, race-detector and all. Its packages map cleanly onto the cross-cutting needs of a realtime fleet:

pkg/ packageResponsibility
configLoadCommon + typed GetString/GetInt/GetBool/GetDuration/GetCSV env readers.

dbpgxpool wrapper; WaitForReachable for workers; typed models over the shared schema.

eventsthe cross-instance pub/sub backplane (Local bus, or Redis when REDIS_URL is set).

Because the foundation exists, a service's main.go is almost entirely boilerplate: load config, open the database, build a validator, build a hub, register handlers, serve /health and /metrics, run until SIGINT/SIGTERM, drain. The events.Bus backplane is what lets the realtime services scale past one replica — an empty REDIS_URL gives an in-process bus; a Redis URL fans messages across instances. That scaling primitive, notably, is language-agnostic: it could be applied to the existing Node services too.

A hermetic build for nine binaries

Nine services need a build that is reproducible and does not depend on each developer having the right Go installed. The operator runbook (go-backend-and-bazel.md) describes two paths, both arranged so Go is not required locally.

Hermetic Go SDK — a toolchain the build system downloads and pins itself, so every build uses the exact same compiler version regardless of what is installed on the machine. Under Bazel, rules_go 0.50.1 ships a hermetic Go 1.23.0 SDK; the build never touches a system go.
Gateway / BFF (edge service) — a single service that sits at the network edge, in front of everything else, and is the one address clients connect to. It routes, enforces the auth trust boundary, and terminates WebSocket upgrades. “BFF” (backend-for-frontend) names the same idea when the edge is tailored to one frontend. The rmhstudios gateway on port 7005 is this edge.

The Makefile's no-Bazel path runs the official Go image in Docker so a contributor needs only Docker: make all expands to tidy → vet → test → build, each running go ... inside golang:1.23-alpine. The Bazel path (make gazelle / build / images) uses the hermetic SDK and a go_service_image macro that emits one oci_image per service, loaded locally as rmhstudios-go-<svc>:dev. Seven services build on a distroless base; vibe-worker uses a Chromium base (it renders thumbnails) and discord-bot a git base. Either way the deliverable is nine containers that slot into PR #121's Helm conventions, reusing its rmhstudios-secrets Secret.

Ported, scaffolded, stubbed — the honest inventory

This is the part to read most carefully, because it is where overstatement would do real harm. The tier mixes three honestly-different states, and the codebase's own scope document (03-scope.html) and risk document (24-risks.html) label each one.

  • Ported — a faithful re-implementation of the Node behavior, tested. Safe to reason about as equivalent (modulo the wire-protocol gap above).
  • Scaffolded — the structure, interfaces, and lifecycle are real and tested, but a representative slice of the behavior is implemented while the long tail is not. A framework with the first example filled in.
  • Stubbed — a placeholder that compiles behind a real interface and runs, but does nothing meaningful yet (marked // TODO(migration):). It exists so the surrounding system can be exercised and so finishing it is additive, not structural.

The discipline that ties these together is the FOUNDATION rule: code must compile with go build ./...; never leave broken code; where a piece is out of scope, leave a clearly-marked stub that still compiles. A stub is not a lie — it is an honestly-empty implementation of a real contract. The rmhbox minigame registry shows it: eight of nine games are a stubGame that auto-completes after one second with everyone ranked zero, while Rhyme Time is a full port.

func (g *stubGame) Start() {
    g.isRunning = true
    // TODO(migration): replace with the real handler. Auto-complete after 1s.
    g.setTimeout(func() { g.ctx.OnComplete(g.ComputeResults()) }, 1000)
}

Honest scope of this tier — read before believing any “done.” The Go fleet is a strangler-fig experiment on an isolated branch, not a production system. (1) The architecture decision was to decline a wholesale rewrite — PR #121 set Go as a Stage-4 option, to be adopted per service, only if metrics prove a bottleneck. (2) The React SSR web tier and Better Auth issuance are kept in JS by design and proxied/validated, not ported. (3) The realtime services are not wire-compatible with live browsers — they speak JSON-over-raw-WebSocket, not Socket.IO v4; the client adapter is designed but unbuilt, so no real browser has ever connected. (4) Several surfaces are scaffolded or stubbed: 8 of 9 rmhbox minigames are compiling stubs, non-KK gamehub games are an empty registry, and /rmhbot's typecheck and PR-creation tools are stubs. (5) Nothing is load-tested or measured — performance claims are structural, not empirical. The codebase's own recommendation: do not adopt wholesale, and not now. Its value is as a tested reference, a source of extracted-good-ideas (the single auth query, the events backplane, the atomic stat-merge), and an option to migrate one service in isolation and measure it — exactly the discipline PR #121 prescribed.

Why this counts as good engineering

It is tempting to read “a full rewrite that we won't deploy” as wasted work. The opposite is closer to the truth, and the reasoning is the real takeaway of the chapter. A big-bang rewrite that shipped would have been the failure: it would have endangered a working platform, discarded the irreplaceable React/Better Auth/Prisma core, and offered no rollback. What was built instead is additive (the web image is unchanged), branch-isolated (no production traffic), interface-honest (every stub compiles behind a real contract), and gateway-fronted (cutover, if it ever happens, is per-prefix and reversible). The risk that a rewrite normally carries — the cliff of a single switch — has been engineered out. That is what disciplined migration looks like: you may build the whole target to learn from it, precisely because you have made it impossible for that target to hurt you.

Summary

  • A microservice tier is many small single-purpose processes communicating over transports, not shared memory. The Go tier mirrors the Node fleet, one binary per service under cmd/, private code under internal/.
  • The strangler-fig pattern replaces a system incrementally behind a stable interface; the gateway (port 7005) is that interface, routing WebSocket prefixes to Go services and everything else to the unchanged Node SSR app. The opposite — a big-bang rewrite — was deliberately declined in PR #121.
  • Some components are not portable by design: the React SSR renderer and Better Auth issuance stay in JS and are integrated, not faked. Go only validates sessions — one SQL query against the shared session table.
  • The shared foundation (pkg/) gives nine services consistent config, logging, db, auth, http, ratelimit, telemetry, an events backplane, and a realtime framework; authors never edit it.
  • Honesty has three grades — ported, scaffolded, stubbed — and every stub still compiles behind a real interface. The tier is a branch-isolated, unmeasured experiment, adoptable later only per-service and only if metrics justify it.
