Every feature in the previous chapters eventually has to remember something: who is logged in, what your high score was, which posts you liked, how many coins you hold. All of that durable memory lives in one PostgreSQL database, reached through one type-safe contract: a Prisma schema of 142 models. This chapter reads that schema closely — not to memorize it, but to learn how a single relational data layer stays legible while a platform grows a hundred features around it.

What an ORM is, and what Prisma is doing

A relational database speaks SQL and stores rows in tables. Application code speaks objects: a User, a Song, a list of comments. Something has to translate between the two worlds, and that something is an object-relational mapper.

ORM (object-relational mapper) — a library that maps database tables to in-language data structures and turns method calls (prisma.user.findUnique(...)) into SQL, so you read and write rows as typed objects instead of hand-writing query strings.

Prisma is the ORM rmhstudios uses. Its source of truth is a single declarative file, prisma/schema.prisma. From that file Prisma generates a TypeScript client whose method names and argument shapes are derived from your models, so a typo in a field name fails at compile time rather than at 3 a.m. in production.

Schema — the prisma/schema.prisma file: a single declarative description of every model, field, relation, index, and enum. It is both the spec the database is migrated to match and the spec the generated client is built from.

The header of the schema tells you the two key choices. The datasource is Postgres, and the generator is configured to run through a JavaScript query engine rather than a native binary, because the database is reached through a connection-pool adapter (more on that below):

generator client {
  provider        = "prisma-client-js"
  // Skip engine binary — the pg adapter uses the JS-based query engine
  previewFeatures = ["relationJoins"]
}

datasource db {
  provider = "postgresql"
}

Model → table, field → column, relation → foreign key

The cleanest way to see the mapping is the authentication core. rmhstudios uses the Better Auth library, and its User model is annotated to map onto a table literally named user:

model User {
  id            String   @id @default(cuid())
  username        String?   @unique // Added for username login
  handle          String?   @unique // Unique user handle for @mentions and URLs
  email           String?   @unique
  emailVerified   Boolean?
  isAdmin         Boolean   @default(false)
  bannedUntil     DateTime?
  createdAt       DateTime  @default(now())
  updatedAt       DateTime  @updatedAt

  sessions Session[]
  accounts Account[]
  // ... ~90 more relation fields ...

  @@index([isBot])
  @@map("user")
}

Read the annotations as a vocabulary:

  • @id marks the primary key; @default(cuid()) generates a collision-resistant string id at insert time. (Some game models instead use @default(uuid()) or Postgres-native gen_random_uuid() — the schema mixes both.)
  • @unique on email, username, and handle becomes a unique index — the database itself refuses a duplicate, so a uniqueness bug can't slip through a race condition.
  • String? (the ?) maps to a nullable column; String to NOT NULL.
  • @@map("user") renames the table. The model is User in code but user in Postgres. Many feature tables go further: the social post model is RMHark in code but @@map("rmheet") in the database, a fossil of an earlier product name.
  • @@index([isBot]) declares a secondary index (covered later).

A relation is how Prisma models a foreign key. Session belongs to a User:

model Session {
  id        String   @id @default(cuid())
  userId    String
  token     String   @unique
  expiresAt DateTime
  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId])
  @@map("session")
}

The userId String field is the actual foreign-key column. The user User @relation(...) line is the virtual field that lets you write session.user in code; it stores nothing extra. onDelete: Cascade tells Postgres that deleting a user deletes their sessions automatically. On the other side, User declares sessions Session[] — the back-reference. Out of 223 @relation declarations in the schema, the vast majority radiate from User like this, which is exactly why User is ~90 relation fields long.

Better Auth's four tables. The auth domain is just User, Session, Account (one row per linked OAuth/password credential, with @@index([providerId, accountId])), and Verification (email/reset tokens). Everything else in the schema is application data hanging off User.id.

The per-game profile pattern, repeated 20+ times

The single biggest reason the schema has so many models is a deliberate, repeated pattern: each game gets its own profile/leaderboard model instead of one shared scores table. Compare two of them:

model AltairPlayer {
  id                String   @id @default(uuid())
  username          String   @unique
  bestTime          Float    @default(0)
  totalKills        Int      @default(0)
  totalXP           Int      @default(0)
  gamesPlayed       Int      @default(1)
  userId String? @unique
  user   User?   @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([bestTime(sort: Desc)], map: "idx_altair_best_time")
  @@index([totalKills(sort: Desc)], map: "idx_altair_kills")
  @@index([totalXP(sort: Desc)],    map: "idx_altair_xp")
}

model VegaPlayer {
  id           String   @id @default(uuid())
  username     String   @unique
  highestLoop  Int      @default(1)
  highestLevel Int      @default(1)
  userId String? @unique
  user   User?   @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([highestLoop(sort: Desc)],  map: "idx_vega_loop")
  @@index([highestLevel(sort: Desc)], map: "idx_vega_level")
}

Each game scores differently — Altair tracks bestTime and totalKills, Vega tracks highestLoop and highestLevel — so each gets exactly the typed columns and indexes it needs. The cost is repetition (20-odd near-identical models); the benefit is that a leaderboard query is a single indexed sort with no game-id filter, and a schema change to one game can never break another. Note userId String? @unique: the link to the auth user is optional (guests can have a leaderboard row) but at most one profile per user per game.

The music cluster shows the more relational style. Rather than store likes as a count on Song, each like is its own row in SongLike with a @@unique([songId, userId]) constraint — the database guarantees a user can like a song at most once. The same join-table pattern appears for ratings, plays, and scores.

Indexing: making reads cheap on purpose

Index — a secondary data structure (in Postgres, usually a B-tree) that lets the database find or order rows by a column without scanning the whole table. It speeds reads but adds write cost and storage, so you add one only where a query needs it.

With 168 @@index declarations, indexing is not an afterthought here; it is shaped to the read patterns. Three patterns recur:

1. Sorted leaderboard indexes. A leaderboard query is "top N by score, descending." Indexing the score column in descending order means Postgres can read the top of the index directly:

@@index([highScore(sort: Desc)], map: "idx_signal_forge_high_score")
@@index([songId, score(sort: Desc)])   // top scores per song

2. Composite keyset-pagination index on the feed. The social feed scrolls newest-first, and uses keyset (cursor) pagination instead of OFFSET. That requires an index on the exact sort tuple:

model RMHark {
  // ...
  @@index([userId])
  @@index([createdAt(sort: Desc)])
  // Composite index backing keyset pagination on (createdAt desc, id desc)
  @@index([createdAt(sort: Desc), id(sort: Desc)])
  @@index([userId, pinnedAt])
}

The composite (createdAt desc, id desc) index lets the feed ask "give me the next 20 posts older than this exact (timestamp, id) cursor" as a single index range scan — the cost stays flat no matter how deep you scroll, where OFFSET 100000 would force Postgres to count past 100,000 rows.

3. Unique composite keys as constraints. A @@unique is both an index and a correctness rule. The daily-puzzle leaderboard uses one row per user, per game mode, per day:

model DailyPuzzleScore {
  userId    String
  gameMode  String   // lights-out | alibi | spectrum | ...
  dateKey   String   // YYYY-MM-DD
  score     Int      @default(0)
  resultJson  Json?
  @@unique([userId, gameMode, dateKey])
  @@index([gameMode, dateKey, score(sort: Desc)])
  @@index([gameMode, dateKey, moves(sort: Asc)])
}

The @@unique([userId, gameMode, dateKey]) makes "one score per player per puzzle per day" a database invariant, and doubles as the index for an upsert. The two @@index lines back two different leaderboards: most games rank by score descending, but lights-out ranks by moves ascending (fewer is better).

Indexes are not free. Each index must be updated on every insert, update, and delete to the table, and it consumes storage. The discipline visible in this schema — indexes that match real query shapes, not "index everything" — is what keeps writes fast. A late migration named add_missing_indexes shows this is tuned over time, not guessed up front.

JSON columns: flexible blobs and their trade-off

JSON / JSONB column — a single column that stores an arbitrary JSON document. In Postgres the jsonb form is stored in a parsed binary layout that supports indexing and querying into the document. It trades relational structure for schema flexibility.

The schema has 38 Json fields, and they cluster around a clear use case: game state whose shape is the game's business, not the database's. A save file, a procedural run, an audio beatmap — these change whenever the game updates, and the data layer has no reason to know their internals.

model TempleOfJoySave {
  id        String   @id @default(cuid())
  userId    String   @unique
  saveData  Json
  updatedAt DateTime @default(now()) @updatedAt
  user User @relation(fields: [userId], references: [id], onDelete: Cascade)
  @@map("temple_of_joy_save")
}

model SignalForgePlayer {
  // ...indexed score columns...
  savedRunState Json?     // mid-run snapshot, shape owned by the game
}

model Song {
  // ...
  analysisData Json?      // beatmap / audio-analysis blob
}

Some models mix the two styles deliberately. VersecraftProgress keeps countable, queryable facts as typed columns and bundles open-ended lists as JSON with sane defaults:

model VersecraftProgress {
  completedChapters Json     @default("[]")
  unlockedEndings   Json     @default("[]")
  totalPoemsWritten Int      @default(0)   // typed: you might sort/aggregate it
  totalPlaytime     Int      @default(0)
}
The trade-off in one sentence. A JSON column lets a game evolve its save format with zero migrations, but the database can no longer enforce, index, or efficiently query inside the blob — so anything you need to sort a leaderboard by, filter on, or constrain should be a real typed column instead. The schema's rule of thumb: scores and identifiers are columns; opaque game state is JSON.

The migration workflow: push, migrate dev, migrate deploy

Migration — a recorded, ordered change to the database structure (a .sql file). The set of applied migrations is tracked in the database so the same schema can be reproduced exactly across dev, CI, and production.

Prisma offers two ways to make the database match the schema, and rmhstudios uses both at different times. They are exposed as db:* scripts in package.json:

"db:push":         "prisma db push --accept-data-loss",
"db:generate":     "prisma generate",
"db:migrate":      "prisma migrate dev",
"db:migrate:prod": "prisma migrate deploy",
"db:reset":        "prisma db push --force-reset",
"db:seed-jobs":    "pnpm exec tsx scripts/seed-jobs.ts",
  • db:push (prisma db push) shoves the current schema straight into the database without writing a migration file. Fast and disposable — good for local prototyping. The --accept-data-loss flag means it will drop columns to converge, so it is a development tool, not a production one.
  • db:migrate (prisma migrate dev) is the authoring command: it diffs the schema against the last migration, writes a new timestamped SQL file under prisma/migrations/, applies it, and regenerates the client.
  • db:migrate:prod (prisma migrate deploy) is the production command: it applies already-authored, version-controlled migrations in order and never invents new ones. This is what runs on deploy.
  • db:reset force-resets the schema (everything is dropped) — strictly local.
  • db:generate regenerates the typed client; it also runs automatically via the postinstall hook.

The prisma/migrations/ directory tells the platform's recent story directly. It starts with a squashed 0_baseline (an 1,883-line SQL file capturing the schema as it stood) and then a run of dated migrations — add_doctrine_engine, add_vibe_page, add_feed_counters_and_keyset, communities, coin_stake, clans, storefront, ai_personas — each one a feature landing. Config lives in prisma.config.ts, which points Prisma at the schema file and the DATABASE_URL:

export default defineConfig({
  schema: path.join('prisma', 'schema.prisma'),
  datasource: { url: process.env.DATABASE_URL! },
});
Scope: schema-side data scripts. Beyond migrations, scripts/ holds one-off TypeScript jobs run with tsx that touch the database directly via Prisma — e.g. assign-handles.ts (backfill user handles), backfill-vibe-thumbs.ts, reconcile-feed-counts.ts (repair the denormalized feed counters described below), and several seeders (seed-news.ts, seed-official-builds.ts). These are operational backfills, not part of the migration chain.

The singleton client and the connection pool

Connection pool — a fixed set of reusable database connections held open by the application. Requests borrow a connection, run a query, and return it. Postgres allows only so many connections, so the pool caps concurrency and avoids the cost of opening a new connection per query.

Every query needs a connection, and Postgres has a hard ceiling on how many it will accept. If each module that needs the database constructed its own PrismaClient — and in a dev server with hot-reload, did so on every reload — you would exhaust connections and crash. The fix is a singleton: build the client once and reuse it. The web app does this in lib/prisma.server.ts:

const globalForPrisma = global as unknown as { prisma: PrismaClient };

function createPrismaClient() {
  const connectionString = process.env.DATABASE_URL;
  if (!connectionString) throw new Error('DATABASE_URL ... required');

  // PrismaPg wraps a pg Pool — configure pool sizing for predictable behaviour under load.
  const adapter = new PrismaPg({
    connectionString,
    max: parseInt(process.env.DATABASE_POOL_SIZE || '10', 10),
    idleTimeoutMillis: 30_000,
    connectionTimeoutMillis: 10_000,
  });
  return new PrismaClient({ adapter, log: /* dev vs prod */ });
}

export const prisma = globalForPrisma.prisma || createPrismaClient();
if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = prisma;

Two things make this work. First, globalForPrisma.prisma || createPrismaClient() reuses an existing client if one is stashed on the global object; the last line stashes it only in non-production, which is precisely the dev hot-reload case where reloads would otherwise leak clients. Second, the PrismaPg adapter wraps a real Postgres connection pool with an explicit max (default 10), an idle timeout, and a connection timeout — so the app's concurrency against the database is bounded and predictable.

The standalone realtime servers (socket-server, rmhbox, rmhtube) get their own factory in server/shared/prisma-client.ts with a smaller default pool (SERVER_DB_POOL_SIZE, default 5), because there are several such processes and their pools add up against the same Postgres ceiling.

One Postgres, and that is the ceiling (forward-ref Ch. 13). Pooling protects the database from connection storms, but it cannot change the fact that there is exactly one Postgres instance behind every domain in this schema. Reads and writes for auth, the feed, leaderboards, the economy, and Doctrine all land on the same machine. That single-instance constraint — and the strategies for living with it, from the denormalized feed counters below to eventual read replicas — is the scaling story Chapter 13 takes up.

A glimpse of scaling pressure: denormalized counters

One concession to the single-Postgres constraint already lives in the schema. The feed could compute a post's like/comment counts by aggregating rows, but at feed scale that is expensive on every read. So RMHark caches them as plain integer columns, updated transactionally on write:

// Denormalized engagement counters (Phase 1: eventual consistency)
// Maintained transactionally by the like/comment/repost/view routes via
// atomic increment/decrement, so the feed read path never needs `_count`.
likeCount    Int @default(0)
commentCount Int @default(0)
repostCount  Int @default(0)
viewCount    Int @default(0)

The trade-off mirrors the JSON one: faster reads in exchange for the risk of drift between the counter and the true row count — which is exactly why scripts/reconcile-feed-counts.ts exists to repair it.

Summary

  • An ORM maps tables to typed objects; Prisma generates a TypeScript client from one schema.prisma file, mapping model→table, field→column, and @relation→foreign key.
  • The 142-model schema is organized into domains (auth, per-game, music, social, economy, content, moderation, Doctrine) by comment banners and naming prefixes; the largest is the deliberate per-game profile pattern, one model per game.
  • Indexes are shaped to queries: descending score columns for leaderboards, a composite (createdAt desc, id desc) index for keyset feed pagination, and @@unique composites that are both constraints and indexes.
  • JSON columns hold opaque game state (saves, run snapshots, beatmaps) for schema-free evolution, at the cost of in-database querying — so scores and identifiers stay typed columns.
  • Migrations split by environment: db push for local prototyping, migrate dev to author migrations, migrate deploy in production.
  • A singleton client over a sized connection pool prevents connection exhaustion; but every pool points at one Postgres instance — the scaling constraint of Chapter 13.
