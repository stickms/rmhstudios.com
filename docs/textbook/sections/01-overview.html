rmhstudios.com is a single TypeScript codebase that behaves like several systems at once: a marketing portfolio, a blog and research site, and a platform for in-browser games and apps with real-time multiplayer. This chapter is the aerial photograph. It names the tiers, shows how they talk, and explains the one design decision that shapes everything else — that the platform is not a single server but a web tier surrounded by a fleet of small, standalone processes. Later chapters land on each tier in turn; for now we stay high enough to see the whole field.

What rmhstudios.com is

From a user's seat, rmhstudios.com is three sites wearing one skin. There is a portfolio and blog — a home page, devlogs, and research/blog posts. There is a games library of original titles that run directly in the browser: rhythm games like Slice It!, the RMHBox multiplayer minigame lobby (Rhyme Time, Emoji Cinema, Undercover Editor, and more), 3D titles such as VELUM 2099 and Forest Explorer built on React Three Fiber and Rapier physics, and strategy/trivia games like Altair and Synapse Storm. And there is a set of apps: RMHTube (video), RMHMusic (collaborative listening rooms), RMHType (typing races), RMHStudy, and RMHCode. Tying it together are social features — feeds, messaging, leaderboards, profiles — and a single sign-on system, RMH Auth, built on Better Auth with Discord, GitHub, and Google login.

From an engineer's seat, all of that lives in one repository, written in TypeScript, rendered with TanStack Start on top of Vite, served at runtime by the Nitro server, and persisted in PostgreSQL through the Prisma ORM. The repository's schema currently declares 142 Prisma models — a useful single number for the breadth of the domain. The interesting structure is not any one of these features but how they are split across processes, which is the subject of the rest of this chapter.

Server-side rendering (SSR) — generating a page's HTML on the server for each request (rather than shipping an empty shell and building the page in the browser) so the client receives meaningful markup immediately. In this platform SSR is produced by TanStack Start and served by Nitro.
Process-split architecture — a design in which one logical application is run as several operating-system processes that can be started, crashed, restarted, and scaled independently, instead of one monolithic server doing everything in a single process.
Scope: a stale map exists. An older internal document, docs/codebase-overview.md, still describes the web tier as "Next.js 16" on port 7000. That is no longer accurate. The authoritative stack — per README.md and the current code — is TanStack Start + Vite + Nitro, with the web tier listening on port 7005 in production. Throughout this book we treat the README and the running code as ground truth and do not repeat the Next.js claim as present tense.

The process-split philosophy

A naïve full-stack app runs as one process: HTTP, WebSockets, background jobs, and rendering all share a single event loop. rmhstudios.com refuses that arrangement. Instead it runs one web tier plus many standalone processes, each with a single responsibility. The cleanest evidence is the dev script in package.json, which launches everything concurrently:

"dev": "concurrently \"vite dev\"
  \"pnpm run socket-server\"
  \"pnpm run rmhbox-server\"
  \"pnpm run rmhtube-server\"
  \"pnpm run discord-bot:dev\"
  \"pnpm run doctrine-worker:dev\"
  \"pnpm run vibe-worker:dev\"
  \"pnpm run bot-worker:dev\"",

That single line tells the whole story. Beyond the Vite dev server (the web tier), there are realtime servers (socket-server, rmhbox-server, rmhtube-server) and background workers (discord-bot, doctrine-worker, vibe-worker, bot-worker). Each is its own Node process under server/<name>/index.ts. The production build script confirms this: after the Vite/Nitro build, esbuild bundles each server entry point separately into dist-server/, and start runs each compiled .cjs file as its own node process.

Worker process — a standalone background process that does not serve user requests directly. It runs scheduled or event-driven work (generating content, decaying reputation, rendering thumbnails) and communicates with the rest of the system through the shared database rather than through HTTP.

Why bother? Three reasons, each a recurring theme in this book:

  • Fault isolation. If the thumbnail worker crashes mid-render, the home page keeps serving. A monolith would take everything down with it.
  • Independent scaling. A trivia night might need three copies of the realtime server but only one web tier. Splitting processes lets you scale exactly the part under load.
  • Independent deployment and runtime. A WebSocket server holds long-lived connections; a request-rendering server is stateless per request. Separating them lets each be restarted, tuned, and (as we will see) rewritten in another language without disturbing the others.

Workers such as discord-bot, doctrine-worker, vibe-worker, and bot-worker publish no host port. They take no inbound user traffic; they reach the world through the database and through outbound calls. That is exactly what makes them workers rather than servers.

The realtime ports are bound to 127.0.0.1 (loopback) — they are reverse-proxied, not exposed directly. The status service is interesting in its own right: it probes the public website URL (so it sees the same path a real user does, through Apache and DNS) and probes the internal services by their compose DNS names, then serves an HTML + JSON dashboard. The repository, top-down

The directory layout mirrors the tiers, which makes the codebase navigable once you know the map. The most load-bearing top-level directories:

app/         TanStack Start routes (file-based): _site, api, rmhbox, altair, ...
components/   React components by feature (594 files per README)
lib/          Shared logic, schemas, utilities (361 files per README)
server/       Standalone processes: socket-server, rmhbox, rmhtube,
              recap, status, discord-bot, doctrine-worker,
              vibe-worker, bot-worker, rmhmusic, shared
prisma/       schema.prisma (142 models) and migrations
stores/       Zustand client state
go-services/  Parallel Go rewrite of the realtime/worker tier (in progress)
deploy/       apache, docker, helm, k8s, terraform, hotswap-web.sh

Two things deserve a callout. First, server/shared/ holds the code every standalone process imports — prisma-client.ts, logger.ts, rate-limit.ts — so the fleet shares one database client and one logging convention rather than reinventing them per process. Second, go-services/ is not dead weight; it is a live migration.

Strangler-fig migration — a strategy for replacing a system gradually: new components grow up alongside the old ones and take over responsibilities one at a time, until the original can be removed. The name comes from the strangler fig, which grows around a host tree and eventually replaces it.
Scope: the Go rewrite is in progress. Under go-services/cmd/<service>/main.go there are Go ports of the realtime and worker processes — gateway, gamehub, rmhbox, rmhtube, rmhmusic, recap, discord-bot, doctrine-worker, vibe-worker — sharing a common pkg/ (config, logging, db) described in go-services/FOUNDATION.md. The platform's process-split shape is precisely what makes this possible: because each realtime server is already an independent process speaking to a shared database, a Go binary can replace a Node one service-by-service without touching the web tier. This rewrite is treated as in-progress throughout the book and is the subject of Chapter 13. The TypeScript fleet remains the production path.

A request, end to end

To see the tiers cooperate, follow a player opening a multiplayer game. The page itself comes from the web tier over HTTP; the live game comes from a realtime server over a WebSocket. Both consult the same database.

Notice that the web tier and the realtime tier never call each other directly in this flow. They coordinate through the database. That loose coupling is what lets the two tiers scale, fail, and be rewritten independently — the recurring payoff of the process split.

Summary

  • rmhstudios.com is one TypeScript codebase serving three faces — portfolio/blog, browser games, and apps — over a stack of TanStack Start + Vite + Nitro, PostgreSQL + Prisma (142 models), and Better Auth.
  • It is organized into five tiers: web/SSR, realtime fleet, data & auth, games & apps, and build & deploy.
  • The defining decision is the process split: one web tier plus many standalone Node processes — three realtime Socket.io servers (socket :7001, rmhbox :7676, rmhtube :7003) and several workers (discord-bot, doctrine-worker, vibe-worker, bot-worker, plus recap and status).
  • The browser holds two connection types at once: HTTP to the web tier and WebSockets to the realtime servers. Tiers coordinate through the shared database rather than calling each other.
  • The split buys fault isolation, independent scaling, and independent deployment — and it is exactly what enables the in-progress strangler-fig migration to Go under go-services/, one service at a time.
  • Watch for stale docs: codebase-overview.md still says "Next.js 16"; the README and running code (TanStack Start, port 7005) are authoritative.
