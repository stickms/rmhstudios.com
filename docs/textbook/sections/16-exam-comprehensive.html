This is the Comprehensive exam, spanning the whole book — the process split and its history, the SSR web tier and client state, the Prisma data layer and RMH Auth, the realtime fleet, the games and content apps, the strangler-fig Go tier, the build toolchain, deployment, and scaling. It is deliberately cross-cutting: the best answers trace a value or a request across subsystems rather than treating each chapter in isolation. The exam has three parts — multiple choice, short answer, and integration/design — for 120 points over 120 minutes. Answer in your own words; where a question names a real file, port, script, model, or event, cite it. Throughout, favour the book's honest-scope discipline: say plainly what is built versus planned, ported versus scaffolded, and which constraint a given mechanism actually relieves. Budget your time roughly in proportion to the points.

Four questions. These are meaty, cross-cutting problems. Draw clearly where asked (a labelled sketch or a text-described flow is fine), state your assumptions, and defend trade-offs against the constraints the book actually names.

Trace the score’s full journey across subsystems: (a) the live in-game updates — which socket process and port carry them, why those updates are throttled, and which HUD they drive; (b) the moment the game ends — the authoritative HTTP path the final score takes (name the route family), the order of gates it passes (rate limit, bound/shape, authenticate, persist), and how the server identifies which user’s row to write without trusting the client; (c) where the score finally rests — the single shared Postgres instance, and how the leaderboard read later finds it cheaply (name the indexing idea). (d) Explain why a CDN, which relieves load for the static surface, is useless for the websocket surface — tie your answer to what a CDN fundamentally caches.
