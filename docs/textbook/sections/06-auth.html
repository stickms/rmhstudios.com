Almost every feature in this book eventually asks the same question: who is making this request, and are they allowed to? A leaderboard write must know whose score it is. A subscription endpoint must refuse to let one user edit another's plan. Chapter 6 reads the one subsystem that answers that question for the entire platform — RMH Auth — built on the better-auth library and wired into the same Prisma database, the same TanStack Start router, and the same SSR pipeline you have already met. The remarkable thing is how little application code it takes: one server config in lib/auth.ts, one client in lib/auth-client.ts, and one catch-all route that hands every auth request to the library.

Authentication versus authorization

Two words that sound alike name two genuinely different jobs, and confusing them is the source of a great many security bugs.

Authentication (authn) — establishing who a principal is. "This request is from the user with id u_abc." It answers identity.
Authorization (authz) — deciding what an already-identified principal may do. "This user may write to their own leaderboard row, but not someone else's." It answers permission.

RMH Auth is overwhelmingly an authentication system: its purpose is to turn an anonymous HTTP request into a known User. Authorization lives after that, scattered across the routes that consume the identity. You can see the seam clearly in two places. In lib/auth.ts, the Stripe subscription plugin defines an authorization rule:

// lib/auth.ts — authorization, not authentication
authorizeReference: async ({ user, referenceId }) => {
  return referenceId === user.id;
},

The user has already been authenticated by the time this runs; the rule only decides whether that known user may manage the subscription whose reference id was passed. And in a game route, the same split appears: first authenticate, then authorize.

// app/routes/api/slice-it/score.ts
const session = await auth.api.getSession({ headers: request.headers });
if (!session?.user) {
  return Response.json({ error: 'Unauthorized.' }, { status: 401 });
}
const userId = session.user.id; // now we know WHO; the row we touch is theirs
The HTTP status code 401 Unauthorized is, by a historical naming accident, an authentication failure ("we don't know who you are"). A pure authorization failure ("we know you, but no") is conventionally 403 Forbidden. The route above returns 401 because a missing session means no identity at all.

The server: one config object

The whole server-side auth surface is a single call to betterAuth(...) exported from lib/auth.ts. Reading its shape tells you almost everything the platform supports.

Better Auth — the authentication library (better-auth) RMH uses. It owns password hashing, session issuance, cookie signing, OAuth flows, and rate limiting; the application supplies configuration and a database adapter.
// lib/auth.ts (abridged)
export const auth = betterAuth({
  baseURL: process.env.BETTER_AUTH_URL,
  database: prismaAdapter(prisma, { provider: "postgresql" }),
  trustedOrigins: [
    "https://rmhstudios.com",
    "https://www.rmhstudios.com",
    "https://staging.rmhstudios.com",
  ],
  advanced: {
    useSecureCookies: process.env.BETTER_AUTH_URL?.startsWith("https"),
  },
  socialProviders: { discord: {...}, google: {...}, github: {...} },
  emailAndPassword: { enabled: true },
  user: { additionalFields: { username, handle, isAdmin, isVerified } },
});

Four design decisions are worth naming. First, the database adapter: prismaAdapter(prisma, ...) tells the library to persist users, sessions, accounts, and verifications through the very same Prisma client the rest of the app uses (Chapter 5). There is no separate auth database. Second, email and password sign-in is enabled. Third, three social providers are configured. Fourth, the user object adds custom fields — username, handle, isAdmin, isVerified — onto the library's base user, so the application's own columns travel with the identity.

Custom / additional fields — application-specific columns (here handle, isAdmin) declared in the auth config so the library knows to read and write them on the User record alongside its built-in fields.

A small but instructive detail: a databaseHooks.user.create.after hook auto-assigns a handle to brand-new users via generateHandle(...). This is application logic running inside the library's lifecycle — proof that "use a library" does not mean "give up control."

The config also registers the @better-auth/stripe plugin (billing, subscriptions) and a per-process rate limiter with stricter caps on /sign-in/email and /sign-up/email. Billing is the subject of a later chapter; here we note only that rate limiting at the auth layer is a defense against credential-stuffing, and that its storage is in-memory unless BETTER_AUTH_RATE_LIMIT_DB=1 coordinates it across processes through the database.

The catch-all route: delegation in one file

A library that owns dozens of endpoints — sign-in, sign-up, callbacks for three providers, session lookup, password reset — could demand dozens of route files. RMH instead exposes all of them through a single catch-all route.

Catch-all route — a router pattern (here the file app/routes/api/auth/$.ts, where $ matches any trailing path) that captures every URL under a prefix and forwards it to one handler instead of declaring a file per endpoint.
// app/routes/api/auth/$.ts
export const Route = createFileRoute('/api/auth/$')({
  server: {
    handlers: {
      GET:     async ({ request }) => withCors(await auth.handler(request), request),
      POST:    async ({ request }) => withCors(await auth.handler(request), request),
      OPTIONS: async ({ request }) => withCors(await auth.handler(request), request),
    },
  },
});

Every method does the same thing: pass the raw Request to auth.handler(request) and wrap the result. The library inspects the URL and method internally and routes /api/auth/sign-in/email, /api/auth/callback/discord, /api/auth/get-session, and the rest to the correct logic. The application's only added responsibility is CORS: withCors echoes Access-Control-Allow-Origin back only for the origins in ALLOWED_ORIGINS, and sets Allow-Credentials: true so the session cookie may ride cross-origin from the platform's own subdomains. Note that the allow-list is hard-coded here and mirrors the trustedOrigins in the config — two gates, same guests.

Session-based authentication, step by step

RMH uses session-based auth (sometimes called stateful auth) rather than self-contained tokens. The server keeps a record of every live session and hands the browser a key to it.

Session — a server-stored record that a particular user is currently logged in. Here it is a row in the session table holding a unique token, the userId, an expiresAt, and forensic fields ipAddress and userAgent.

Follow an email/password login. On the client, the login page calls the library:

// app/routes/login.tsx
await authClient.signIn.email(
  { email, password, callbackURL },
  { onSuccess: () => { window.location.href = callbackURL; } }
);

That fires a POST /api/auth/sign-in/email, which the catch-all hands to auth.handler. Server-side the library: (1) looks up the Account for that email, (2) verifies the submitted password against the stored hash, (3) on success inserts a Session row with a fresh random token and an expiresAt, recording the request's ipAddress and userAgent, and (4) returns a Set-Cookie header carrying that token. Because useSecureCookies is true under HTTPS, the cookie is sent Secure and HttpOnly — invisible to JavaScript, which is what keeps a stray XSS bug from reading it.

On every subsequent request the browser automatically attaches the cookie. Server code that needs identity calls auth.api.getSession({ headers: request.headers }): the library reads the token from the cookie, finds the matching unexpired Session row, joins to its User, and returns { user, session } — or null if the token is missing, forged, or expired. The client mirror is authClient.useSession(), the React hook the login page uses to redirect an already-logged-in visitor away.

Because the session is a server row, the token in the cookie is meaningless on its own — deleting or expiring the row instantly invalidates the cookie everywhere. This is the chief advantage of stateful sessions over self-contained tokens, which remain valid until they expire because the server holds no record to revoke.

OAuth 2.0: signing in with Discord, Google, GitHub

The three social buttons do not collect a password. Instead they start an OAuth 2.0 authorization-code flow, in which RMH asks an external identity provider to vouch for the user.

OAuth 2.0 — a delegation protocol that lets a user authorize one site (RMH) to receive an identity assertion from another (Discord/Google/GitHub) without sharing a password with RMH.
Identity provider (IdP) — the external service that authenticates the user and issues the assertion. Here Discord, Google, and GitHub act as IdPs; RMH is the relying party.

The client kickoff is one call:

// app/routes/login.tsx
await authClient.signIn.social({ provider: 'discord', callbackURL });

From there the redirect dance runs: the library redirects the browser to Discord's authorize URL with RMH's clientId and a redirect URI pointing back at /api/auth/callback/discord. The user approves on Discord's own page (RMH never sees their Discord password). Discord redirects the browser back with a short-lived authorization code. The catch-all route receives the callback, and the library exchanges that code — server to server — for an accessToken (and often a refreshToken and idToken), fetches the user's profile, then finds or creates a User and an Account row before issuing a session exactly as the password flow does. From the browser's point of view, it ends logged in with a session cookie either way.

// lib/auth.ts — each provider supplies credentials from the environment
socialProviders: {
  discord: { clientId: ..., clientSecret: ..., overrideUserInfoOnSignIn: true },
  google:  { clientId: ..., clientSecret: ..., overrideUserInfoOnSignIn: true },
  github:  { clientId: ..., clientSecret: ..., overrideUserInfoOnSignIn: true },
}

The overrideUserInfoOnSignIn: true flag means that on each social sign-in the provider's current profile (name, image) refreshes the stored User — the IdP stays the source of truth for that data.

CSRF, briefly. Because the session cookie is sent automatically on every request to RMH's origin, a malicious third-party page could try to forge a state-changing request on the user's behalf — a cross-site request forgery. Better Auth defends the OAuth flow with a state parameter that must round-trip through the IdP, and the route's strict origin allow-list (trustedOrigins plus ALLOWED_ORIGINS) blocks credentialed requests from unknown origins. CSRF protection is the library's and the route's responsibility, not each feature's.

The data model: User, Session, Account, Verification

Four Prisma models in prisma/schema.prisma hold the entire auth state. They are deliberately small and they map (via @@map) to the lower-cased table names Better Auth expects.

ModelHoldsKey fieldsCardinality to User
UserThe identity itselfid, email, password, handle, isAdmin, isVerified—
SessionOne active logintoken (unique), expiresAt, ipAddress, userAgentmany per user
AccountOne sign-in method / linked providerproviderId, accountId, accessToken, refreshToken, idToken, passwordmany per user
VerificationPending email/reset tokensidentifier, value, expiresAtnone (keyed by identifier)

The shape repays study. A User has many Session rows, so one person can be logged in on a phone and a laptop simultaneously, each row carrying its own ipAddress and userAgent. A User also has many Account rows — that is how the same person can link Discord and GitHub and an email/password to one identity. For social accounts the OAuth tokens live on the Account (accessToken, refreshToken, idToken, scope); for the email/password method the hashed password lives on the Account's password column. Both Session and Account declare onDelete: Cascade, so deleting a user atomically sweeps away their logins and linked providers.

// prisma/schema.prisma (excerpt)
model Session {
  token     String   @unique
  userId    String
  expiresAt DateTime
  ipAddress String?
  userAgent String?
  user User @relation(fields: [userId], references: [id], onDelete: Cascade)
  @@map("session")
}
model Account {
  userId       String
  accountId    String
  providerId   String
  accessToken  String?
  password     String?  // hashed, for the email/password "account"
  @@index([providerId, accountId])
  @@map("account")
}
Notice that the application never stores a plaintext password and never writes hashing code. The password column on Account holds a hash produced and verified entirely inside Better Auth. Rolling your own password hashing is a classic, dangerous mistake; here it is simply not the application's concern.

How games use auth for leaderboards

Most of the games library (Chapter coverage of the games comes later) is playable anonymously. Auth becomes relevant the moment a game wants to attribute something to a person — a high score, progression, a ranked match. The pattern is uniform: a server route calls getSession, refuses the write if there is no user, and otherwise keys the persisted row on session.user.id.

// app/routes/api/slice-it/score.ts — auth-gated leaderboard write
const session = await auth.api.getSession({ headers: request.headers });
if (!session?.user) {
  return Response.json({ error: 'Unauthorized.' }, { status: 401 });
}
const userId = session.user.id;
const existingProfile = await prisma.player.findFirst({ where: { userId } });
// ...increment totalScore / gamesPlayed on the row that belongs to THIS user

This is the authn-then-authz split from the opening section, made concrete: the session establishes who, and keying every read and write on userId enforces that they only touch their own row. The same route also rate-limits by client IP and rejects implausible scores before it ever consults the session — auth is one layer of defense among several, not the only one.

"Optional auth" is a real third mode worth naming. A route may call getSession and simply branch: write to the global leaderboard when a user is present, fall back to a local or anonymous experience when not. Whether a given game requires a session or merely prefers one is decided route by route; the slice-it score endpoint above requires it, returning 401 to anonymous callers.

Summary

  • Authentication establishes who; authorization decides what. RMH Auth does the former; routes and plugin rules like authorizeReference do the latter.
  • The entire server surface is one betterAuth(...) config in lib/auth.ts: a Prisma adapter, email/password, three social providers, custom user fields, and a Stripe plugin.
  • A single catch-all route, app/routes/api/auth/$.ts, forwards every /api/auth/* request to auth.handler, adding only CORS.
  • Sessions are stateful: login writes a Session row with a unique token and returns it as an HttpOnly, Secure cookie; later requests are identified by getSession reading that cookie.
  • OAuth 2.0 sign-in runs a redirect dance — authorize, approve, code, server-side exchange — and ends by upserting User + Account and issuing a session, never touching the user's IdP password.
  • The model is four tables: User (one) to many Session and many Account, plus standalone Verification. Password hashing and CSRF defenses are the library's job.
  • Games gate attributed writes by calling getSession and keying rows on session.user.id.
