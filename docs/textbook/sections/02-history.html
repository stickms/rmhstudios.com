Learning Objectives

  • Reconstruct the evolution of rmhstudios.com from its git history, dividing it into six design eras and naming the pressure that drove each transition.
  • Explain why a starter template (Create Next App) becomes inadequate as a project grows, and what the first real product code adds on top of it.
  • Contrast the Next.js App Router era with the TanStack Start + Vite + Nitro era, and state the concrete reasons a team leaves a framework it already ships on.
  • Describe how a single web process splits into a fleet of realtime and worker processes, and why that split is the precondition for later infrastructure work.
  • Distinguish a reversible infrastructure migration (Compose to k3s) from a strangler-fig service migration (Node to Go), and recognize hermetic builds as an enabling technology for both.

A production system is an archaeological site. Its current shape records every pressure it has survived: the day the starter template ran out of road, the day one server process became three, the day "it works on my machine" stopped being good enough. This chapter reads the rmhstudios.com git history — roughly 1,470 commits, from the first Initial commit from Create Next App to a live migration toward Kubernetes — as a sequence of six eras. The dates here are not invented: they are the author and commit timestamps recorded in the repository. Where ordering is clear but a precise day is not load-bearing, we describe the order rather than overclaim a calendar.

Reading a system's history from its commits

Before the eras, a word on method. Architecture documents tell you what a system is; the commit log tells you how it became that, including the dead ends. Three commands carry most of the weight in this chapter. git log --oneline --reverse | head exposes the genesis — the template, the first product code, the first feature. git log --oneline | grep -iE "next|tanstack|vite|bazel|k3s|helm|terraform|migrat" dates the migrations by surfacing the commits whose messages announce them. And reading a migration's own design docs (the repository keeps several under docs/) recovers the reasoning a one-line commit message compresses away.

Era — in this book, a stretch of a project's history during which the dominant architectural decisions are stable. An era ends when a new design pressure (scale, framework limits, deployment pain) forces a structural change large enough to rename the system's shape.

Era 0 — "Create Next App": the empty lot

Every system begins as somebody else's defaults. The first commit, on 2026-01-27, is literally the output of a scaffolding tool:

851692a6 Initial commit from Create Next App
4dbf6afe Initial code for rmh studios

The gap between those two commits — about two hours, same evening — is the entire content of Era 0. The first is a template: a Next.js project with no product in it. The second is the first line the team actually owns. Within a day, interactivity appears (feat: implement mouse position context and update components for enhanced interactivity), then the personality of the site arrives in a burst: a homepage with content, projects, and merch sections; an interactive text effect; a ParticleField background; a BouncyCard with 3D hover built on Framer Motion; a "hyperpop" theme in globals.css. None of this is load-bearing architecture. It is identity — the visual signature that distinguishes "a Next.js app" from "rmhstudios.com." That is the real work of Era 0: replacing the template's anonymity with a product's voice.

Why start from a template at all? A starter gives you a working build, routing, and a dev server on minute one, so the first commit can be about your idea rather than webpack configuration. The cost is that you inherit the template's assumptions — here, the Next.js App Router — and those assumptions become the thing a later era has to renegotiate.

Era 1 — the Next.js App Router monolith

Era 1 is the longest-lived shape in the early history and the one captured by the repository's docs/codebase-overview.md, which describes a Next.js 16 App Router application: the web app on port 7000 and a separate Node HTTP + Socket.io server on port 7001, sharing a PostgreSQL database through Prisma. This era is where rmhstudios.com stops being a portfolio site and becomes a platform. Three things arrive that change its nature.

Games, and therefore realtime

In mid-February the games begin: Slice It (a rhythm game), Echoes/Altair, Laundry Sort, Vega, Signal Forge, Temple of Joy, House Always Wins. Single-player games need only a leaderboard endpoint. But multiplayer needs a server that holds live state, and on 2026-02-19 the commit log records exactly that being stood up:

feat: establish socket server for multiplayer game communication
feat: implement socket server for real-time game interactions

This is the first structural fork in the system: a second long-running process, speaking Socket.io on port 7001, living beside the Next.js app on port 7000. The web app renders pages; the socket server owns lobbies, score relays, and host reassignment. The same overview enumerates the event vocabulary — join_lobby, select_song, start_game, score_update for Slice It, and an ndw:-prefixed set for Neon Driftway — a convention that an event name carries the name of the game it belongs to. Era 1 is where the system first becomes two processes.

Realtime server — a long-running process that holds mutable session state in memory (lobbies, live scores) and pushes updates to clients over a persistent connection such as a WebSocket, rather than answering one stateless HTTP request at a time.

Identity, and therefore a database

Auth and persistence arrive together. The same February cluster of commits sets up Prisma and the initial migrations, and shortly after, feat: enforce auth across all games and apis (2026-02-18) makes login a precondition rather than a feature. Authentication is Better Auth with a Prisma adapter — email/password plus Discord OAuth — and the database grows a model per game profile (AltairPlayer, VegaPlayer, SignalForgePlayer, and so on) alongside the auth tables (User, Session, Account). This is the seed of what becomes, by the present day, a 142-model schema.

An honest detour worth noticing. The log briefly records the data layer oscillating: refactor database interactions from Prisma to direct SQL using @neondatabase/serverless (2026-02-18), then features migrating back onto Prisma models days later (migrate Signal Forge to auth-based leaderboard with Prisma model). Real history is not monotonic. Teams try direct SQL for hot paths, then return to the ORM for consistency. The commit log preserves the indecision that a tidy architecture diagram erases.

Era 2 — leaving Next.js for TanStack Start

On 2026-03-06 a single commit renames the system's foundation:

be48ea33 feat: migrate from Next.js to TanStack Start

This is the most counterintuitive transition in the history — the team leaves a framework it was already shipping on. The current README.md records where it landed: TanStack Start for routing and full-stack glue, Vite as the bundler and dev server, and Nitro as the server runtime that produces the production output. The flurry of fix-up commits immediately after the migration commit shows the real cost of such a move — it is never just one commit:

22b13f7f Externalize heavy packages and optimize SSR bundling in Vite config
11a63caa externalize Prisma from Nitro server bundle to fix .prisma/client resolution
3764a479 correct deploy.sh paths for Nitro output and use .cjs for esbuild bundles

Why leave? The evidence points at the build and server boundary rather than the UI. The web port moves from Next's 7000 to 7005 (the present BETTER_AUTH_URL=http://localhost:7005/), the bundler becomes Vite, and the production server becomes Nitro emitting a portable .output/ directory that PM2 can run on a plain VPS. A framework like Next.js couples your app tightly to its own build, server, and (historically) deployment model; TanStack Start + Vite + Nitro is a more unbundled stack where each layer — router, bundler, server runtime — is a component you can reason about and swap. For a system already running a second non-Next process (the socket server), a foundation that treats "the server" as an ordinary, ejectable artifact is a better fit than one that assumes it owns deployment.

SSR (server-side rendering) — rendering a page's HTML on the server for the first response, then "hydrating" it into an interactive React app in the browser. Both Next.js and the TanStack Start + Nitro stack do SSR; the migration changed which runtime produces and serves that HTML, not the fact of it.

Era 3 — the process-split fleet grows

Era 3 is not a clean break so much as a thickening. The two-process shape of Era 1 (web + socket) keeps subdividing as new realtime products land. RMHBox — a Jackbox-style party-game platform — appears around 2026-02-22; RMHTube (synchronized watch parties) and RMHMusic (collaborative listening rooms) follow. The log shows these services being born, sometimes merged into the unified socket server and sometimes split back out:

10ae25b0 feat(rmhmusic): add Spotify OAuth API routes and socket server
44f4810c refactor(rmhmusic): merge socket server into main unified server

By the time the infrastructure docs are written, the platform runs as eight long-running Node processes from a single shared image: web (SSR + ~130 API routes), socket (games and realtime apps), rmhbox, rmhtube, recap (a daily Discord recap), discord-bot, doctrine-worker (puzzle generation and decay), and vibe-worker (headless-Chromium thumbnails) — with rmhmusic riding inside the socket process under an event-name prefix. This proliferation matters for a reason that only pays off in Eras 4 and 5: the services are already decoupled, already speak well-defined transports, and already authenticate by querying the shared session table directly rather than calling into the Better Auth library. A system that is process-split before it is orchestrated has done the hardest part of the work in advance.

Era 4 — Docker Compose to k3s, Helm, and Terraform

A fleet of eight processes is fine until you have to run it reproducibly. Era 4 modernizes the run layer. On 2026-06-17, PR #121 lands:

ebcb92f5 feat(deploy): add Terraform + Helm (k3s) migration

The shape of the change is declarative orchestration. A Helm chart renders the eight processes as Kubernetes Deployments from one image; a Prisma migration runs in a pre-upgrade hook with atomic auto-rollback; a Terraform module manages DNS; and a registry switch leaves a path open to multi-node. Crucially, PR #121 kept Docker Compose intact as a reversible fallback. That is the defining discipline of this era: a migration that can be undone is a migration you can ship on a Tuesday.

Reversible migration — a change introduced alongside the old path rather than in place of it, so that rolling back is a configuration flip rather than a re-engineering project. PR #121's retention of the Compose path is the canonical example.

PR #121 also did something rarer: it wrote down the decision it was not making. Its recorded roadmap placed any rewrite of the backend in Go at "Stage 4 — only if metrics demand," and argued explicitly against a wholesale rewrite, on the grounds that the app was already process-split, the web tier must stay JavaScript (React SSR), and the data/auth layers (Prisma + Better Auth) had no Go equivalents. We will see in Era 5 that this stated boundary is exactly what the next era chooses to cross — deliberately, and on an isolated branch.

Era 5 — Bazel hermetic builds and an experimental Go tier

Era 5 has two intertwined strands, both landing within a few days in mid-June 2026.

A Go microservices tier (PR #122)

On 2026-06-18, the Go fleet arrives:

613405ae feat(go): complete Go microservices migration of the realtime/worker tier
46462fb9 Merge PR #122: Go microservices migration (additive; web image unchanged)

Read the parenthetical carefully: additive; web image unchanged. Nine Go binaries (gateway, gamehub, rmhmusic, rmhtube, rmhbox, recap, and three workers) mirror the Node realtime/worker tier rather than replacing it. A new gateway becomes the single ingress and reverse-proxies the WebSocket prefixes to the Go services while passing everything else to the still-JavaScript Node web app. Session issuance stays in the Node + Better Auth tier; the Go services only validate sessions against the shared session table. This is a textbook strangler-fig: a new system is grown around the old one, intercepting traffic at a boundary, able to take over one service at a time — not a big-bang rewrite. The honest framing the docs maintain is that this branch executes the complete Go target at once for evaluation, even though the design permits incremental adoption; the deliverable is a working, tested reference, not a claim that the rewrite was necessary.

Strangler-fig migration — replacing a system gradually by placing a new implementation behind a shared entry point (here, the gateway), routing slices of traffic to it as each slice is proven, until the old system can be removed. Named for the fig that grows around a host tree and eventually stands in its place.

A Bazel hermetic build (PR #142)

Two days later, on 2026-06-20, the build system is rebuilt around Bazel:

fe9be66b build(bazel): scaffold bzlmod workspace with hermetic Go SDK
aa7c0655 build(bazel): gazelle-generated BUILD files; all 9 Go services build hermetically
db0cc58b Merge pull request #142 from stickms/feat/bazel-build-system

The operator runbook makes the intent explicit: Bazel is pinned to 7.4.1 via .bazelversion, Go is not required locally because rules_go ships a hermetic Go 1.23 SDK, and each service builds into a distroless OCI image. A nine-service backend is exactly the kind of codebase where ad-hoc per-service build scripts rot; a hermetic build that pins its own toolchains is what makes "all 9 Go services build the same way on any machine" a true statement rather than a hope.

Hermetic build — a build that depends only on inputs it declares and toolchains it pins (here, Bazel 7.4.1 and a bundled Go 1.23 SDK), so the same commit produces the same artifact regardless of what happens to be installed on the developer's machine.

Summary

  • rmhstudios.com's history splits into six eras, reconstructable directly from ~1,470 commits; every date in this chapter is a real git timestamp, all in 2026.
  • Era 0 turns a Create Next App template into a product identity; Era 1 adds games, a Socket.io realtime server, Better Auth, and Prisma, becoming a two-process platform.
  • Era 2 leaves Next.js for TanStack Start + Vite + Nitro — a more unbundled stack where router, bundler, and server runtime are separable, fitting a system that already ran a second server process.
  • Era 3 grows the fleet to eight decoupled Node processes sharing one image and one session table; that decoupling is the precondition for what follows.
  • Era 4 (PR #121) makes the run layer declarative with k3s + Helm + Terraform while keeping Docker Compose as a reversible fallback; Era 5 (PR #122, #142) adds an experimental strangler-fig Go tier behind a gateway and a hermetic Bazel build.
