Chapter 1 · The rmhstudios Platform at a Glance

1.1 Answer: A process-split architecture runs one logical application as several independent operating-system processes, each with a single responsibility, instead of one monolithic server. In rmhstudios.com this means one web/SSR tier (Nitro) plus a fleet of standalone processes — three Socket.io realtime servers and several background workers. Two distinct benefits (any two): (1) fault isolation — if one process (e.g. the vibe-worker rendering thumbnails) crashes, the web tier and the rest keep serving, whereas a monolith would fail as a unit; (2) independent scaling — you can run extra copies of just the part under load (e.g. more realtime servers during a tournament) without duplicating everything; (3) independent deployment/runtime — long-lived WebSocket servers and stateless request renderers can be restarted, tuned, or even rewritten in another language separately.
1.2 Answer: The stale claim is that the web tier is "Next.js 16" (running on port 7000). The current web tier is TanStack Start + Vite + Nitro, listening on port 7005 in production. The two authoritative sources the chapter uses instead are README.md and the running code itself (e.g. package.json scripts and docker-compose.yml).
1.3 Answer: The dev script launches eight things via concurrently:
  • vite dev — web tier (TanStack Start dev server).
  • socket-server — realtime server.
  • rmhbox-server — realtime server.
  • rmhtube-server — realtime server.
  • discord-bot:dev — worker (no inbound user traffic).
  • doctrine-worker:dev — worker.
  • vibe-worker:dev — worker.
  • bot-worker:dev — worker.
So: one web tier, three realtime servers, four workers. (Note recap and status are not in the dev script but appear in the production build/start and in compose.)
1.4 Answer: The browser holds two connections at once (Figure 1.3):
  1. HTTP to the web tier (Nitro SSR) — fetches the rendered game page and its JS, and serves API routes. In production this is the web service (port 7005).
  2. WebSocket to a realtime server — for joining the lobby and receiving live game events. For most games this is socket-server (port 7001); for the minigame lobby it is rmhbox (7676).
What they have in common is the shared PostgreSQL database: both the web tier and the realtime server read/write it, and that is the only thing connecting the two halves — they do not call each other directly.
1.5 Answer: Publishing a host port means a service accepts inbound network connections from users (via the reverse proxy). socket, rmhbox, and rmhtube publish loopback ports (7001 / 7676 / 7003) because they are realtime servers that clients connect to over WebSockets. The four bot/worker services publish no port because they are background workers: they take no inbound user traffic and instead reach the rest of the system through the shared database and outbound calls (e.g. Discord, DeepSeek). The presence or absence of a published port is therefore a reliable signal of "server vs. worker."
1.6 Answer: A correct sketch keeps the Figure 1.1 core — Browser connecting by HTTP to the Nitro web tier and by WebSocket to the realtime servers, with every server pointing into PostgreSQL — and adds a status node off to the side. From status, draw dashed arrows (probes, not data dependencies) to the web tier and to each realtime server. A strong answer notes the asymmetry from the compose file: status probes the web tier via its public URL (through Apache/DNS, so it sees the real user path) and probes the internal services by their compose DNS names.
1.7 Answer: A strangler-fig migration replaces a system incrementally: new components grow alongside the old ones and take over responsibilities one at a time until the original can be removed (named after the strangler fig that grows around and eventually replaces its host tree). It is feasible here because the realtime tier is already split into independent processes that coordinate only through the shared database — they do not call each other in-process. The load-bearing property is the loose coupling between tiers (communication via the database rather than direct in-process calls). Because a Go binary under go-services/cmd/<service>/ can speak to the same PostgreSQL, it can replace one Node service at a time, with the web tier and the remaining Node services none the wiser. A tightly coupled monolith would have to be cut over all at once.
1.8 Answer: Because RMHBox is its own process (rmhbox on port 7676), the natural response is to run multiple instances of just the rmhbox service during tournaments and load-balance new connections across them, leaving the single web tier untouched. A single-process monolith cannot do this: every responsibility shares one process, so scaling RMHBox would mean duplicating the home page, blog rendering, and every other server too — wasteful and possibly impossible if the monolith holds shared in-process state. The new problem the split introduces is stateful WebSocket connections: a lobby's players must reach the same instance that holds their room. Naïve round-robin would scatter a lobby across instances. A strong answer proposes sticky sessions / room-aware routing, or sharing room state out of the process (e.g. via Redis or the database) so any instance can serve any room.
1.9 Answer: One image, many commands: Advantage — the image is built once and shared by every service (no redundant rebuilds, identical dependencies and code across the fleet, and rollback is a single tagged image). Drawback — every service carries the full payload even if it only needs a slice of it (e.g. workers ship the whole web build; the vibe-worker drags in everything although only it needs headless Chromium), so images are larger and a vulnerability anywhere in the shared image affects all services. Per-service minimal images would be leaner and have smaller attack surfaces, at the cost of more build complexity and more artifacts to manage.
1.10 Answer: A correct diagram (essentially Figure 1.2 with storage added) groups services into tiers and connects them to one PostgreSQL node:
  • Web/SSR: web (Nitro, port 7005/7015) — user-facing.
  • Realtime: socket (7001), rmhbox (7676), rmhtube (7003) — user-facing (publish loopback ports).
  • Workers: discord-bot, doctrine-worker, vibe-worker, bot-worker, recap — not user-facing (no published port).
  • Ops: status (7008) — published, but its arrows to the other services are dashed probes.
  • Storage: minio / minio-init (9000/9001) S3-compatible object store, plus the shared PostgreSQL all tiers read and write.

Chapter 2 · A Brief History

2.1 Answer: An era is a stretch of a project's history during which the dominant architectural decisions stay stable. The boundary between two eras is a structural change large enough to rename the system's shape — typically forced by a new design pressure such as scale, framework limitations, or deployment pain. Example boundaries from the chapter: the TanStack migration (foundation rename) and PR #121 (run-layer rename).
2.2 Answer: Commit be48ea33 feat: migrate from Next.js to TanStack Start, dated 2026-03-06. It ends the Next.js App Router era (Era 1) and begins the TanStack Start + Vite + Nitro era (Era 2).
2.3 Answer: Two evidence-backed reasons: (1) Unbundling the build/server boundary. The post-migration fix-up commits (externalizing heavy packages and Prisma from the Nitro bundle, correcting deploy.sh paths for Nitro's .output/) show the team wanted a stack where the bundler (Vite) and server runtime (Nitro) are separable, ejectable components rather than parts of a framework that owns deployment. (2) Fit with an already-multi-process system. The app already ran a second non-Next process (the socket server); a Nitro server that emits a portable .output/ directory PM2 can run on a plain VPS fits that reality better than a framework that assumes it owns the server and deploy model. (The web port also moves to 7005, consistent with a new server runtime.)
2.4 Answer: Orchestration (Helm/k3s) and a strangler-fig rewrite both require services that are independently deployable, communicate over well-defined transports, and don't share in-process state. In Era 3 the fleet already met all three conditions: eight processes built from one image, each speaking HTTP/Socket.io/Discord, and — critically — each authenticating by querying the shared session table directly rather than calling into the Better Auth library. Because that decoupling was already done, Era 4 only had to wrap the existing processes in Deployments, and Era 5 only had to put a gateway in front of them; neither had to first untangle a monolith.
2.5 Answer: The Era 4 migration is the reversible one. PR #121 introduced the k3s + Helm + Terraform run layer alongside Docker Compose, keeping the Compose path fully intact as a fallback. Rolling back is therefore a configuration flip (run via Compose instead of the chart), not a re-engineering effort. The Era 5 Go migration is also designed to be safe — it is additive (the web image is unchanged) and lives on an isolated branch behind a gateway — but it is a parallel rewrite of services rather than the same code run two ways, so it is "reversible" in the sense of being unmerged/unrouted rather than a single config toggle on one running system.
2.6 Answer: A strangler-fig migration replaces a system gradually by placing a new implementation behind a shared entry point and routing slices of traffic to it as each is proven, until the old system can be removed. The component that makes rmhstudios.com fit the pattern is the gateway: it becomes the single ingress, reverse-proxies the WebSocket prefixes to the new Go services, and passes everything else (notably the React SSR web tier) through to the still-running Node stack. Because routing happens at this one boundary, services can be cut over one at a time.
2.7 Answer: A correct answer keeps the Figure 2.2 fleet (web, socket, rmhbox, rmhtube, recap, discord-bot, doctrine-worker, vibe-worker, all reading Postgres) and inserts a single gateway process in front of the client-facing services. The gateway intercepts the arrows from "Browsers and Discord" that previously hit the realtime services directly; it routes the WebSocket prefixes (/socket/, /rmhbox-ws/, /rmhtube-ws/, /rmhmusic-ws/) to the corresponding Go services and proxies all other traffic (e.g. the SSR routes) to the Node web tier. Session issuance still flows to the Node + Better Auth tier; the Go services only validate against the shared session table.
2.8 Answer: A defensible answer engages both sides. For: PR #121's roadmap placed Go at "Stage 4 — only if metrics demand," so building the full fleet up front with no metrics arguably inverts the stated discipline. Against that objection: the work was done on an isolated branch, made additive (web image unchanged), kept production on the proven Node path, and produced a tested reference implementation plus a Helm chart that slots into PR #121's infrastructure — value that holds regardless of whether the rewrite ships. A sound position: building the complete target "for evaluation" is legitimate precisely because it is branch-isolated and reversible; the risk it would otherwise carry (a big-bang cutover) is neutralized by never routing production traffic to it until metrics justify per-service adoption. The key constraint — production must not be endangered — is satisfied by the additive, gateway-fronted, branch-only design.
2.9 Answer: The oscillation: in mid-February a commit refactors database interactions from Prisma to direct SQL via @neondatabase/serverless (a leaderboard hot path); within days, features migrate back onto Prisma models (e.g. "migrate Signal Forge to auth-based leaderboard with Prisma model"). The general lesson: real git history is not monotonic. Teams experiment — trying raw SQL for performance, then returning to the ORM for consistency and maintainability — and the commit log preserves that back-and-forth. A clean architecture diagram shows only the endpoint; the log shows the decisions and reversals that produced it, which is why reading commits (not just the final state) is essential to reconstructing real history.
2.10 Answer: A hermetic build depends only on inputs it declares and toolchains it pins, so the same commit produces the same artifact regardless of the developer's machine. PR #142 pins Bazel to 7.4.1 (via .bazelversion) and, through rules_go, ships a bundled Go 1.23 SDK so that Go need not be installed locally. This is what makes "all 9 services build the same way on any machine" true rather than aspirational: if each developer used their own locally installed Go, builds could diverge by version or platform. By making the toolchain a declared, pinned input fetched by the build system itself, the only variable left is the source at a given commit — eliminating the "works on my machine" class of build differences across the nine services.

Chapter 3 · The Web Tier: SSR with TanStack Start and Nitro

3.1 Answer: Server-side rendering is the per-request execution of the React component tree on the server, producing a finished HTML string that is sent to the browser so content is visible on the first response. Hydration is the subsequent browser-side phase in which React attaches event listeners and component state to the server-rendered DOM that already exists, rather than recreating it. In the window between first paint and the end of hydration, the page is visible but not interactive: text, images, and the correct (e.g. signed-in) shell are painted, but clicks, form inputs, and other event handlers do nothing yet because React has not yet wired them up.
3.2 Answer: (1) Browser sends GET / with the session cookie. (2) The Nitro server (.output/server/index.mjs) matches the URL against the generated route tree: __root → _site → _site/index. (3) Loaders run on the server — the root route's getInitialUser (in app/routes/__root.tsx) reads the cookie via getRequest() and queries Better Auth (auth.api.getSession); the index route's fetchSidebarData (in app/routes/_site/index.tsx) calls getSidebarData(). (4) React renders the matched component tree (sidebar + feed + correct signed-in shell) to an HTML string. (5) Nitro returns that HTML plus the serialized loader data embedded in the document. (6) The browser paints content, then downloads the client bundle. (7) Hydration: React re-runs the components in the browser, matches the existing DOM, and attaches handlers and state. The page is now interactive. (Matches Figure 3.1.)
3.3 Answer:
  • app/routes/_site.tsx → serves no URL of its own; it is a pathless layout (the leading _) at /_site that adds the sidebar shell.
  • app/routes/_site/index.tsx → / (the home feed), a page route nested inside the _site layout.
  • app/routes/blog.$slug.tsx → /blog/<slug>, a page route with a dynamic $slug parameter.
  • app/routes/api/search.ts → /api/search, a server route.
You can tell search.ts is not a page route two ways: its extension is .ts (no JSX, so no React component), and it exports a server.handlers object (a GET function returning a Response) rather than a component.
3.4 Answer: Layout is determined purely by position in the route tree. app/routes/_site/index.tsx lives inside the _site folder, so it nests under the pathless _site layout and renders into that layout's <Outlet />, inheriting the LeftSidebar. By contrast app/routes/altair.tsx sits at the top level of app/routes/, not under _site/, so it is never wrapped by that layout and renders full-screen with no sidebar. The file comment in _site.tsx states this explicitly: game routes, /secret, and /login are deliberately kept out of the _site layout. No runtime switch is involved — moving the file in or out of the folder is the entire mechanism.
3.5 Answer: Three concrete arguments against a pure SPA: (1) Personalized first paint — getInitialUser reads the session cookie on the server so the shell renders signed-in immediately; an SPA would flash a signed-out state until the client session loaded, which is exactly the bug the comment in __root.tsx says SSR avoids. (2) Crawlability/SEO — blog, news, and public builds must be indexable; sitemap[.]xml.ts emits real URLs for crawlers, and a crawler hitting an SPA shell sees no content. (3) Time-to-content — a content document should show text on the first response rather than after a large JS bundle executes. The legitimate concession: the games do not want server rendering — three, pixi.js, monaco-editor, tone are browser-only and huge. The config already supports this client-only lane via heavyExternals/traceDeps (kept out of the server bundle), manualChunks (split into vendor-* chunks), and stubServerFiles. So the right answer is SSR by default with client-only escape hatches, not a wholesale SPA.
3.6 Answer: The GET handler first computes the client IP via getClientIp(request) and calls rateLimit(ip, { limit: 60, windowMs: 60_000, prefix: 'search' }). When the caller exceeds the limit, allowed is false and the handler immediately returns Response.json({ error: 'Too many requests' }, { status: 429 }) — HTTP 429 with a JSON body { "error": "Too many requests" } — without touching the database. (Note also that q=ab is length 2, so even if allowed it would proceed; but the rate-limit branch returns first.) The contrast: a server route's handler returns a Web Response (status + headers + body) that the client consumes as an HTTP response, whereas a page route's loader returns plain data that TanStack Start serializes into the rendered document and exposes to the component via Route.useLoaderData() — there is no explicit status code or Response object in the loader path.
3.7 Answer: A backend-for-frontend (BFF) is a server layer built to serve one specific frontend, shaping and aggregating data exactly as that UI needs (applying auth, rate limits, and per-page selection) rather than exposing a generic, reusable API. In this codebase two mechanisms implement it together: (1) the server routes under app/routes/api/ (e.g. search.ts, explore.ts), which expose HTTP endpoints, and (2) server functions created with createServerFn (e.g. getInitialUser, fetchSidebarData) called from route loaders. The description "the same file-based router, just exporting handlers instead of a component" is accurate because API endpoints are ordinary files in the same app/routes/ tree, matched by the same router; they simply export a server.handlers object returning a Response rather than exporting a React component — same routing machinery, different output.
3.8 Answer: Pipeline (compare Figure 3.2): app/routes/* (route files on disk) → the TanStack Start plugin generates app/routeTree.gen.ts (the typed route tree, imported by app/router.tsx) → vite build (driven by the build:frontend script with NODE_OPTIONS=--max-old-space-size=8192) compiles client + Nitro SSR → output entry .output/server/index.mjs → the container runs node .output/server/index.mjs → the process listens on port 7005 (per the Dockerfile EXPOSE and deploy.sh PORT_WEB). The web entry is produced by either script, but build:frontend is the minimal one (just the Vite/Nitro build); the full build script additionally bundles the standalone WebSocket/worker servers (server/socket-server, server/rmhbox, server/rmhtube, recap, status, discord-bot, and the workers) via a separate esbuild command into dist-server/.
3.9 Answer: Two distinct problems: (a) Server bundle bloat / unrunnable server — these packages are browser-only (canvas/WebGL/native), so if Nitro's Rolldown bundler tried to include them the server bundle would balloon (monaco alone is ~2.6 MB) or fail to build. Solution: list them in heavyExternals and use Nitro's traceDeps to externalize them from the production server bundle (and trace truly server-needed natives like Prisma into .output/node_modules). (b) Client over-download — a content visitor who never opens a game should not download megabytes of game code. Solution: manualChunks splits them into named vendor-three, vendor-pixi, vendor-monaco-* chunks loaded only when needed. Separately, stubServerFiles replaces *.server.ts imports with empty stubs on the client so Node-only deps (e.g. pg) never enter the browser bundle. ssr.external alone is insufficient because, as the config comment states, Nitro's bundler ignores Vite's ssr.external — it only affects Vite's dev SSR bundling, so production externalization must go through the nitro() plugin's traceDeps/rollupConfig.external.
3.10 Answer: The server cannot know the user's chosen theme (it lives in localStorage under rmh-style), so it renders a generic default shell. If the theme class were applied only inside a React effect that runs after hydration, the user would see a flash of the wrong theme (FOUC): the default-themed page paints first, then visibly snaps to the correct theme a moment later once JavaScript runs. The inline themeScript avoids this by running synchronously before React hydrates — it reads localStorage, adds the style-* class and background color to <html> immediately, so the very first paint already shows the correct theme. suppressHydrationWarning is needed because this means the server-rendered markup and the (script-modified) client markup intentionally differ; without it React would warn about the hydration mismatch the script deliberately introduces.

Chapter 4 · State, Data Fetching, and the Client

4.1 Answer: The deciding question is: does the value have an authoritative source of truth on the server (the database)? If unplugging the network can never make the value "wrong," it is client state; if another user or another device changing the database can make your copy stale, it is server state. Classifications: the chosen site theme is client state (lives in themeStore / localStorage, no server truth); the puzzle leaderboard is server state (owned by the DB, fetched via useDoctrineReputationLeaderboard and refetched); a dropdown's open/closed flag is local component state (one component, useState).
4.2 Answer: set is the writer handed to the store's setup callback; calling set({ style }) merges that partial object into the store's state and notifies subscribers, triggering re-renders. A component should select with useThemeStore((s) => s.style) because Zustand subscribes the component only to the selected slice: it re-renders when style changes but not when some unrelated field changes. Reading the whole store object instead would re-render the component on any store change (and, with a new object identity each time, defeat bail-out optimisations).
4.3 Answer: Following takeDamage in game-store.ts: reduced = Math.max(1, 30 - 5) = 25; newHp = Math.max(0, 20 - 25) = 0; set({ hp: 0 }). Because newHp <= 0 and revivalsRemaining > 0 (it is 1), the branch calls get().revive() — not die(). revive() then sets phase: 'playing', restores hp to maxHp, and decrements revivalsRemaining to 0. So hp is written to 0 first, then immediately healed to maxHp by the revive action.
4.4 Answer: Per-run game state (HP, XP, weapons, the current run's coins) is meaningful only during a run and is fully reset by goToMenu/startRun; persisting it would resurrect a half-finished run on reload, which is incoherent and exploitable. Keybinds and volumes are stable user preferences that should survive reloads, so the settings store wraps its setup in persist(...). partialize is a whitelist: it returns the subset of state to write to localStorage (theme, keybinds, the volume/toggle fields), so transient or derived fields are never serialised. The key name: 'altair-settings' is the storage key under which that subset is saved.
4.5 Answer: Both components share the one QueryClient from Providers.tsx and use the same query key ['doctrine', 'reputation'], so React Query deduplicates: exactly one network request to /api/doctrine/reputation fires. Whichever mounts second subscribes to the same in-flight/cached entry and sees the result without a second fetch. Ninety seconds later the cached value is older than staleTime (60s), so it is stale; React Query will refetch in the background on the next trigger. Independently, refetchInterval: 120_000 schedules an automatic refetch every two minutes regardless, keeping the value fresh.
4.6 Answer: The rejection-path sequence: User submits → prependItem shows the post optimistically → POST /api/rmharks returns 4xx → the UI must roll back, calling removeItem(tempId) (and typically showing an error toast). Because the post was rejected, no rmhark.created SSE echo arrives, so there is nothing to dedupe. The line that otherwise guards against a duplicate is the early return in prependItem: if (state.items.some((i) => i.id === item.id)) return state; — without it, an optimistic insert plus the server's later SSE echo (same id) would appear twice.
4.7 Answer: This is server state (the count is owned by the DB), so the right tool is React Query: a query keyed e.g. ['notifications', 'unread-count'] with a modest refetchInterval (say 30–60s) so it stays "roughly current" without hammering the API; the live SSE notification stream could also invalidate or directly update it for instant accuracy. To avoid a hydration flash, seed the first render from the server — either render a neutral placeholder (no number) until the query resolves, or, better, hydrate an initial count from the same server loader that resolves the session (as initialUser does) so the first painted HTML already shows the correct number. Trade-off: a short interval costs requests but feels live; seeding from the server avoids a wrong-number flash at the cost of plumbing the count through SSR.
4.8 Answer: Three pieces cooperate. (1) The inline theme script in __root.tsx runs before React hydrates and synchronously applies the saved theme class and background colour, so the live DOM already matches the server HTML. (2) The server-rendered HTML and the client's first React render must be identical; the Zustand theme store still holds "default" on that first render because localStorage has not been read yet, which matches what the server emitted. (3) The isFirstRun guard makes the theme-sync effect a no-op on its first run, so React does not overwrite the class/colour the inline script already set. If the guard were removed, the first effect run would write based on the still-default store value, momentarily applying the wrong theme — a visible one-frame flash (and on iOS Safari, the wrong status-bar tint derived from the body background).
4.9 Answer: Generalising "persist to localStorage + sync to DB" to all server data is a poor default. The pattern fits the Altair meta-store because progression is read constantly during play and benefits from instant, offline-capable reads; but as a general server-state strategy it reintroduces the cache-invalidation problems React Query exists to solve. With two devices editing the same persisted-and-synced store you get a last-write-wins race: each device holds its own localStorage copy, both push to the DB, and one device's changes silently clobber the other's; there is no shared notion of "current server truth" until a reload. A pure React Query cache avoids this because the DB is always the single source of truth — the cache is explicitly a copy with a TTL, refetches reconcile it to the server, and there is no durable client-owned write to diverge. Verdict: keep the meta-store as a deliberate exception; use React Query for ordinary server data.
4.10 Answer: In receiveCreated, with filter === "friends" the code enters the friends branch and prepends only posts where delivery.followed || delivery.own is true. A stranger's event has followed: false and own: false, so the branch returns without adding it — the post is dropped (strangers' posts never belong on the Following surface). On the "For You" surface (e.g. filter === "all") the same stranger event takes the other branch: since delivery.own is false, it is not prepended immediately but pushed into pendingItems, surfacing behind the "N new posts" pill so the scroll position is not disturbed. The deciding code is the if (filter === "friends") { ... return; } block versus the final delivery.own ? prependItem : push to pendingItems branch.

Chapter 5 · The Data Layer: Postgres and Prisma at Scale

5.1 Answer: An ORM (object-relational mapper) is a library that maps database tables to in-language objects and translates method calls into SQL, so you read and write rows as typed objects instead of hand-written query strings. @@map("rmheet") tells Prisma that the model called RMHark in the code corresponds to a Postgres table physically named rmheet. Names differ here because the table was created under an earlier product name and renaming a live table is risky; @@map lets the code adopt a cleaner model name while the underlying table keeps its original name and data.
5.2 Answer: onDelete: Cascade is a foreign-key rule enforced by Postgres itself. When a User row is deleted, the database automatically deletes every Session row whose userId referenced it (and likewise every other child with a cascading relation). The application doesn't have to delete sessions manually because the constraint lives in the database, so the cleanup happens atomically in the same transaction and can't be skipped by a forgetful code path or lost in a crash between two separate deletes.
5.3 Answer: @@unique([userId, gameMode, dateKey]) is a correctness constraint: it guarantees at most one score row per user, per game mode, per day, and it doubles as the lookup index for the daily upsert. The column order puts the most selective identity first (userId) and is the natural key for "this user's entry today." @@index([gameMode, dateKey, score(sort: Desc)]) backs the leaderboard read: it leads with gameMode, dateKey so the database can jump straight to one day's one game, then has score descending so the top scores are already in order — the query becomes an index range scan with no separate sort step.
5.4 Answer: The high score and best lap time should be typed columns (Int and Float/Int ms): you will sort leaderboards by them, so each should get a descending index, e.g. @@index([highScore(sort: Desc)]) and an ascending index on lap time (lower is better). The opaque save blob should be a Json column, because its shape is owned by the game team and will change often; making it JSON means those changes need no database migration. This follows the schema's rule of thumb seen in SignalForgePlayer (indexed highScore + floorReached alongside savedRunState Json?): scores and identifiers are columns; opaque game state is JSON. Anything you need to sort, filter, or constrain on must be a real column, since the database can't efficiently query inside a JSON blob.
5.5 Answer: prisma db push --accept-data-loss shoves the current schema directly into the database without writing a migration file, dropping columns as needed to converge. It is appropriate for local prototyping where the database is disposable. Run in production it would be destructive and unaudited — --accept-data-loss could silently drop a column of real user data, and no migration record would exist to reproduce or review the change.
prisma migrate deploy applies the already-authored, version-controlled migrations in order and never invents new ones; it is the production deploy command. If you ran it locally while authoring a new schema change, nothing would happen to your in-progress edits — it only applies existing migration files, so you'd be missing the change you were trying to test (you needed migrate dev to author it first).
5.6 Answer: likeCount is maintained by atomically incrementing/decrementing it in the same transaction as the like/unlike write, so the feed read path never aggregates. It can drift from the true count of RMHarkLike rows in two ways: (1) a write that updates one side but not the other — e.g. a RMHarkLike row is created/deleted but the counter update fails or is skipped outside a transaction; and (2) bulk or out-of-band changes, such as a cascade delete of likes (when a user or post is removed) or a direct database edit that removes rows without going through the counter logic. The repair script is scripts/reconcile-feed-counts.ts.

The poll uses ||--o| because RMHark declares poll RMHarkPoll? — at most one poll per post. Likes and comments are ||--o{ (one-to-many).
5.8 Answer: The singleton is stashed on the global object only in non-production to survive development hot-reload. In dev, the module that creates the client is re-evaluated on every code change; without a global cache, each reload would build a fresh PrismaClient with a fresh connection pool, and the old pools would leak, quickly exhausting Postgres connections. Caching on global lets each reload reuse the one already created. In production the module is loaded once and never hot-reloaded, so there's no reload leak to guard against; stashing on the global there would be unnecessary, and keeping production state on a mutable global is best avoided, so the code deliberately skips it.
5.9 Answer: Worst-case total = 1 web process × 10 + 3 realtime servers × 5 = 10 + 15 = 25 connections. With a 60-connection ceiling, you have 60 − 25 = 35 spare connections, which at 5 per realtime process is 35 ÷ 5 = 7 more realtime server processes (reaching 25 + 35 = 60) before you must change the Postgres configuration. (A prudent engineer would stop earlier to leave headroom for admin connections and migrations.) Two options at the ceiling: (1) raise Postgres max_connections (costs memory per connection); or (2) put a connection pooler such as PgBouncer in front so many app connections multiplex onto fewer real Postgres connections. A third is to lower each pool's max.
5.10 Answer: For one model per game: each game gets exactly the typed columns and indexes its scoring needs (Altair's bestTime/totalKills vs Vega's highestLoop/highestLevel); leaderboard queries are a single indexed sort with no gameId filter; the database enforces real constraints (one profile per user per game); and a schema change to one game can't break another. For a shared GameScore table: far fewer models, no repeated boilerplate, and adding a game needs no migration. The cost is that scores live in an unindexable JSON payload, you can't enforce per-game shapes, and every query carries a gameId filter that competes for the same index.
Choice for 50 more games: a hybrid is most defensible — keep the per-game model pattern (it's what the schema does and it scales reads well), but reduce boilerplate by generating the repetitive models from a template rather than hand-writing each. If forced to pick one pure approach, the per-game model wins for a leaderboard-heavy platform precisely because sortable scores must be typed, indexed columns, which the shared-JSON design cannot provide.

Chapter 6 · Authentication and Identity (RMH Auth)

6.1 Answer: Authentication establishes who a request is from (identity); authorization decides what that already-known principal may do (permission). RMH Auth is almost entirely authentication — it turns an anonymous request into a known User. The clearest authorization line in lib/auth.ts is the Stripe plugin's authorizeReference: async ({ user, referenceId }) => referenceId === user.id. It is authorization because by the time it runs the user is already authenticated; the rule only decides whether that known user may manage the subscription whose referenceId was passed (only their own).
6.2 Answer: (1) login.tsx calls authClient.signIn.email({ email, password, callbackURL }), which fires a POST /api/auth/sign-in/email. (2) The request hits the catch-all route app/routes/api/auth/$.ts, whose POST handler wraps the call in withCors and passes the raw Request to auth.handler(request). (3) Better Auth looks up the email/password Account, verifies the submitted password against the stored hash, and on success inserts exactly one new Session row (unique token, userId, expiresAt, plus ipAddress and userAgent). (4) It returns a Set-Cookie header carrying the token, sent HttpOnly and (under HTTPS, since useSecureCookies is true) Secure. The single created row is the Session row.
6.3 Answer: The $ makes app/routes/api/auth/$.ts a catch-all (splat) route: it matches any path beneath /api/auth/, not a single endpoint. All three method handlers do the same thing — forward the unmodified Request to auth.handler(request) and wrap the response in withCors. Better Auth then inspects the URL and method internally and dispatches /sign-in/email, /callback/discord, /get-session, etc. to the right logic. So one route can serve every auth endpoint because the library owns the sub-routing; the application only provides one delegating entry point plus CORS.
6.4 Answer: A correct sequence (mirroring Figure 6.3 with Google as IdP): 
Browser -> RMH:    signIn.social({ provider: 'google' })
RMH -> Browser:    302 redirect to Google authorize (clientId, redirect_uri, state)
Browser -> Google: user approves on Google's own page
Google -> Browser: 302 back to /api/auth/callback/google?code=...&state=...
Browser -> RMH:    GET the callback (catch-all route)
RMH -> Google:     exchange code for tokens (server-to-server)
Google -> RMH:     access token + id token + profile
RMH -> Postgres:   upsert User, upsert Account, INSERT Session
RMH -> Browser:    Set-Cookie session + redirect to callbackURL
RMH first learns the user's Google identity at the server-to-server token exchange, when Google returns the profile (the preceding authorization code is opaque and not yet identity). RMH never sees the user's Google password.
6.5 Answer: A User has a one-to-many relation to Account, where each Account is one sign-in method or linked provider. Concretely, the same person could sign in with Discord, link GitHub, and also set an email/password — three Account rows, one User (distinguished by providerId/accountId). On Account, an OAuth access token is stored in the accessToken column (with refreshToken, idToken, scope), while the email/password method's hashed password is stored in the password column.
6.6 Answer: auth.api.getSession({ headers: request.headers }) reads the token from the cookie, fails to find a matching unexpired Session row (it has expired), and returns null. In app/routes/api/slice-it/score.ts the guard if (!session?.user) is therefore true, and the route returns Response.json({ error: 'Unauthorized.' }, { status: 401 }). The score is never written because the route never reaches const userId = session.user.id.
6.7 Answer: Because each login is its own Session row keyed by userId, "log out of all other devices" is a single delete of every session for the current user except the one making the request: conceptually DELETE FROM session WHERE userId = :me AND token != :currentToken (in Prisma, prisma.session.deleteMany({ where: { userId, NOT: { token: currentToken } } })). It works instantly because sessions are stateful: the cookie only holds a key into the table, so once the row is gone, the next getSession on that other device finds nothing and treats the request as anonymous — the cookie is dead even though the browser still holds it. The stored ipAddress/userAgent can be surfaced to let the user recognize which sessions they are ending.
6.8 Answer: Hashing happens entirely inside Better Auth; the application contains no hashing code. The result is stored in the password column of the email/password Account row (never plaintext, and never on the request path the app handles). "Never write your own password hashing" is sound because correct password hashing requires a vetted, deliberately slow, salted algorithm with constant-time verification and safe defaults; small mistakes (fast hashes, missing salts, timing leaks) silently weaken security, so delegating to a maintained library is the safe default.
6.9 Answer: Stateful sessions store a row server-side and put only an opaque key in the cookie; self-contained tokens carry their claims and are trusted on signature alone with no server lookup. Advantage of self-contained tokens: no per-request database read, so they scale statelessly. Advantage of stateful sessions: instant, reliable revocation — there is a row to delete. RMH chose stateful sessions, which is exactly why 6.7 is easy: deleting the other Session rows revokes those logins immediately, whereas self-contained tokens would stay valid until expiry because there is no server-side record to remove (you would need an extra denylist to revoke them).
6.10 Answer: A correct sketch: 
User ||--o{ Session : has      (one user, many sessions)
User ||--o{ Account : links     (one user, many accounts)
Verification                    (no relation to User)
User is the parent of both Session (via Session.userId) and Account (via Account.userId), each one-to-many with onDelete: Cascade. Verification stands alone — it is keyed by identifier (e.g. an email for a pending verification or reset) and has no foreign key to User.

Chapter 7 · Realtime Architecture: The Socket Server Fleet

7.1 Answer: (1) HTTP is client-initiated: the server can only reply to a request, so it cannot push another player's move when it happens. The workaround is polling — re-asking every few hundred ms — which adds latency equal to the poll interval, wastes bandwidth, and still cannot carry a 60 Hz feed. (2) HTTP is stateless: each request is meant to stand alone, but a lobby is an evolving shared world (who is here, who is ready, the chosen level) that must persist between messages. The workaround is to reload full state on every request, which is both expensive and racy. WebSockets fix both by keeping one connection open and bidirectional.
7.2 Answer: A room is a server-side label grouping a set of sockets so the server can broadcast to exactly that group with io.to(room).emit(...) (e.g. ndw:${roomId}); a namespace is a higher-level partition of the connection itself (like /admin). rmhstudios.com generally does not use namespaces — RMHbox's middleware is "applied server-wide — no namespace." Isolation between Blackjack and Neon Driftway comes instead from (a) event-name prefixes (bj: vs ndw:), so a handler only listens for its own verbs, and (b) rooms, so broadcasts for one lobby never reach another. In index.ts all handlers are registered against the same socket on connection, but their disjoint prefixes keep them from colliding.
7.3 Answer: (1) Client→server ndw:joinLobby — server creates the lobby (first joiner becomes host), records the player, and the socket joins room ndw:ABC. (2) Server→room ndw:lobbyState via broadcastLobby. (3) Client→server ndw:toggleReady → server→room ndw:lobbyState (updated). (4) Client→server ndw:startGame (only honored if lobby.hostId === socket.id and status is WAITING); status becomes COUNTDOWN. (5) Server→room ndw:startCountdown with countdownSeconds: 3. The one timer: a setTimeout(..., 3000); on fire (if still COUNTDOWN) status becomes PLAYING and (6) server→room ndw:gameStarted is emitted, followed by a final ndw:lobbyState.
7.4 Answer: socket.to(room).emit(...) broadcasts to everyone in the room except the sending socket; io.to(room).emit(...) broadcasts to everyone including the sender. ndw:playerUpdate uses socket.to because the sending client already knows its own position — echoing it back would be wasted bandwidth and could cause jitter. ndw:startCountdown uses io.to because the countdown is a shared signal every client (the host included) must see and act on simultaneously; excluding the host would leave the player who pressed start without a countdown.
7.5 Answer: Server-authoritative means the server holds canonical state and recomputes anything affecting the outcome, treating client messages as requests. In rmhtype:game:progress the client sends only raw position (characters typed) and errors, both clamped with Math.max(0, …). The server refuses the event unless the room is in state TYPING and room.roundStartTime is set, and it ignores progress from a player already finished. It then computes WPM itself as (position / 5) / (elapsedMs / 60000) where elapsedMs = Date.now() - room.roundStartTime — the server's own clock. A client cannot announce "400 WPM" because WPM is derived on the server from elapsed time the server measured; the only lever the client has is how many characters it claims, and that is bounded by the round timer it does not control.
7.6 Answer: Benefits: (1) isolation / blast-radius — a crash in rmhbox does not take down the race server on 7001 or the video rooms on 7003, and Docker's restart: unless-stopped revives only that one container; (2) independent scaling — different load shapes (typing surge vs. watch-party surge vs. a Chromium-heavy worker) can get different resources without competing for one event loop; (3) independent deployment / lifecycle — each service declares its own shutdown contract. Cost: more moving parts to build, deploy, monitor, and reason about than a single server. A concrete per-service shutdown contract: the Discord bot's stop_grace_period: 90s (to finish in-flight AI edits), the vibe worker's 30s (to finish Chromium captures), versus a socket server draining within SHUTDOWN_TIMEOUT_MS of 10s.
7.7 Answer: The function scans ndwLobbies for the lobby containing the socket. Mutations, in order: lobby.players.delete(socket.id) (now 3 players), then socket.leave(`ndw:${roomId}`). Since players.size !== 0, it takes the else branch. Because the leaver was the host, lobby.hostId === socket.id is true, so it executes the reassignment line: lobby.hostId = lobby.players.keys().next().value || '' — the new host is the first remaining entry in the players map (insertion order ≈ join order). It then emits ndw:playerDisconnected with reason: 'disconnect' to the room and calls broadcastLobby, which emits an updated ndw:lobbyState (re-flagging isHost). If instead the leaver was the last player, players.size === 0 and the function runs ndwLobbies.delete(roomId) — the lobby object is destroyed and no events are emitted.
7.8 Answer: The MAX_ENTRIES cap defends against a memory-exhaustion attack: without it, an attacker could create unbounded distinct socketId:eventName keys (e.g. by flooding many event names or churning connections) and grow the counter map without limit; at capacity the limiter evicts the oldest entry, keeping memory predictable. Treating unlisted events as unlimited is a deliberate convenience — only events that need throttling appear in SOCKET_RATE_LIMITS — but it is also a gap: a hot event nobody added a rule for is unthrottled, so the limiter is only as good as its rule coverage. The asymmetry between altair:game:input (1200/min) and altair:lobby:create (3/min) reflects intent vs. cost: gameplay input is legitimately high-frequency (≈20/s), so a tight cap would break the game, whereas lobby creation is rare, cheap to abuse, and resource-heavy (each spins up state), so 3/min throttles spam without hurting real users.
7.9 Answer: What breaks: lobby state lives in each process's ndwLobbies Map, which is not shared. If the load balancer routes player A to replica 1 and player B to replica 2, each replica creates its own lobby for room code ABC; io.to('ndw:ABC').emit(...) on replica 1 reaches only sockets connected to replica 1, so A and B never see each other despite "joining" the same code. Required changes: (1) at the proxy — enable sticky sessions so each client stays pinned to one replica for the life of its WebSocket connection (needed for the handshake even with a backplane); (2) in Socket.io — install a backplane (e.g. the Redis adapter) so room broadcasts are forwarded between instances; (3) where state lives — move authoritative lobby state out of the per-process Map into shared storage (Redis or Postgres), or shard so that all members of one lobby are routed to the same instance. The two key terms: backplane (adapter) and sticky session. This is the Chapter 13 horizontal-scaling problem.
7.10 Answer: A correct flowchart shows browsers connecting to an Apache reverse proxy that routes three path prefixes to three client-facing Socket.io processes — /socket/ → socket-server :7001, /rmhbox-ws/ → rmhbox :7676, /rmhtube-ws/ → rmhtube :7003 — and those arrows from browser to proxy to process are the WebSocket upgrades. The four no-port processes — discord-bot (which instead holds an outbound connection to the Discord gateway), doctrine-worker, vibe-worker, bot-worker — have no inbound proxy arrow. All seven processes plus the proxy-fronted ones draw a plain database connection arrow to a single Postgres node. (The status :7008 process additionally draws dashed "probe" arrows to the other services and to the public site.) Matches Figure 7.1.

Chapter 8 · The Games Platform

8.1 Answer: Server-authoritative scoring means the server, not the browser, is the final authority over what score is allowed to affect persistent state. The client may compute and display a score for responsiveness, but every submission is re-authenticated, re-validated, and bounded by the server before it can touch a leaderboard. The browser's number cannot be trusted because the player owns the machine running the JavaScript: handleHit() runs in their process, they can edit it, replay it, or skip it entirely and call POST /api/slice-it/score directly with any number. The arithmetic being correct on screen is irrelevant — the server never observed the keystrokes, so the submitted number is a claim, and treating a claim as truth is the whole vulnerability.
8.2 Answer: In order, the gates in app/routes/api/slice-it/score.ts are: (1) Rate limit by IP — rateLimit(ip, { limit: 5, windowMs: 60_000, prefix: 'slice-score' }); fails with 429 plus a Retry-After header. (2) Payload shape — username must be a string and score a number; fails with 400 ("Invalid score data."). (3) Username sanitization — stripped to an allow-list and capped at 24; if under 2 chars, 400 ("Invalid username"). (4) Score bound — must be 0 … 1_000_000_000, else 400 ("Invalid score"). (5) Speed check — playSpeed < 1.0 is rejected as unranked with 400. (6) Authentication — auth.api.getSession(); no session.user yields 401 ("Unauthorized."). Only after all six does it write: increment Player.totalScore/gamesPlayed (or create the profile, with a 409 if the username is taken), and if songId matches a real Song and the score beats the existing personal best, upsert a SongLeaderboard row. A thrown error returns 500.
8.3 Answer: Writes mutate persistent, publicly visible state (a leaderboard) and have side effects — incrementing totalScore, creating rows. A flood of forged writes corrupts the rankings everyone sees and grows the database, so the cost of abuse is high and permanent; capping at 5/min sharply limits how fast a single IP can attempt forgeries. Reads (leaderboard GET) only return existing data, are cacheable, and leave no residue; the worst a flood does is consume CPU and bandwidth, which is a denial-of-service concern handled more cheaply elsewhere. So the generous 20/min read limit trades a little DoS exposure for a smooth UI (refreshing the board, paging songs), while the tight 5/min write limit protects the integrity of the data itself — the asymmetry mirrors the asymmetry in consequences.
8.4 Answer: The range check only rejects values that are obviously impossible (negative, or above a billion). It does nothing to a cheater who submits a forged-but-plausible number — say a legitimate-looking 480,000 for a song where a strong human might score 250,000. Such a submission passes every gate: it is a well-formed number in range, played at speed ≥ 1.0, from an authenticated session, under the rate limit. The server cannot tell it apart from an honest play because it never received the evidence of the play — the per-note hit timings against the beatmap. To actually re-validate, the server would need to receive (and re-simulate) the input trace: the sequence of (lane, timestamp) presses plus the seed/beatmap id, then recompute the score with the same HIT_WINDOWS and calculateScoreMultiplier() logic server-side and confirm it matches. That is the deterministic-replay design the deterministic RNG (8.6) is a precondition for; bounding only limits the blast radius.
8.5 Answer: The blob (savedRunState Json?) carries the full state needed to resume an in-progress roguelike run — deck, relics, floor layout, enemy state — which is too rich and too client-evolving to model relationally; storing it whole lets the client change its state shape without a database migration. But Postgres cannot efficiently ORDER BY a field buried inside a JSON document, so the blob is useless for ranking. The derived scalar columns highScore and floorReached exist precisely for that: they are indexed (@@index([highScore(sort: Desc)])) so the leaderboard's findMany … orderBy desc is a fast index scan. The leaderboard query cannot read the blob because doing so would mean deserializing every player's full run and sorting in application code — O(players) JSON parses on every page load. So each representation does what the other cannot: the blob resumes, the columns rank.
8.6 Answer: createSeededRandom(seed) is deterministic because its entire internal state is initialized from the seed string (hashed into h) and then advanced by a pure integer recurrence (the mulberry32 step) with no external entropy — no Date.now(), no Math.random(), no I/O. Given the same seed, the closure produces the identical sequence of numbers every time, on any machine. The feature this enables, named in the chapter, is a seeded-run leaderboard (and the closely related deterministic replay): everyone playing seed S faces the exact same generated content, so their scores are comparable, and a stored seed lets the server reconstruct the run for validation (see 8.4). Neon Driftway's spec records seed = timestamp XOR random in run stats for exactly this reproducibility.
8.7 Answer: For a window where the left channel is much louder, BeatMapGenerator.generateFromBuffer() hits the branch if (maxL > maxR * 1.2) lane = 0, so the Slice is assigned lane 0 (top) — the engine reads a left-dominant transient as a "top" note. (Symmetrically, maxR > maxL * 1.2 gives lane 1.) The slice is only emitted if its peak exceeds the threshold and at least minDistance (0.2s) has passed since the last beat, preventing spam. When the channels are balanced (neither exceeds the other by the 1.2× factor), the lane is chosen by the seeded generator — lane = rng() > 0.5 ? 1 : 0 — so balanced beats still get a lane, but a reproducible one tied to the track id, not a fresh Math.random().
8.8 Answer: A correct flowchart keeps the validation spine of Figure 8.2 and adds a parallel, non-persistent branch during play. Example: 
flowchart TD
  P[Play: GameEngine.update] -->|throttled ~3x/sec| MP[mp.updateScore lobbyId]
  MP --> FLEET[Socket fleet: opponent HUD]
  P -->|on FINISHED| SUB[POST score.ts]
  SUB --> RL{rate limit 5/60s}
  RL -->|429| STOP1[reject]
  RL -->|ok| VAL{sanitize + bound + speed}
  VAL -->|400| STOP2[reject]
  VAL -->|ok| AUTH{getSession}
  AUTH -->|401| STOP3[reject]
  AUTH -->|ok| DB[(write Player + SongLeaderboard)]
The mp.updateScore branch leaves update() mid-play and goes only to the socket fleet to drive the live opponent HUD; it carries { score, combo } for presence, is throttled (Math.random() < 0.05), and is never authoritative. It does not reach the leaderboard write because the database is the record of truth and only the final, authenticated, validated score.ts submission may mutate it — realtime sync is for presence, not truth.
8.9 Answer: (1) Infrastructure layer: put a trusted reverse proxy / CDN (nginx or Cloudflare) in front of the app and have it overwrite — not append — the x-forwarded-for header with the real connecting IP, and drop any client-supplied value. This restores the assumption getClientIp() documents, so the spoof in the warning callout no longer works; the proxy can also enforce its own edge rate limiting and bot challenges before requests reach the Node process. (2) Application layer: rate-limit and account on a server-trusted identity rather than IP — key the limiter (or a second limiter) on the authenticated session.user.id, which the client cannot forge because it comes from auth.api.getSession(). Per-account limits (e.g. N ranked submissions per minute, plus anomaly checks like rejecting a new personal best that exceeds the previous one by an implausible margin) bind abuse to a real, revocable account regardless of network path. Either measure is independent of IP rate limiting and survives its bypass.
8.10 Answer: Altair is a survivors-like — it can have hundreds of enemies, projectiles, and particles on screen at once, i.e. a very high draw count ("the heaviest draw load," per the chapter). Canvas 2D issues one immediate-mode draw call per object on the CPU and stalls at that volume, so Altair pays for a hand-written WebGL2 SpriteBatch that packs many sprites into one GPU draw call. Slice-It, by contrast, draws a handful of lanes and notes, so Canvas 2D is plenty. What would not change if Slice-It were ported to WebGL is everything above the renderer: the loop body GameEngine.update() still reads the audio clock and resolves notes against HIT_WINDOWS; submitInput(), the seeded RNG, the beatmap, the score pipeline, and the MultiplayerFactory all stay identical. The loop-update-render pattern is renderer-agnostic — only the final "draw the current state" step is swapped.

Chapter 9 · Apps and Content Systems

9.1 Answer: Object storage is a service that holds arbitrary blobs addressed by a string key inside a named bucket, exposed over an HTTP (S3) API rather than SQL. Two reasons to use the rmh-media bucket instead of a Postgres column: (1) Size and opacity — images are megabytes of bytes the database never parses, indexes, or joins on; storing them in rows bloats backups and replication and competes for the connection pool that serves real queries. (2) Cheap, cacheable serving — the store is built to return immutable blobs by key, so the serve route can hand back the object with a year-long immutable cache header (and later a CDN/R2 in front), which a database row cannot do as naturally. A bonus reason: portability — the same S3 interface lets MinIO in dev swap for Cloudflare R2 in production by changing only the endpoint and keys.
9.2 Answer: Order of checks in app/routes/api/rmharks/image.ts before putObject: (1) require a valid Better Auth session; (2) apply the rate limit (10 requests/minute); (3) cap the request at 4 images; (4) cap each file at 5 MB; (5) verify each file's leading bytes with validateImageBuffer. Only then is the key minted (${userId}-${Date.now()}-${rand}${ext}) and written via putObject(feedImageKey(filename), buffer, contentTypeForFilename(filename)). The magic-byte check cannot be replaced by trusting the extension because the filename is attacker-controlled: anyone can name an executable or HTML payload cat.png. Reading the real content-type from the first bytes ensures the bucket only ever receives genuine images, which matters because the serve route later returns those bytes to other users' browsers with a permissive content type.
9.3 Answer: The deciding fact is who owns the playhead. RmhMusic delegates playback to an external Spotify player (the Web Playback SDK) on each client; the server need only rebroadcast an authoritative position every five seconds and let each Spotify instance hold and advance the timeline itself — so no fine-grained extrapolation, clock-offset estimation, or drift correction is required. RmhTube embeds the player (YouTube/Twitch/direct file) and is therefore responsible for keeping many independent embedded players in step; that demands a shared extrapolate/reanchor timeline, an NTP-lite clock to compare client and server time fairly, and two-tier drift correction (nudge then hard seek) to pull stragglers back. In short: the more the underlying player gives you for free, the less sync machinery you must build.
9.4 Answer: Non-host viewers do not freeze. The server keeps the canonical videoState with { currentTime, playbackRate, updatedAt }. Every two seconds the heartbeat fires and emits a freshly reanchored state computed by extrapolate: since playing is true, it returns currentTime + (serverNow - updatedAt)/1000 × playbackRate. Because serverNow advances with wall-clock time regardless of whether the host's tab sent a report, each heartbeat carries a larger position than the last, so viewers' players keep moving forward smoothly for the full ten seconds. The host's stalled reports only mean the server's updatedAt is not being refreshed by new host input — but extrapolation from the last known anchor is exactly what bridges the gap. When the host tab wakes and resumes reporting, its next host-state report re-anchors the canonical state, and any small divergence is corrected by the drift logic.
9.5 Answer: Fan-out-on-write (push): at post time the system writes the post (or a reference) into a materialized per-follower timeline. Reading is cheap (scan one pre-built list); posting is expensive. Hazard: the whale problem — an author with millions of followers triggers millions of writes per post. Fan-out-on-read (pull): the post is stored once; each timeline is assembled on demand by querying the authors the viewer follows. Posting is cheap (one insert); reading is expensive. Hazard: a viewer who follows very many accounts produces a huge IN (...) query whose plan and latency degrade as the follow list grows.
9.6 Answer: The proof clause is where: { userId: { in: followingIds }, ... } — the timeline is computed at read time by querying every followed author's posts; no per-follower timeline is ever written, which is the definition of fan-out-on-read. The keyset cursor encodes (createdAt, id) and the next page's WHERE is createdAt < cursor.createdAt OR (createdAt = cursor.createdAt AND id < cursor.id) under an ORDER BY createdAt DESC, id DESC. This anchors paging to the content of the last row seen rather than to a numeric offset. A post inserted mid-scroll changes which rows exist but not the boundary value the cursor stores, so the next page still begins strictly after the last row already shown — nothing is skipped (as offset paging would skip when rows shift) and nothing is duplicated (the tie-break on id disambiguates rows sharing a createdAt). The matching index @@index([createdAt(sort: Desc), id(sort: Desc)]) makes the walk efficient.
9.7 Answer: The constraint that makes the split worthwhile is the asymmetry of follower counts: most authors have small follower sets (cheap to fan out on write), but a few "whales" have enormous ones (catastrophic to fan out on write). Why not write-fan-out for everyone: a single whale post would generate millions of timeline writes, spiking latency and storage and risking write amplification that starves the rest of the system — the classic Twitter whale problem. Why not leave whales on the write path: precisely to avoid that explosion; flagging them isHighFanout lets the system skip their write fan-out and instead merge their (few, well-known) accounts into each reader's timeline at read time, where the cost is bounded by the small number of whales a person follows. The hybrid therefore pays the cheap write cost for the many normal authors (fast reads) and pays a small, bounded read-merge cost for the few whales (no write explosion) — getting the best of both strategies by routing on the one variable that distinguishes them. Note this is designed in docs/feed/plan.md but not built; today everything is pull.
9.8 Answer: The count is denormalized for read performance: the feed renders many posts per page, and recomputing each engagement total with a COUNT(*) over rmheet_like/rmheet_comment/etc. on every read would be far more expensive than reading a single integer column. So the column is maintained inline — a like both inserts the join row and increments likeCount in one transaction. The cron exists to repair drift: any path that mutates likes/comments/reposts/views without the matching counter update — a partial failure, a bug, a manual deletion, a race, a backfill — leaves the stored count out of step with the underlying rows. scripts/reconcile-feed-counts.ts periodically re-derives all four counters from the source tables with one set-based UPDATE, writing only the rows that have drifted. This is the standard "denormalize for speed, reconcile for correctness, tolerate eventual consistency" trade-off.

Chapter 10 · The Go Microservices Tier

10.1 Answer: A microservice owns one slice of behavior, runs as its own process/container, and communicates only over well-defined transports (HTTP, WebSocket, a database, a message bus) rather than by sharing in-process memory. A monolith puts all behavior in one process where modules call each other by direct function calls. In the Go tier the rule from FOUNDATION.md is “every service is a binary under cmd/<service>/main.go” — e.g. cmd/rmhbox/main.go compiles to one binary, packaged as one container (rmhstudios-go-rmhbox:dev), deployed as one unit. A service author may write only under cmd/<svc>/ and internal/<svc>/; they must not edit pkg/ or go.mod.
10.2 Answer: Strangler-fig migration replaces a system incrementally: a stable interface is placed in front of the old system, the replacement is built behind it one piece at a time, and traffic is routed to each new piece only once it is proven; old and new run side by side, so you can always route back. A big-bang rewrite switches everything at once — no incremental validation and no partial rollback, so a failed switch loses both systems. The enabling component is the gateway (port 7005). Because it is the single ingress and selects an upstream by longest-matching path prefix — falling back to the Node web tier for everything Go doesn't own — cutting over one more service is just adding one prefix to the routes slice in internal/gateway/proxy.go, and rolling back is removing it. Routing decisions live at exactly one boundary, so services migrate one at a time.
10.3 Answer: Session issuance — creating a session: OAuth callbacks, cookie signing, writing the session row — is framework-bound work that stays in Better Auth in the React tier, because there is no Go equivalent and porting it would mean rewriting the frontend's auth stack. Session validation — checking that a presented token corresponds to a live session — is just a database read. The Node services already did validation without the Better Auth SDK, by one SQL lookup. pkg/auth's ValidateSession reproduces that query verbatim: SELECT s."userId", s."expiresAt", u."name", u."image", u."isAdmin" FROM "session" s JOIN "user" u ON u."id" = s."userId" WHERE s."token" = $1 LIMIT 1, returning ErrUnauthenticated on no row or an expired expiresAt. So the Go fleet only validates against the shared session table; it never issues.
10.4 Answer: The request reaches the gateway's catch-all handler, which delegates to Router.ServeHTTP → Router.pick(path). pick walks the prefix list (sorted longest-prefix-first so “most specific wins”) and matches /rmhbox-ws/, returning the rmhbox upstream. The path lands on the Go rmhbox service at port 7676, which serves that exact prefix via hub.ServeWS — the full path is proxied unchanged, no prefix stripping. The gateway does not strip Connection/Upgrade because Go 1.23's httputil.ReverseProxy detects the WebSocket upgrade, switches to bidirectional-copy mode on a 101 Switching Protocols response, and manages those hop-by-hop headers itself; stripping them manually would break the upgrade. The one reason this does not work with a real browser today: the browser client speaks Socket.IO v4, while the Go service speaks JSON-over-raw-WebSocket — same semantics, different wire format, not interoperable — and the client adapter is designed but not built.

10.9 Answer: The rebuttal: the branch delivers value whether or not it ships. (2) Extracted good ideas that are language-agnostic and adoptable now: the single auth-validation SQL query, the events.Bus Redis backplane (which lets even the existing Node realtime services scale past one replica), and the atomic stat-merge fix. The risk a rewrite normally carries is neutralized by design: the work is additive (the web image is unchanged), branch-isolated (no production traffic ever routed to it), interface-honest (every stub compiles behind a real contract), and gateway-fronted (any cutover is per-prefix and reversible). The failure mode would have been shipping a big-bang rewrite; building the target to learn from it, with no way for it to endanger production, is the opposite of waste.

Chapter 12 · Deployment and Operations

12.1 Answer: A zero-downtime deployment releases a new version with no interval in which the service is unavailable: the old version keeps answering requests until the new one is proven able to take over, then traffic is switched. The guarantee hotswap-web.sh provides is that if the new web container never becomes healthy, traffic is never moved and the old container keeps serving. The new color is brought up on a spare port and polled with curl for up to HEALTH_TIMEOUT (60s); if it exits or never returns a success, the script removes the new container (or leaves it for diagnosis), never writes the active-port file, and the old color stays live. A bad build therefore degrades to "old version stays up," never to an outage.
12.2 Answer: The "process manager" role is filled by Docker Compose (not PM2). The two docker-compose.yml settings that express it are (1) restart: unless-stopped — Compose/Docker restarts a crashed service automatically and brings it back after a reboot, and (2) the logging driver json-file with options: max-size: "10m", max-file: "3" — capping each service's logs the way a process manager would. In the migration target the same role is played by Kubernetes Deployments.
12.3 Answer: In order:
  1. GitHub sends a push event to /webhook, which Apache proxies to webhook-server.cjs on 127.0.0.1:7002.
  2. webhook-server.cjs verifies the X-Hub-Signature-256 HMAC against WEBHOOK_SECRET, maps refs/heads/main → production, answers GitHub immediately, and spawns deploy.sh production detached (the script path comes from DEPLOY_SCRIPT, defaulting to deploy.sh).
  3. deploy.sh takes the flock lock, does git fetch + reset --hard, (re-execs itself if it changed), re-asserts public/ permissions, runs dc build web, SHA-tags the image, runs the Prisma migration, and brings up all services except web with --scale web=0.
  4. It then calls deploy/hotswap-web.sh, which starts the new web container on the spare blue/green port, health-checks it, and writes the active port into /home/rmhstudios/rmhstudios-web-active.conf.
  5. A root-owned systemd path-unit watching that file (or apachectl graceful directly) reloads Apache, whose ProxyPass / http://localhost:${WEB_UPSTREAM_PORT}/ now points at the new port. Apache serves the new version.
12.4 Answer: Blue (7005) is live, so the target is green on 7015.
  1. Start green with docker compose run on 127.0.0.1:7015, overriding PORT=7015.

Live user traffic moves at step 3, the instant Apache completes its graceful reload pointing at port 7015 (existing connections drain on blue; new requests go to green).

Chapter 13 · Scaling and the Road Ahead

13.1 Answer: Vertical scaling (scale up) gives a single machine or pod more resources — CPU, RAM, disk — so the same one process handles more load; its ceiling is the largest machine available and it offers no redundancy. Horizontal scaling (scale out) runs more replicas and spreads load across them; it is effectively unbounded and fault-tolerant. Horizontal scaling is clean only when replicas are interchangeable — when any replica can serve any request. A stateless service like web keeps no client state in memory: every page render re-reads from Postgres, so if a request lands on a different replica nothing breaks. A stateful service like rmhbox holds the live lobby in the memory of the replica that owns the connection; a second replica has never heard of that lobby, so the two replicas are not interchangeable and scaling out breaks the game unless extra machinery (backplane + stickiness) is added.
13.2 Answer: Constraint A — a single Postgres instance. Every tier reads and writes one database; each new process opens its own pool of connections, and Postgres runs out of connection slots (a fixed worker pool, each backend costing memory) long before it runs out of CPU. This bites as soon as you scale anything out, because more replicas of any service means more connections to the same database. Constraint B — un-CDN-able, single-replica websockets. The realtime services hold per-lobby state in memory, so they cannot be cached at a CDN edge and cannot run past one replica without a backplane and sticky sessions. This bites the moment you try to scale the realtime tier specifically: two players in one lobby may land on different replicas that cannot see each other. A relieves a constraint that grows with the whole fleet's size; B is a hard wall on the realtime tier in particular — hence the chart pins those services at replicas: 1.
13.3 Answer: A CDN caches a response at an edge node and replays it to many users; that only works for content that is identical for everyone and changes slowly (built JS, CSS, images). A websocket is a long-lived, bidirectional, per-connection conversation whose contents are unique to one lobby at one instant — there is nothing to cache and replay, and the connection must reach a live origin process that holds the lobby state. So the CDN can front the static site, but the realtime hostnames (socket., box., tube.) must bypass the cache and hit a live origin.
13.4 Answer: Sticky sessions pin one client to the same backend replica for the life of its connection, so all of that client's websocket frames hit the replica that already holds its connection (connection integrity). The Redis backplane is a shared pub/sub bus every replica subscribes to: when one replica gets an event for a room, it publishes on the bus and the other replicas holding members of that room forward it to their clients (cross-replica room coherence). Failure scenarios: (stickiness, no backplane) — each client stays on its replica, so single connections are coherent, but two players in one lobby stuck to different replicas never see each other's moves, because no replica relays room events to the others. (backplane, no stickiness) — the bus could relay room events, but without affinity a single client's successive frames scatter across replicas; a replica receives a frame for a connection (and lobby membership) it does not hold, so the connection itself becomes incoherent and handshake/upgrade state breaks. You need both: stickiness for the connection, the backplane for the lobby.
13.5 Answer: X triggers an event in the browser → it travels over X's websocket through the sticky load balancer, which pins X to rmhbox replica 1 → replica 1 applies the event to the lobby it holds and, because the event concerns a room with members elsewhere, publishes it on the Redis backplane → rmhbox replica 2, subscribed to the backplane and holding player Y, receives the published event → replica 2 forwards it down Y's websocket (Y is stuck to replica 2) → Y's browser renders the update. The backplane is the hop that crosses the replica boundary; stickiness guarantees the first and last legs stay on the correct replica.
13.6 Answer: Choose PgBouncer. The stated symptom is connection count — the database is running out of connection slots, not out of read throughput or CPU. PgBouncer directly relieves that: it terminates the fleet's many connections and multiplexes them onto a small, fixed set of real backend connections (transaction pooling returns a backend per-transaction), so the same Postgres tolerates a much larger fleet with no new hardware. Read replicas relieve a different symptom — read load/throughput and read latency — by serving reads from copies; they would not fix slot exhaustion (in fact each replica's clients add connections) and they bring replica-lag/consistency complexity. With connection count as the binding symptom this quarter, PgBouncer is the targeted fix; read replicas come later (Stage 3) when read volume or geographic latency is the problem.
13.7 Answer: An HPA scales a Deployment up or down based on an observed metric (CPU or a custom metric like active connections) between a floor and ceiling — it is, by definition, a control loop over data. If that metric is not collected, the autoscaler has nothing to act on and cannot function; you literally cannot autoscale on a signal you do not have. The same is true of human capacity decisions: justifying "add a node" requires evidence that the current nodes are saturated, which means metrics, logs, and traces. So observability is not parallel to Stage 2 — it is the input Stage 2's autoscaling and every subsequent capacity decision consume. Scaling blind is guessing; observability is what turns it into a decision.
13.8 Answer: Safe from a stale replica: a blog/devlog page render, a research post, or a leaderboard view. These are read-heavy, change slowly, and a few seconds of replication lag is imperceptible — no user action depends on seeing their own just-written value instantly. Not safe: a read that immediately follows the same user's write and must reflect it — e.g. a user posts a message or submits a score and the very next page load must show it (read-your-writes), or any flow where a write and a confirming read are paired. If that read is routed to a lagging replica, the user sees stale data and concludes their write was lost. The distinguishing reasoning is the read-your-writes guarantee: reads with no causal dependence on a recent write by the same session tolerate replica lag; reads that must observe a just-issued write must go to the primary (or use a read-your-writes routing rule), because async replication offers no freshness guarantee.
13.9 Answer: Neither binding constraint is a per-connection-cost problem, which is the only thing a Go rewrite primarily improves. Constraint A is connection count against one Postgres — relieved by PgBouncer pooling and read replicas, both language-agnostic; a Go service still opens connections to the same single database. Constraint B is shared lobby state across replicas — relieved by a Redis backplane and sticky sessions, which a Go realtime tier needs in exactly the same form (the Go performance notes themselves say it "scales horizontally" only "with the Redis backplane enabled" behind "sticky-session ingress"). So the backplane, not the language, is the load-bearing mechanism, and rewriting in Go changes none of the staged work. The narrow justified condition: adopt Go for a specific service only if collected metrics (Stage 2 observability) prove that service is bottlenecked on per-connection cost — memory per idle connection or cold-start — that a single fatter replica cannot absorb. That is the metrics-driven, per-service, strangler-fig condition; absent that data, Go is an optimization in search of a problem, not a scaling strategy.
13.10 Answer: A correct flowchart is a left-to-right chain of four nodes with the relieving pressure labelled on each arrow: 
Stage 0 (single-node, scaled up)
   | --[hits Constraints A and B: realtime pinned at 1 replica;
   |    single DB throttled by connection count]-->
Stage 1 (CDN + Redis backplane + sticky + PgBouncer)
   | --[now realtime scales out and the DB tolerates the fleet,
   |    so spreading across machines finally pays off]-->
Stage 2 (multi-node k3s + HPA + observability)
   | --[machines exhausted near users; geographic latency is the
   |    remaining cost]-->
Stage 3 (regional clusters + latency DNS + read replicas)
The stages cannot be reordered because each depends on the prior one: Stage 2's multi-node and autoscaling are pointless (and unsafe for the realtime tier) until Stage 1 supplies a backplane and pooling; Stage 3's regional reads need Stage 2's observability and the read/write split's groundwork. The labels make the dependency explicit — every arrow names the constraint the next stage exists to relieve.
