Learning Objectives

  • Distinguish vertical from horizontal scaling, and stateless from stateful services, and explain why each combination scales differently.
  • Name the two global-scale constraints that actually bind rmhstudios.com — a single Postgres instance and in-memory per-lobby websocket state — and explain why each one resists naive replication.
  • Explain why a stateful realtime service cannot be served from a CDN and cannot run past one replica without a backplane and sticky sessions.
  • Describe connection pooling (PgBouncer) and the read/write split with read replicas as two independent ways to relieve a single database.
  • Walk the project's agreed three-stage scaling roadmap and say, for each stage, what it adds and which constraint it relieves.
  • Justify why this system scales by reaching for a Redis backplane, a CDN, and database replicas — not by rewriting services in another language.

Every chapter so far has read the system as it is built today: a single-node k3s cluster, one Postgres, a fleet of realtime processes each holding game state in memory. This closing chapter asks the next question — what happens when one node is not enough? — and answers it the way the team has agreed to answer it. Most of what follows is a plan, not shipped code. The discipline of the chapter is to reason from the two real constraints the architecture imposes, derive a staged roadmap that relieves them in order, and resist the tempting-but-wrong move of treating "scale" as a synonym for "rewrite."

Two kinds of scaling, two kinds of service

There are only two ways to make a system handle more load. You can give a machine more resources — a bigger CPU, more memory, faster disk — and let the same single process do more. Or you can run more copies of the process and spread the load across them. The first is vertical scaling; the second is horizontal. Vertical scaling is the easy first move because it changes nothing about the software: bump the VPS to a larger instance and the single-node k3s deploy from Chapter 12 keeps working unchanged. Its ceiling is the largest machine you can rent, and its failure mode is total — one node, one outage.

Vertical scaling (scale up) — increasing the resources (CPU, RAM, disk) of a single machine or pod so one instance handles more load. Simple and fast, but bounded by the largest available machine and offering no redundancy.
Horizontal scaling (scale out) — running more replicas of a service across machines and distributing load among them. Effectively unbounded and fault-tolerant, but it only works cleanly for services that hold no per-request state in memory.

That last clause is the whole problem. Horizontal scaling assumes any replica can serve any request — that the replicas are interchangeable. Whether that assumption holds depends on whether a service is stateless or stateful.

Stateless service — one that keeps no client-specific state in process memory between requests; everything it needs is either in the request itself or in shared backing storage (the database, Redis, object storage). Two replicas of a stateless service are interchangeable, so it scales out trivially.
Stateful service — one that holds live, client-specific state in its own memory (an open game lobby, a connected socket's room membership). A second replica does not share that memory, so two replicas are not interchangeable; scaling out requires extra machinery to make the in-memory state visible across replicas.

Map this onto the rmhstudios fleet from Chapter 1. The web SSR tier and the workers (doctrine-worker, vibe-worker, bot-worker, recap, discord-bot) are essentially stateless with respect to client connections: a page render reads from Postgres and returns HTML, and any replica can render any page. The realtime tier is different. socket, rmhbox, and rmhtube are Socket.io servers that hold per-lobby game state — who is in which room, whose turn it is, the current round — in the memory of the process that owns the connection (Chapters 5–7). That single design fact is what makes scaling this platform interesting rather than routine.

The interchangeability test. Before scaling any service horizontally, ask: "If I kill this replica and a client's next packet lands on a different replica, does anything break?" For web the answer is no — it re-reads from Postgres. For rmhbox the answer is yes — the new replica has never heard of that lobby. That asymmetry drives the entire roadmap below.

The two constraints that actually bind

A scaling discussion can drift into a wishlist. The honest version names the constraints that bite first and treats everything else as secondary. For rmhstudios.com there are exactly two, and both come straight from the architecture as built.

Constraint A — a single Postgres instance

Every tier ultimately reads and writes one Postgres database (Chapter 8). The Helm chart provisions one data volume; the deploy syncs one connection string. As the fleet grows — more web replicas, more realtime replicas, more workers — every new process opens its own pool of connections to that one database. Postgres serves connections from a fixed-size worker pool, and each idle connection still costs memory and a backend process. The database does not fall over because it is slow; it falls over because it runs out of connection slots long before it runs out of CPU. This is the constraint that bites as soon as you scale anything out, because more replicas means more connections to the same database.

Connection pool — a bounded set of reusable database connections shared by an application, so that many requests take turns on a few connections rather than each opening its own. Pooling caps the number of connections the database must service at once.

Constraint B — un-CDN-able, single-replica websockets

The realtime services hold per-lobby state in memory. Two consequences follow, and they are the analytical heart of this chapter. First, they cannot be served from a CDN. A CDN caches and replays a response from an edge node near the user; that only works for content that is the same for everyone and changes slowly (static assets, images, built JS bundles). A websocket is a long-lived, bidirectional, per-connection conversation whose contents are unique to one lobby at one instant — there is nothing to cache and replay. The CDN can sit in front of the static site, but the realtime path must reach a live origin process. Second, and more sharply, each realtime service is pinned to one replica. If you naively run two rmhbox pods behind a load balancer, two players in the same lobby may land on different pods. Pod A holds the lobby; pod B has never heard of it. The players cannot see each other. The deploy README states this constraint in plain terms:

> Stateful realtime note: socket/rmhbox/rmhtube hold in-memory game/lobby
> state. Scaling them past 1 replica needs sticky sessions at the ingress AND a
> shared backplane (e.g. socket.io Redis adapter) — otherwise clients on
> different replicas can't see each other. web and the workers scale freely;
> the realtime tier needs that backplane first.

So the production chart sets every realtime service to replicas: 1, and the prod values file comments that web "is the scale target" while the realtime tier is held at one. That is not an oversight; it is the architecture telling you the order in which scaling work must happen. You cannot scale the realtime tier out until you give it a way to share state across replicas — a backplane — and a way to keep each conversation on one replica — sticky sessions.

Backplane (Redis pub/sub) — a shared message bus (here, the Socket.io Redis adapter) that every replica of a realtime service subscribes to. When one replica receives an event for a room, it publishes it on the backplane; the other replicas holding members of that room receive it and forward it to their own clients. The backplane makes replicas cooperate so an event reaches everyone in a logical room regardless of which replica holds each connection.
Sticky session (session affinity) — a load-balancer policy that pins a given client to the same backend replica for the life of its connection (by cookie or source-hash). It guarantees a client's many websocket frames all hit the replica that already holds its connection, rather than scattering across replicas mid-conversation.
Backplane and sticky sessions are complementary, not redundant. Sticky sessions keep one client on one replica (so a single connection is coherent). The backplane lets different clients in the same lobby, who may be stuck to different replicas, still exchange events. You need both: stickiness for connection integrity, the backplane for cross-replica room coherence.

Stage 1 — relieve the two constraints in place

The first real step does not add nodes or regions. It removes the two constraints from a single-node deploy that has been scaled up as far as is reasonable, so that the realtime tier can finally run more than one replica and the database stops being throttled by connection count. Three changes, each targeting one pressure.

A CDN in front of the static surface

The static half of the site — built JS bundles, CSS, images, the marketing and blog pages that change rarely — is moved behind a CDN. The team already fronts DNS with Cloudflare (the Terraform cloudflare_record resources support a proxied flag), so the edge is one configuration away. The CDN absorbs the bulk of byte-volume traffic at the edge, near users, and only cache misses and the dynamic SSR routes reach the origin. Crucially, the websocket hostnames (socket., box., tube.) are not proxied through the cache — they must reach a live origin, per Constraint B.

CDN / edge cache — a network of geographically distributed cache nodes that store copies of static responses close to users. The first request for an asset fills the cache from the origin; subsequent requests are served from the nearest edge node, cutting latency and offloading the origin. Effective only for content identical across users.

A Redis backplane plus sticky sessions for the realtime tier

This is the change that unlocks horizontal scaling of socket, rmhbox, and rmhtube. Enable the Socket.io Redis adapter so the replicas share a pub/sub bus, and configure the ingress for session affinity so each client stays on its replica. With both in place, the prod values file's replicas: 1 can rise:

kubectl -n rmhstudios scale deploy/rmhstudios-socket --replicas=3
kubectl -n rmhstudios scale deploy/rmhstudios-rmhbox --replicas=3

Each new replica registers on the backplane; a lobby event published by one replica reaches members held by the others. The realtime tier moves from "pinned at one" to "scales out like any other service" — but only because the backplane made its in-memory state effectively shared.

PgBouncer in front of Postgres

With more replicas of everything, the connection count against the single Postgres climbs. PgBouncer is a lightweight connection pooler that sits between the application fleet and Postgres: the many application connections terminate at PgBouncer, which multiplexes them onto a small, fixed set of real backend connections. In transaction pooling mode, a backend connection is borrowed only for the duration of a transaction and returned immediately, so hundreds of mostly-idle application connections share a handful of real ones. This directly relieves Constraint A without touching the database's size: the same Postgres now tolerates a much larger fleet because it never sees more than PgBouncer's pool of real connections.

PgBouncer — a lightweight PostgreSQL connection pooler. Applications connect to PgBouncer instead of directly to Postgres; it maintains a small pool of real backend connections and hands them out per-transaction, so a large, mostly-idle client fleet is served by far fewer database connections.
Why Stage 1 comes first. It is the cheapest stage — no new nodes, no new regions — and it removes the two binding constraints. After Stage 1 the realtime tier scales horizontally and the database survives the larger fleet. Only now does adding nodes make sense, because there is finally something that benefits from more of them.

Stage 2 — multi-node k3s, autoscaling, observability

With the constraints relieved, the deploy can spread across multiple nodes. The infrastructure is already wired for this, which is the point of the registry seam from Chapter 12: setting the REGISTRY environment variable flips deploy-k8s.sh from importing the image into one node's containerd (pullPolicy: Never) to pushing it to a registry that every node pulls from (pullPolicy: IfNotPresent). Two storage seams, both flagged MULTI-NODE SEAM in the chart, must also be addressed: the local-path data volume is node-local ReadWriteOnce and must move to a ReadWriteMany class (NFS/Longhorn/Rook) for pods to share /app/db across nodes, and the discord-bot host-path repo mount must be pinned with a nodeSelector or moved onto the shared volume.

Once load is spread, replica counts should follow demand rather than being set by hand. A Horizontal Pod Autoscaler watches a metric (CPU, or a custom metric like active connections) and adjusts replica count between a floor and a ceiling automatically. The stateless tiers — web and the workers — are the natural first targets; the realtime tiers can join only because Stage 1 gave them a backplane and stickiness.

Horizontal Pod Autoscaler (HPA) — a Kubernetes controller that automatically scales a Deployment's replica count up or down based on observed metrics (CPU utilization or custom metrics), within configured minimum and maximum bounds, so capacity tracks load without manual intervention.

None of this autoscaling is safe to enable blind. You cannot scale on a metric you do not collect, and you cannot trust an HPA you cannot watch. That is why observability is a Stage-2 prerequisite, not a nicety: metrics (request rate, connection count, pool saturation), logs (aggregated across replicas and nodes), and traces (a request followed across services). The Go-migration reference implementation already emits /metrics endpoints; the operational work is a Prometheus scrape config, dashboards, and alerts. Observability is listed alongside the autoscaling work precisely because it is the input the autoscaler — and every human decision after it — depends on.

Observability — the practice of instrumenting a system with metrics, logs, and traces so its internal state can be inferred from the outside. It is a prerequisite for scaling: autoscaling needs metrics to act on, and capacity decisions need data to be made rather than guessed.

Stage 3 — regional clusters, latency DNS, read replicas

The final stage attacks the last fixed cost: the speed of light. A user far from the single VPS pays round-trip latency on every request. Stage 3 stands up regional clusters — a full k3s deploy in more than one geography — and uses latency-based DNS to route each user to the nearest healthy region. The Terraform DNS surface is provider-agnostic and already the seam where this policy would live; today it returns one A record per host, and latency routing replaces that with a geo-aware answer.

Latency-based DNS — a DNS routing policy that resolves a hostname to different IP addresses depending on which region can serve the requester with the lowest measured latency, steering each user to the nearest healthy cluster.

Multiple regions reopen Constraint A in a new form: there is still one writable Postgres, and a region far from it pays cross-region latency on every query. The standard answer is the read/write split with read replicas. The primary database takes all writes and streams its changes to read-only replicas placed near each region. Read-heavy queries (page renders, feeds, leaderboards) are served by the local replica; writes still travel to the single primary. Most web traffic is reads, so this removes most of the cross-region penalty while keeping a single source of truth for writes.

Read replica — a read-only copy of a database that asynchronously follows a primary's write stream. Read queries are directed to replicas (often a nearby one) to spread read load and reduce latency; all writes still go to the single primary.
Read/write split — routing read-only queries to replicas and write queries to the primary, so the primary's capacity is reserved for writes while reads scale across replicas.
Replicas are not free consistency. A read replica lags the primary by some replication delay. A user who writes and immediately reads from a different replica can see stale data ("read-your-writes" violation). The realtime tier sidesteps this — it broadcasts over the backplane rather than polling the database — but ordinary web reads must tolerate, or route around, replica lag. Splitting reads from writes buys throughput at the cost of a weaker consistency guarantee, and that trade has to be made deliberately per query.

The roadmap, stage by stage

The three stages are ordered by cost and by dependency: each relieves a constraint that the next stage would otherwise hit, and each is only safe once the prior one is in place.

StageWhat it addsWhy — which constraint it relieves
1 — Backplane & pooling CDN for static assets; Socket.io Redis backplane + sticky sessions; PgBouncer in front of Postgres. Unblocks Constraint B (realtime can finally run multiple replicas) and Constraint A (the single DB tolerates a larger fleet). No new nodes — cheapest first step.
2 — Multi-node & autoscale Multi-node k3s via the REGISTRY seam + RWX storage; Horizontal Pod Autoscalers; metrics/logs/traces. Spreads load across machines and makes capacity track demand. Observability is the prerequisite — you cannot autoscale on metrics you do not collect.
3 — Multi-region Regional k3s clusters; latency-based DNS; Postgres read replicas with a read/write split. Attacks geographic latency and reopens Constraint A across regions; read replicas serve local reads while writes stay on the single primary.

Why scaling here is not a language rewrite

A recurring temptation, when a realtime system needs to scale, is to rewrite it in a faster language — and the team did build a complete Go reference implementation of the realtime tier (Chapter on the Go migration). It is important to be precise about what that does and does not buy. A Go realtime service uses less memory per connection and starts faster, so a single replica can hold more idle connections. But notice that none of the two binding constraints is a per-connection-cost problem. Constraint A is connection count against one database — relieved by PgBouncer and read replicas, which are language-agnostic. Constraint B is shared room state across replicas — relieved by a Redis backplane and sticky sessions, which a Go rewrite still needs in exactly the same form. The Go performance notes say as much: the realtime tier "scales horizontally" only "with the Redis backplane enabled," behind "sticky-session ingress." The backplane is the load-bearing mechanism in either language.

That is why the project declined the wholesale Go rewrite as a scaling strategy and adopted the strangler-fig stance from Chapter 2 instead: keep the proven Node tier in production, and adopt Go per-service later, only if collected metrics prove a specific service is bottlenecked on per-connection cost. The rewrite is a possible optimization of one stage's constituent services, not a substitute for the staged roadmap. Scaling this system means a backplane, a CDN, connection pooling, and database replicas — applied in order, justified by observability — regardless of what language the services are written in.

Scope: this is a roadmap, not a shipped system. As of this writing the production deploy is single-node k3s + Helm + Terraform with every realtime service at replicas: 1 and one Postgres, kept reversibly alongside Docker Compose (PR #121). The multi-node path is wired (the REGISTRY env var and the MULTI-NODE SEAM storage flags exist) but unused; the Redis backplane, sticky-session ingress, PgBouncer, HPAs, observability stack, regional clusters, latency DNS, and read replicas are planned, not deployed. No load testing has been performed, so the ordering here is argued from the architecture's constraints, not from measured benchmarks. Treat the stages as the team's agreed direction and the proof points (load tests, metrics baselines) as explicit future work.

Summary

  • Vertical scaling grows one machine; horizontal scaling adds replicas. Horizontal scaling is clean only for stateless services whose replicas are interchangeable.
  • Two constraints bind this system: (A) a single Postgres, throttled by connection count; (B) realtime services that hold per-lobby state in memory, which makes them un-CDN-able and pinned to one replica.
  • A backplane (Redis pub/sub) plus sticky sessions together let the realtime tier scale out: stickiness keeps a connection coherent, the backplane keeps a lobby coherent across replicas.
  • Stage 1 relieves both constraints in place (CDN + backplane/sticky + PgBouncer). Stage 2 adds multi-node k3s, HPAs, and observability. Stage 3 adds regional clusters, latency DNS, and read replicas with a read/write split.
  • Scaling this platform is achieved with backplanes, edge caches, pooling, and replicas — not a language rewrite. A Go realtime tier still needs the same backplane and stickiness, so Go is a per-service optimization gated on metrics, not a scaling plan.
