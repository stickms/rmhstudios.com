  • Describe toolchain pinning — how Bazel 7.4.1 and a hermetic Go 1.23 SDK are nailed down so no system Go is required.
  • Explain the monorepo idea: one repository, many deployables, with pnpm workspaces and a single Bazel module.
  • Read a Bazel BUILD target and explain how Gazelle generates the Go ones automatically.
  • Trace how a service becomes a distroless container image through rules_oci and rules_pkg.
  • Explain why the frontend is a coarse leaf and what that buys: new frontend files need no BUILD edits.

Every other chapter has treated the codebase as something that already runs. This chapter asks the duller but load-bearing question: how does source on a developer's laptop become a Go binary, a JavaScript bundle, and a container image that a cluster can pull and start? rmhstudios.com answers that question with two deliberately different tools. The Go service fleet is built by Bazel, which goes to extreme lengths to be hermetic and reproducible — it ships its own Go compiler so that no two machines disagree about what "build" means. The TypeScript frontend is built by Vite through a thin shell script that reuses whatever node_modules the developer already has — fast and convenient, deliberately not hermetic. Understanding why the same repository chooses both, and where the seam between them sits, is the point of this chapter.

What a build system does, and why

A build turns sources into runnable artifacts: compiling, linking, bundling, generating code, and packaging. For a one-file program you do this by hand. For rmhstudios.com — nine Go binaries, a TypeScript SSR app, nine standalone WebSocket/worker servers, a generated Prisma client, and per-service container images — doing it by hand is hopeless. You need a tool that knows the dependency graph between artifacts so it can build them in the right order, skip work whose inputs have not changed, and produce the same result on your laptop and in CI.

Build system — a tool that, given a description of artifacts and their dependencies, computes and executes the minimal set of actions to produce up-to-date outputs from sources. It owns ordering, caching, and incrementality so humans do not have to.

The repository draws a clean line. Bazel owns the hermetic core: the Go binaries and the OCI images. A top-level Makefile is the ergonomic face — a thin set of human-friendly targets that mostly delegate to Bazel, plus a few edges Bazel does not touch (the frontend leaf, the special base images, the Helm deploy). The operator runbook states the split plainly: "Bazel (via bazelisk) owns the hermetic core (Go binaries + container images); this Makefile is the ergonomic face and orchestrates the edges."

Hermetic and reproducible builds

The strongest idea in this chapter is hermeticity. A build is hermetic when its inputs are fully declared and pinned, so it does not depend on whatever happens to be installed on the machine running it. A reproducible build goes one step further: the same inputs always yield byte-for-byte the same outputs.

Hermetic build — a build whose every input (compiler, libraries, source) is declared and version-pinned, so it ignores the host's installed tools and produces the same result everywhere. The opposite — a non-hermetic build — reaches out to whatever the host provides (a system Go, the developer's node_modules) and can therefore differ machine to machine.
Toolchain pinning — fixing the exact version of each tool a build uses (here: Bazel 7.4.1 and a downloaded Go 1.23 SDK) so upgrades are explicit and intentional rather than accidental drift from whatever a laptop has on its PATH.

Bazel achieves this in three layers. First, the version of Bazel itself is pinned. The repo does not call bazel directly; it calls bazelisk, a launcher that reads .bazelversion and downloads exactly that release. Both files agree on the version:

# .bazelversion
7.4.1

# .bazeliskrc
USE_BAZEL_VERSION=7.4.1

Second, the Go compiler is pinned and downloaded by Bazel — not taken from the host. MODULE.bazel declares the rule sets and asks rules_go to fetch a specific Go SDK:

# MODULE.bazel (excerpt)
bazel_dep(name = "rules_go", version = "0.50.1")
bazel_dep(name = "gazelle", version = "0.39.1")
bazel_dep(name = "rules_oci", version = "2.0.0")
bazel_dep(name = "rules_pkg", version = "1.0.1")

go_sdk = use_extension("@rules_go//go:extensions.bzl", "go_sdk")
go_sdk.download(version = "1.23.0")

Because rules_go ships the toolchain, the runbook can make a striking claim: "Go is NOT required locally — rules_go ships a hermetic Go 1.23 SDK." A contributor can build all nine Go services without ever installing Go. Third, .bazelrc tightens the sandbox so actions cannot silently read the host environment, and sets a shared disk cache so unchanged work is reused:

# .bazelrc
common --enable_bzlmod
build --disk_cache=~/.cache/bazel-disk
build --incompatible_strict_action_env
test --test_output=errors

# Cross-compile Go services for Linux containers.
build:linux_amd64 --platforms=@rules_go//go/toolchain:linux_amd64
build:linux_arm64 --platforms=@rules_go//go/toolchain:linux_arm64

--incompatible_strict_action_env scrubs the environment passed to build actions so a stray variable on one laptop cannot change the output; --enable_bzlmod turns on the modern MODULE.bazel dependency system. The two build:linux_amd64 / build:linux_arm64 config groups let the same Go source be cross-compiled for either container architecture by passing --config=linux_amd64 at build time — the developer's CPU is irrelevant.

Why pin so hard? The payoff is that "works on my machine" stops being a category of bug. CI, a teammate's laptop, and a release box all run Bazel 7.4.1 and Go 1.23, so a green build in one place is a green build everywhere. Pinning makes upgrades a deliberate one-line change rather than invisible drift.

The monorepo: one repo, many deployables

rmhstudios.com is a monorepo: a single Git repository holding the web app, the Go fleet, the Node WebSocket servers, shared libraries, the database schema, and the deploy configuration. The alternative — a repo per service — trades atomic cross-cutting changes for independent versioning. A monorepo keeps everything in lockstep: one commit can change a shared type and every consumer of it at once.

Monorepo — one version-controlled repository that contains many independently deployable units (here: the SSR web app, nine Go services, nine Node servers/workers). It favors atomic cross-service changes and a single source of truth over the isolation of many small repos.

On the JavaScript side, the monorepo is held together by pnpm workspaces. The pnpm-workspace.yaml file declares which built dependencies are allowed to run install-time scripts and which are skipped — a security-and-speed control over native packages:

# pnpm-workspace.yaml (excerpt)
ignoredBuiltDependencies:
  - '@prisma/engines'
  - prisma
  - sharp
  - unrs-resolver
onlyBuiltDependencies:
  - esbuild
  - '@swc/core'

A companion .npmrc hoists the generated Prisma client to the top-level node_modules/.prisma/ so the production server bundle can resolve it at runtime — a concrete example of monorepo plumbing the build must get right:

# .npmrc
public-hoist-pattern[]=.prisma*

On the Go side, the whole module lives under //go-services with a single go.mod (module github.com/rmhstudios/rmh-go, go 1.23), and Bazel treats the entire repository as one module named rmhstudios. One repo, but the build can target any subset of it: //go-services/cmd/... for the binaries, //:frontend for the web app.

BUILD targets and Gazelle

Bazel describes the world as targets written in BUILD.bazel files. A target names an artifact and lists its sources and dependencies. Here is a real, hand-trimmed one for the gateway service:

# go-services/cmd/gateway/BUILD.bazel
go_library(
    name = "gateway_lib",
    srcs = ["main.go"],
    importpath = "github.com/rmhstudios/rmh-go/cmd/gateway",
    deps = [
        "//go-services/internal/gateway",
        "//go-services/pkg/auth",
        "//go-services/pkg/config",
        # ...
    ],
)

go_binary(
    name = "gateway",
    embed = [":gateway_lib"],
    visibility = ["//visibility:public"],
)
BUILD target — a named unit in a BUILD.bazel file (e.g. //go-services/cmd/gateway:gateway) that declares an artifact, its sources, and its dependencies on other targets. Bazel builds the dependency graph these targets form.

Writing that deps list by hand for every package would be tedious and error-prone — exactly the kind of bookkeeping a tool should do. Gazelle reads the Go source, follows its import statements, and generates and updates the go_library / go_binary targets automatically. The root BUILD.bazel wires it up:

# BUILD.bazel (root)
load("@gazelle//:def.bzl", "gazelle")

# gazelle:exclude node_modules
gazelle(name = "gazelle")
Gazelle — a Bazel tool that reads Go source and generates/updates the corresponding BUILD.bazel targets (libraries, binaries, dependency lists) so developers do not maintain them by hand. Run via make gazelle after adding or moving Go files.

So the Go workflow is: add a .go file, run make gazelle, and the BUILD files catch up. The gazelle:exclude node_modules directive keeps it from wandering into the JavaScript tree. The MODULE.bazel even notes that the external Go dependency list is "populated by bazelisk mod tidy" — another generation step rather than hand-editing.

From binary to distroless image

A Go binary is not deployable on its own; the cluster wants a container image. Bazel builds those too, via rules_pkg (to tar the binary into a layer) and rules_oci (to assemble the image). The repository factors the repetition into one reusable macro, go_service_image, in bazel/defs.bzl:

# bazel/defs.bzl (excerpt)
def go_service_image(name, binary, base = "@distroless_base", ...):
    pkg_tar(
        name = name + "_layer",
        srcs = [binary],
        package_dir = "/usr/local/bin",
    )
    oci_image(
        name = name + "_image",
        base = base,
        tars = [":" + name + "_layer"],
        entrypoint = ["/usr/local/bin/" + bin_name],
        env = env or {},
    )
    oci_load(name = name + "_load",
        image = ":" + name + "_image",
        repo_tags = ["rmhstudios-go-" + name + ":dev"])
    oci_push(name = name + "_push",
        image = ":" + name + "_image",
        repository = registry_repo or
            ("ghcr.io/rmhstudios/rmhstudios-go-" + name))

For each service the macro generates three targets: <name>_image (the image), <name>_load (load it into local Docker, tagged rmhstudios-go-<name>:dev), and <name>_push (push it to a registry, defaulting to ghcr.io/rmhstudios/rmhstudios-go-<name>). The call sites live in go-services/images/BUILD.bazel, kept in their own package so Gazelle's generated cmd/* files stay untouched.

The base image is distroless. Seven of the nine services sit on @distroless_base, pinned by digest in MODULE.bazel to gcr.io/distroless/static-debian12. A distroless image contains the application and its runtime files but no shell, no package manager, and no general-purpose userland — a smaller, harder-to-attack surface.

Distroless image — a container base that ships only what the program needs to run (libraries, CA certs) and omits a shell and package manager. Smaller and more secure than a full OS image, at the cost of being harder to poke at interactively.

Two services need more than the static base, and the macro's base argument handles the exceptions: vibe-worker runs on @base_chromium (headless Chromium, with PLAYWRIGHT_CHROMIUM_EXECUTABLE_PATH set in the image env) and discord-bot runs on @base_git. Those two bases are not pinned by digest — they are pulled by tag from GHCR and built separately by make base-images, which is why the runbook warns you to build them first.

The frontend: a coarse, non-hermetic leaf

The TypeScript frontend is built differently on purpose. Rather than model every .tsx file as a Bazel target, the root BUILD.bazel exposes the whole frontend as a single coarse target that shells out to pnpm:

# BUILD.bazel (root)
# Coarse frontend leaf — `bazel run //:frontend` builds the Vite app via
# pnpm. Non-hermetic by design (reuses repo node_modules).
sh_binary(
    name = "frontend",
    srcs = ["scripts/frontend-build.sh"],
    visibility = ["//visibility:public"],
)

The script it runs is small and honest about being non-hermetic. It changes into the source tree, installs dependencies and generates the Prisma client only if node_modules is absent, then builds:

# scripts/frontend-build.sh (excerpt)
cd "${BUILD_WORKSPACE_DIRECTORY:-$PWD}"
if [ ! -d node_modules ]; then
  pnpm install --frozen-lockfile --ignore-scripts
  pnpm exec prisma generate
fi
pnpm run build:frontend   # → .output/
Coarse leaf — a single Bazel target that wraps an entire external build (here, the whole Vite frontend) as one opaque step, instead of modeling its internal files as targets. It trades fine-grained incrementality for the freedom to add files without editing BUILD definitions.

The phrase to remember from the runbook is: "New frontend files need no BUILD changes." Because the leaf is opaque, a developer can add a hundred React components and never open a BUILD.bazel file. The cost is that this step is not hermetic — it reuses the repo's node_modules (which include native packages like @napi-rs/canvas and @resvg/resvg-js) and runs in the source tree, so Bazel cannot guarantee the byte-for-byte reproducibility it gives the Go side. The team accepts that trade deliberately: the JS ecosystem's native-dependency reality makes a fully hermetic frontend expensive, and the frontend is a leaf with no Bazel consumers downstream.

What Vite and esbuild actually produce

Under that leaf, two bundlers do the real work. Vite (configured in vite.config.ts) builds the SSR web app: it runs the TanStack Start plugin, Tailwind, React, and the Nitro server plugin, emitting a runnable server into .output/. build:frontend in package.json is just build-vibe-packages followed by vite build with an 8 GB Node heap.

Bundler — a tool that resolves a module graph and packs many source files into a few optimized output files. Vite (on Rolldown) bundles the web app and its SSR server; esbuild bundles the standalone Node WebSocket/worker servers.

The nine standalone WebSocket/worker servers under server/* are not built by Vite. The full build script in package.json first runs vite build, then invokes esbuild directly to bundle each server entry into dist-server/ as a CommonJS .cjs file:

# package.json "build" (esbuild stage, trimmed)
pnpm exec esbuild \
  server/socket-server/index.ts server/rmhbox/index.ts \
  server/rmhtube/index.ts server/recap/index.ts \
  server/discord-bot/index.ts server/vibe-worker/index.ts ... \
  --bundle --platform=node --target=node20 \
  --outdir=dist-server --format=cjs \
  --out-extension:.js=.cjs --packages=external \
  --tree-shaking=true --tsconfig=tsconfig.server.json

So the frontend leaf hides two distinct bundlers with different jobs: Vite/Nitro for the browser app and its SSR server (.output/), and a separate esbuild pass for the long-lived realtime/worker processes (dist-server/). Vite externalizes heavy native and client-only packages (three.js, monaco-editor, Tone.js, and others) via traceDeps and ssr.external so they are not pulled into the wrong bundle.

Scope: the leaf vs. the core. The Go services are fully Bazel-built and hermetic; that path is production. The frontend leaf is a real, used target (make build calls bazel run //:frontend), but it is intentionally non-hermetic and exists mostly to give the frontend a single Bazel entry point — its actual work is delegated to pnpm/Vite/esbuild, which the same scripts drive in CI and in the Dockerfile.

Putting it together: the Makefile

The Makefile is where a human meets all of this. Its targets are short and mostly delegate. The key ones:

# Makefile (excerpt)
build:   ## all Go binaries + frontend bundle
	@$(GUARD)
	$(BAZEL) build //go-services/cmd/...
	$(BAZEL) run //:frontend

images:  ## build + load every service image into Docker
	@for svc in $(SERVICES); do \
	  $(BAZEL) run --config=$(PLATFORM) //go-services/images:$${svc}_load; \
	done

gazelle: ## regenerate Go BUILD.bazel files
	$(BAZEL) run //:gazelle

Notice the division of labor in build: bazel build //go-services/cmd/... compiles the hermetic Go core, and bazel run //:frontend fires the non-hermetic leaf — the two worlds, one command. images loops over the nine services, passing --config=$(PLATFORM) (defaulting to linux_amd64) so each image is built for the target container architecture. A guard, ./scripts/preflight.sh --guard, runs first and hard-fails if Docker is missing, since no image can be built without it. Note make dev is the exception — it runs pnpm run dev and Bazel is not involved at all.

Summary

  • A build system owns ordering, caching, and incrementality so artifacts come from sources reliably. rmhstudios.com splits the job: Bazel for the hermetic core, a Makefile as the ergonomic face.
  • Hermetic builds pin every input. Bazel is pinned to 7.4.1 via bazelisk, and rules_go ships a Go 1.23 SDK, so no system Go is required and outputs match across machines.
  • It is a monorepo: one repo, many deployables, with pnpm workspaces on the JS side and a single Bazel module plus one go.mod on the Go side.
  • Gazelle generates the Go BUILD targets from imports, so adding a Go file means running make gazelle, not hand-editing deps.
  • The go_service_image macro turns each binary into a distroless OCI image via rules_pkg + rules_oci, exposing _load and _push targets; vibe-worker and discord-bot override the base.
  • The frontend is a deliberately coarse, non-hermetic leaf (//:frontend → scripts/frontend-build.sh → Vite + a separate esbuild pass). New frontend files need no BUILD changes.
