Learning Objectives

  • Distinguish local component state, client state in a global store, and server state held in a cache, and pick the right tool for a given piece of data.
  • Read a real Zustand store built with create(...) and explain how its slice of state, its actions, and its selectors fit together.
  • Explain the hydration boundary — how server-rendered HTML becomes a live client tree — and why a mismatch between the two causes a "flash" or a hydration warning.
  • Describe how TanStack React Query turns a fetch into a cached, deduplicated, auto-refetching query cache entry, and when staleTime matters.
  • Trace an optimistic update through the feed store and say what guarantees it makes (and breaks) versus a server round-trip.
  • Justify why rmhstudios keeps durable server data and ephemeral client state in separate systems rather than one global blob.

Every interactive page is a negotiation between three kinds of memory: the state that lives and dies inside one component, the state that the whole browser tab shares, and the state that really belongs to the server and is only borrowed by the client. This chapter reads how rmhstudios.com draws those lines — with React's own hooks for the first, Zustand stores for the second, and TanStack React Query for the third — and how all of it is stitched onto server-rendered HTML at the hydration boundary.

Three kinds of state

It is tempting to think of "state" as one undifferentiated bag of variables. In practice a production frontend has at least three populations of state with very different lifetimes and owners, and conflating them is the source of a large fraction of frontend bugs.

Local component state — data scoped to a single React component instance, created with useState/useReducer and discarded when the component unmounts. Example: whether a dropdown is open.
Client state — data that is owned by the browser session and shared across many components, but has no authoritative copy on the server. Example: which of the 30 site themes the visitor picked, or the live HP of the game you are playing.
Server state — data whose source of truth is the database behind an API. The client never owns it; it holds a possibly-stale cache of it. Example: the puzzle leaderboard, your reputation score, today's daily puzzles.

The tells are ownership and freshness. If unplugging the network can never make the value "wrong," it is client state. If the value can drift out of date because someone else changed the database, it is server state and needs a cache with a refetch policy — not a hand-rolled variable. rmhstudios uses one library per category, as its README.md "Tech Stack" line states plainly: "State: Zustand, TanStack React Query."

Local state and custom hooks

The cheapest state is the kind you never share. When a value only matters to one component, React's own useState is the whole answer. In the puzzle page app/routes/strategies/puzzles/$mode.tsx, the "play again" replay puzzle and a remount key are exactly this — local, throwaway, owned by the page:

const [replayPuzzle, setReplayPuzzle] = useState<PuzzleData | null>(null);
const [replayKey, setReplayKey] = useState(0);

When a pattern of local state recurs, the repo extracts it into a custom hook under hooks/ — a plain function whose name starts with use and that calls other hooks. These are not global state; each call gets its own independent state. hooks/useDebounce.ts is a tidy example that delays a value until typing stops:

export function useDebounce<T>(value: T, delay: number): T {
  const [debouncedValue, setDebouncedValue] = useState(value);
  useEffect(() => {
    const timer = setTimeout(() => setDebouncedValue(value), delay);
    return () => clearTimeout(timer);
  }, [value, delay]);
  return debouncedValue;
}

hooks/useLocalStorage.ts is subtler: it is local state that persists across reloads by mirroring into localStorage. Crucially, its comment notes it is "SSR-safe — returns initialValue on the server, hydrates from localStorage on mount." That phrase is the whole next section in miniature: the server cannot read the browser's localStorage, so the first render must use the default, and the real value is read inside a useEffect that only runs in the browser.

Why useEffect for storage reads? Effects run only on the client, after the first paint. Reading localStorage directly in the render body would either crash on the server (no localStorage global) or produce HTML that disagrees with the client's first render. Deferring the read to an effect keeps the first render identical on both sides.

Global client state with Zustand

When many unrelated components need the same value — the whole app's theme, or every HUD widget reading the player's HP — lifting state up through props becomes unworkable. rmhstudios reaches for Zustand, a tiny store library. A store is created with the create function, which takes a setup callback receiving set (to write) and get (to read), and returns a React hook you call from any component.

Store — a single object holding a slice of client state plus the actions (functions) that mutate it, created once at module load and shared by every component that imports its hook. In Zustand the store hook doubles as a selector: pass a function and the component re-renders only when that selected slice changes.

The smallest store in stores/ is themeStore.ts. After a long SITE_STYLES table (the 30 named themes), the store itself is four lines:

export const useThemeStore = create<ThemeStore>((set) => ({
  style: "default",
  setStyle: (style) => set({ style }),
}));

A component selects just the field it needs — const style = useThemeStore((s) => s.style) in components/Providers.tsx — so changing an unrelated field would not re-render it. This is the payoff of a store over React context: fine-grained subscriptions instead of re-rendering every consumer on any change.

A real game store: useAltairGameStore

Games are where global client state earns its keep, because dozens of systems read and write the same run. lib/altair/stores/game-store.ts holds the entire per-run session: phase, hp, xp, level, kills, coins, the chosen weapons and passives, and a long list of actions. Its header comment is explicit that this is "per-run state (NOT persisted)" — it is wiped between runs, which is why the store has a goToMenu action that resets every field back to its starting value.

The actions show the two ways Zustand lets you write state. The simplest take a fresh value; others read the current state with get() or with the functional form of set to derive the next value from the previous one:

takeDamage: (amount) => {
  const s = get();
  const reduced = Math.max(1, amount - s.effectiveStats.armor);
  const newHp = Math.max(0, s.hp - reduced);
  set({ hp: newHp });
  if (newHp <= 0) {
    if (s.revivalsRemaining > 0) {
      get().revive();
    } else {
      get().die();
    }
  }
},

heal: (amount) => {
  set((s) => ({ hp: Math.min(s.maxHp, s.hp + amount) }));
},

Notice takeDamage calls get().revive() and get().die() — actions can invoke other actions, so the store encapsulates not just data but the rules that govern it (armor reduces damage, zero HP triggers a revive or death). The HUD does not implement any of that logic; it only reads hp and renders a bar. That separation — fat store, thin component — is the point.

Combo / derived state — the game store keeps a kills counter and a separate totalKills, and an addKill action that bumps both then calls checkKillMilestones() to award coins at thresholds (500, 1500, 3000, 5000). Derived rewards live in the store, not the component, so every caller sees consistent milestone behaviour.

Persisting client state: the persist middleware

Some client state should outlive the tab. Zustand ships a persist middleware that serialises a store to localStorage automatically. The Altair settings store (lib/altair/stores/settings-store.ts) wraps its setup in persist(...) and declares both a storage key and a partialize function that whitelists which fields to save:

export const useAltairSettingsStore = create<AltairSettingsState>()(
  persist(
    (set, get) => ({
      theme: 'dark',
      keybinds: { ...DEFAULT_KEYBINDS },
      /* ...volumes, toggles, actions... */
    }),
    {
      name: 'altair-settings',
      partialize: (s) => ({
        theme: s.theme,
        keybinds: s.keybinds,
        masterVolume: s.masterVolume,
        /* ...only the durable fields... */
      }),
    }
  )
);

The sibling meta-store.ts goes one step further: its header notes it is "persisted to localStorage + synced to DB," and an action calls a save endpoint with keepalive on unload. That is the boundary case where a store straddles client and server state — the localStorage copy is the fast, offline-friendly cache, and the database is the durable backup. It is worth flagging honestly:

The Altair meta-store is the one store that blurs the client/server line on purpose: durable progression (coins, unlocked classes, bestiary) is held client-side for instant reads and pushed to the database opportunistically. Treat it as the exception that proves the rule, not the pattern to copy for ordinary server data.

Server state with TanStack React Query

Data that lives in the database needs different machinery. The client should fetch it, remember it briefly so the next component doesn't re-fetch, refresh it when it goes stale, and re-render when it arrives — all without hand-written loading flags scattered across components. That is exactly what TanStack React Query provides.

Query cache — React Query's in-memory store keyed by a query key (an array like ['doctrine', 'reputation']). The first useQuery with a given key runs its queryFn; later mounts with the same key get the cached value instantly and only refetch when it is older than staleTime. Identical keys mounted at once are deduplicated into a single network request.

The cache is configured once, at the app root, by wrapping the tree in a QueryClientProvider. In components/Providers.tsx a single QueryClient sets sensible defaults:

const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 60_000, // 1 minute
      retry: 1,
    },
  },
});

That Providers component is mounted by app/routes/__root.tsx around the router's <Outlet />, so every route in the app shares one cache. A query itself is just a hook. hooks/useDoctrineReputation.ts wraps a fetch in useQuery, supplying a key, an async queryFn, and per-query overrides:

export function useDoctrineReputation() {
  return useQuery({
    queryKey: ['doctrine', 'reputation'],
    queryFn: async () => {
      const res = await fetch('/api/doctrine/reputation');
      if (!res.ok) throw new Error('Failed to fetch reputation');
      return res.json();
    },
    staleTime: 60_000,
    refetchInterval: 120_000,
  });
}

The hook returns the familiar { data, isLoading, error } shape; the component just reads data. The refetchInterval of two minutes means the leaderboard slowly self-heals as other players' scores change — a polling cadence chosen per query, not globally. A query key can carry parameters: useDoctrineReputationLeaderboard(limit) keys on ['doctrine', 'reputation', 'leaderboard', limit], so different limits are cached separately. The puzzle page reads today's puzzles the same way and then layers local replay state on top — a clean illustration of server state and client state coexisting in one component:

const { data: puzzles } = useQuery({
  queryKey: ['doctrine', 'puzzles', 'today'],
  queryFn: async () => {
    const res = await fetch('/api/doctrine/puzzles/today');
    return res.json();
  },
  staleTime: 60_000,
});
const dailyPuzzle = puzzles?.find((p) => p.mode.toLowerCase() === mode);
const activePuzzle = replayPuzzle ?? dailyPuzzle;
Reads vs. writes in this codebase. rmhstudios uses React Query's useQuery for reads, but it does not use the matching useMutation helper for writes. Writes go through a plain fetch and then update a Zustand store directly — as the feed does below. So the "optimistic update" pattern here is implemented in the store, not via React Query's mutation cache.

Optimistic updates in the feed store

When you post an "RMHark," waiting for the server before showing it would feel sluggish. The feed applies the change to the UI first, assuming success, and lets the server confirm afterward.

Optimistic update — a write that immediately mutates local state to the expected post-write value, before (or instead of) waiting for the server's response, trading a small risk of having to roll back for a UI that feels instant.

stores/feedStore.ts is a Zustand store that also caches a server resource — the infinite feed — because the feed needs realtime behaviour (cursor paging, dedup, a "buffer" of incoming posts) that goes beyond a simple query. Its prependItem action is the optimistic insert, and it guards against duplicates so an optimistic add and the later realtime echo do not double up:

prependItem: (item) => {
  set((state) => {
    // Skip if item already exists (prevents SSE + local double-add)
    if (state.items.some((i) => i.id === item.id)) return state;
    return { items: [item, ...state.items] };
  });
},

The store distinguishes your action from everyone else's. In receiveCreated, a streamed post that you authored is prepended immediately ("self-action consistency"), while a stranger's post is buffered into pendingItems and surfaced behind an "N new posts" pill so your scroll position is never yanked. The real comment in the file makes the intent explicit:

// For You surface: your own post appears immediately (self-action
// consistency); everyone else's is buffered behind an "N new posts" pill
// so the scroll position is never yanked.
if (delivery.own) {
  get().prependItem(item);
} else {
  set((state) => ({ pendingItems: [item, ...state.pendingItems] }));
}

The store even maintains a second cache for cross-cutting display data: userDisplayStore.ts caches each user's avatar and name by id and merges updates so "a stale null doesn't clobber a valid image." This is a deliberate, hand-built cache because the same user appears across many feed items and should look consistent everywhere.

The hydration boundary

rmhstudios server-renders pages with TanStack Start over Nitro (Chapter 2). The server produces HTML so the first paint is fast and crawlable; then the same React tree boots in the browser and "takes over" the static markup, attaching event handlers and state.

Hydration boundary — the moment the client's React tree adopts the server-rendered HTML. For it to succeed, the client's first render must produce markup identical to what the server emitted; any difference is a hydration mismatch, which React must repair (a visible "flash" or a console warning).

This is the precise reason server data and client state are kept apart and seeded carefully. The server cannot know browser-only facts — localStorage, the live session cookie's resolved user — at the instant it renders, so naively reading them on the client's first render would diverge from the server HTML. The repo handles this with two complementary tactics, both visible in components/Providers.tsx.

First, an inline theme script in __root.tsx applies the saved theme class and background colour before React hydrates, so the document already matches. The Providers effect that syncs theme then guards its very first run to avoid touching anything React already agreed on — its comment warns that "touching ANYTHING here would flash the wrong colour":

// Hydrate style from localStorage on mount
useEffect(() => {
  const stored = localStorage.getItem("rmh-style") as SiteStyle | null;
  if (stored && SITE_STYLES.some((s) => s.id === stored)) {
    useThemeStore.getState().setStyle(stored);
  }
}, []);

Second, the session is seeded from the server so the shell renders signed-in on the very first frame. __root.tsx reads an initialUser in its loader and passes it to Providers, which uses it as the initial state — the comment is unambiguous: it makes "the very first painted HTML (and the first client render) ... already signed-in — no 'signed out' flash on refresh." The live better-auth session then resolves a moment later and "always wins," self-healing if the seed was stale.

The mismatch trap. Reading localStorage, window, or the live session directly in a render body is the classic hydration bug: the server has no such value, so the markup differs and React tears down and rebuilds the subtree. The fix is always the same — render a server-safe default first, then reconcile to the real value inside a useEffect (or seed the real value from the server loader, as the session does).

Putting the lines in the right place

The discipline that makes this codebase legible is that each piece of state lives in exactly one of the three systems, and the choice follows from ownership:

  • One component, throwaway? useState — e.g. replayKey.
  • Shared across the tab, no server truth? A Zustand store — theme, game run, settings. Add persist if it should survive a reload.
  • Owned by the database? React Query, keyed and given a staleTime — reputation, leaderboards, daily puzzles.
  • A server resource needing realtime/optimistic behaviour? A purpose-built Zustand store that caches it (the feed), updated optimistically and reconciled by SSE.

Keeping durable server data out of the global client store means a stale write can never silently become the source of truth, and keeping ephemeral client state out of the query cache means a background refetch can never wipe your in-progress UI. The separation is not bureaucracy; it is what lets server rendering, realtime streams, and instant interactivity coexist on the same page.

Summary

  • Frontend state splits into local (one component), client (the tab, no server truth), and server (a cache of the database). rmhstudios maps these to React hooks, Zustand, and TanStack React Query respectively.
  • Zustand stores are made with create((set, get) => ...); they bundle a state slice with the actions that mutate it, and components select just the fields they need. useAltairGameStore holds a whole game run; persist mirrors stores like settings to localStorage.
  • React Query keeps a query cache keyed by a query key, deduplicates concurrent requests, and refetches after staleTime. One QueryClient in Providers.tsx serves the whole app.
  • The feed performs optimistic updates in a Zustand store, prepending your post instantly and deduplicating the later SSE echo by id; this repo uses useQuery for reads but plain fetch-plus-store for writes (no useMutation).
  • The hydration boundary requires the client's first render to match the server HTML. Browser-only state is rendered as a default first and reconciled in a useEffect, or seeded from a server loader (the session) so there is no flash and no mismatch.
