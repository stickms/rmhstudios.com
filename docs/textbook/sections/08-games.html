Learning Objectives

  • Identify the three recurring building blocks of an in-browser game engine — the game loop, a deterministic RNG, and a beatmap — and trace each through the real lib/game/ Slice-It engine.
  • Explain why a score that is computed entirely on the client cannot be trusted, and describe the server-authoritative validation that app/routes/api/<game>/score.ts performs before a number reaches a leaderboard.
  • Justify why every score endpoint sits behind an in-memory IP rate limiter, and read the limit/window pairs the codebase actually uses.
  • Compare the two save strategies in the platform — an opaque JSON save blob (Temple of Joy, Signal Forge) versus derived high-water-mark columns (leaderboards) — and state the trade-offs of each.
  • Distinguish the 2D rendering paths (Canvas 2D, WebGL) from the 3D stack (React Three Fiber + Rapier) and explain why the same game-loop pattern survives across both.
  • Connect single-player play to the realtime socket fleet of Chapter 7 through the MultiplayerFactory abstraction.

A games library is a stress test for everything else in this book. Each game is a tiny real-time system: it runs a loop at sixty frames per second, generates its own content, talks to a realtime socket fleet for multiplayer, and — the moment a player wants their score remembered — crosses the same authentication, persistence, and abuse-prevention boundaries as the rest of the platform. rmhstudios.com ships a suite of these (the README lists Slice It!, Signal Forge, Neon Driftway, Altair, Temple of Joy, the 3D titles VELUM 2099 and Forest Explorer, and more). This chapter reads the real engine code in lib/game/ and several per-game libraries, then follows the single most security-sensitive path in the whole subsystem: how a score travels from a browser nobody controls to a leaderboard everybody can see.

The anatomy of an in-browser engine

Strip away the art and every game in the library reduces to the same skeleton: a game loop that advances state once per animation frame, a renderer that draws the current state, an input path that mutates state in response to the player, and some source of content — a level, a deck, a beatmap. The clearest reference implementation in the repository is the Slice-It rhythm engine, whose logic lives in lib/game/GameEngine.ts rather than in lib/slice-it/ (that directory holds only upload-validation.ts, the helper that vets user-uploaded songs).

Game loop — the function a game calls on every frame to advance one tick of simulation: read the clock, update state, resolve collisions and inputs, then hand the new state to the renderer. In Slice-It the loop body is GameEngine.update(), invoked by the Canvas component every frame.

Notice what update() does not do: it does not own a clock. A rhythm game must stay synchronized to the music, so the engine reads time from the audio subsystem instead of from a frame counter. The loop's first act is to ask the audio manager where the song is, then subtract a user-calibrated offset:

public update() {
    // Called every frame by the Canvas loop
    if (useGameStore.getState().status !== 'PLAYING' || useGameStore.getState().isPaused) return;

    const offsetSeconds = (useGameStore.getState().audioOffset || 0) / 1000;
    const rawTime = this.audioManager.getCurrentTime();
    const currentTime = rawTime - offsetSeconds;
    const duration = this.audioManager.getDuration();

    // Complete check
    if (duration > 0 && currentTime >= duration) {
        useGameStore.getState().setStatus('FINISHED');
        this.audioManager.stop();
        if (this.lobbyId) this.mp.finishGame(this.lobbyId, this.score);
    }
    // ... resolve holds, scan for missed notes, throttle multiplayer sync ...
}

Two design decisions in this excerpt recur across the whole library. First, the loop is a pure consumer of an external clock (here, audio playback time), so the simulation cannot drift even if frames are dropped. The arcade racer Neon Driftway makes the same choice differently: its design spec (specs/neon-driftway.md, internally titled "REDLINE") mandates a variable-but-clamped timestep — dt = min(dt, 0.033) — so that a long frame can never tunnel a car through a wall. Second, the loop never blocks: long-running work (decoding audio, loading a track) happens in loadMap() before play starts, leaving the per-frame path cheap.

Input as a state mutation, not an event handler

Player input enters through submitInput(lane). Two details make it robust. A per-lane cooldown (INPUT_COOLDOWN_MS = 50) guarantees one press maps to at most one note, and the engine only allows the next sequential unhit note in a lane to be struck — found by getTargetedSlice(lane). The timing windows that grade a hit are constants in lib/game/types.ts:

export const HIT_WINDOWS = {
  MARVELOUS: 0.020,    // 20ms
  PERFECT: 0.033333,   // ~33ms
  GREAT: 0.108333,     // ~108ms
  GOOD: 0.158333,      // ~158ms
  BAD: 0.191666,       // ~191ms
};

Deterministic randomness

Games need randomness — which lane a beat lands in, which card a shop offers, where traffic spawns — but raw Math.random() is a liability: it cannot be reproduced, so a bug, a replay, or a "seeded run" leaderboard is impossible. The fix is a deterministic RNG: a small pseudo-random generator seeded from a string, so the same seed always yields the same sequence. Both GameEngine.ts and BeatMapGenerator.ts carry an identical mulberry32 implementation:

Deterministic RNG — a pseudo-random number generator whose output is fully determined by an initial seed. Re-running with the same seed reproduces the same sequence, which is what makes replays, deterministic level generation, and seeded-run leaderboards possible.
function createSeededRandom(seed: string): () => number {
    let h = 0;
    for (let i = 0; i < seed.length; i++) {
        h = Math.imul(31, h) + seed.charCodeAt(i) | 0;
    }
    return () => {
        h |= 0; h = h + 0x6D2B79F5 | 0;
        let t = Math.imul(h ^ h >>> 15, 1 | h);
        t = t + Math.imul(t ^ t >>> 7, 61 | t) ^ t;
        return ((t ^ t >>> 14) >>> 0) / 4294967296;
    };
}

The beatmap generator seeds it with the track id (createSeededRandom(`${id}-${120}`)) so that the lane assignment for a balanced stereo beat is stable for a given song. Neon Driftway's spec goes further, recording seed = timestamp XOR random in each run's stats "for debugging and replay." The discipline is worth naming as a rule: any randomness whose result you might later need to reproduce should flow through a seeded generator, never the global RNG.

Scope: not every randomness path is deterministic yet. The simulated opponents in lib/game/useOpponents.ts advance with raw Math.random() on a 100 ms interval, and the map modifiers in loadMap() (converting ~5% of slices to BOMB or ~15% to SWITCH) also use Math.random(). These are cosmetic or single-session, so reproducibility is not required — but a future seeded-replay feature would have to route them through createSeededRandom too.

Beatmaps and procedural content

A beatmap is the score sheet a rhythm game plays against: an ordered list of notes, each with a timestamp, a lane, and a type. In this engine the unit is a Slice (lib/game/types.ts): { id, time, type, lane, duration?, ... }, where type is one of seven SliceTypes (STANDARD, LONG, SWITCH, BOMB, and so on). A BeatMap bundles slices with track metadata, and crucially its slices field can be either a flat array or a Record<Difficulty, Slice[]> — one chart per difficulty.

Beatmap — the timed list of notes a rhythm game scores the player against. Each note carries a time (seconds from track start), a lane, and a type; the player's job is to act on each note inside its hit window.

The platform generates beatmaps two ways. MapGenerator.generate() builds a chart arithmetically from BPM — secPerBeat = 60 / bpm, one slice per beat from beat 4 on, promoting every 8th beat to a MOVING note and every 16th to a LONG — which is cheap and perfectly regular. BeatMapGenerator.generateFromBuffer() instead does onset detection on an uploaded song: it walks the decoded AudioBuffer in 0.1 second windows, finds peaks above a threshold, enforces a minimum spacing so beats cannot spam, and assigns a lane from stereo balance (louder-left → lane 0, louder-right → lane 1, balanced → the seeded RNG decides):

const peak = Math.max(maxL, maxR);
if (peak > threshold) {
    const currentTime = i / sampleRate;
    if (currentTime - lastBeatTime >= minDistance) {
        let lane = 0;
        if (maxL > maxR * 1.2)      lane = 0; // Top
        else if (maxR > maxL * 1.2) lane = 1; // Bottom
        else                        lane = rng() > 0.5 ? 1 : 0; // balanced → seeded
        slices.push({ id: `slice-${slices.length}`, time: currentTime, type: 'STANDARD', lane });
        lastBeatTime = currentTime;
    }
}
Where do beatmaps live? Nowhere on disk. The data/ directory holds content for the RMHbox party games (e.g. data/rmhbox/emoji-cinema/movies.json), not Slice-It charts. A Slice-It beatmap is generated at runtime from a track in lib/game/tracks.ts or from a freshly uploaded file — it is never persisted, which is why the score endpoint, not a stored chart, is the system of record for what a player achieved.

The score pipeline: why scoring must be server-authoritative

Here is the central lesson of the chapter. The browser computes the score — handleHit() adds points, multiplies by combo, applies modifier multipliers from calculateScoreMultiplier() — but that number cannot be trusted. The player owns the machine running it; a determined user can call the endpoint directly with any number they like. The only defense is to make the server the authority over what counts.

Server-authoritative scoring — a design in which the client may compute and display a score for responsiveness, but the server independently authenticates, validates, and bounds every submission before it is allowed to affect persistent state such as a leaderboard. The client's number is a claim; the server decides what is true.

rmhstudios.com cannot fully re-simulate a play on the server (it never received the keystrokes), so it does the next best thing: it treats every submission as hostile and gates it through a fixed sequence of checks. Reading app/routes/api/slice-it/score.ts top to bottom, the gates are: (1) rate-limit by IP, (2) shape and bound the payload, (3) authenticate the session, then (4) persist only what survived. The order matters — the cheapest, most abusable check runs first:

POST: async ({ request }) => {
    const ip = getClientIp(request);
    const { allowed, retryAfter } = rateLimit(ip, { limit: 5, windowMs: 60_000, prefix: 'slice-score' });
    if (!allowed) return Response.json({ error: 'Too many requests' },
        { status: 429, headers: { 'Retry-After': String(retryAfter) } });

    const { username, score, accuracy, maxCombo, songId, speed, modifiers } = await request.json();

    if (!username || typeof username !== 'string' || typeof score !== 'number')
        return Response.json({ error: 'Invalid score data.' }, { status: 400 });

    const cleanUsername = username.trim().replace(/[^a-zA-Z0-9_\-. ]/g, '').slice(0, 24);
    if (cleanUsername.length < 2) return Response.json({ error: 'Invalid username' }, { status: 400 });
    if (score < 0 || score > 1_000_000_000) return Response.json({ error: 'Invalid score' }, { status: 400 });

    const playSpeed = typeof speed === 'number' ? speed : 1.0;
    if (playSpeed < 1.0) return Response.json({ error: 'Scores at speeds below 1.0x are unranked' }, { status: 400 });

    const session = await auth.api.getSession({ headers: request.headers });
    if (!session?.user) return Response.json({ error: 'Unauthorized.' }, { status: 401 });
    // ... only now do we touch the database ...
}

Every line here is a guardrail. The username is sanitized to an allow-list of characters and capped at 24, so it cannot smuggle markup or grow unboundedly. The score is range-checked (0 … 1_000_000_000) so an obviously impossible value is rejected even though a plausible forged value still could not be caught this way — bounding limits the blast radius, it does not prove honesty. Sub-1.0x plays are declared unranked at the server, not merely hidden in the UI. And the auth.api.getSession() call ties the submission to a real account, so a leaderboard entry always has an owner. Only after all of this does the handler write to Postgres, incrementing the player's totalScore and, if a songId matches a real Song, recording a per-song personal best.

This shape is uniform across the library. Every game has a sibling pair app/routes/api/<game>/score.ts and .../leaderboard.ts — Slice-It, Signal Forge, Neon Driftway, Altair, Vega, Void Breaker, and more. They differ only in the columns they keep. Neon Driftway's handler, for instance, bounds score to ten million, clamps level to the valid range 1…3, and stores high-water marks (highScore, bestDistance, bestTimeMs) with Math.max(existing, incoming) so a worse run never overwrites a better one.

Rate limiting the write path

The first gate, rateLimit() from lib/rate-limit.ts, is an in-process fixed-window counter keyed by client IP. It is deliberately humble: a bounded Map (MAX_STORE_SIZE = 10_000) swept by a 60-second GC timer, with FIFO eviction if it ever fills. There is no Redis, no shared store — which means each server process keeps its own counts.

Rate limiting — capping how many requests a given client (here, an IP address) may make in a time window. Slice-It allows 5 score POSTs per 60_000 ms but 20 leaderboard GETs, reflecting that writes are far more abusable than reads.

The limit/window pairs encode a threat model. Reads are generous (leaderboard GET: 20/min); writes are tight (score POST: 5/min). Save endpoints sit in between (Temple of Joy save: 20/min; Signal Forge save: 10/min). A blocked request returns 429 with a Retry-After header so a well-behaved client can back off.

The IP is only as trustworthy as the proxy. getClientIp() reads the first entry of x-forwarded-for. The file's own comment is explicit: this "is only safe behind a trusted reverse proxy (nginx, Cloudflare) that sets/overwrites this header. Without a trusted proxy, clients can spoof this header to bypass rate limits." The limiter is a speed bump that depends on the deployment topology of Chapter 7, not a cryptographic guarantee.

Save systems: opaque blobs versus derived columns

Two of the games persist a player's entire progress, not just a number. The idle game Temple of Joy and the deckbuilder Signal Forge each store a save blob: a single JSON document the client serializes and ships up whole. The Prisma schema models this with a Json column — TempleOfJoySave.saveData Json and SignalForgePlayer.savedRunState Json?:

Save blob — a player's complete game state serialized to one opaque JSON document and stored as a single column, rather than decomposed into relational rows. The server treats it as a value to hold for that user, not a structure to query.
model TempleOfJoySave {
  id        String   @id @default(cuid())
  userId    String   @unique
  saveData  Json
  updatedAt DateTime @default(now()) @updatedAt
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  @@map("temple_of_joy_save")
}

The save handler (app/routes/api/temple-of-joy/save.ts) is a textbook upsert-per-user: authenticate, rate-limit, reject a non-object body, cap the payload at 500 KB (bodyStr.length > 500_000), then prisma.templeOfJoySave.upsert keyed on userId — one save row per player, last write wins. Signal Forge's save.ts adds a little more semantics: because its blob is a roguelike run, the handler peeks inside it to validate essential fields (typeof runState.floor !== 'number' is rejected as malformed) and to maintain derived leaderboard columns alongside the blob:

const currentScore  = typeof runState.score === 'number' ? runState.score : 0;
const newHighScore  = existing ? Math.max(existing.highScore, currentScore) : currentScore;
const newFloorReached = existing ? Math.max(existing.floorReached, currentFloor) : currentFloor;

await prisma.signalForgePlayer.upsert({
  where: { userId: session.user.id },
  update: { savedRunState: runState, highScore: newHighScore, floorReached: newFloorReached, username },
  create: { userId: session.user.id, username, savedRunState: runState, highScore: currentScore, floorReached: currentFloor },
});

The contrast is the lesson. The blob is dead simple to write and lets the client evolve its state shape without a migration — but the database cannot index, query, or validate its interior, the server must trust the client's serialization, and a 500 KB cap is the only bound on its growth. The derived column (highScore, floorReached, indexed @@index([highScore(sort: Desc)])) is what the leaderboard reads, because you cannot ORDER BY a field buried inside a JSON document efficiently. So Signal Forge keeps both: the blob to resume a run, the columns to rank it. This split — opaque state for resume, extracted scalars for queries — is the pragmatic answer whenever you must both store and rank.

Scope: leaderboards reuse the same dual idea. Slice-It's global board reads Player.totalScore (a scalar column), while its per-song board reads SongLeaderboard rows keyed @@unique([songId, userId]) and indexed @@index([songId, score(sort: Desc)]). The leaderboard GET simply findMany … take 10, orderBy score desc. No blob is ever read on the ranking path.

The rendering split: 2D, WebGL, and the 3D stack

The README's tech stack lists "React Three Fiber, Rapier Physics, Pixi.js, Howler.js," and it is tempting to assume every game uses all of them. It does not — and reading the source corrects the assumption. The five games detailed here are all 2D, and they reach for different renderers:

2D vs 3D split — a 2D game draws a flat scene with a canvas drawing API (Canvas 2D) or a 2D-batched GPU layer (WebGL/Pixi), updating positions directly. A 3D game describes a scene graph of meshes that React Three Fiber renders and Rapier simulates with real physics. The same game-loop pattern drives both; only the renderer and physics differ.

Slice-It, Signal Forge, and Neon Driftway draw with the plain Canvas 2D context (canvas.getContext('2d')); Slice-It synthesizes its audio through the Web Audio API, while Neon Driftway renders pixel-art sprites (setPixelArt(ctx)) and clamps its timestep as its spec demands. Altair is also 2D but renders through a hand-written WebGL2 batch renderer (lib/altair/engine/webgl/, with initWebGL() and a SpriteBatch) and uses howler for audio — it is the survivors-like with the heaviest draw load, so it pays for a GPU path. Temple of Joy renders no canvas at all: it is a pure React-DOM idle game whose engine (lib/temple-of-joy/engine.ts) is a set of pure compute functions like computeTotalHPS driven by a Zustand store and an applyTick step.

Scope: where the 3D stack actually runs. @react-three/fiber, @react-three/rapier, and three are real dependencies, and the README assigns them to the 3D titles (VELUM 2099, "a 3D cyberpunk game"; Forest Explorer, "a 3D exploration game"). None of the five games dissected in this chapter import them — a useful reminder to read source rather than the dependency list. The pattern (loop → update → render) is identical across 2D and 3D; only the renderer (a canvas context vs. an R3F scene graph) and the physics (manual AABB checks vs. Rapier) change.

The component layout is just as uniform. Each game is a top-level shell (GameCanvas.tsx, SignalForgeGame.tsx, NeonDriftwayGame.tsx, AltairShell.tsx, TempleOfJoyGame.tsx) wrapping a HUD (HUD.tsx / CombatHUD.tsx / GameHUD.tsx) and a folder of screens (a menu/landing screen, a game-over screen, a pause overlay, a settings panel, a leaderboard panel). Multiplayer through the socket fleet

Single-player play is self-contained, but the moment two players share a lobby the game must talk to the realtime fleet of Chapter 7. Slice-It does this through MultiplayerFactory (lib/game/MultiplayerFactory.ts), a singleton that wraps one socket.io-client connection behind a small action/event API. The engine never touches a socket directly; it calls the factory, and the factory speaks the fleet's protocol.

private mp = MultiplayerFactory.getInstance();
// inside update(), throttled to ~3 times/sec:
if (this.lobbyId && Math.random() < 0.05) {
    this.mp.updateScore(this.lobbyId, { score: this.score, combo: this.combo });
}

The factory connects to VITE_SOCKET_URL (default http://localhost:7001) over the path: "/socket/" WebSocket transport — the same fleet endpoint Chapter 7 describes. It exposes outbound actions (joinLobby, selectSong, startGame, updateScore, finishGame) and a local pub/sub for inbound GameEvents ('lobby_update', 'player_update', 'player_finished', 'match_results', …). When a 'lobby_update' arrives it reconciles the opponent list into the Zustand store — adding present players, removing absent ones. The abstraction is the point: by funneling all realtime traffic through one factory, the engine stays testable, the socket lifecycle is owned in one place, and a different transport could be swapped in without touching GameEngine.

Authoritative scoring still applies in multiplayer. The throttled updateScore messages drive the live opponent HUD only; they are not the record of truth. A final, ranked score still travels the server-authoritative score.ts pipeline of Figure 8.2. Realtime sync is for presence; the leaderboard is for truth.

Summary

  • Every game reduces to the same skeleton — a game loop, a renderer, an input path, and a content source. Slice-It's GameEngine.update() reads its clock from audio playback, so the simulation stays in sync even when frames drop.
  • A deterministic RNG (the shared mulberry32 seeded generator) makes lane assignment, level generation, and seeded-run leaderboards reproducible; raw Math.random() remains only on cosmetic, single-session paths.
  • A beatmap is a timed list of Slice notes, built arithmetically by MapGenerator or by onset detection in BeatMapGenerator.generateFromBuffer(); it is generated at runtime, never stored.
  • Scoring is server-authoritative: score.ts rate-limits by IP, sanitizes and bounds the payload, authenticates the session, and only then writes — because the client's number is a claim, not a fact.
  • Rate limiting (lib/rate-limit.ts) caps writes tightly (5/min) and reads loosely (20/min), but trusts x-forwarded-for, so it is only as strong as the proxy in front of it.
  • Save blobs (Json columns) make progress easy to persist and evolve but impossible to query; the platform pairs them with derived, indexed columns (highScore) for ranking.
  • The library spans Canvas 2D, WebGL2, and pure DOM rendering, with the R3F + Rapier 3D stack reserved for the 3D titles — yet the loop-update-render pattern is invariant across all of them.
  • Multiplayer flows through the MultiplayerFactory singleton to the Chapter 7 socket fleet for live presence, while final scores still cross the authoritative pipeline.
