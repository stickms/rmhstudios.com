Writing code is only half of shipping software; the other half is getting that code onto a server, running it, and replacing the old version without anyone noticing. This chapter reads the operations layer of rmhstudios.com closely: the single deploy script that pulls, builds, migrates, and swaps; the small blue-green script that flips live traffic between two running copies of the web container; the reverse proxy that fronts everything; and the in-progress migration from Docker Compose to Kubernetes. Where earlier chapters described what the code does, this one describes how it gets to production and stays up while it changes.

Deployment — the process of taking a specific version of the source code, turning it into runnable artifacts, and making those artifacts the live, traffic-serving version of the system on its servers.

What deployment is, and what makes it hard

rmhstudios.com runs on a single VPS (virtual private server) under the deploy user rmhstudios, from a checkout at /home/rmhstudios/rmhstudios.com. A deploy must do four things in order: get the new code, turn it into a runnable image, bring the database schema up to date, and switch the running services over. Each of these can fail, and the dangerous moment is the switch — if you stop the old version before the new one is ready, the site is down for the gap. Most of the engineering in this chapter exists to shrink or eliminate that gap.

The whole pipeline is one Bash script, deploy.sh at the repository root (~34 KB). Despite its size it is linear: a header that picks the environment, a block of helper functions, and then a sequence of numbered steps from "Step 1: Pull latest code" to "Step 6: Prune stale images." Its own header comment calls it the "Docker-based deploy script," and that is the key fact about the current production system: the run-layer is Docker Compose, not a bare process manager.

Scope: process management here is Docker, not PM2. A common Node deployment uses PM2, a userspace process manager that keeps Node processes alive and restarts them on crash. rmhstudios.com does not use PM2. The same job — keep every service process running, restart it on failure, cap its logs — is done by Docker Compose with restart: unless-stopped and a json-file logging driver capped at max-size: "10m" (see docker-compose.yml). When this chapter says "process manager," read it as the run-layer that supervises long-lived processes; in production that role is filled by Docker/Compose, and in the migration target by Kubernetes.
Deployment pipeline — the fixed, repeatable sequence of stages a release passes through on its way to production. Here: pull → build → migrate → bring up services → hotswap web → health-check → prune.

The build → migrate → swap pipeline

The script starts by choosing an environment from its first argument (production maps to branch main, env file .env.production, Compose project rmhstudios-prod, and the port block 7005/7001/7676/7003/7008; staging is currently disabled and exits early). It then takes a per-environment lock with flock so two deploys can never run at once — a second trigger queues its PID and waits, and if a newer request arrives while it waits, it notices it has been superseded and exits rather than running a stale build.

From there the steps run in order. A subtle but important one comes right after the pull: the script hashes deploy.sh before and after git reset --hard, and if the script itself changed, it re-execs the new version so the rest of the deploy runs with updated logic (a DEPLOY_SELF_RESTARTED env var guards against an infinite loop).

Build

The build is a single command, dc build web, where dc() is a helper that wraps docker compose -p "$PROJECT_NAME" --env-file "$ENV_FILE". One multi-stage Dockerfile produces one image, rmhstudios-prod-app, shared by all services; the realtime servers and workers just override the container CMD. After a successful build the script tags the image with the short git SHA so any past release can be brought back instantly:

# ── Step 2: Build Docker image ──
step_start "Building Docker image..."
if ! dc build web; then
    log "ERROR: Docker build failed."
    update_deploy_status fail "docker build failed"
    exit 1
fi

# Tag with git SHA for instant rollback (docker compose up with the old tag)
"$DOCKER_BIN" tag "${IMAGE_NAME}:latest" "${IMAGE_NAME}:${GIT_SHA}" 2>/dev/null || true

Because the build runs on the VPS itself, the script spends real effort on disk hygiene around it: before building it prunes dangling images and stale SHA-tagged rollback images, and if free space is under 10 GB it wipes the BuildKit cache entirely rather than die mid-build with "no space left on device." After building it trims the build cache back toward a 5 GB target and hard-caps it at 8 GB. None of this changes what ships; it exists because there is no separate build farm.

Reversible migration — a schema change applied in a way that can be undone or retried, with a recorded baseline, so a half-applied or failed migration does not leave the database in an unrecoverable state.

Migrate

Before swapping any code, the database schema is brought forward with Prisma. The script first proves the database is reachable — it retries prisma migrate status up to ten times, distinguishing a real connection failure (retry) from a non-zero exit that merely reports a pending or failed migration (proceed). It then runs baseline-resolve, an optional rolled-back-resolution for a previously failed migration, and migrate deploy — all collapsed into one container invocation to avoid paying Node+Prisma startup three times:

MIGRATE_CMD='npx prisma migrate resolve --applied 0_baseline 2>/dev/null || true'

FAILED_MIGRATION=$(echo "$MIGRATE_OUTPUT" | grep -oP '(?<=resolve --rolled-back ")[^"]+' || true)
if [ -n "$FAILED_MIGRATION" ]; then
    MIGRATE_CMD="$MIGRATE_CMD; npx prisma migrate resolve --rolled-back \"$FAILED_MIGRATION\" || true"
fi
MIGRATE_CMD="$MIGRATE_CMD; npx prisma migrate deploy"

Marking 0_baseline as applied is what makes the migration reversible and idempotent: it tells Prisma "these tables already exist, do not try to recreate them," so a re-deploy onto an already-migrated database is a safe no-op rather than an error. The same logic reappears verbatim in the Kubernetes run-layer, as a Helm pre-upgrade hook (below) — the two run-layers share one migration contract.

Swap

The swap is split deliberately. Every service except web is brought up with dc up -d --no-build --remove-orphans --scale web=0. The --scale web=0 is the crucial detail: Docker Compose recreates a container in place by stopping the old one and then starting the new one, and that multi-second gap on the web port is exactly what once produced a user-facing Cloudflare 520. So Compose is told never to manage web; a brief blip on a websocket or worker reconnect is invisible, but the page must never go dark. The web container is instead handed to a separate blue-green script.

# ── Step 4: Bring up containers (everything EXCEPT web) ──
if ! dc up -d --no-build --remove-orphans --scale web=0; then
    log "ERROR: docker compose up failed."
    update_deploy_status fail "docker compose up failed"
    exit 1
fi

# ── Step 4b: Zero-downtime hotswap of the web container ──
if ! PROJECT_NAME="$PROJECT_NAME" ENV_FILE="$ENV_FILE" \
     IMAGE_NAME="$IMAGE_NAME" IMAGE_TAG="latest" \
     BLUE_PORT="$PORT_WEB" GREEN_PORT="${PORT_WEB_GREEN:-$((PORT_WEB + 10))}" \
     NETWORK="${PROJECT_NAME}_default" DOCKER_BIN="$DOCKER_BIN" \
     bash "${REPO_DIR}/deploy/hotswap-web.sh"; then
    log "ERROR: web hotswap failed."
    exit 1
fi

Zero-downtime hotswap (blue-green)

Zero-downtime deployment — releasing a new version with no interval during which the service is unavailable, by keeping the old version answering requests until the new one is proven able to take over.
Blue-green deployment — running two complete copies of a service, "blue" and "green," with only one live at a time. A release starts the idle color, health-checks it, then switches traffic to it; the previously live color becomes the standby (and the instant rollback target).

The file deploy/hotswap-web.sh implements blue-green for one container. Production reserves two ports for it: 7005 (blue, PORT_WEB) and 7015 (green, PORT_WEB_GREEN). The script reads which color is currently live from an Apache include file, brings up the other color on its spare port, and only moves traffic once the new container actually answers HTTP.

It starts the new container with docker compose run rather than a hand-rolled docker run — an important choice, because Compose's parser strips quotes and applies the service's env_file, extra_hosts, volumes, and network exactly as the real web service has them (a naive docker run --env-file passes a quoted DATABASE_URL through literally and can't reach Postgres). It overrides only PORT and publishes that port on loopback:

"$DOCKER_BIN" compose -p "$PROJECT_NAME" --env-file "$ENV_FILE" \
    run -d --no-deps \
    --name "$new_container" \
    -p "127.0.0.1:${target_port}:${target_port}" \
    -e "PORT=${target_port}" \
    -e "HOSTNAME=0.0.0.0" \
    web >/dev/null || die "failed to start ${new_container}"

It then polls the new port for up to HEALTH_TIMEOUT (60s), curling http://127.0.0.1:${target_port}/ until it returns success, and bails early if the container exits during startup. The guarantee here is the whole point: if the new container never becomes healthy, traffic is never moved and the old one keeps serving — a bad build degrades to "old version stays up," never to an outage.

healthy=false
for _ in $(seq 1 "$HEALTH_TIMEOUT"); do
    if curl -fsS -o /dev/null --max-time 3 "http://127.0.0.1:${target_port}/"; then
        healthy=true; break
    fi
    if [ "$("$DOCKER_BIN" inspect -f '{{.State.Running}}' "$new_container" \
            2>/dev/null || echo false)" != "true" ]; then
        die "${new_container} exited during startup — keeping ${old_color} live."
    fi
    sleep 1
done

Only after the new container is healthy does the script flip the reverse proxy, drain in-flight requests for DRAIN_SECONDS (8s), and then prove the flip landed before deleting anything: it stops (not removes) the old container and probes Apache itself on :80 with the real Host header. If the site still serves, the flip genuinely routed to the new port and the old container is safe to delete; if not, it restarts the old container, reverts the port file, and aborts — converting "503 on every route until a human notices" into a bounded blip that self-heals back to the old version.

The reverse proxy: Apache out front

Reverse proxy — a server that sits in front of one or more application processes, accepting client requests and forwarding them to the right backend, then relaying the response. It centralizes TLS, routing, caching, and (here) the traffic flip between blue and green.

Apache fronts the whole site (deploy/apache/rmhstudios.conf). TLS is terminated at Cloudflare, so Apache listens on :80 only. It does three jobs. First, it is a self-hosted CDN: the heavy static directories /library, /music, /models, and /sprites are served straight off the host disk via Alias directives and excluded from the proxy with ProxyPass /library !, so those requests never touch Node. (This is why deploy.sh re-asserts world-read permissions on public/ on every deploy — a fresh git reset can land files Apache's www-data can't read, 403-ing every asset.) Second, it routes the WebSocket upstreams (/socket/→7001, /rmhbox-ws/→7676, /rmhtube-ws/→7003) and the /webhook endpoint. Third, and central to the hotswap, it proxies everything else to the active web port:

IncludeOptional /home/rmhstudios/rmhstudios-web-active.conf
<IfDefine !WEB_UPSTREAM_PORT>
    Define WEB_UPSTREAM_PORT 7005
</IfDefine>
...
    ProxyPass / http://localhost:${WEB_UPSTREAM_PORT}/
    ProxyPassReverse / http://localhost:${WEB_UPSTREAM_PORT}/

The flip is just a one-line file. The hotswap script writes Define WEB_UPSTREAM_PORT 7015 (or 7005) into the included rmhstudios-web-active.conf, and Apache picks up the new upstream on a graceful reload — which finishes in-flight requests on the old worker set before swapping. The include lives under the deploy user's home, not /etc, for a concrete security reason: the webhook that triggers deploys runs sandboxed (ProtectSystem=strict + NoNewPrivileges) and cannot write /etc or use sudo, so a root-owned systemd path-unit watches the home file and runs apachectl graceful when it changes. The deploy only writes a file; the privileged reload is delegated.

Path coupling. The IncludeOptional path in the vhost MUST equal ACTIVE_CONF in hotswap-web.sh (/home/rmhstudios/rmhstudios-web-active.conf). If they diverge — say the include points at /etc/... — the deploy writes the home file while Apache keeps reading the stale one, the flip never lands, and every route 503s the instant the old container stops. The flip-verification probe exists precisely to catch this and roll back.

Webhook-driven deploys

Nobody SSHes in to deploy. A tiny Node HTTP server, webhook-server.cjs, listens on 127.0.0.1:7002 (Apache proxies /webhook to it) and turns a GitHub push into a deploy. It is deliberately minimal and security-first: it verifies the GitHub X-Hub-Signature-256 HMAC against a shared WEBHOOK_SECRET before doing anything, refuses to start if the secret is unset or left at change-me, and never logs the computed signature or secret. It maps refs/heads/main to production and (when enabled) refs/heads/staging to staging, ignoring every other ref.

const sig = req.headers['x-hub-signature-256'];
const expected = 'sha256=' + crypto.createHmac('sha256', SECRET)
    .update(body).digest('hex');
if (sig !== expected) {
    logMsg(`WARN: Invalid signature — rejected. bodyLen=${body.length} hasSig=${!!sig}`);
    res.writeHead(401);
    return res.end('Unauthorized');
}
...
const child = spawn('bash', [DEPLOY_SCRIPT, environment], {
    detached: true,
    stdio: ['ignore', fs.openSync(LOG_FILE, 'a'), fs.openSync(LOG_FILE, 'a')],
    cwd: PROJECT_DIR, env: { ...process.env, PATH: '...' }
});
child.unref();

The deploy is spawned detached and unref'd so it outlives the HTTP request — the webhook answers GitHub immediately with "Deploying production" and the long build runs independently. Note the indirection that makes the run-layer migration painless: the script to run is process.env.DEPLOY_SCRIPT || '.../deploy.sh'. Cutting over to Kubernetes is one environment variable.

Two run-layers and a reversible fallback

The platform is mid-migration between two ways of running the same image. The build is identical in both; only the run-layer differs.

Process manager / run-layer — the component that launches the application's processes, keeps them running, restarts them on failure, and governs their lifecycle. Here it is Docker Compose today and Kubernetes (k3s) in the migration target.

The current/default layer is Docker Compose + the hotswap script, driven by deploy.sh. The migration target (PR #121) is single-node k3s + Helm, driven by deploy/deploy-k8s.sh. The k8s script keeps the build on Compose (docker compose build web, same caching and build-args), then re-tags the image to rmhstudios:<sha>, imports it into k3s's containerd, syncs a Secret from .env.production, and runs helm upgrade --install ... --atomic --wait. The --atomic flag is the k8s analogue of the hotswap's rollback: if any hook or readiness probe fails, Helm rolls the whole release back automatically.

Reversible fallback — keeping the previous, working deployment mechanism fully in place during a migration so that cutting back to it is a small, well-understood operation rather than an emergency rebuild.

The migration is deliberately reversible. Per deploy/README.md, cutover is just pointing the webhook at the new script (DEPLOY_SCRIPT=.../deploy/deploy-k8s.sh) and restarting it; rolling back to Compose is unsetting that variable, restarting the webhook, and running docker compose ... up -d. The old Compose stack (deploy.sh + docker-compose.yml) "stays in place as a fallback." This is why the webhook reads the script path from an env var instead of hard-coding it — the whole switch, and its undo, is one variable.

Why keep a fallback at all? The migration target is new and single-node; the Compose path has served production for the life of the site. Keeping both means a Kubernetes problem discovered in production is a one-variable revert to a known-good system, not a multi-hour scramble. The cost is maintaining two run-layers briefly; the benefit is that the risky change is cheap to undo.

Infrastructure as Code: Helm and Terraform

Infrastructure as Code (IaC) — describing servers, networking, DNS, and cluster resources in version-controlled declarative files, so infrastructure is created and changed by applying code rather than by clicking in a console or running ad-hoc commands.
Helm chart — a templated, versioned package of Kubernetes manifests. Values (image tag, replica counts, hostnames) are supplied separately, so one chart renders into the concrete manifests for a given environment with helm upgrade --install.

The k3s layer is defined as code under deploy/helm/rmhstudios. The chart renders all eight services from one image and includes a service.yaml, an ingress.yaml, a pvc.yaml for the shared /app/db volume, and a migrate-job.yaml. That migration Job is a Helm hook annotated "helm.sh/hook": pre-install,pre-upgrade — it runs the same Prisma reachability-check, baseline-resolve, rolled-back-resolution, and migrate deploy sequence as deploy.sh Step 3, so the two run-layers cannot drift apart on the one operation that touches persistent state. Replica counts, ports, resource floors, and the storage class all live in values.yaml, overridden by values-prod.yaml.

Networking is Terraform. deploy/terraform/dns.tf declares the four Cloudflare A-records (app, socket, rmhbox, rmhtube) as a for_each over a map, all pointing at var.vps_ip; versions.tf pins Terraform >= 1.6.0 and the cloudflare/cloudflare provider. To move the site to a new IP you change one variable and apply — the DNS records are not manual console edits but reviewable code.

Scope: where the migration honestly stands. The k3s/Helm/Terraform layer is real and runnable but is the migration target, not yet the default production path — the webhook still defaults to deploy.sh. The build deliberately stays on the VPS: there is no external CI or image registry in the default path; the image is imported straight into k3s's containerd with k3s ctr. Multi-node support is wired but inert: setting a REGISTRY env var makes deploy-k8s.sh push to a registry and flip the chart to pullPolicy: IfNotPresent instead of importing locally, and the chart marks the two remaining seams (the node-local local-path PVC and the discord-bot host-path mount) with MULTI-NODE SEAM comments. There is also a parallel deploy/deploy-go.sh for a Go microservice fleet. Genuine multi-node scaling — a registry, shared RWX storage, and a realtime backplane — is the subject of Chapter 13; here we note only that the seams exist and are switched by one variable.

Summary

  • Deployment turns a code version into the live, traffic-serving system. rmhstudios.com does it with one script, deploy.sh, that runs a fixed pull → build → migrate → swap pipeline under a flock lock.
  • The build produces one shared image, SHA-tagged for instant rollback; the migrate step runs idempotent, reversible Prisma migrations gated on database reachability.
  • The web tier is swapped zero-downtime by hotswap-web.sh: start the idle blue/green color on a spare port, health-check it, flip Apache to it, verify the flip, then retire the old container. A bad build degrades to "old version stays up."
  • Apache is the reverse proxy; the traffic flip is a one-line included file plus a graceful reload delegated to a root-owned systemd path-unit (because the deploy webhook is sandboxed).
  • Deploys are webhook-driven: an HMAC-verified GitHub push spawns a detached deploy. The script to run is an env var, which is also the migration switch.
  • Two run-layers coexist — Docker Compose (default) and single-node k3s + Helm (PR #121) — sharing one build and one migration contract. The Compose path is kept as a reversible fallback.
  • The k3s layer is Infrastructure as Code (Helm chart + Terraform DNS). Build stays on the VPS by default; multi-node registry support is wired behind a REGISTRY env var (Ch. 13).
