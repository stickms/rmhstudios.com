The doctrine-worker is a headless background scheduler — a faithful port of server/doctrine-worker/index.ts — that drives the Doctrine Engine's daily content. It owns three independent jobs (puzzle generation, Sahur activation, weekly reputation decay) and exposes no client-facing HTTP at all, only /health and /metrics. Its defining property is bit-exact determinism: the PRNG and seeding logic are ported so that the puzzle the worker records for a date is the same puzzle the web app will regenerate from that seed on demand.

Service shape: a port with no client surface

The Node original was a Node process that ran three timers off setTimeout/setInterval. The Go port keeps exactly that structure but expresses each timer as a goroutine ticker draining into a shared stop channel. cmd/doctrine-worker/main.go wires the common foundation (config, DB wait-for-reachable, telemetry) and then launches the worker; the only HTTP it serves is the operational pair on METRICS_ADDR:

metrics := telemetry.New("doctrine-worker")
go func() {
    mux := http.NewServeMux()
    mux.HandleFunc("/health", httpx.Health("doctrine-worker", nil))
    mux.Handle("/metrics", metrics.Handler())
    if err := http.ListenAndServe(cfg.MetricsAddr, mux); err != nil {
        // A bind failure leaves the pod unprobeable (k8s liveness hits
        // /health here), so fail loudly rather than run blind.
        logger.Fatal("metrics server", "error", err)
    }
}()

w := doctrine.New(database, logger, metrics)
w.Start(ctx)        // launches goroutine tickers
httpx.WaitForSignal() // block until SIGINT/SIGTERM
cancel()
w.Stop()
There is no listener on a service port. The Doctrine Engine's read/write API lives in the web app; this worker only produces rows the app consumes. That keeps the worker trivially horizontally-isolated — it is a pure side-effecting scheduler.

The three jobs

Worker.Start runs generateDailyPuzzles once immediately (the startup catch-up run), then spawns three loop goroutines. Each has its own cadence and its own gating rule, faithful to the Node startup():

Self-rescheduling midnight loop

The Node code recursively re-armed a setTimeout(schedulePuzzles, …) at each midnight. The Go port computes the duration to the next UTC midnight, arms a one-shot timer, runs the job, then loops to recompute — the same recursive self-rescheduling, but leak-free under cancellation:

func (w *Worker) runMidnightLoop(ctx context.Context) {
    defer w.wg.Done()
    for {
        d := millisUntilMidnightUTC(time.Now().UTC())
        timer := time.NewTimer(d)
        select {
        case <-ctx.Done():
            timer.Stop(); return
        case <-w.stop:
            timer.Stop(); return
        case <-timer.C:
            w.runJob(ctx, "daily_puzzles", w.generateDailyPuzzles)
        }
    }
}

The decay loop is the inverse: it ticks hourly but the Monday gate lives in the scheduler, not the job, matching the Node setInterval(... if (isMonday()) ..., 3_600_000):

case <-ticker.C:
    if time.Now().UTC().Weekday() == time.Monday {
        w.runJob(ctx, "reputation_decay", w.applyReputationDecay)
    }

Determinism highlight: a bit-exact PRNG port

This is the heart of the worker and the reason the port had to be exact. The web app and the worker must agree on puzzle content: the worker stores a seed, and the app later regenerates the full puzzle (board layout, clue ordering, etc.) by feeding that same seed into the same PRNG. If the Go PRNG diverged from the original Node mulberry32 by even one bit, the worker's stored seed would reconstruct a different puzzle in the browser.

The challenge is that JavaScript's number semantics — | 0 truncation, Math.imul signed 32-bit multiply, >>> unsigned shift — must be reproduced in Go's typed integer world. The port does this with deliberate int32/uint32 conversions:

func Mulberry32(seed int) func() float64 {
    s := int32(seed) // `seed | 0` truncates to signed 32-bit.
    return func() float64 {
        s = s + int32(0x6D2B79F5)           // wrapping 32-bit add
        us := uint32(s)
        // t = Math.imul(s ^ (s >>> 15), 1 | s)
        t := imul(int32(us^(us>>15)), int32(uint32(1)|us))
        // t = (t + Math.imul(t ^ (t >>> 7), 61 | t)) ^ t
        ut := uint32(t)
        t = (t + imul(int32(ut^(ut>>7)), int32(uint32(61)|ut))) ^ t
        // return ((t ^ (t >>> 14)) >>> 0) / 4294967296
        ut = uint32(t)
        return float64(ut^(ut>>14)) / 4294967296.0
    }
}

func imul(a, b int32) int32 { return int32(uint32(a) * uint32(b)) }

The seed itself comes from a string hash over "<date>:<mode>:doctrine" using the classic (hash << 5) - hash + c rolling hash with 32-bit wrap, then Math.abs. Mode names are lowercased at the call site to match the Node mode.toLowerCase():

func GetSeedForDate(date, mode string) int {
    var hash int32
    input := date + ":" + mode + ":doctrine"
    for i := 0; i < len(input); i++ {
        hash = (hash<<5 - hash + int32(input[i]))
    }
    if hash < 0 {
        return int(-int64(hash)) // guard against MinInt32
    }
    return int(hash)
}
Golden-vector tested. seed_test.go locks both functions to reference values produced by running the actual Node code. For 2026-06-19 the seeds must be alibi=489762987, spectrum=1864840003, outcast=1313135021, chainlink=49484481, impostor=1589193561, sahur_special=1827053367; Mulberry32(0)'s first three draws must equal 0.26642920868471265, 0.00032974570058286, 0.22327202744781971 (tolerance 1e-15, i.e. float-literal rounding only). These are not approximate checks — the integer arithmetic is identical, so the outputs are identical.

Difficulty by weekday

Difficulty is a fixed map over the UTC weekday (0 = Sunday, matching JS Date.getUTCDay), with a ?? 3 default. SAHUR_SPECIAL derives from the base via min(5, base+1):

func DifficultyForWeekday(weekday int) int {
    switch weekday {
    case 1: return 1   // Mon
    case 2: return 2   // Tue
    case 3: return 3   // Wed
    case 4: return 3   // Thu
    case 5: return 4   // Fri
    case 6: return 4   // Sat
    case 0: return 5   // Sun
    default: return 3
    }
}

func SahurDifficulty(base int) int { return int(math.Min(5, float64(base+1))) }

The five regular modes are an ordered set; SAHUR_SPECIAL is intentionally generated separately (it is not in the Modes slice), exactly as in the source:

var Modes = []string{"ALIBI", "SPECTRUM", "OUTCAST", "CHAINLINK", "IMPOSTOR"}
const SahurSpecialMode = "SAHUR_SPECIAL"

Each generation run targets tomorrow (now + 1 day, sliced to YYYY-MM-DD), computes one difficulty for the day, and for each mode checks existence before inserting — so re-running (startup after a restart) is idempotent. The persisted JSONB mirrors the Node shape exactly: { "generated": true, "seed": <seed> }.

Per-timezone Sahur activation

Every 60 seconds, checkSahurActivation pulls the distinct non-null doctrineTimezone values from the user table and, for each timezone whose local hour is currently 3 AM, opens a sahur_session for that (today, timezone) pair if one does not exist. Unknown timezones return false, matching the Node try/catch fallback:

func isSahurHour(timezone string, t time.Time) bool {
    loc, err := time.LoadLocation(timezone)
    if err != nil {
        return false
    }
    return t.In(loc).Hour() == 3
}

Weekly reputation decay

On Mondays, decay finds reputations with lastActiveAt < now-7d and totalXp > 0, multiplies XP by 1 - 0.05 (Go integer truncation equals Math.floor for non-negatives), updates the row, and writes a WEEKLY_DECAY ledger entry recording the (negative) delta:

newXp := int(float64(rep.totalXp) * (1 - weeklyDecayRate))
delta := newXp - rep.totalXp
// UPDATE doctrine_reputation SET totalXp ...
// INSERT doctrine_reputation_ledger (action=WEEKLY_DECAY, xpDelta=delta, ...)
Scope — UTC vs local cutoff. The Node code computed oneWeekAgo with a local Date. The Go port deliberately uses UTC (time.Now().UTC().AddDate(0,0,-7)) so the whole worker is consistent with its UTC-based scheduling. In practice this only shifts the 7-day boundary by the host's offset; for a worker that runs in a UTC container the behavior is identical.

Scope — UUID primary keys vs cuid. The Prisma schema issued cuid() IDs. The Go worker generates uuid.NewString() for every inserted row. Both are opaque, collision-resistant string PKs and the columns are plain text, so nothing in the schema or the consuming app depends on the format — but it is an honest substitution worth noting: rows created by the Go worker are visibly UUIDs, rows created by the legacy Node path are cuids.

Observability

Every job run increments JobRuns{job, ok|error} via the runJob wrapper (replacing the Node .catch(console.error) pattern), and every query increments DBQueries{ok|error}. Combined with /health, this gives k8s liveness and Prometheus enough to alert on a stalled scheduler even though the worker has no request traffic of its own.
