The realtime package is the heart of the migration: a single shared WebSocket framework that replaces the four Node Socket.IO deployments (socket-server, rmhbox, rmhtube, rmhmusic). It defines one frame shape — the Envelope — and a small runtime of three cooperating objects: a per-service Hub, a per-client Conn, and a per-broadcast-group Room. Around them sit two support pieces ported directly from Node idioms: GraceTimers (the setTimeout disconnect-grace / room-GC pattern) and a cross-replica Bus fan-out (the Redis adapter that Socket.IO used for horizontal scale). This section documents the protocol, the concurrency design, and the one explicit wire-compatibility gap honestly.

The Envelope protocol

Every frame in either direction is one JSON object — the Envelope. It is deliberately tiny: an event name, an opaque payload, an optional server-assigned sequence number, and a timestamp.

// Envelope is the single frame shape exchanged in both directions.
//
//   { "event": "room:join", "payload": {...}, "seq": 42, "ts": 1718700000000 }
//
// Seq is server-assigned and monotonic per room for mutation broadcasts,
// reproducing the ordering guarantee the Node delta protocol relied on.
type Envelope struct {
    Event   string          `json:"event"`
    Payload json.RawMessage `json:"payload,omitempty"`
    Seq     uint64          `json:"seq,omitempty"`
    TS      int64           `json:"ts,omitempty"`
}

The payload is held as json.RawMessage so the framework never needs to know a service's concrete types: the hub routes on Event, and the handler lazily decodes with Bind. Construction goes through NewEnvelope (which stamps TS with time.Now().UnixMilli() and marshals the payload) or the panic-on-error MustEnvelope convenience used with statically-safe structs.

func NewEnvelope(event string, payload any) (Envelope, error) {
    e := Envelope{Event: event, TS: time.Now().UnixMilli()}
    if payload != nil {
        b, err := json.Marshal(payload)
        if err != nil {
            return Envelope{}, err
        }
        e.Payload = b
    }
    return e, nil
}

func (e Envelope) Encode() []byte { b, _ := json.Marshal(e); return b }

func Decode(b []byte) (Envelope, error) {
    var e Envelope
    err := json.Unmarshal(b, &e)
    return e, err
}

func (e Envelope) Bind(v any) error {
    if len(e.Payload) == 0 {
        return nil
    }
    return json.Unmarshal(e.Payload, v)
}

The round-trip is exercised directly in tests — TestEnvelopeRoundTrip marshals a payload struct, encodes, decodes, and re-binds it; TestEnvelopeNilPayload asserts that a nil payload (heartbeats, acks) produces no payload key yet still yields valid JSON because of the omitempty tag.

Mirroring the Node S2C mutation semantics

The Node servers exposed two delta-style server-to-client broadcasts — S2C.ROOM_ACTION and S2C.GAME_ACTION — each carrying a payload plus a monotonically increasing sequence number so that clients could detect gaps and apply mutations in order. The Envelope reproduces this exactly: Seq is assigned server-side, per room, and monotonic. Anything sent through BroadcastSeq gets a fresh sequence stamp; ordinary events (chat, presence, acks) leave Seq at zero and are omitted from the wire. The mapping is one-to-one:

The Socket.IO compatibility gap

Scope — the one explicit known gap. Legacy browser clients speak the Socket.IO v4 wire protocol (Engine.IO framing, packet-type prefixes, namespace/ack envelopes, the HTTP long-poll handshake that upgrades to WebSocket). This package speaks a clean JSON object over a raw WebSocket — no Engine.IO, no packet prefixes, no long-poll fallback. The two wire formats are not interoperable. An unmodified Socket.IO client cannot connect to this server, and the Go server does not implement the Socket.IO handshake. This is the single deliberate break in the migration, and it is called out at the top of the package doc comment rather than hidden.

The design choice is intentional. Re-implementing Engine.IO framing in Go would import all of Socket.IO's accidental complexity (polling transport, binary attachment splitting, ack callbacks) for a transport we control on both ends. Instead the package preserves the semantics the clients actually depend on — named events plus seq-numbered mutation broadcasts — and closes the wire gap with a thin TypeScript client adapter.

Migration path

The adapter presents the existing socket.on(name, cb) / socket.emit(name, data) surface to game code, but underneath it opens a raw WebSocket and translates:

  • Outbound: emit(name, data) → JSON.stringify({event: name, payload: data}) over the socket.
  • Inbound: each message is JSON.parsed into an Envelope; the adapter dispatches to listeners registered for env.event, passing env.payload (and exposing env.seq for the mutation channels that already consume it).
  • Heartbeat: handled by the browser's native WebSocket ping/pong against the server's writePump ticker (below), replacing Engine.IO's heartbeat.

Because the event names are unchanged, game logic and reducers are untouched; only the transport shim is swapped. The migration sequence is: ship the Go hub on a new path (/rmhbox-ws/ etc.), point the adapter at it behind a feature flag, cut traffic over per service, then retire the Node Socket.IO processes.

Not yet built: the TS adapter is the documented bridge, not shipped code in this package. The Go side is complete and tested; the client shim is the remaining migration deliverable and is the only thing standing between this transport and the existing clients.

Hub — the per-service runtime

One Hub backs one service. It upgrades connections, performs soft auth, routes inbound events to registered handlers, and owns the room registry and cross-instance broadcast.

type Hub struct {
    origin    string
    logger    *log.Logger
    metrics   *telemetry.Metrics
    validator *auth.Validator
    bus       events.Bus

    upgrader websocket.Upgrader

    mu       sync.RWMutex
    conns    map[string]*Conn
    rooms    map[string]*Room
    handlers map[string]Handler

    onConnect    func(*Conn)
    onDisconnect func(*Conn)

    // RequireAuth rejects anonymous connections at handshake when true
    // (rmhmusic requires auth; the legacy games do not).
    RequireAuth bool

    ctx context.Context
}

ServeWS: upgrade and soft auth

ServeWS is the http.HandlerFunc mounted at the service's WebSocket path. It resolves identity from a token (query ?token=, the Authorization header, or a session cookie) but treats authentication as soft: an absent or expired token yields an anonymous connection rather than a rejection, because the legacy games allow anonymous play. Only when RequireAuth is set (rmhmusic) is an anonymous handshake refused.

token := r.URL.Query().Get("token")
if token == "" {
    token = httpx.SessionToken(r)
}
var (
    id   auth.Identity
    anon = true
)
if token != "" && h.validator != nil {
    resolved, err := h.validator.ValidateSession(r.Context(), token)
    switch {
    case err == nil:
        id, anon = resolved, false
    case errors.Is(err, auth.ErrUnauthenticated):
        // Expired/unknown token — legitimately anonymous, no log.
    default:
        // A real DB/transport error must be observable: otherwise an auth
        // outage silently downgrades every user to anonymous.
        h.logger.Warn("session validation error", "error", err.Error())
    }
}
if h.RequireAuth && anon {
    http.Error(w, "unauthorized", http.StatusUnauthorized)
    return
}

Note the deliberate three-way switch: a clean validation is authenticated, an ErrUnauthenticated is silently downgraded to anonymous (this is the common, expected case), but any other error — a database or transport fault — is logged at WARN so an auth outage cannot quietly demote every user to anonymous without anyone noticing. After the switch, the connection is upgraded, registered, counted in metrics, the onConnect hook fires, and the two pumps start:

ws, err := h.upgrader.Upgrade(w, r, nil)
if err != nil {
    return // upgrader already wrote the error
}
c := newConn(h, ws, id, anon)

h.mu.Lock()
h.conns[c.ID] = c
h.mu.Unlock()
if h.metrics != nil {
    h.metrics.ActiveConnections.Inc()
}
if h.onConnect != nil {
    h.onConnect(c)
}
go c.writePump()
go c.readPump()

Handler registration is setup-phase only

On, OnConnect, and OnDisconnect mutate hub state with no lock. This is safe only because of a documented contract: they must all be called during service setup, before ServeWS is mounted and starts accepting connections.

// On/OnConnect/OnDisconnect MUST be called during service setup, before ServeWS
// is mounted and starts accepting connections. The handler map is treated as
// immutable once serving begins, so no lock guards dispatch reads — registering
// after start would be a data race.
func (h *Hub) On(event string, handler Handler) { h.handlers[event] = handler }

The pay-off is a lock-free read path in dispatch: once serving begins the handler map is effectively immutable, so the hot path never contends on a mutex to look up a handler.

Dispatch with panic-recover fault isolation

Every inbound frame from a connection's read pump lands in dispatch. It records a per-event metric, looks up the handler (silently dropping unknown events), and runs it inside a recover so a panic in one handler cannot crash the whole service — porting the "try/catch around every handler" discipline from the Node rmhbox server.

func (h *Hub) dispatch(c *Conn, e Envelope) {
    if h.metrics != nil {
        h.metrics.MessagesTotal.WithLabelValues(e.Event).Inc()
    }
    handler, ok := h.handlers[e.Event]
    if !ok {
        return
    }
    // Fault isolation: a panic in one handler must not take down the service
    // (ports the try/catch-around-every-handler discipline from rmhbox).
    defer func() {
        if rec := recover(); rec != nil {
            h.logger.Error("handler panic", "event", e.Event, "panic", rec)
        }
    }()
    handler(c, e)
}

Conn — pumps, heartbeat, and the send channel

Each client is a Conn with exactly two goroutines: a read pump and a write pump. gorilla/websocket forbids concurrent writes to the socket, so every outbound frame funnels through a single buffered channel that only the write pump drains. No other goroutine ever touches the socket for writing.

const (
    writeWait      = 10 * time.Second
    pongWait       = 60 * time.Second
    pingPeriod     = (pongWait * 9) / 10
    maxMessageSize = 1 << 20 // 1 MiB, matching the Socket.IO maxHttpBufferSize
    sendBuffer     = 256
)

Read pump and the ping/pong heartbeat

The read pump sets a read limit and an initial read deadline, installs a pong handler that pushes the deadline forward on every pong, then loops decoding frames into the dispatcher. Malformed frames are skipped, not fatal. Any read error (including a missed pong deadline) exits the loop and triggers close via the deferred call.

func (c *Conn) readPump() {
    defer c.close()
    c.ws.SetReadLimit(maxMessageSize)
    _ = c.ws.SetReadDeadline(time.Now().Add(pongWait))
    c.ws.SetPongHandler(func(string) error {
        return c.ws.SetReadDeadline(time.Now().Add(pongWait))
    })
    for {
        _, raw, err := c.ws.ReadMessage()
        if err != nil {
            return
        }
        env, err := Decode(raw)
        if err != nil {
            continue // ignore malformed frames
        }
        c.hub.dispatch(c, env)
    }
}

The write pump owns the heartbeat: a pingPeriod ticker (54s, nine-tenths of the 60s pongWait) emits WebSocket pings. The peer's pongs reset the read deadline; if pongs stop, the read pump's deadline lapses and the connection is reaped. This reproduces Socket.IO's heartbeat so dead peers are detected within a bounded window.

func (c *Conn) writePump() {
    ticker := time.NewTicker(pingPeriod)
    defer func() {
        ticker.Stop()
        _ = c.ws.Close()
    }()
    for {
        select {
        case <-c.closed:
            // Graceful teardown: tell the peer and stop. Buffered frames are
            // dropped — realtime favors a clean disconnect over backlog flush.
            _ = c.ws.SetWriteDeadline(time.Now().Add(writeWait))
            _ = c.ws.WriteMessage(websocket.CloseMessage, nil)
            return
        case msg := <-c.send:
            _ = c.ws.SetWriteDeadline(time.Now().Add(writeWait))
            if err := c.ws.WriteMessage(websocket.TextMessage, msg); err != nil {
                return
            }
        case <-ticker.C:
            _ = c.ws.SetWriteDeadline(time.Now().Add(writeWait))
            if err := c.ws.WriteMessage(websocket.PingMessage, nil); err != nil {
                return
            }
        }
    }
}

Slow-consumer eviction

The send channel is buffered at 256 frames. Send is non-blocking: if the buffer is full, the consumer is too slow and the connection is torn down rather than allowed to wedge a shared room broadcast. A single stuck client must never block delivery to everyone else in the room.

The concurrency design (teaching highlight)

A real blocking bug, found in senior review. The naive design — "close the send channel to shut a connection down" — is a classic Go footgun. The send channel has many concurrent producers: Send called from heartbeat, grace timers, and room-broadcast goroutines all write to it. In Go, closing a channel that still has concurrent senders panics ("send on closed channel"), and selecting to write on a closed channel can also race. The first cut shipped this hazard; review caught it before it reached production.

The fix: a separate closed signal + closeOnce

The rule is: the channel carrying data is never closed. Shutdown is signalled out of band by closing a dedicated, payload-free closed chan struct{}, exactly once, via sync.Once. Producers select on closed to know they must not send; the write pump selects on it to exit.

type Conn struct {
    // ...
    send chan []byte
    // closed is closed exactly once when the connection is torn down. Producers
    // (Send / Room.deliverLocal, called from heartbeat/timer/broadcast
    // goroutines) select on it so they never write to a dead connection, and
    // writePump selects on it to exit cleanly. We deliberately NEVER close the
    // `send` channel — multiple goroutines write to it, and closing a channel
    // with concurrent senders panics. Closing `closed` + closing the socket is
    // the safe shutdown signal instead.
    closed chan struct{}
    // ...
    closeOnce sync.Once
}

The single safe send path is trySend. It checks closed first (cheap fast-fail), then does a three-way non-blocking select: deliver if there's room, abort if closed mid-send, or — if the buffer is full and the connection is still live — evict the slow consumer. Critically, none of these branches can panic, because no branch ever closes send.

func (c *Conn) Send(e Envelope) { c.trySend(e.Encode()) }

func (c *Conn) trySend(raw []byte) {
    select {
    case <-c.closed:
        return
    default:
    }
    select {
    case c.send <- raw:
    case <-c.closed:
    default:
        c.close()
    }
}

Teardown is idempotent and ordered. closeOnce guarantees the body runs once even if the read pump error and a slow-consumer eviction race to call it. It closes closed (unblocking every producer and the write pump), closes the socket (unblocking the read pump's blocking ReadMessage), and deregisters from the hub. The send channel is left for the garbage collector — never closed.

func (c *Conn) close() {
    c.closeOnce.Do(func() {
        close(c.closed)
        _ = c.ws.Close()
        c.hub.remove(c)
    })
}
Why this is correct. A chan struct{} can be safely read by any number of goroutines after it's closed (a closed channel always returns immediately), so it is the canonical Go broadcast-signal primitive. Closing it has exactly one writer (guarded by sync.Once), so there is no "close on closed" or "close with concurrent senders" hazard. The data channel keeps its many writers and is simply abandoned. This is the textbook "never close a channel from the receiver side / with multiple senders" pattern, applied precisely.

Room — members, sequencing, and local delivery

A Room is a broadcast group (a lobby, watch-party, music room, or game room). It holds the locally-connected members on this replica, a monotonic sequence counter, and — when a Bus is configured — one subscription so broadcasts from other replicas reach local members.

type Room struct {
    ID string

    mu      sync.RWMutex
    members map[string]*Conn // connID -> conn (this replica only)

    seq uint64

    // busCancel tears down the room's bus subscription when it empties.
    busCancel func()
}

// NextSeq returns the next monotonic sequence number for ordered mutation
// broadcasts (the seq field of S2C.ROOM_ACTION / GAME_ACTION).
func (r *Room) NextSeq() uint64 { return atomic.AddUint64(&r.seq, 1) }

NextSeq uses atomic.AddUint64, not the room mutex, so sequence assignment never contends with membership operations. TestRoomSeqMonotonic fires 1000 concurrent NextSeq calls and asserts all 1000 results are unique, verifying the ordering guarantee under contention.

Local delivery snapshots the member set under a read lock, then sends outside the lock so a slow trySend (or its eviction path) cannot block membership changes:

func (r *Room) deliverLocal(raw []byte) {
    r.mu.RLock()
    members := make([]*Conn, 0, len(r.members))
    for _, c := range r.members {
        members = append(members, c)
    }
    r.mu.RUnlock()
    for _, c := range members {
        c.trySend(raw)
    }
}

Room lifecycle: lazy create, teardown when empty

Rooms are created lazily on first Join and reaped on the Leave that empties them. The empty-check is re-validated under the write lock to avoid racing a concurrent Join that might re-populate the room between the read and the delete:

func (h *Hub) Leave(c *Conn, roomID string) {
    h.mu.RLock()
    room := h.rooms[roomID]
    h.mu.RUnlock()
    if room == nil {
        return
    }
    empty := room.remove(c.ID)
    c.removeRoom(roomID)
    if empty {
        h.mu.Lock()
        // Re-check under the write lock to avoid racing a concurrent Join.
        if r := h.rooms[roomID]; r != nil && r.Size() == 0 {
            if r.busCancel != nil {
                r.busCancel()
            }
            delete(h.rooms, roomID)
        }
        h.mu.Unlock()
    }
}

GraceTimers — porting the setTimeout disconnect grace

The Node room managers leaned on setTimeout for two recurring behaviors: a disconnect grace period (don't remove a player the instant their socket drops — give them seconds to reconnect: rmhtube ~120s, rmhmusic ~30s) and an empty-room GC reap. GraceTimers is the faithful Go port: keyed, cancellable, delayed actions where re-scheduling a key cancels its prior timer so the most recent call always wins.

func (g *GraceTimers) Schedule(key string, d time.Duration, fn func()) {
    g.mu.Lock()
    defer g.mu.Unlock()
    if t, ok := g.timers[key]; ok {
        t.Stop()
    }
    g.timers[key] = time.AfterFunc(d, func() {
        g.mu.Lock()
        delete(g.timers, key)
        g.mu.Unlock()
        fn()
    })
}

func (g *GraceTimers) Cancel(key string) bool {
    g.mu.Lock()
    defer g.mu.Unlock()
    if t, ok := g.timers[key]; ok {
        stopped := t.Stop()
        delete(g.timers, key)
        return stopped
    }
    return false
}

The reconnect pattern reads naturally: on disconnect, Schedule(userID, grace, removePlayer); on reconnect within the window, Cancel(userID) stops the pending removal. TestGraceTimerCancel confirms a cancelled timer never fires; TestGraceTimerReschedule confirms that scheduling the same key twice runs only the second action (reschedule cancels the prior). CancelAll drains every pending timer on shutdown.

Cross-replica broadcast via the bus

Because each replica only holds its own local members, a room spanning multiple pods needs a fan-out path — the role Socket.IO filled with its Redis adapter. Here it is the events.Bus. Broadcast does two things: deliver to local members immediately, and publish the same bytes to the room's bus topic for remote replicas.

func (h *Hub) Broadcast(roomID string, e Envelope) {
    raw := e.Encode()
    h.mu.RLock()
    room := h.rooms[roomID]
    h.mu.RUnlock()
    if room != nil {
        room.deliverLocal(raw)
    }
    _ = h.bus.Publish(h.ctx, busTopic(roomID), raw)
}

func (h *Hub) BroadcastSeq(roomID string, e Envelope) {
    h.mu.RLock()
    room := h.rooms[roomID]
    h.mu.RUnlock()
    if room != nil {
        e.Seq = room.NextSeq()
    }
    h.Broadcast(roomID, e)
}

BroadcastSeq is the ordered mutation path: it stamps the envelope with the room's next sequence number before broadcasting, so the seq is assigned once (on the origin replica) and travels with the bytes to every replica — preserving the global ordering that S2C.ROOM_ACTION / GAME_ACTION clients depend on.

Origin echo-skip

The subtlety is avoiding double-delivery: the origin replica already delivered locally in Broadcast, so when its own publish loops back through the bus it must be ignored. Each room's subscription drops any message whose Origin matches this hub's origin:

func (r *Room) subscribeBus(ctx context.Context, h *Hub) {
    ch, cancel, err := h.bus.Subscribe(ctx, busTopic(r.ID))
    if err != nil {
        h.logger.Warn("bus subscribe failed", "room", r.ID, "error", err.Error())
        return
    }
    r.busCancel = cancel
    go func() {
        for msg := range ch {
            if msg.Origin == h.origin {
                continue // already delivered locally by Broadcast
            }
            r.deliverLocal(msg.Payload)
        }
    }()
}

func busTopic(roomID string) string { return "rt:room:" + roomID }

The room teardown path (above) calls busCancel when the last local member leaves, so an empty room on this replica stops consuming the bus topic — the subscription lifecycle is tied exactly to local membership.
