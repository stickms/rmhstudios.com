The recap service ports server/recap/index.ts — the Lights Out daily-recap runner. Every five minutes it finds Discord channels whose recap is due, builds a rich embed summarising that guild's players for the recap date (medals, ratings, ranking), posts it, and clears the schedule. It also holds a Discord gateway connection so the bot shows as online. It serves /health and /metrics on RECAP_PORT (default 7004).

The 5-minute due-recap loop

CHECK_INTERVAL_MS = 5 * 60 * 1000 becomes checkInterval = 5 * time.Minute. Start opens the gateway, runs one recap pass immediately (mirroring the Node processDueRecaps() kick before the interval), then ticks. Each tick calls processDueRecaps, which fetches every due channel and processes them one at a time, logging — but not aborting on — per-guild failures.

The "always clear" behavior is the subtle, faithful part. Whether the post succeeds, the bot is forbidden (403), the channel is gone (404), or the transport itself errors, the recap schedule for that channel is cleared so it never retries forever:

status, postErr := r.poster.Post(ctx, ch.ChannelID, msg)
switch {
case postErr != nil:
    r.log.Warn("recap post error", "guildId", ch.GuildID, "error", postErr)
case status == 403 || status == 404:
    r.log.Info("bot lacks access — clearing recap", "guildId", ch.GuildID, "status", status)
default:
    r.log.Info("posted recap", "guildId", ch.GuildID, "channelId", ch.ChannelID)
}
// Always clear the recap schedule (faithful to the Node comment).
return r.repo.ClearRecap(ctx, ch.ID)

The due-channel query is the literal port of the Node predicate (activity='lights-out' with a non-null, past recapDueAt and a non-null recapDateKey):

SELECT "id", "guildId", "channelId", "recapDateKey"
  FROM "discord_activity_channel"
 WHERE "activity" = 'lights-out'
   AND "recapDueAt" IS NOT NULL
   AND "recapDueAt" <= $1
   AND "recapDateKey" IS NOT NULL

Building the embed: medals, ratings, ranking

Participants come from discord_daily_participant ordered by moves ASC NULLS LAST (completers with the fewest moves first; NULL-move players last), matching Prisma's orderBy moves:'asc' on Postgres. buildRecapEmbed then reproduces the Node embed exactly: the top three completers get 🥇🥈🥉, fourth onward get a plain bullet, each line shows the rating emoji (default 💡) and optional rating label, and a "did not finish" line counts the players still playing:

for i, p := range completed {
    medal := "▪️" // plain bullet for 4th+
    if i < len(medals) {
        medal = medals[i] // 🥇🥈🥉
    }
    emoji := "\U0001F4A1" // 💡 default
    if p.RatingEmoji != nil && *p.RatingEmoji != "" {
        emoji = *p.RatingEmoji
    }
    line := fmt.Sprintf("%s **%s** — %s %d move%s", medal, p.Username, emoji, moves, plural(moves))
    if p.RatingLabel != nil && *p.RatingLabel != "" {
        line += fmt.Sprintf(" (%s)", *p.RatingLabel)
    }
    lines = append(lines, line)
}

The description is assembled with blank separators then run through filterNonEmpty, which removes all empty strings — the exact behavior of the Node .filter(Boolean) over the whole array. The embed color is the amber 0xf59e0b from the source, and when an APP_ID is configured a "Play today's puzzle" link is appended pointing at discord.com/activities/<APP_ID>. The leaderboard image URL is rebuilt the same way the Node runner did, from SITE_URL plus the guild and date:

siteURL + "/api/discord/activity-image?type=leaderboard&guildId=" +
    guildID + "&dateKey=" + dateKey + "&recap=1"

buildRecapEmbed returns nil for an empty participant list, so the caller just clears the schedule — the Node if (participants.length === 0) early-out. Pluralization is the faithful n !== 1 ? 's' : '' rule (the test asserts "1 move" but never "1 moves").

Porting the Lights Out puzzle math

The recap header shows the day's shape and the optimal move count, which means the runner must recompute the puzzle from the date. lightsout.go ports just the pieces needed for (shapeLabel, optimalMoves): the Mulberry32 RNG, the SHAPES table (order matters — the seed indexes into it), generatePuzzle (difficulty scaling, the 100-attempt retry loop, deterministic fallback), and a brute-force solvePuzzle for the minimum solution:

func computePuzzleMeta(dateKey string) puzzleMeta {
    y, m, d, ok := parseDateKey(dateKey)
    if !ok {
        return puzzleMeta{shapeLabel: dateKey, optimal: -1}
    }
    seed := getDateSeed(y, m, d) // y*10000 + m*100 + d
    shape := getDailyShape(seed) // seed % len(shapes)
    label := getShapeLabel(shape)
    puzzle := generatePuzzle(createSeededRng(seed), shape)
    optimal := getOptimalMoves(puzzle, shape) // -1 == omit "Optimal" line
    return puzzleMeta{shapeLabel: label, optimal: optimal}
}
Scope — only the recap-facing helpers are ported. The interactive play helpers (cell toggling for live play, share/persistence) are deliberately out of scope: the runner only needs to describe the day's puzzle, not let anyone play it. recap_test.go verifies computePuzzleMeta is deterministic and that the seeded RNG reproduces its sequence (same seed → identical draws in [0,1)).

Gateway presence: discordgo replaces hand-rolled opcodes

This is the clearest improvement over the Node runner. The original hand-rolled the Discord gateway: op-10 hello, op-1 heartbeat on a setInterval, op-2 identify, op-7 reconnect, op-9 invalid-session, and a 30-second reconnect timer it had to maintain itself. The Go port hands all of that to discordgo:

session, err = discordgo.New("Bot " + botToken)
session.Identify.Intents = discordgo.IntentsNone // index.ts uses intents: 0
session.Identify.Presence = discordgo.GatewayStatusUpdate{
    Status: "online",
    Game:   discordgo.Activity{Name: "RMHBox", Type: discordgo.ActivityTypeGame},
}
// ... later, in Runner.Start:
if err := r.session.Open(); err != nil { ... }
Improvement. session.Open() manages the handshake, heartbeats, reconnect, and RESUME. The op-10/op-1/op-7/op-9 logic, the heartbeat interval, and the manual reconnect timer that index.ts maintained are gone. Posting uses the same library's REST path (ChannelMessageSendComplex), and the production Poster recovers the HTTP status from a *discordgo.RESTError so 403/404 are treated as non-fatal skips — preserving the exact Node semantics with far less code. When no bot token is set, the runner still serves health/metrics and simply skips posting (the Node if (!BOT_TOKEN) guard, minus the exit).
