The Go fleet shares one PostgreSQL database with the React SSR app — the same DSN, the same tables, the same Prisma-managed schema. Go does not own migrations and does not use an ORM; it reads and writes a well-defined subset of tables with hand-written, type-safe pgx queries. This section explains the data model boundary and how Go binds to it.

Who owns what

Prisma's schema has 98 models. The Go services do not model all of them — they model only the tables backing the extracted services. Everything else (the social graph, feed, blog, the bulk of game tables, build galleries) remains the React app's territory, accessed through Prisma. The shared session, user, and account tables are the contact surface: Go reads them to authenticate; Better Auth (in JS) writes them.

The access layer: pgx, not an ORM

All database access goes through one pgxpool.Pool built by pkg/db from the same DATABASE_URL the Node services use, with the pool size mirroring SERVER_DB_POOL_SIZE. This replaces both the Prisma client (used by the Node socket/worker services via server/shared/prisma-client.ts) and the ad-hoc raw pg.Pool clients embedded in the Node auth code — one uniform pool for the whole binary.

Each service keeps its SQL behind a small Repo interface so the business logic is unit-testable with a fake and the real implementation is a thin pgx layer. Column names and casing are taken verbatim from prisma/migrations so the quoted identifiers bind correctly.

// rmhtube: optimistic persistence behind a Repo interface
const q = `UPDATE "vibe_page"
           SET "thumbnailStale" = false, "thumbnailUrl" = $1
           WHERE "id" = $2 AND "updatedAt" = $3`
tag, err := pool.Exec(ctx, q, url, id, seenUpdatedAt) // 0 rows = lost the race

Typed models

pkg/db/models.go defines Go structs for the Go-owned tables, mapping Postgres types to Go: TIMESTAMP(3) → time.Time (*time.Time when nullable), JSONB → json.RawMessage, DOUBLE PRECISION → float64, SMALLINT → int16. Nullable scalars use pointers.

Authentication against the shared schema

The migration's keystone insight is that authentication needs no Better Auth library on the Go side — it needs one SQL query. Both the original Node socket services and the Go fleet validate a session identically:

SELECT s."userId", s."expiresAt", u."name", u."image", u."isAdmin"
FROM "session" s JOIN "user" u ON u."id" = s."userId"
WHERE s."token" = $1 LIMIT 1     -- then reject if expiresAt < now()

The token arrives as a websocket query parameter, a Bearer header, or the better-auth.session_token cookie (the value before the dot). Discord-Activity clients present a Discord user id instead, mapped to a platform user through the account table (providerId = 'discord'). Issuance, refresh, and OAuth remain entirely in Better Auth.

Migrations stay with Prisma

Go writes rows; it never alters the schema. PR #121's Helm pre-upgrade migration hook remains the single authority for schema evolution. The Go services assume the schema is already migrated — exactly as the Node workers did — and fail fast on a missing column rather than attempting DDL. This keeps one source of truth for the schema and avoids two systems racing to migrate the same database.

Concurrency-safe writes

Two write patterns deserve note. First, optimistic concurrency (vibe-worker): clear a flag only if the row's updatedAt is unchanged since it was read, so a concurrent edit is never clobbered. Second, atomic read-modify-write (rmhbox profile stats): the Node code read a JSONB blob, merged in memory, and wrote it back — a lost-update hazard under concurrent games. The Go port performs the merge inside a transaction with SELECT … FOR UPDATE, closing the race. Both are detailed in their service sections.
