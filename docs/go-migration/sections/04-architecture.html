The Go fleet is a single module that compiles into nine independent binaries, fronted by one gateway, sharing one PostgreSQL database and one Redis backplane. This section gives the bird's-eye view: topology, the request and realtime data paths, the deployment shape, and the design principles that hold it together.

Service topology

Every service is a cmd/<name>/main.go binary built from the same module, packaged as its own container, and run as its own Kubernetes Deployment. The gateway is the only externally-reachable service; it reverse- proxies websocket prefixes to the realtime services and everything else to the React SSR app. Workers hold no client connections at all.

One module, many binaries

A common question is whether "microservices" implies many repositories or modules. It does not. The fleet is a single Go module (github.com/rmhstudios/rmh-go) with shared libraries under pkg/ and one main package per service under cmd/. Each binary links only what it imports, ships as its own image, scales independently, and fails independently — the operational properties that define a microservice — while a single module keeps the shared contracts type-checked at compile time and refactors atomic.

The HTTP request path

A normal page or API request enters the gateway, which establishes the auth trust boundary (strip forged identity headers, validate the session, inject trusted headers) and forwards to the appropriate upstream. The gateway never blocks anonymous traffic — the SSR app renders its own auth pages.

The realtime data path

Realtime services upgrade the websocket at the gateway (HTTP 101 pass-through), then run on the shared realtime.Hub. Within a replica, a broadcast fans out to local room members directly; across replicas, it is published to the Redis backplane and re-delivered by every other replica's room subscription (skipping the origin to avoid duplicates). This is what lets a realtime service run more than one pod.

Design principles

PrincipleHow it shows up
Shared foundation, not copy-pasteNine services, one pkg/; auth/db/log/realtime written once.
One trust boundaryThe gateway is the only place identity headers are asserted.
Validate, don't reimplement, authGo trusts the Better Auth session table via one SQL query.
Scale-ready by constructionThe events.Bus abstraction flips single-replica → N-replica with one env var.
Fail independentlyPer-handler recover(); one bad game handler can't crash a service.
Compile-time honestyUnfinished work is a compiling stub behind a real interface, never broken code.
Reuse the existing infraHelm chart reuses PR #121's Secret, PVC, and conventions.

Deployment shape

In Kubernetes the fleet renders to nine Deployments, five ClusterIP Services (the ported services with client ports), a Redis Deployment+Service, and an optional Ingress — all from one Helm chart. Workers expose only /health and /metrics on a metrics port for probes and Prometheus. Section 19 details the chart; Figure 4.1 above shows the runtime wiring.
