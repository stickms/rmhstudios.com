Some concerns span every service: how identity flows, how the fleet scales, how it is observed, and how configuration and secrets are handled. These are solved once in the shared foundation and applied uniformly, which is the main payoff of the single-module design.

Authentication & the trust boundary

There is exactly one place identity is asserted: the gateway. It strips every inbound X-Rmh-* header (by prefix, so future headers can't be forged), validates the Better Auth session, and injects trusted headers downstream. Realtime services additionally validate the token themselves at websocket handshake (defense in depth, since a client connects to a WS prefix the gateway proxies). Both paths use the same auth.Validator and the same SQL.

A real DB/transport error during validation is logged and metered rather than silently treated as "anonymous" — so an auth-database outage is observable instead of quietly downgrading every user. (This distinction was added during senior review.)

Horizontal scaling & the backplane

The realtime services hold per-room state in memory, so a naive second replica would split a room across processes that can't see each other. The events.Bus abstraction solves this: a broadcast is delivered to local room members and published to a topic; every other replica's room subscription re-delivers it locally, skipping its own origin to avoid duplicates. Flipping between single- and multi-replica is one environment variable.

REDIS_URLBusRealtime replicas
unsetevents.Local (in-process)1 (today's behavior)
setevents.Redis (pub/sub)N, with sticky sessions at ingress

This is precisely the "socket.io Redis adapter + sticky sessions" work PR #121 named as Stage 1 and "the real first scaling investment." Here it is a first-class part of the foundation, language-agnostic in spirit and required by no service-specific code.

Observability

Every service constructs a per-service Prometheus registry via pkg/telemetry and exposes /metrics — on the client port for HTTP/WS services, on a dedicated metrics port for workers. Standard instruments ship out of the box: rmh_active_connections (gauge), rmh_messages_total{event}, rmh_db_queries_total{outcome}, and rmh_job_runs_total{job,outcome}. Structured JSON logs (pkg/log) match the Node logger's field shape so existing log tooling keeps working. This is the instrumentation half of PR #121's Stage 2; wiring dashboards/alerts is the remaining operational step.

Configuration & secrets

All configuration is environment-driven through pkg/config, which reads the same variables the Node services used — DATABASE_URL, SERVER_DB_POOL_SIZE, the PORT_* family, the Discord and DeepSeek keys. In Kubernetes the values come from PR #121's rmhstudios-secrets Secret, mounted as envFrom on every Deployment; the Go chart adds no new secret and never puts secrets in git or Helm history. REDIS_URL and the gateway's *_UPSTREAM values are the only genuinely new variables, and both have sensible defaults.

Resilience patterns

  • Fault isolation: every client-driven handler runs under recover(); a panic in one game or one connection cannot crash a service.
  • Graceful shutdown: HTTP/WS services drain on SIGTERM within a grace period; workers stop their tickers and cancel their context; the discord-bot honors a 90s grace for in-flight AI work.
  • Fail-fast startup: services ping the database on boot (or wait for it, for workers) and exit non-zero on misconfiguration, so a broken pod restarts rather than serving errors.
  • Backpressure: slow websocket consumers are evicted rather than allowed to wedge a shared room broadcast.
