The rmhtube service is the Go port of the standalone RmhTube watch-party server (server/rmhtube/*). It hosts synchronized video rooms: a group of viewers share one playback head, a media queue, and a chat. One leader's player state is authoritative and the server fans the projected position out to everyone on a heartbeat. It serves the realtime hub at /rmhtube-ws/ (default port 7003, RMHTUBE_PORT) plus /health and /metrics, and restores its active rooms from the database before it starts listening so a restart does not vaporize a session. The package lives in internal/rmhtube/ (events, types, manager, room, sync, queue, chat, snapshot, repo, utils) and is wired in cmd/rmhtube/main.go.

Service shape and startup

The binary is a single WebSocket runtime. main.go loads the common config, opens the database optionally (a failed open degrades to a fully in-memory service, exactly as the legacy server started anyway when DB restore failed), constructs the shared realtime.Hub with RequireAuth: false (anonymous viewers are allowed), registers the manager's handlers, and — critically — restores active rooms before the listener accepts connections:

mgr := rmhtube.NewManager(ctx, hub, repo, logger, metrics)
mgr.Register()
defer mgr.Stop()

// Restore active rooms before listening
// (mirrors restoreRoomsFromDb().then(listen)).
if err := mgr.RestoreRoomsFromDB(); err != nil {
    logger.Error("db_restore_failed", "error", err) // start anyway
}

mux.HandleFunc("/rmhtube-ws/", hub.ServeWS)
addr := ":" + config.GetString("RMHTUBE_PORT", "7003")

Because auth is optional, identity falls back gracefully. userOf returns the authenticated account id when present and the per-connection id otherwise, so the room machinery functions for guests; nameOf defaults to "Guest".

Wire protocol: events, snapshot, and seq deltas

The hub speaks a JSON envelope whose Event field carries the exact strings ported 1:1 from lib/rmhtube/events.ts, so the existing TypeScript client adapter maps cleanly onto the Go server. Events are grouped into room lifecycle, video sync, queue, reactions, and chat. A representative slice of the client→server (C2S) and server→client (S2C) constants:

// Client -> Server (excerpt)
c2sRoomCreate    = "rmhtube:room:create"
c2sRoomJoin      = "rmhtube:room:join"
c2sSyncHostState = "rmhtube:sync:host_state"
c2sSyncPlay      = "rmhtube:sync:play"
c2sSyncSeek      = "rmhtube:sync:seek"
c2sSyncSetSpeed  = "rmhtube:sync:set_speed"
c2sQueueAdd      = "rmhtube:queue:add"
c2sQueueVoteSkip = "rmhtube:queue:vote_skip"
c2sQueueShuffle  = "rmhtube:queue:shuffle"

// Server -> Client (excerpt)
s2cRoomStateSnapshot = "rmhtube:room:state_snapshot"
s2cRoomAction        = "rmhtube:room:action"
s2cSyncState         = "rmhtube:sync:state"
s2cSyncMediaChanged  = "rmhtube:sync:media_changed"

The protocol is snapshot + seq-numbered delta. On join the server sends a full room:state_snapshot built by buildClientState; every subsequent mutation goes out as a room:action envelope carrying a monotonically increasing per-room sequence number. The single broadcast primitive is broadcastAction, which bumps the room's Seq under the room lock and fans the envelope to the whole hub room:

// broadcastAction bumps the room seq and fans S2C.ROOM_ACTION
// {type,payload,seq,timestamp} to the whole room. The caller MUST
// hold the room lock (so seq stays consistent with snapshots).
func (m *Manager) broadcastAction(r *room, actionType string, payload any) {
    r.Seq++
    m.hub.Broadcast(r.ID, realtime.MustEnvelope(s2cRoomAction, map[string]any{
        "type":      actionType,
        "payload":   payload,
        "seq":       r.Seq,
        "timestamp": nowMs(),
    }))
}

Holding the room lock while incrementing Seq is what keeps the seq on a snapshot consistent with the seqs on the deltas that follow it: a client can discard any delta whose seq it has already applied and detect gaps. A unit test (TestBroadcastActionSeqMonotonic) drives 200 actions and asserts the seq lands at exactly 200.

Leader-authoritative video synchronization

This is the heart of the service and a faithful port of server/rmhtube/sync-engine.ts. The model: one member is the leader (initially the host). The leader's local player is the source of truth. The leader's client reports its player state roughly once a second via SYNC_HOST_STATE; the server stores it as the canonical videoState but does not rebroadcast it. Instead, a 2-second heartbeat broadcasts the projected state to every multi-member room that has something playing. Explicit leader actions (play / pause / seek / set-speed) broadcast immediately.

The projection is the key computation. When playing, the live position is the stored position plus the wall-clock seconds elapsed since updatedAt, scaled by playback rate — so a viewer receiving a heartbeat sees where the leader actually is, not where the leader was one report ago. Paused state is returned unchanged (the position is frozen), and a clock-skew report from the future is clamped so a viewer can never be rewound:

func projectedState(v videoState, now int64) videoState {
    out := v
    if v.Playing {
        elapsed := float64(now-v.UpdatedAt) / 1000.0
        if elapsed < 0 {
            elapsed = 0
        }
        out.CurrentTime = v.CurrentTime + elapsed*v.PlaybackRate
        out.UpdatedAt = now
    }
    return out
}

Three unit tests pin this behavior: a 1× projection 5s later yields 10 + 5 = 15; a 2× projection 4s later yields 10 + 8 = 18; a paused video stays at 42; and a negative elapsed does not rewind below 10. Authority is enforced by leaderRoom, which returns the user's room only if r.LeaderUserID == userID — every sync handler short-circuits for a non-leader. The handlers store under the lock, then release before any websocket send (websocket writes never happen under the lock):

func (m *Manager) onSeek(c *realtime.Conn, e realtime.Envelope) {
    var p struct{ Time float64 `json:"time"` }
    if err := e.Bind(&p); err != nil { return }
    ent, r, ok := m.leaderRoom(userOf(c))
    if !ok { return }
    r.VideoState.CurrentTime = p.Time
    r.VideoState.UpdatedAt = nowMs()
    r.LastActivity = nowMs()
    roomID := r.ID
    ent.mu.Unlock()
    m.broadcastExcept(roomID, c.ID, s2cSyncSeek, map[string]any{"time": p.Time})
}

Play / pause / seek use broadcastExcept (the leader applies them locally, so it is excluded), while SET_SPEED goes to the whole room — matching the TS io.to vs socket.to distinction. The 2s heartbeat tick (tickHeartbeat) snapshots the registry, then per room broadcasts projectedState only when CurrentItem != nil and there is more than one member (no point synchronizing an audience of one). When media changes, onMediaChanged resets the videoState to a clean paused state at t=0, rate=1 and emits SYNC_MEDIA_CHANGED.

Rooms, queue, and chat

A room owns the full watch-party state: members, queue, current item, video state, chat ring buffer, skip votes, played history, bans, and invite links. Room creation generates a 6-character code from an unambiguous alphabet (retrying up to 10 times on collision), installs the creator as host + leader + first member, joins the hub room, persists the row fire-and-forget, and sends the creator both room:created and a full snapshot.

The queue (port of media-queue.ts) supports add, remove, reorder, play-item, skip, vote-skip, queue-voting, and shuffle. Permissions are role-gated: adding requires host or AllowMemberQueue; reorder / play-item / skip / shuffle require the leader; removing your own item is allowed. Adding the first item with AutoPlay on auto-starts playback. Vote-skip computes a ceiling-of-half threshold against the active member count and advances when met:

r.SkipVotes[userID] = struct{}{}
active := r.activeCount()
votesNeeded := (active + 1) / 2 // ceil(active/2)
voters := setToSlice(r.SkipVotes)
m.broadcastAction(r, "VOTE_SKIP_UPDATED", map[string]any{
    "voters": voters, "votesNeeded": votesNeeded, "totalMembers": active,
})
if len(r.SkipVotes) >= votesNeeded {
    m.broadcastAction(r, "VOTE_SKIP_PASSED", map[string]any{})
    m.advanceQueue(r)
}

Reorder uses moveItem, which clamps the target position into range and reindexes positions; tests cover the move, the high/low clamps, and the no-op-on-missing case. Shuffle is a Fisher–Yates over the items after the current index (so the now-playing item is never displaced). Chat (port of chat-handler.ts) sanitizes and truncates content to 300 chars, resolves replies, validates mentions against current members, appends to a 200-message ring buffer, and supports typing indicators (with a 3s auto-expiring timer), per-message emoji reactions (toggle add/remove), and host-only pinning. Reactions and typing fan out with broadcastExcept so the sender does not echo their own event.

Concurrency model

The brief is “per-room mutex; many goroutines,” and the manager implements it precisely. A top-level registryMu guards the global maps (rooms, userRoomIndex, userConns); each roomEntry carries its own mutex, and every read/write of a room's state happens while holding it. The discipline is: acquire the room lock, mutate, collect the data to send, release, then broadcast — so websocket writes and DB calls never run under the lock. Grace and GC timers run on their own goroutines and re-acquire locks as needed.

type roomEntry struct {
    mu sync.Mutex
    r  *room
}

Disconnect grace. On disconnect the member is marked offline and a 120s grace timer (via the shared realtime.GraceTimers) is armed; if the member reconnects within the window, rejoin cancels the timer and sends a fresh snapshot. Otherwise the member is removed. Empty rooms are scheduled for disband after a 5-minute timeout, and a 60-second GC ticker sweeps for rooms that are empty-past-timeout or idle-past-30-minutes as a safety net.

Review fixes carried in the port. Two correctness fixes from review are visible in the code. First, removeMember prunes the departing user's votes — it deletes them from SkipVotes and from every QueueVotes set — so vote thresholds (computed against activeCount) are never measured against ghosts: 
delete(r.SkipVotes, userID)
for itemID, voters := range r.QueueVotes {
    delete(voters, userID)
    if len(voters) == 0 {
        delete(r.QueueVotes, itemID)
    }
}
Second, disbandRoom holds the room lock across the entire teardown (broadcast, index cleanup, and registry delete) so a concurrent onRoomJoin cannot add a member to a room being deleted. Correspondingly onRoomJoin re-verifies the room is still registered after it acquires ent.mu, because a concurrent disband may have removed it from the registry while the join was blocked on the lock — adding a member to a disbanded entry would strand it.

Persistence behind a Repo

Unlike the games hub (which keeps its core DB-free), rmhtube persists, so the room logic depends only on a Repo interface — the room / sync / queue / chat handlers are exercisable in tests with the in-memory NopRepo{}. All Repo calls are made fire-and-forget through a persist helper that logs failures and counts a metric, so a slow or failing database never blocks a real-time mutation (faithful to the TS .catch(...) pattern around every Prisma call).

The production PgxRepo runs raw pgx against the Prisma-managed, quoted-identifier schema (column names kept verbatim with the TS DDL). Startup restore selects every not-closed room, hydrates its live members (joined to "user" for name/image), its unplayed queue (ordered by position), and recent chat, and skips rooms with no live members — mirroring restoreRoomsFromDb. Restored members start disconnected and reconnect over the websocket. A join miss for a room not in memory triggers a single on-demand LoadRoom. Membership uses an upsert on (roomId,userId) that clears leftAt on re-join, and position rewrites run in a transaction.

YouTube URL parsing is pure string ops

utils.go ports lib/rmhtube/utils.ts with no network access whatsoever. detectMediaType classifies a URL as youtube / twitch / direct (or "" for unsupported) by host and a file-extension regex; extractYouTubeID pulls the id from youtu.be/<id> or the ?v= query parameter; and the thumbnail URL is templated against img.youtube.com. There is no YouTube Data API call — title resolution that the TS version stubbed ("YouTube Video (<id>)") is reproduced exactly. Room codes use crypto/rand over the unambiguous alphabet; opaque ids use a nanoid analog.

Scope: what is faithful vs simplified. The load-bearing watch-party core — leader-authoritative sync and its projection, the queue (add/remove/reorder/play/skip/vote-skip/shuffle/loop), chat with replies/mentions/reactions/typing/pinning, the snapshot+seq-delta protocol, the disconnect-grace + GC lifecycle, and room/member/queue persistence and restore — is a faithful, behavior-for-behavior port. The niche Phase-4 social features are intentionally simplified: the entry gate for bans is kept real (a banned user is rejected at join with BANNED), but the surrounding ban / unban / invite-link bookkeeping is in-memory only and not persisted, and the invite codes are generated but not enforced as a join path. Chat persistence is wired as an optional chatWriter extension a Repo may implement rather than part of the required interface — the in-memory ring buffer remains authoritative for live rooms either way. None of this is invented behavior; it is the deliberate faithful-core / simplified-periphery split called out in the source comments.
