A migration of this magnitude only earns trust if it is precise about what it actually achieves. This section states the goals, the explicit non-goals, and a per-component honesty table separating faithful 1:1 ports from working frameworks with stubbed long tails.

Goals

  • Re-implement every Node realtime/worker process as a compiling, tested Go service.
  • Preserve external behavior: same ports, same event names, same database tables, same auth model.
  • Provide a shared foundation so the services are consistent, not nine snowflakes.
  • Ship a deployment story that builds on PR #121 (Helm conventions, reused Secret/PVC) rather than replacing it.
  • Implement the Stage-1 scaling primitive (Redis backplane) so realtime services can run more than one replica.
  • Verify rigorously: build, vet, race-detector unit tests, and end-to-end against real infrastructure.

Non-goals

  • Rewriting the React SSR web app. The TanStack Start renderer and its ~130 API routes stay in JavaScript. The Go gateway proxies to it.
  • Replacing Better Auth. Session issuance, OAuth callbacks, and cookie signing remain in Better Auth. Go services only validate sessions.
  • Replacing Prisma migrations. The schema and PR #121's migration hook remain authoritative; Go uses hand-written pgx queries against the same tables.
  • Porting every minigame and every RMHBot tool in one pass. Those ship as frameworks + representative implementations (see below).
  • Production cutover. This is a branch deliverable with a documented cutover/rollback plan, not a live switch.

Why this matters

A migration document that claims total completion would be dishonest and dangerous: someone would deploy it expecting nine fully-equivalent services and be bitten by the gaps. By stating the boundaries explicitly — and by making every stub a compiling stub behind a real interface — the codebase stays trustworthy: it builds and runs today, and the remaining work is additive (implement a stub against an existing contract) rather than structural.
