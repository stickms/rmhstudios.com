Performance was not the stated motivation for this migration — correctness and faithful behavior were — but a Go rewrite of a realtime tier brings real efficiency characteristics worth recording, along with the specific optimizations baked into the implementation.

The concurrency model

Each websocket connection runs exactly two goroutines (a read pump and a write pump) over a buffered send channel — the idiomatic gorilla pattern. Go's scheduler multiplexes thousands of such goroutines onto a handful of OS threads, so a single replica can hold a large number of idle connections cheaply, where the Node event loop served the same role single-threaded. Per-room state is guarded by fine-grained per-room mutexes rather than one global lock, so unrelated rooms never contend.

Footprint

The runtime footprint is modest by construction: a static Go binary on Alpine, a few MB of RSS at idle, and image sizes in the tens of MB. The Helm defaults request 25m CPU / 32Mi memory per pod with 1 CPU / 256Mi limits — deliberately small, on the assumption that connection-bound realtime work is cheap per connection and scales by replica count rather than per-pod size.

Scaling characteristics

The dominant scaling axis is concurrent websocket connections. With the Redis backplane enabled, a realtime service scales horizontally: add replicas behind sticky-session ingress and the bus keeps rooms coherent across them. Workers scale differently — they are singletons by nature (one scheduler, one poller), so they scale up (more DB pool, larger batches) rather than out, and the chart keeps them at one replica.

Honest caveat: no load testing was performed as part of this work. The performance claims here are structural (what the design enables), not measured benchmarks. Establishing connection-count and latency baselines is explicit future work (Section 25).

Versus the Node services

The Go services should use less memory per connection and start faster than the Node equivalents, and the per-room locking gives true parallelism across rooms on multiple cores (where Node's single event loop did not). These are reasons a Go realtime tier is attractive if connection volume ever justifies it — which is precisely the metrics-driven condition PR #121 set for Stage 4. This migration provides the implementation; the metrics to justify adopting it are the missing input.
