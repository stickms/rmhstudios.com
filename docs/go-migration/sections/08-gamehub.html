The gamehub service is the Go port of the unified Socket.IO games server (server/socket-server): one WebSocket runtime hosting every mini-game relay behind a single shared realtime.Hub on the path /socket/, listening on SOCKET_PORT (default 7001). Auth is soft — anonymous connections are allowed, matching the legacy soft-auth middleware. Kowloon Knockout is ported faithfully and standalone (kk.go); the remaining games are represented by the generic RelayGroup framework (relay.go) plus a clearly-marked, compiling registry of stubs (manager.go).

Role: one WebSocket runtime for every game

The Node deployment ran a single Socket.IO process (server/socket-server) whose index.ts registered every game's handlers on connect and called every game's disconnect handler on disconnect. gamehub reproduces that shape on top of the shared realtime.Hub. The entrypoint (cmd/gamehub/main.go) builds the hub with RequireAuth: false — soft-auth, anonymous allowed — and serves it on /socket/ alongside /health and /metrics:

hub := realtime.NewHub(ctx, realtime.Options{
    Origin:       host,
    Validator:    validator,
    AllowOrigins: config.GetCSV("SOCKET_CORS_ORIGIN"), // empty => allow all
    RequireAuth:  false,                               // soft-auth: anonymous allowed
})
mgr := gamehub.NewManager(hub, logger)
mgr.Register()
mux.HandleFunc("/socket/", hub.ServeWS)
addr := ":" + config.GetString("SOCKET_PORT", "7001")
The session DB is treated as optional. The legacy server allowed connections even when auth was unavailable, so a failed DB open here degrades to fully-anonymous (“db unavailable, running anonymous-only”) rather than being fatal.

The connection registry

Host-authoritative relays must deliver a frame to a specific peer (the host, or the guest) rather than broadcast to a room. The hub dispatches handlers per-Conn but exposes no public “send to a connection id” lookup, so the package keeps its own connRegistry mapping connID -> send callback (relay.go). The Manager owns one shared registry and feeds it from a single OnConnect/OnDisconnect pair, fanning disconnect cleanup out to every game (mirroring index.ts):

m.hub.OnConnect(func(c *realtime.Conn) {
    conn := c
    m.reg.add(conn.ID, func(e realtime.Envelope) { conn.Send(e) })
})
m.hub.OnDisconnect(func(c *realtime.Conn) {
    m.kk.Cleanup(c.ID)
    for _, g := range m.relays { g.Cleanup(c.ID) }
    m.reg.remove(c.ID)
})

Keeping the sender behind a func(realtime.Envelope) callback rather than a live *realtime.Conn is what makes the relay cores unit-testable: tests register plain string ids with an in-memory capture callback, no websocket required.

Kowloon Knockout: faithful 1v1 host-authoritative relay

Kowloon Knockout (kk.go) is a faithful Go port of server/socket-server/handlers/kowloon-knockout.ts. It is a simple 1v1 host-authoritative relay: the host creates a room, a guest joins, then input frames (guest -> host) and game_state frames (host -> guest) are relayed; a rematch handshake re-arms the match. It is ported standalone rather than via RelayGroup because of the rematch handshake and fighter-class bookkeeping, which are kept verbatim against the source of truth.

State

KKManager holds a rooms map (code -> *KKRoom), a connToRoom reverse index (connID -> code), and a sync.Mutex guarding both, plus the shared connRegistry. A KKRoom mirrors the TS interface: the host and (initially empty) guest connection ids, each side's fighter class, and the pending rematch-handshake state (HostReady/GuestReady and PendingHostClass/PendingGuestClass). The default fighter is stone_tiger.

Relay direction is enforced

The relays are strictly directional and seat-checked. Input only relays guest -> host (it returns early unless room.GuestConnID == connID); GameState only relays host -> guest (requires room.HostConnID == connID and a seated guest). Relayed frames reuse the inbound payload bytes verbatim (realtime.Envelope{Event: kkInput, Payload: payload}), so the gateway / hub never re-serializes game data.

Rematch handshake

FighterReady records the caller's pending fighter selection, sets that seat's ready flag, and notifies the opponent with kk:opponent_ready. When both seats are ready it commits the pending classes onto the room, clears the ready/pending state, and re-arms the match by emitting a fresh kk:room_joined to both players. Resetting the flags is what prevents a later single-side ready from spuriously re-triggering a rematch.

Room codes

Codes are 6 characters from the alphabet ABCDEFGHJKLMNPQRSTUVWXYZ23456789 (matching the socket-server config), generated with crypto/rand instead of Math.random. uniqueCode ports generateUniqueCode: up to 20 attempts, falling back to the last candidate rather than panicking — collisions in the 326 space are astronomically unlikely, so it degrades gracefully where the legacy code threw.

The generic RelayGroup framework

The legacy Node server hand-rolled the same 1v1 “host creates / guest joins / relay input+state” shape across several handlers. Rather than copy that boilerplate per game, RelayGroup (relay.go) captures it once and lets each game register onto the hub with its own event names. A RelayConfig declares the full wire surface — the inbound CreateRoom/JoinRoom/Input/State/Leave events and the outbound RoomCreated/RoomJoined/Disconnected/Errorf events — with every name fully qualified so the mapping to the legacy Socket.IO constants stays explicit and greppable.

RelayGroup implements only the common create/join/input/state/leave flow over a two-seat RelayRoom (Code, Host, optional Guest), with the same mutex + rooms/connToRoom structure and the same directional relay and seat checks as KK. Multiple games can run as separate RelayGroup instances sharing one connRegistry and hub. Games that need extra semantics — like KK's rematch handshake — are ported standalone instead.

Scope — what is faithful vs framework+stub. Kowloon Knockout is a faithful, verbatim port with full game semantics and a lifecycle test suite. The other games are framework + stub: RelayGroup is real, reusable, and tested-by-construction, but the per-game registrations are intentionally not wired with real event names or rules yet. buildRelayRegistry returns an empty slice today, so only KK is exposed on the wire; each remaining game is documented as a precise plug-in point.

manager.go enumerates the remaining games as a TODO(migration) registry — Slice It, Neon Driftway, Synapse Storm, Blackjack (noted as multi-seat, needing a richer model than the two-seat RelayRoom), plus Holdem, Baccarat, Roulette, Lights Out, Doctrine, RMH Type/Study, Altair, and RMH Music. The Kowloon Knockout lifecycle test

kk_test.go exercises the KK core directly on connection-id strings plus a capture sink that records the envelopes a fake connection receives — no live websocket. newKKHarness wires a KKManager around a registry with two fake conns (host-conn, guest-conn). The suite covers the full lifecycle:

  • TestKKCreateAndJoin — create yields kk:room_created with a non-empty code; join delivers kk:room_joined to both seats with the right isHost flag and the committed hostClass/guestClass (iron_crane / jade_fox).
  • TestKKJoinErrors — an unknown code returns “Room not found”; a third connection joining an occupied room returns “Room is full”.
  • TestKKRelay — guest input relays to host and host game_state relays to guest, while the reverse directions are asserted to not relay (directionality enforced).
  • TestKKRematchHandshake — one side ready notifies the opponent but does not start a rematch; both ready commits the pending classes (new_host / new_guest) and re-emits kk:room_joined; and the handshake flags are confirmed reset so a later single-side ready does not re-trigger.
  • TestKKDisconnectNotifiesOpponentAndCleansUp — host cleanup sends kk:opponent_disconnected to the guest, empties both the rooms and connToRoom maps, and confirms a subsequent stale input is a no-op.
Because the relay cores resolve peers through the registry callback rather than a live *realtime.Conn, this entire lifecycle runs in-memory and under the race detector, giving high-confidence coverage of the one game that is fully ported.
