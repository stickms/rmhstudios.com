This report documents the complete migration of rmhstudios.com's realtime and background-worker tier from Node.js/TypeScript to a fleet of Go microservices. It records what was built, the mechanisms behind each piece, how the work was verified, and — candidly — what is a faithful 1:1 port versus what is a scaffolded framework awaiting incremental completion.

What was delivered

A new Go module, github.com/rmhstudios/rmh-go, containing a shared foundation (pkg/) and nine independently-deployable service binaries (cmd/): an edge gateway, four realtime websocket services (gamehub, rmhbox, rmhtube, rmhmusic), and four worker/utility services (doctrine-worker, recap, vibe-worker, discord-bot). Each builds into its own minimal container image and deploys through a Helm chart (deploy/helm/rmhstudios-go) that deliberately mirrors the conventions established in PR #121, reusing the same Kubernetes Secret and PVC and adding a Redis backplane for horizontal scaling.

Why this is notable (and the honest caveat)

PR #121 — the k3s/Helm/Terraform infrastructure migration — recorded an explicit architecture decision not to rewrite the backend in Go, on the grounds that the app was already process-split, the web tier is React SSR (must stay JavaScript), and the auth/data layers are Better Auth + Prisma. This project deliberately overrides that decision on an isolated feature branch. The report does not pretend the decision was wrong; it documents an alternative universe in full engineering detail and reconciles it with PR #121's chart.

Faithful vs scaffolded. Wherever a component is genuinely portable, it is ported faithfully and tested (auth validation, the realtime framework, doctrine determinism, watch-party/music sync, the Discord chat bot, thumbnail rendering). Wherever it is not — the React SSR renderer and Better Auth's OAuth/cookie issuance — it stays in JavaScript and the Go services integrate with it. A few large surfaces (the rmhbox minigame catalogue, the RMHBot agentic editor's tool set) ship as a working framework plus one real implementation and clearly-marked compiling stubs. Every such boundary is called out in its section with a purple scope note.

Verification at a glance

This is not a paper design. The entire module compiles (go build ./...), passes go vet ./..., passes every unit and integration test under the race detector (go test -race ./...), and passes an end-to-end suite that launches the real service binaries against a real PostgreSQL and drives them with real websocket clients. The Helm chart lints and validates under kubeconform -strict; the container image builds and runs. A three-part senior engineering review was run before this document was written; its findings — including a genuine data race the race detector caught — were fixed and re-verified. Section 22 records that loop in full.
