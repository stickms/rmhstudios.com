The fleet has a CI pipeline that gates every change on build, vet, race-tested units, formatting, end-to-end tests, and chart validation; a deploy script that mirrors PR #121's single-node-or-registry switch; and a documented, reversible cutover plan.

Continuous integration

.github/workflows/go-microservices.yml runs three jobs on changes under go-services/ or the chart:

JobSteps
build-testgo build · go vet · go test -race · gofmt check
e2ePostgres service container → load schema.sql → run.sh --no-docker (build + launch binaries + ws-client tests)
helmhelm lint · helm template render
The --no-docker mode of the e2e script exists specifically for CI: GitHub Actions already provides a Postgres service container and a Go toolchain, so the script builds and launches the binaries directly on the runner instead of nesting Docker. (This branch in run.sh was added during review when the CI/script contract didn't match.)

The deploy script

deploy/deploy-go.sh is the Go analog of PR #121's deploy-k8s.sh. It builds one image per service (with the right EXTRA_PKGS per service), then either imports them straight into k3s's containerd for single-node, or pushes to a registry for multi-node — the same switch PR #121 introduced — and finishes with an atomic Helm upgrade.

# single-node k3s (imports into containerd):
./deploy/deploy-go.sh production
# multi-node (push to a registry):
REGISTRY=registry.example.com ./deploy/deploy-go.sh production

helm upgrade --install rmhstudios-go deploy/helm/rmhstudios-go \
  -f values-prod.yaml --set image.tag=$SHA --atomic --wait

Cutover & rollback

Because the gateway proxies anything it doesn't own back to the existing Node SSR app, cutover can be incremental and is fully reversible:

  1. Deploy rmhstudios-go beside the existing stack; it shares the DB and Secret.
  2. Smoke-test the gateway and each realtime service in-cluster.
  3. Shift one websocket prefix at a time to the Go service at the ingress (e.g. start with /socket/ → gamehub), leaving the rest on the Node services.
  4. Watch metrics; if a service misbehaves, route that prefix back to the Node equivalent — both run against the same database.
  5. Full rollback: point the ingress back at the Node stack and helm uninstall rmhstudios-go. No data migration is involved because both stacks use the same tables.
The realtime services do not yet speak the Socket.IO wire protocol (see Section 6). A real cutover of a websocket prefix requires the client-side adapter described there. Until then, the Go realtime services are exercised by the Go e2e clients and the new-protocol path, not by the existing browser clients — this is the single most important cutover caveat and is stated plainly.

Operational notes

  • Migrations: unchanged — PR #121's Prisma hook owns the schema; the Go deploy assumes it has already run.
  • Scaling: kubectl scale deploy/rmhstudios-go-gamehub --replicas=N once Redis + sticky sessions are in place.
  • Secrets: rotate via the existing rmhstudios-secrets flow; the Go fleet picks them up on next rollout.
