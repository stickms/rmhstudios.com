The discord-bot service is the Go port of server/discord-bot/*.ts: a long-running Discord gateway bot (no client-facing HTTP) built on bwmarrin/discordgo. It powers two features — the /chat AI persona ("Alex Wu"), which is faithfully ported end-to-end, and the /rmhbot agentic website editor, which is ported as an honest scaffold. This section documents the gateway lifecycle, command registration, interaction routing, the DeepSeek client, the chat embed-packing logic, the rmhbot agentic tool loop, and the security guards — quoting the real code throughout.

Gateway lifecycle & bootstrap

Unlike every other service in this report, the bot exposes no client port. It connects outbound to Discord's gateway and serves only /health and /metrics on MetricsAddr. The main() entrypoint (cmd/discord-bot/main.go) follows the FOUNDATION worker skeleton: resolve config from env, wait for the database, start the metrics server, construct services, then block on Run(ctx) until a signal cancels the context.

// cmd/discord-bot/main.go — metrics-only HTTP; bind failure is fatal.
go func() {
    mux := http.NewServeMux()
    mux.HandleFunc("/health", httpx.Health("discord-bot", nil))
    mux.Handle("/metrics", metrics.Handler())
    if err := http.ListenAndServe(cfg.MetricsAddr, mux); err != nil {
        logger.Fatal("metrics server", "error", err) // unprobeable => k8s restart
    }
}()

botCfg := discordbot.Config{
    Token:       firstNonEmpty(config.GetString("DISCORD_BOT_TOKEN", ""),
                               config.GetString("DISCORD_ACTIVITY_BOT_TOKEN", "")),
    DevGuildID:  config.GetString("DISCORD_DEV_GUILD_ID", ""),
    OwnerID:     config.GetString("OWNER_ID", ""),
    DeepSeekKey: config.GetString("DEEPSEEK_API_KEY", ""),
    DeepSeekMod: config.GetString("DEEPSEEK_MODEL", "deepseek-chat"),
    WorktreesDir: config.GetString("RMHBOT_WORKTREES_DIR", ""),
    GithubToken:  config.GetString("GITHUB_TOKEN", ""),
}

The token resolution mirrors the TS code: DISCORD_BOT_TOKEN wins, falling back to DISCORD_ACTIVITY_BOT_TOKEN. The DB is used for /chat session persistence; the chat handler tolerates a nil DB but main still waits for reachability like the other workers.

The session is created with the Guilds intent only — the bot reacts to interactions (slash commands, buttons, modals), which do not require message-content or member privileged intents:

// bot.go — New()
session, err := discordgo.New("Bot " + cfg.Token)
session.Identify.Intents = discordgo.IntentsGuilds
b := &Bot{cfg: cfg, logger: logger, session: session, chat: chat, rmhbot: rmhbot}
session.AddHandler(b.onReady)
session.AddHandler(b.onInteraction)

Run records the lifecycle context, opens the gateway, blocks on ctx.Done(), then closes the session cleanly — graceful shutdown:

func (b *Bot) Run(ctx context.Context) error {
    b.ctxMu.Lock(); b.lifecycleCtx = ctx; b.ctxMu.Unlock()
    if err := b.session.Open(); err != nil { return fmt.Errorf("open gateway: %w", err) }
    b.logger.Info("discord gateway opened")
    <-ctx.Done()
    b.logger.Info("shutdown_start")
    if err := b.session.Close(); err != nil { b.logger.Warn("session close", "error", err) }
    b.logger.Info("shutdown_complete")
    return nil
}

Command registration on ready

On the READY event the bot bulk-overwrites its slash commands. Registration is guild-scoped when DISCORD_DEV_GUILD_ID is set (instant propagation for development) and global otherwise. A subtle but important detail is preserved from the TS: a bulk overwrite cannot delete Discord's Entry Point command (type 4, used by Activities) and will fail with error 50240 if you try. So the bot first fetches existing commands and re-appends any Entry Point command to the desired set:

// bot.go — onReady()
guildID := b.cfg.DevGuildID // "" => global
desired := slashCommands()

existing, err := s.ApplicationCommands(s.State.User.ID, guildID)
if err == nil {
    for _, c := range existing {
        if int(c.Type) == 4 { // PrimaryEntryPoint — bulk overwrite cannot remove it
            desired = append(desired, c)
        }
    }
}
if _, err := s.ApplicationCommandBulkOverwrite(s.State.User.ID, guildID, desired); err != nil {
    b.logger.Error("command_registration_failed", "error", err)
    return
}

Interaction routing & owner-gating

All interactions enter through onInteraction. A review-fix here is that each interaction runs under a context derived from the bot's lifecycle context with a per-interaction timeout, rather than being detached via context.Background(). This bounds in-flight agent loops and DeepSeek calls and cancels them on shutdown:

const interactionTimeout = 10 * time.Minute // generous: rmhbot loop runs many rounds

func (b *Bot) onInteraction(s *discordgo.Session, i *discordgo.InteractionCreate) {
    b.ctxMu.RLock(); parent := b.lifecycleCtx; b.ctxMu.RUnlock()
    if parent == nil { parent = context.Background() } // handler may fire before Run records ctx
    ctx, cancel := context.WithTimeout(parent, interactionTimeout)
    defer cancel()
    switch i.Type {
    case discordgo.InteractionApplicationCommand: b.routeCommand(ctx, s, i)
    case discordgo.InteractionMessageComponent:   b.routeComponent(ctx, s, i)
    case discordgo.InteractionModalSubmit:        b.routeModal(ctx, s, i)
    }
}

Buttons and modals carry a custom ID of the form action:ownerId, split by splitCustomID. Every component and modal handler is owner-gated: the interacting user must equal the embedded owner id, or be the configured OWNER_ID admin override. Otherwise an ephemeral denial is sent:

func (b *Bot) ownerOK(s, i, userID, ownerID, denyMsg string) bool {
    if userID == ownerID { return true }
    if b.cfg.OwnerID != "" && userID == b.cfg.OwnerID { return true } // admin override
    _ = s.InteractionRespond(i.Interaction, &discordgo.InteractionResponse{
        Type: discordgo.InteractionResponseChannelMessageWithSource,
        Data: &discordgo.InteractionResponseData{Content: denyMsg,
            Flags: discordgo.MessageFlagsEphemeral},
    })
    return false
}

The button/modal actions form two flows: rmhbot_continue / chat_continue buttons open a paragraph-input modal (showModal), whose submit (rmhbot_continue_modal / chat_continue_modal) re-enters the same handler with isNew=false; the rmhbot_push button calls HandlePush directly. The user id is read via interactionUser, which handles both guild (i.Member.User) and DM (i.User) shapes.

The DeepSeek client

The TS code used the openai SDK pointed at DeepSeek's OpenAI-compatible API. Per the migration's no-SDK rule, the Go port re-implements a minimal client over net/http against https://api.deepseek.com, model from DEEPSEEK_MODEL (default deepseek-chat). It is non-streaming — a single blocking POST to /v1/chat/completions, rendered once — a deliberate simplification of the TS live-streaming embed edits.

// deepseek.go — single non-streamed completion round.
func (c *DeepSeekClient) complete(ctx, messages []ChatMessage, tools []Tool) (*completionResult, error) {
    if c.apiKey == "" { return nil, fmt.Errorf("DEEPSEEK_API_KEY is not set") }
    raw, _ := json.Marshal(chatCompletionRequest{
        Model: c.model, Messages: messages, Stream: false, Tools: tools})
    req, _ := http.NewRequestWithContext(ctx, http.MethodPost,
        c.baseURL+"/v1/chat/completions", bytes.NewReader(raw))
    req.Header.Set("Content-Type", "application/json")
    req.Header.Set("Authorization", "Bearer "+c.apiKey)
    resp, err := c.http.Do(req) // http.Client{Timeout: 60s}
    ...
    body, _ := io.ReadAll(io.LimitReader(resp.Body, 4<<20)) // 4 MiB cap
    if resp.StatusCode < 200 || resp.StatusCode >= 300 {
        return nil, fmt.Errorf("deepseek HTTP %d: %s", resp.StatusCode, string(body))
    }
    ...
}

The message and tool-call structs (ChatMessage, ToolCall, ToolCallFunc, Tool) mirror the OpenAI schema. Tool-call fields are omitempty so a plain /chat message serializes identically to the original TS payload, while the rmhbot loop can round-trip tool_calls / tool_call_id. Chat() is a thin wrapper for the persona (one round, no tools, returns the reply text).

Session persistence

Sessions live in an in-memory cache keyed by Discord user id, backed by the discord_chat_session table via raw pgx. loadSession returns cache → DB → fresh persona-seeded session; on any DB error (including no rows) it keeps the fresh session, matching the TS .catch(() => null). saveSession upserts by discordUserId:

// chat.go — saveSession upsert (best-effort, like the TS .catch(() => {})).
_, err = s.db.Pool.Exec(ctx,
  `INSERT INTO "discord_chat_session"
     ("discordUserId","username","history","lastMessageId","updatedAt")
   VALUES ($1,$2,$3,$4,$5)

     "username"=EXCLUDED."username", "history"=EXCLUDED."history",

The 6000-char embed packing (ported 1:1)

Discord caps an embed at 6000 characters total across title, footer, and field names+values, with ≤ 25 fields and ≤ 1024 chars per field value. buildChatEmbed ports the TS budget-packing logic exactly: it subtracts title+footer from the 6000 budget, gives the echoed user prompt first claim, then splits the reply across as many fields as fit, using alexFieldName ("🤙 Alex") for the first and a zero-width space for continuations. If it runs out of budget or fields, the last field is ellipsis-marked:

// chat.go — reply field-packing loop (runes, not bytes).
remaining := []rune(replyText)
if len(remaining) == 0 { remaining = []rune("(no response)") }
for len(remaining) > 0 {
    if len(fields) >= maxFields { truncated = true; break }
    name := alexFieldName
    if replyFieldCount > 0 { name = continuationFieldName } // zero-width space
    valueBudget := min(fieldValueMax, budget-runeLen(name))
    if valueBudget <= 0 { truncated = true; break }
    take := valueBudget
    if take > len(remaining) { take = len(remaining) }
    value := string(remaining[:take])
    fields = append(fields, embedField{Name: name, Value: value})
    budget -= runeLen(name) + runeLen(value)
    remaining = remaining[take:]
    replyFieldCount++
}

All arithmetic is done in runes rather than bytes, so the emoji-heavy persona never splits a multi-byte character mid-rune — a Go-specific correctness improvement over the JS .length semantics it emulates.

/rmhbot — the agentic website editor (scaffold)

What is real: command/interaction structure; the in-memory conversationState (branch, worktree path, history, deletedFilesThisSession, lockedAt); 5-minute auto-unlock with an injectable clock; per-session locking that fixes a TOCTOU race; the safePath traversal guard and safety guards; git ops via os/exec; the MAX_TOOL_ROUNDS = 40 loop; real read_file / list_directory / write_file / delete_file tools. What is stubbed: run_typecheck (needs the JS toolchain), createPullRequest (needs the GitHub REST call), search_code (needs grep wiring), recursive listing, attachment ingestion, and the live progress reporter.

A session is created on /rmhbot by cloning the target repo into a worktree and checking out a fresh branch. The repo is env-driven: resolveRepo() parses GITHUB_REPO ("owner/name"), falling back to stickms/rmhstudios.com. The branch name is deterministic under the injectable clock: rmhbot-<sanitized-username>-<unix>.

The agentic tool loop

runAgentLoop drives up to maxToolRounds = 40 non-streamed DeepSeek rounds. Each round, if the model returns tool calls, the bot records the assistant turn, dispatches each call through executeTool, truncates the result to maxToolResultLen = 8000, and round-trips it back as a role:"tool" message. A round with no tool calls is the final answer:

// rmhbot.go — runAgentLoop core.
for round := 0; round < maxToolRounds; round++ {
    res, callErr := s.deepseek.complete(ctx, state.history, rmhbotTools())
    if callErr != nil { return "", changedFiles, callErr }
    if len(res.ToolCalls) > 0 {
        state.history = append(state.history, ChatMessage{
            Role: roleAssistant, Content: res.Content, ToolCalls: res.ToolCalls})
        for _, tc := range res.ToolCalls {
            var parsed map[string]any
            argsStr := tc.Function.Arguments; if argsStr == "" { argsStr = "{}" }
            result := ""
            if jErr := json.Unmarshal([]byte(argsStr), &parsed); jErr != nil {
                result = "Error: " + jErr.Error()
            } else {
                result = s.executeTool(ctx, tc.Function.Name, parsed, state, &changedFiles)
            }
            if len(result) > maxToolResultLen { result = result[:maxToolResultLen] + "\n[truncated]" }
            state.history = append(state.history, ChatMessage{
                Role: roleTool, ToolCallID: tc.ID, Content: result})
        }
        continue
    }
    summary = res.Content // no tool calls => done
    state.history = append(state.history, ChatMessage{Role: roleAssistant, Content: res.Content})
    return summary, changedFiles, nil
}

On a successful loop with changes, the bot stages all (git add -A), checks status --porcelain, commits with a Co-Authored-By: RMHBot trailer, and best-effort pushes the branch with --force-with-lease (mirroring the TS .catch). /rmhbot-push would then run typecheck and open a PR — both currently stubbed, so push fails closed on typecheck.

Per-session locking (TOCTOU review fix)

The original guard was an unsynchronized exists && isSessionLocked check, which let two concurrent interactions from the same user both pass. The Go port closes this with a per-session mutex (conversationState.mu) and a single check-and-lock critical section, tryLock. The map-level s.mu only guards the sessions map, not per-session state:

// rmhbot.go — atomic check-and-lock (closes the TOCTOU window).
func (s *RmhbotService) tryLock(state *conversationState) bool {
    state.mu.Lock(); defer state.mu.Unlock()
    if s.isSessionLockedLocked(state) { return false } // already locked
    now := s.clock.Now(); state.lockedAt = &now
    return true
}

Auto-unlock assumes a crashed prior run: a lock older than sessionLockTTL = 5 * time.Minute is cleared. The comparison is strict (>), so exactly 5 minutes is still locked — matching the TS:

func (s *RmhbotService) isSessionLockedLocked(state *conversationState) bool {
    if state.lockedAt == nil { return false }
    if s.clock.Now().Sub(*state.lockedAt) > sessionLockTTL { // auto-unlock
        state.lockedAt = nil
        return false
    }
    return true
}

Security guards

Path-traversal guard. Every filesystem tool resolves its path through safePath, ported 1:1 from command-handler.ts. It strips leading slashes, cleans/joins against the worktree base, and rejects any result that escapes the base. Crucially it uses a separator-aware prefix check, so a sibling like /srv/clone-evil (which shares the /srv/clone string prefix) is rejected:

func safePath(basePath, userPath string) (string, error) {
    normalized := leadingSlashes.ReplaceAllString(userPath, "")
    full := filepath.Clean(filepath.Join(basePath, normalized))
    base := filepath.Clean(basePath)
    if full != base && !strings.HasPrefix(full, base+string(filepath.Separator)) {
        return "", fmt.Errorf("Path traversal detected: %s", userPath)
    }
    return full, nil
}

Write/delete safety. Beyond traversal, writes are blocked on protected paths (.env, server/discord-bot, and the .env. / server/discord-bot/ prefixes) and on content matching secret patterns (DEEPSEEK_API_KEY, DISCORD_*TOKEN, GITHUB_TOKEN, BETTER_AUTH_SECRET, TOKEN_ENCRYPTION_KEY). Deletes are capped at 10 files per session. These mirror the persona's HARD LIMITS in the rmhbot system prompt, enforced in code rather than trusted to the model.

Git token handling caveat. gitConfigArgs() injects the GITHUB_TOKEN via an inline url.https://x-access-token:<token>@github.com/.insteadOf rewrite passed on the git -c command line. This is functional and scoped per-invocation (with GIT_TERMINAL_PROMPT=0 to prevent interactive prompts), but the token appears in the process argument list, so it may be visible via /proc or process listings on the host while git runs. A hardened version should pass the credential via an askpass helper or a transient credential file instead.

Tests

discordbot_test.go covers the three load-bearing, pure pieces — no Discord or DeepSeek mock needed:

  • safePath traversal guard — allows in-tree paths (including leading-slash and .), rejects ../etc/passwd, app/../../etc/passwd, .., and the sibling-prefix ../clone-evil/x.
  • Embed packing — a 20 000-char reply packs into multiple fields and never exceeds the 6000 total or the 1024/field cap; a 30 000-char reply ends in the ellipsis; a short reply yields exactly the You + Alex fields; an empty reply yields the (no response) placeholder.
  • 5-minute auto-unlock — driven by a fakeClock: nil lockedAt is unlocked; 2 min stays locked; 5 min + 1 s auto-unlocks and clears lockedAt; exactly 5 min stays locked (strict >). TestMakeBranchName asserts the deterministic sanitized branch rmhbot-cool-user-1700000000.
The faithfully-ported surface (gateway lifecycle, command registration, interaction routing & owner-gating, DeepSeek client, the full /chat persona with DB persistence and 1:1 embed packing) is complete and tested. The /rmhbot scaffold compiles, runs its real tools and git flow under the locking/safety guards, and marks every stub (run_typecheck, createPullRequest, search_code) with an explicit TODO(migration) that fails closed rather than faking success.
