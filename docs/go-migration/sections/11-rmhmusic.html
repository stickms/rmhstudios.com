The rmhmusic service is the Go port of the Node rmhmusic feature: synchronized group music-listening rooms where the host controls playback and members share a queue and chat. In Node this lived inside the unified socket-server behind the rmhmusic: event prefix; in the Go world it is extracted into its own standalone service. It serves the realtime hub at /rmhmusic-ws/ (default port 7002, RMHMUSIC_PORT) plus /health and /metrics. Unlike rmhtube, it sets RequireAuth: true on the hub — only authenticated users may connect — which makes the database mandatory (it backs both session validation and the one persisted row). The package lives in internal/rmhmusic/ (events, types, manager, room, sync, queue, chat, repo, util) and is wired in cmd/rmhmusic/main.go.

Service shape and the auth requirement

The binary is a single authenticated WebSocket runtime. Because auth is mandatory, main.go opens the database with db.Open and Fatals on failure (no in-memory degrade path, unlike rmhtube), then constructs the hub with RequireAuth: true and a session validator:

database, err := db.Open(ctx, cfg.DatabaseURL, cfg.DBPoolSize)
if err != nil { logger.Fatal("db", "error", err) }

v := auth.NewValidator(database.Pool)
hub := realtime.NewHub(ctx, realtime.Options{
    Validator:   v,
    RequireAuth: true, // rmhmusic requires auth
    // ...
})
mgr := rmhmusic.NewManager(hub, logger, rmhmusic.NewPgRepo(database))
mgr.Register()
mgr.Start(ctx) // GC reaper + 5s playback heartbeat

Because every connection is authenticated, c.UserID() is always a real account id — there is no guest fallback to reason about. The Go port composes the four Node managers (RoomManager, SyncEngine, QueueManager, ChatHandler), which in Node shared one RoomManager instance, onto a single Manager struct that holds the shared state directly.

Snapshot + seq-delta protocol

Event names are ported verbatim from lib/rmhmusic/events.ts, carrying the rmhmusic: prefix so the existing client adapter speaks them byte-for-byte. As with rmhtube, the protocol is full snapshot on join + seq-numbered deltas thereafter. On join the server sends a room:state_snapshot from toClientState; every mutation then goes out as a room:action envelope with a per-room monotonic sequence. The delta types are a compact set:

actionMemberJoined       = "MEMBER_JOINED"
actionMemberLeft         = "MEMBER_LEFT"
actionMemberDisconnected = "MEMBER_DISCONNECTED"
actionHostTransferred    = "HOST_TRANSFERRED"
actionChatMessage        = "CHAT_MESSAGE"
actionNowPlaying         = "NOW_PLAYING"
actionQueueItemAdded     = "QUEUE_ITEM_ADDED"

The broadcast primitive broadcastAction increments room.Seq under room.mu and broadcasts the {type,payload,seq,timestamp} envelope:

func (m *Manager) broadcastAction(room *ServerRoom, typ string, payload any) {
    room.Seq++
    room.LastActivityAt = m.now()
    m.hub.Broadcast(room.ID, realtime.MustEnvelope(s2cRoomAction, map[string]any{
        "type":      typ,
        "payload":   payload,
        "seq":       room.Seq,
        "timestamp": m.now(),
    }))
}
Seq-stamping consistency (per review). The doc comment is explicit that the inner payload seq — the room's own counter, what the Node delta carried and what snapshots report — is the single authoritative sequence. This is why broadcastAction uses plain hub.Broadcast and not a BroadcastSeq helper: a second, envelope-level seq stamp would be redundant and could diverge from the inner one. This deliberately aligns rmhmusic with rmhtube. TestSeqOrderedDeltaNumbering drives three actions and asserts both the room counter and the snapshot's reported seq land at 3.

Host-only playback and drift correction

Playback control is a faithful port of server/rmhmusic/sync-engine.ts and is strictly host-only: a non-host play is rejected with NOT_HOST, while non-host pause / seek / skip are silently ignored, exactly as the Node SyncEngine behaved. The ephemeral playback head is a small struct — track URI, position in milliseconds, a playing flag, and the updatedAt stamp that anchors drift correction:

type Playback struct {
    TrackURI   string `json:"trackUri"`
    PositionMs int64  `json:"positionMs"`
    IsPlaying  bool   `json:"isPlaying"`
    UpdatedAt  int64  `json:"updatedAt"`
}

A 5-second heartbeat (runHeartbeat) emits sync:heartbeat to each playing room carrying the drift-corrected position. The correction is the same idea as rmhtube's but in integer milliseconds and with an implicit 1× rate: while playing, the projected position is the stored position advanced by the wall-clock elapsed since updatedAt; while paused, the stored value is returned unchanged:

func projectedPositionMs(p Playback, nowMs int64) int64 {
    if !p.IsPlaying {
        return p.PositionMs
    }
    return p.PositionMs + (nowMs - p.UpdatedAt)
}

Three subcases of TestProjectedPositionMs pin the math: playing advances by elapsed (1000 + 3500), zero-elapsed returns the stored position, and paused returns the stored position regardless of how much time passed. The onPlay handler shows the locking discipline that prevents out-of-order track changes: the playback mutation and the NOW_PLAYING delta are stamped in one critical section (so a concurrent host control cannot interleave between the state change and the broadcast), and the plain music:play fan-out is emitted only after the unlock:

room.mu.Lock()
track := p.Track
room.CurrentTrack = &track
room.Playback = Playback{
    TrackURI: p.TrackURI, PositionMs: p.PositionMs,
    IsPlaying: true, UpdatedAt: m.now(),
}
m.broadcastAction(room, actionNowPlaying, map[string]any{"track": track})
room.mu.Unlock()

m.broadcast(room.ID, s2cMusicPlay, map[string]any{
    "trackUri": p.TrackURI, "positionMs": p.PositionMs, "track": track,
})

onSkip pops the queue head, reindexes, resets the playback head to position 0 for the next track (or clears it and emits track_changed: nil when the queue is empty), then broadcasts music:play and queue:updated.

Queue and chat

The queue (port of queue-manager.ts) is deliberately not host-gated — any member may add, remove, or reorder, matching the Node QueueManager. Items are Spotify tracks (URI, title, artist, album art, duration, preview URL). Add appends to a 50-item-capped queue, broadcasts queue:updated with the full queue, then a QUEUE_ITEM_ADDED delta. Remove and reorder splice the slice and reindex positions 0..n-1. Chat (port of chat-handler.ts) sanitizes and truncates content to 300 chars, appends to a 100-message ring buffer, and broadcasts a CHAT_MESSAGE delta — a compact subset of rmhtube's chat (no replies, mentions, reactions, or pinning here, faithful to the simpler Node service).

Host auto-reassignment and the memberOrder list

When the host leaves a non-empty room, the host role is auto-reassigned to the first remaining member in insertion order. In Node this fell out naturally from room.members.values().next().value over an insertion-ordered JS Map. Go maps randomize iteration order, so the port maintains an explicit memberOrder []string alongside the members map; every join appends to it and every removal filters the user out, and firstMember walks it:

// firstMember returns the first member in insertion order (host
// reassignment target), matching room.members.values().next().value
// over an insertion-ordered JS Map.
func (r *ServerRoom) firstMember() *ServerMember {
    for _, uid := range r.memberOrder {
        if mm, ok := r.Members[uid]; ok {
            return mm
        }
    }
    return nil
}

The reassignment happens inside removeMember: if the room is now empty it is deleted immediately; otherwise MEMBER_LEFT is broadcast, and when the departing user was the host the new host is set and HOST_TRANSFERRED announced:

if hostLeft {
    if newHost := room.firstMember(); newHost != nil {
        room.HostUserID = newHost.UserID
        m.broadcastAction(room, actionHostTransferred, map[string]any{
            "newHostUserId": newHost.UserID, "newHostUserName": newHost.UserName,
        })
    }
}

TestHostAutoReassignOnHostLeave builds a room ordered [host, bob, amy], removes host, and asserts the new host is bob (the first remaining member) and the room survives with two members. A companion test confirms a solo room is reaped immediately on last-member leave.

Concurrency and lifecycle

The manager holds a top-level sync.RWMutex over its index maps (rooms, connToRoom, userToConn); each ServerRoom additionally carries its own mu, and all mutation of a room's fields happens under it — so events on different rooms never contend and events on the same room serialize. broadcastAction and toClientState both document that the caller must hold room.mu.

Disconnect grace. On disconnect the member is marked offline, a MEMBER_DISCONNECTED delta is broadcast, and a 30-second grace timer (shared realtime.GraceTimers) is armed; a reconnect within the window cancels it. The disconnect handler guards against a stale connection replacing a newer one by checking member.ConnID == c.ID before acting, and only clears userToConn if the disconnecting conn is still the user's active one. A 60-second GC ticker reaps rooms that have been empty for more than 300 seconds — a safety net, since empty rooms are normally deleted immediately on last-member removal. The clock is injected (now func() int64) so tests can run deterministically.

Persistence: one row, faithfully

This is the sharpest contrast with rmhtube. The Repo interface has a single method, CreateRoom, and that is the only database write the service ever makes. Members, queue, playback, and chat live entirely in memory and are intentionally never written — replicating the Node service exactly. The write is fire-and-forget: a failure is logged, never surfaced to the client.

// Repo persists the durable part of a room. NOTE (faithful to Node):
// the original code only persists the room ROW on create -- members,
// queue, playback and chat live entirely in memory.
type Repo interface {
    CreateRoom(ctx context.Context, r RoomRow) error
}

The persisted row is the rmh_music_room table (id, code, name, hostId, isPublic, password, maxMembers, createdAt, updatedAt), inserted by PgRepo with quoted, camelCase column names matching the Prisma / better-auth schema convention. NopRepo{} is the no-op used in tests so the room logic — which is fully in-memory and works regardless — is exercisable without a database. Room ids use a 21-char nanoid-style alphabet (crypto/rand-backed); join codes use the 6-char unambiguous alphabet.

Scope: faithful to the simpler Node service. rmhmusic is a faithful port of a deliberately leaner feature than rmhtube. The full core — host-only playback with drift-corrected heartbeat, the open (non-host) queue, chat, the snapshot+seq-delta protocol, host auto-reassignment via the explicit memberOrder, and the grace + GC lifecycle — is ported behavior-for- behavior. The single-row persistence is not a shortcut but a faithful replication of the Node service, which also kept members / queue / playback / chat purely in-memory; consequently there is no startup room restore (a restart drops live rooms, as it did in Node). The only intentional cross-service alignment beyond the Node original is the seq-stamping consistency note above, which standardizes the inner-payload-seq convention with rmhtube. Nothing here is invented.
