The vibe-worker renders gallery thumbnails for AI-generated "vibe pages" — arbitrary interactive HTML (esbuild-bundled React, canvas, three.js from esm.sh). It ports server/vibe-worker/index.ts and lib/rmhvibe/vibe-screenshot.server.ts: every 10 seconds it picks up to five stale pages, renders each headlessly in Chromium, writes a downscaled PNG to the shared db/vibe-thumbs/<slug>.png volume, and clears the stale flag with optimistic concurrency. There is no client HTTP surface — only /health and /metrics on METRICS_ADDR.

Poll → render → optimistic clear

The worker decouples headless Chromium from the web app's request path. Its loop runs one pass immediately on start (the Node void processStale() kick) then ticks every pollInterval. Each pass selects the oldest stale pages, renders each, and clears the flag only if the page hasn't changed underneath the render:

Reentrancy guard

A batch of five Chromium renders can easily outlast the 10-second tick, so the worker carries a running flag (the Node running boolean) guarded by a mutex. A tick that fires while a batch is in flight is a no-op:

func (w *Worker) processStale(ctx context.Context) {
    w.mu.Lock()
    if w.running {
        w.mu.Unlock()
        return // overlapping tick — skip
    }
    w.running = true
    w.mu.Unlock()
    defer func() { w.mu.Lock(); w.running = false; w.mu.Unlock() }()
    ...
}

The select query is the literal port of the Prisma findMany({ where: thumbnailStale, orderBy: updatedAt asc, take: 5 }):

SELECT "id", "slug", "html", "updatedAt"
  FROM "vibe_page"
 WHERE "thumbnailStale" = true
 ORDER BY "updatedAt" ASC
 LIMIT $1

Optimistic concurrency clear

The page's HTML can be edited by a user while we are rendering the previous version. The worker reads updatedAt with the row and, after capturing, only clears the stale flag when updatedAt is still unchanged — a compare-and-set in SQL:

UPDATE "vibe_page"
   SET "thumbnailStale" = false, "thumbnailUrl" = $1
 WHERE "id" = $2 AND "updatedAt" = $3   -- expected updatedAt

If no row was affected, the content changed mid-render. The worker still records the freshly-captured URL (so the gallery shows something) via SetThumbnailURL but leaves the flag set so the newest content re-renders next tick:

cleared, err := w.repo.ClearStale(ctx, page.ID, url, page.UpdatedAt)
if cleared {
    return true // clean render
}
// Page changed mid-render: record URL, keep it stale.
if err := w.repo.SetThumbnailURL(ctx, page.ID, url); err != nil {
    return false
}
return true

Batch outcome is recorded honestly: ok when the batch was empty or every render succeeded, error when at least one was attempted and none succeeded, and partial when some succeeded and some failed — feeding JobRuns{vibe_thumbnails, ok|partial|error}.

Headless capture with chromedp

The Capturer interface hides the rendering so the loop can be tested with a fake. The production implementation drives headless Chromium via chromedp with container-safe flags mirroring the Node launch args:

err := chromedp.Run(runCtx,
    chromedp.EmulateViewport(viewportWidth, viewportHeight, chromedp.EmulateScale(thumbDeviceScale)),
    chromedp.Navigate(dataURL),
    chromedp.Sleep(settleDelay),       // modules + first paint + intro animations
    chromedp.CaptureScreenshot(&png),
)
Scope — chromedp replaces Playwright. The Node worker drove Chromium through Playwright. The Go port uses chromedp (the CDP client) to the same end: navigate, settle, screenshot. The container-safe flags (NoSandbox, DisableGPU, disable-dev-shm-usage, headless) mirror the Playwright launch args; execPath honors PLAYWRIGHT_CHROMIUM_EXECUTABLE_PATH in production and is empty locally so chromedp discovers an installed Chrome.
Runtime requirement. A Chromium/Chrome binary must be present at runtime — there is no fallback renderer. The production Dockerfile installs Chromium and points PLAYWRIGHT_CHROMIUM_EXECUTABLE_PATH at it; without a browser, every capture fails and pages stay stale.

Review fix: base64-encoded data URL

The page is rendered from a data: URL rather than a temp file, so the worker never touches the filesystem for input. The HTML is arbitrary and routinely contains # (a fragment delimiter) and % (a percent-escape introducer), both of which truncate or mis-decode in a plain data: URL. The payload is therefore base64-encoded so Chromium decodes the exact bytes:

dataURL := "data:text/html;charset=utf-8;base64," +
    base64.StdEncoding.EncodeToString([]byte(html))
Review fix. Base64 makes the URL opaque, eliminating the #/% truncation class of bugs entirely — a correctness improvement over a naive inline data URL.

A real per-render timeout

Animation-heavy pages may never reach network-idle, and a sub-resource that never finishes could hang a render forever. Each capture runs under a real deadline (captureTimeout = 10s), comfortably above the settleDelay of 1.5s so a render is never cut off mid-settle. On timeout the run is cancelled and the page is left stale to retry — the batch is never blocked indefinitely:

runCtx, cancelRun := context.WithTimeout(taskCtx, captureTimeout)
defer cancelRun()

Downscaling: Chromium device scale instead of sharp

Scope — substitution for sharp. The Node code downscaled the raw screenshot with the sharp image library (resize to thumbWidth, PNG compression level 9). With no sharp-equivalent in the allowed dependency set, the Go port instead asks Chromium itself to rasterize smaller: it sets a fractional device scale factor (thumbWidth/viewportWidth) on the emulated viewport. The layout viewport stays 1200×750 so pages lay out identically, but the PNG comes out ~640px wide. This achieves the faithful intent — a small gallery PNG — with chromedp's own scaling rather than a post-capture resize.
const thumbDeviceScale = float64(thumbWidth) / float64(viewportWidth)
// EmulateViewport(1200, 750, EmulateScale(thumbDeviceScale))

On success the PNG is written to <thumbDir>/<slug>.png and the worker returns a cache-busted URL the web app serves at /api/vibe/thumb/{slug}?v=<unixMillis>.
