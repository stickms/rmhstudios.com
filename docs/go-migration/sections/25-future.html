If this branch were to move forward, the work breaks into a clear, ordered backlog. The architecture was built so that each item is additive — a stub implemented against an existing interface, or an operational step — rather than a structural change.

1. Close the Socket.IO gap

Build the client-side adapter described in Section 6: a thin TypeScript layer that presents the existing Socket.IO event API to the React components while speaking the new JSON-envelope protocol underneath. This unblocks cutting over any single websocket prefix and is the highest-leverage next step.

2. Fill the scaffolded surfaces

  • Port the remaining eight rmhbox minigames against the existing Minigame interface.
  • Register the other gamehub games onto the generic RelayGroup.
  • Complete the RMHBot tool set (typecheck, PR creation, code search) behind the existing tool-loop.

3. Realize the scaling path

Enable the Redis backplane in production, add sticky-session ingress, and load- test to establish connection-count and latency baselines per service. Then wire HorizontalPodAutoscalers — completing PR #121's Stage 2 for the Go tier.

4. Complete observability

The instrumentation exists (Section 17); the remaining work is operational: Prometheus scrape config for the new /metrics endpoints, dashboards for connections/messages/job-runs, and alerts on the auth-error and bus-publish signals already emitted.

5. Quality hardening

  • Restore DeepSeek streaming in the discord-bot for chat UX parity.
  • Visual-diff the vibe-worker thumbnails against the sharp output and tune if needed.
  • Add contract tests that assert the Go event payloads match what the React clients expect, byte for byte.
  • Add per-render and per-job tracing spans (OpenTelemetry) on top of the existing metrics.

6. Decide, with data

Finally — and most importantly — the adoption decision should be made against real metrics, per service, exactly as the Stage-4 condition states. The low-risk services (doctrine-worker, recap, gateway) could be adopted first as a trial; the realtime services follow only if connection volume justifies the investment and after the protocol adapter exists.

The through-line: nothing in this backlog requires re-architecting what was built. The interfaces, the backplane, the chart, and the verification harness are in place; the future work is to fill them in, measure, and decide.
