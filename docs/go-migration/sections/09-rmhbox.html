The rmhbox service is the Go port of the Node server/rmhbox Jackbox-style party-game platform: short multiplayer minigames played from a phone or a Discord Activity, with a shared host-driven lobby. It serves the realtime hub at /rmhbox-ws/ on port 7676, built on the shared realtime.Hub (replacing Socket.IO) rather than on a bespoke transport. This section documents the lobby manager and its room model, the per-lobby lifecycle FSM and game coordinator, the pausable timer/state-sync layer, the minigame framework, the per-player scoped state that prevents info-leaks, persistence, and the concurrency model — and is honest about what is fully ported (one minigame) versus scaffolded (eight).

Purpose & entrypoint

One rmhbox.Manager wires together five subsystems onto a single realtime.Hub: the LobbyManager (lobby CRUD, host controls, garbage collection), the GameCoordinator (the per-lobby minigame lifecycle FSM), the StateSync service (heartbeats and phase timers), the minigame framework + registry, and the persistence Repo. The cmd/rmhbox/main.go entrypoint is deliberately thin — it opens an optional database (a failed open degrades to in-memory play with persistence disabled), constructs the hub, then registers handlers and starts background goroutines.

hub := realtime.NewHub(ctx, realtime.Options{
    Origin: host, Logger: logger, Metrics: metrics, Validator: validator,
    Bus: bus, AllowOrigins: config.GetCSV("SOCKET_CORS_ORIGIN"), RequireAuth: false,
})
mgr := rmhbox.NewManager(hub, database, logger)
mgr.Register()   // bind connect/disconnect + every inbound rmhbox:* handler
mgr.Start()      // launch heartbeat + lobby GC goroutines
defer mgr.Stop() // tear down goroutines and cancel all grace/disband timers

mux.HandleFunc("/rmhbox-ws/", hub.ServeWS)   // addr := ":" + RMHBOX_PORT (7676)

Every participating action requires an identity. The Manager guards each handler with requireUser, which sends an UNAUTHENTICATED error and bails for anonymous connections, so the hub itself runs with RequireAuth: false but the application layer enforces identity per event. Inbound events are bound from the envelope into typed payload structs and delegated to the appropriate subsystem; small adapters (lobbyOnly, hostTarget, away) cut boilerplate for the common {lobbyId} / {lobbyId, targetUserId} shapes.

Faithful-port note. Socket.IO's sub-rooms (lobby:{id}, :players, :spectators, :player:{userId}) are modelled as logical member sets on the Lobby and broadcast through the hub's single rmhbox:{id} room, filtering per-recipient where the Node code targeted a sub-room. Sequenced GAME_ACTION mutations go through hub.BroadcastSeq so seq numbering matches the legacy delta protocol. Every minigame handler call is wrapped in recover() for fault isolation, porting the try/catch-around-every-handler discipline.

The LobbyManager owns all lobbies plus the same indices the Node manager kept. It is the single source of truth for membership and the only thing that talks to the hub for broadcasts.

Broadcast routing

Two distinct broadcast paths exist. broadcastAction emits a sequenced rmhbox:game:action envelope via hub.BroadcastSeq — the hub stamps a monotonic per-room seq so clients can detect gaps and reorder deltas. Non-mutating events (settings_opened, countdown, round_results, disbanded) go through the non-sequenced broadcastRaw. Targeted sends resolve the user's live conn through userConn; broadcastToPlayers/broadcastToSpectators iterate the logical member set and fan out individual sends.

Test seam. Every broadcast/send routes through an optional function field (broadcastSeqFn, broadcastFn, sendFn). When set (in tests) they capture envelopes without a live WebSocket; testSeq mirrors the hub's atomic per-room seq stamping so ordering behaviour is verifiable. In production these are nil and calls hit the hub directly.

Grace, succession, disband & GC

On disconnect, a player is marked disconnected and unready and given a 120-second grace timer; a spectator is removed immediately (spectators have no grace). Reconnecting cancels the grace timer. If a host leaves, succeedHost promotes the earliest-joined remaining player (or, failing that, promotes a spectator into a player slot); if nobody remains, the lobby disbands. A 60-second GC ticker disbands lobbies that are idle in WAITING past 15 min, fully disconnected past 30 min, or empty past 2 min.

Review fix — grace closures re-resolve the live lobby. A grace/preload/ingame timer closure captures a *Lobby that may be stale by the time it fires (the lobby could have been disbanded and replaced). Every such closure now re-resolves by id and bails if the live lobby differs: cur := m.GetLobby(l.ID); if cur == nil || cur != l { return }. Likewise removePlayer only clears userToLobby[userID] if it still points at this lobby, so a delayed grace expiry can't evict a user who has since rejoined a different lobby.

Lobby FSM

The coordinator drives a single forward path through the lobby's LobbyState. The happy path from a host pressing "Start" to results is WAITING → (GAME_SETTINGS) → INSTRUCTIONS → PRELOADING → COUNTDOWN → PLAYING → ROUND_RESULTS → WAITING. GAME_SETTINGS is entered only in post-vote mode (a direct host start skips straight to INSTRUCTIONS). VOTING, SESSION_RESULTS and DISBANDED are branch/terminal states.

State changes are always paired: a sequenced STATE_CHANGED action carries the canonical state, and a per-phase raw event carries phase payload (game:instructions, game:preload_start, game:countdown, game:started, game:round_results). After each transition the coordinator calls broadcastFullSync so every member receives a fresh scoped snapshot at phase boundaries.

Two phases are gated rather than purely time-driven. PRELOADING waits for every connected player to emit rmhbox:game:ready_to_render; the coordinator tracks readiness in lifecycleState.readyPlayers and advances to COUNTDOWN as soon as all are ready, with a 30-second preload grace timer as a force-proceed fallback. The host can also short-circuit any pre-play phase via force_skip, or abort entirely with force_end.

GameCoordinator & a game round

The GameCoordinator is the FSM engine. It holds a per-lobby lifecycleState (the active TimerHandle, the preload readiness set, and an optional phase-cancel func) plus a spectatorTargets map. When startPlaying runs it snapshots the player set, instantiates the minigame handler from the serverRegistry, and builds a MinigameContext — the handler's entire outward communication surface (broadcast/send closures, host-id getter, OnComplete/OnError callbacks).

Player input is routed only while PLAYING, only from actual players (spectator input is silently dropped), and only into a non-nil handler — then dispatched under a panic recover so one bad handler can't crash the goroutine:

func (gc *GameCoordinator) OnInput(userID, lobbyID, action string, data map[string]any) {
    // ... resolve lobby, verify PLAYING + handler present, verify isPlayer ...
    func() {
        defer func() {
            if rec := recover(); rec != nil {
                gc.logger.Error("game_input_error", "lobbyId", lobbyID,
                    "userId", userID, "panic", rec)
            }
        }()
        handler.HandleInput(userID, action, data)
    }()
}

Host controls map onto the FSM directly. OnForceSkip advances the current non-playing phase (instructions→preload, preload→countdown, countdown→playing, results→waiting). OnForceEnd tears down the handler with safeCleanup and returns to WAITING. OnPauseTimer toggles either the lifecycle TimerHandle (for instructions/countdown/results) or the handler's own phase timer (during play). On game completion the coordinator applies round scores cumulatively, records a ServerMatchSummary in the lobby's match history, builds live session standings, transitions to ROUND_RESULTS with an infinite (host-advanced) timer ring, and kicks off fire-and-forget persistence.

StateSync & the TimerHandle data-race fix

StateSync has two jobs. First, a 10-second heartbeat that walks every lobby and, for those in PLAYING, pushes a fresh per-player scoped full snapshot to every connected member — so a client that missed deltas self-heals every 10s. Second, the pausable 1-second TIMER_TICK broadcasts that drive the phase rings: startTimerBroadcast emits a TIMER_START, then ticks TIMER_TICK each second until the countdown crosses below zero, at which point it fires onComplete. broadcastInfiniteTimer shows a full ring with no ticking for host-advanced results.

Review fix — the TimerHandle stop-channel data race (found by -race). The broadcast goroutine reads h.stop without the lock. Cancel originally nil'd t.stop, which raced against that lockless read. The fix: the goroutine captures the channel once before it starts (set before the goroutine, never reassigned — happens-before via goroutine creation, so reading it needs no lock), and Cancel closes it exactly once behind a cancelled guard and never nils it.
// Cancel stops the timer broadcast.
func (t *TimerHandle) Cancel() {
    t.mu.Lock()
    if t.cancelled { t.mu.Unlock(); return }
    t.cancelled = true
    // Closing the channel is the signal; the `cancelled` guard makes this
    // close-once. We deliberately do NOT nil t.stop — the broadcast goroutine
    // reads it without the lock, so mutating it here would be a data race.
    if t.stop != nil { close(t.stop) }
    t.mu.Unlock()
}

// in startTimerBroadcast:
stop := h.stop          // capture once; never reassigned -> lockless read is safe
go func() {
    for {
        select {
        case <-stop:    return
        case <-t.C:      /* decrement under h.mu, broadcast TIMER_TICK */
        }
    }
}()

Minigame framework

The Node abstract BaseMinigame class becomes a Go interface (Minigame) plus an embeddable helper (BaseGame) supplying shared lifecycle plumbing: tracked pausable one-shot timers, the pausable phase-timer broadcast, default no-op hooks, and panic isolation. Every concrete game embeds BaseGame and overrides the gameplay methods.

Review fix — baseGame() replaces a concrete type-switch. The coordinator needs the embedded *BaseGame to drive phase pause/resume generically. An earlier version used a type-switch over each concrete game, which had to be kept in sync as games were added. Now the Minigame interface declares an unexported baseGame() *BaseGame method that BaseGame itself implements, so every embedder satisfies it automatically and asBaseGame(h) is a single, total call — the compiler enforces coverage.

The phase-timer pause is non-trivial because in-flight one-shot timers must also freeze. pausePhaseTimer snapshots each unfired pausable timer's remaining delay and stops it; resumePhaseTimer re-arms each with its saved remainder via time.AfterFunc. Every fired callback runs through fire, which dedupes (pt.fired) and recovers panics into ctx.OnError. BaseGame is concurrency-safe by design: the host may pause/resume from one goroutine while game timers fire on others, all guarded by b.mu.

Rhyme Time — the one fully-ported game

rhymeTimeGame is the faithful port of server/rmhbox/minigames/rhyme-time/handler.ts. Players get a root word each round and submit rhymes within a time limit; scoring rewards rarity (fewer players found the word), multi-syllable rhymes, and speed (first to a non-unique word), with a penalty for known non-rhyming words. Each round runs a four-phase sub-FSM ROUND_START → INPUT → SCORING → INTERMISSION across three rounds, driven by setTimeout chains and startPhaseTimer broadcasts. End-of-game computeAwards produces superlatives (Wordsmith, Diamond in the Rough, Syllable Surfer, Quick Draw, Overachiever).

Scope: ported vs stubbed.
  • Fully ported (1): rhyme-time — full phase FSM, input validation, rarity/speed/multi-syllable scoring, per-player state masking, and awards. Caveat: the Node handler validated rhymes against the CMU pronouncing dictionary (the rhyming-part package). To stay dependency-free the Go port substitutes a deterministic phonetic-suffix heuristic (rtRhymeKey) over a small bundled word set; the structure, scoring, masking and awards are faithful, the phonetic accuracy is approximate.
  • Compiling stubs (8): undercover-agent, category-crash, wiki-race, wit-war, fact-or-friction, undercover-editor, minimalist-masterpiece, emoji-cinema. Each is a stubGame that auto-completes after 1s with all players ranked at zero, so the full lobby/coordinator lifecycle stays exercisable end-to-end against every registered game. Real handlers are tracked as TODO(migration) in registry.go.
The 9 games are registered with correct static metadata (min/max players, join-in-progress policy, spectator mode) in minigameRegistry and serverRegistry; only the gameplay handlers differ in fidelity.

Per-player scoped state (info-leak prevention)

The single exit point for all lobby state is LobbyManager.buildClientState(l, userID). It strips internal fields and, critically, scopes the active minigame state by the recipient's role: a player gets GetStateForPlayer(userID) (which includes only their own private data), while a spectator gets GetStateForSpectator() (no private submissions). For Rhyme Time this means mySubmissions is populated only for the owning player and is empty for spectators during the INPUT phase — the load-bearing distinction that prevents one player seeing another's in-progress answers. Competitive spectators may follow a chosen player via GetSpectatorSnapshot, which for competitive-individual games returns the target player's scoped state.

if myRole == "player" {
    gameState = safeStateForPlayer(l.CurrentGame.Handler, userID)
} else {
    gameState = safeStateForSpectator(l.CurrentGame.Handler)
}

Both safeStateForPlayer and safeStateForSpectator wrap the handler call in recover(), defaulting to an empty map on panic. Host-only data (kick/transfer/settings, the ban/kick controls) is enforced at the handler layer — non-host callers receive a NOT_HOST error and the mutation is rejected before any broadcast.

Persistence & leaderboard

Persistence is fire-and-forget: on game complete the coordinator spawns a goroutine (panic-recovered) that calls repo.PersistMatch; a DB error is logged but never blocks or fails the game flow. When the database is absent (repo == nil) the whole path is a no-op and leaderboard reads return empty. PersistMatch writes a rmhbox_match row and one rmhbox_match_player row per player inside a single transaction.

Atomic minigameStats merge — fixes a Node concurrency hazard. The Node code read a player's profile, merged the new match into the minigameStats JSONB in application code, then wrote it back — a find-then-update with no lock, so two matches finishing for the same user could clobber each other. The Go port performs the read-modify-write atomically under a row lock inside the transaction:

SELECT "id","minigameStats","currentWinStreak","bestWinStreak"
FROM "rmhbox_profile" WHERE "userId"=$1 FOR UPDATE   -- row locked for the tx
-- then mergeMinigameStats(...) in Go, then UPDATE ... in place

The pure merge function mergeMinigameStats (games played, wins, best score, total score, total/average rank) is extracted so the math is unit-tested directly without a database. The leaderboard read maps the caller's metric onto a fixed whitelist of quoted column literals ("totalScore" / "totalWins" / "totalGamesPlayed") before concatenation, so no user input ever reaches the ORDER BY — the query is injection-proof by construction.

Concurrency model

  • Per-lobby mutex. Each Lobby carries its own sync.Mutex; every read or write of lobby fields takes it. Many goroutines touch a lobby concurrently — timers, the GC ticker, the heartbeat, and every connection's read pump — so the lock is the lobby's single serialization point. The manager's own RWMutex guards the global indices.
  • Snapshot-then-broadcast. Handlers consistently take the lock, copy out the minimal data they need (ids, names, a settings value), release the lock, and only then broadcast — broadcast helpers re-take the lock themselves, so locks are never held across a fan-out.
  • Grace closures re-resolve. As noted, every deferred timer re-resolves its lobby by id and bails if it was disbanded/replaced (the review fix).
  • Panic-recover fault isolation. Every minigame handler entry point (Start, HandleInput, HandlePlayerDisconnect, ForceEnd, the state getters, Cleanup) and the async persist goroutine are wrapped in recover(), routing failures to ctx.OnError/handleGameError or a log line rather than crashing the service.
