The fleet ships with two orchestration paths that mirror the existing project: a Docker Compose file for local/single-host parity with the Node stack, and a Helm chart that deliberately follows PR #121's conventions so it slots into the same k3s cluster.

Docker Compose

docker-compose.go.yml is the Go analog of the project's root docker-compose.yml. It builds one image per service from the shared Dockerfile, runs a Redis instance as the realtime backplane, reads the same .env.production, and exposes the gateway as the single ingress. The gateway's upstreams are wired to the Compose service names; the catch-all points back at the host's Node web app via host.docker.internal.

docker compose -f go-services/docker-compose.go.yml \
  --env-file .env.production up -d --build

The Helm chart

deploy/helm/rmhstudios-go renders the whole fleet. It reuses PR #121's patterns: a generic Deployment template that ranges over a services map, a Service template for the ported services, a Traefik Ingress, and — crucially — the same rmhstudios-secrets Secret (via envFrom) and rmhstudios-data PVC. The one deliberate departure from PR #121's single-image chart is noted in the chart description: because each Go service is a distinct static binary, each is a distinct image (<registry>/<prefix>-<svc>:<tag>).

The services map

The chart is data-driven: each service is an entry in values.services declaring its client port (if any), replica count, ingress path, and whether it is a worker (no Service, probes hit /health on the metrics port) or needs the shared storage. Probes target the right port automatically via a helper.

Gateway upstream wiring

The gateway's reverse-proxy upstreams default to Compose-style hostnames, which do not resolve in Kubernetes. The chart therefore injects the correct in-cluster Service DNS names into the gateway's environment (GAMEHUB_UPSTREAM=http://rmhstudios-go-gamehub:7001, etc., and WEB_UPSTREAM for the Node SSR Service). This was a blocking deploy bug caught in senior review and fixed before this document was written.

The backplane in-cluster

When redis.enabled (default), the chart deploys a single in-memory Redis (no persistence, no PVC) and points every realtime service at it via REDIS_URL. With it, the realtime services can scale beyond one replica behind sticky-session ingress; without it, they run single-replica exactly as today.

Validated rendering

The chart was validated, not just written: helm lint passes for both default and production values, and the rendered manifests — 35 resources across the default and prod value sets — pass kubeconform -strict with zero invalid resources.

$ helm lint deploy/helm/rmhstudios-go         # 0 failed
$ helm template ... | kubeconform -strict     # 35/35 valid, 0 invalid
