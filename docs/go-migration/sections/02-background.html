To understand this migration you must understand what came immediately before it. PR #121 moved the platform's run layer from Docker Compose to single-node k3s + Helm, with Terraform for DNS — and, in the same breath, recorded a deliberate decision against the very thing this branch now does.

The starting point: a process-split Node monolith

rmhstudios.com is a TanStack Start (React SSR) application backed by Prisma on PostgreSQL and Better Auth for identity. Beyond the web app, the system was already split into eight long-running Node processes, orchestrated by Docker Compose from a single shared image:

(rmhmusic ran inside the socket process under an event-name prefix.) This process split is the single most important precondition for the migration: the services were already decoupled, communicated over well-defined transports, and — critically — already authenticated by querying the shared session table with raw SQL rather than going through the Better Auth library.

What PR #121 did

PR #121 introduced a declarative run layer: a Helm chart rendering eight Deployments from one image, a Prisma migration pre-upgrade hook with atomic auto-rollback, a Terraform DNS module, and a registry switch wiring the path to multi-node. It kept the Compose path intact as a reversible fallback and was verified end-to-end on a real (kind) cluster. Its scaling roadmap named the genuinely hard next steps: a Redis backplane + sticky sessions for realtime horizontal scaling (Stage 1), multi-node + autoscaling + observability (Stage 2), and multi-region (Stage 3).

The architecture decision this branch overrides

From PR #121, verbatim in spirit: "We explicitly chose not to rewrite the backend in Go microservices. The app is already process-split into 8 services; the web tier is React SSR (must stay JS) and the data/auth layers are Prisma + Better Auth (no Go equivalents). A wholesale rewrite is the largest possible source of tech debt." PR #121 placed any Go adoption at Stage 4 — "only if metrics demand," and only as incremental per-service extraction.

This branch is that Stage 4 — executed in full, at once, on request, and on an isolated branch where it can be evaluated without endangering production. The honest framing is maintained throughout: this is an exploration of the complete Go target, not a claim that the rewrite was necessary. The value it delivers is real regardless: a working, tested reference implementation of the Go fleet, a Helm chart that slots into PR #121's infrastructure, and — because the Go services share the Postgres session table — a design that could in principle be adopted one service at a time rather than big-bang.

How the two reconcile

Rather than competing with PR #121, the Go fleet is designed to live beside it. The Go Helm chart (rmhstudios-go) reuses PR #121's rmhstudios-secrets Secret and rmhstudios-data PVC, reads the same .env.production, and routes all traffic through a single gateway that proxies anything it does not own back to the still-JavaScript React SSR web tier. The Redis backplane PR #121 deferred to Stage 1 is implemented here as the realtime services' fan-out primitive.
