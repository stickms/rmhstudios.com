This section is the deliberate counterweight to the rest of the document. A migration report that only lists accomplishments is untrustworthy. Here is a frank assessment of what is risky, what is incomplete, and whether this work should actually be adopted.

The Socket.IO wire-protocol gap (largest risk)

The existing browser clients speak the Socket.IO v4 protocol. The Go realtime services speak a clean JSON-over-raw-WebSocket protocol with the same semantics but a different wire format. They are not interoperable. A production cutover of any websocket prefix requires a client-side adapter (mapping existing event names onto the new transport) that is designed but not built. Until then, the Go realtime services are validated by the new-protocol e2e clients, not by the live browser clients.

This is the single most important limitation and is stated everywhere it is relevant. It does not affect the workers (doctrine, recap, vibe) or the discord-bot, which have no browser websocket clients.

Behavioral substitutions

  • vibe-worker downscaling: Chromium device-scale replaces Node's sharp. Visual parity of thumbnails should be eyeballed before relying on it.
  • recap presence: discordgo's gateway replaces a hand-rolled opcode loop — an improvement, but a behavioral change in how presence is maintained.
  • IDs: Go generates UUID primary keys where Prisma used cuid; both are opaque TEXT, but the format differs.
  • rmhbox rhyme check: a phonetic-suffix heuristic substitutes for the Node CMU-dictionary lookup; scoring may differ at the margins.

What is unverified

  • No load testing. Performance claims are structural, not measured.
  • No live-traffic validation. The services have never seen production clients (see the protocol gap).
  • chromedp under load. Headless Chromium stability for the thumbnail worker at scale is untested in Go.
  • DeepSeek streaming. The Go chat client is non-streaming where the Node version streamed; UX differs.

Should this be adopted?

Honest recommendation. Not wholesale, and not now — for the same reasons PR #121 gave. The Node services work; the React/Better Auth/ Prisma core must stay JavaScript regardless; and the biggest realtime scaling win (the Redis backplane + sticky sessions) is language-agnostic and could be applied to the existing Node services. This Go fleet is most valuable as: (1) a complete, tested reference for what a Go target looks like; (2) a source of genuinely good ideas already extracted here — the single auth-validation query, the events.Bus backplane abstraction, the atomic stat-merge fix; and (3) an incremental option, since the shared session table means one service could be migrated and measured in isolation. Adopting it should be metrics-driven and per-service — exactly the Stage-4 condition that was set.

Where the risk is low

Conversely, several pieces are low-risk and could be adopted on their own merits: the doctrine-worker (bit-exact, fully ported, deterministic), the recap runner, and the gateway as an auth-enforcing edge in front of the existing stack. These have no Socket.IO dependency and clear, tested behavior.
